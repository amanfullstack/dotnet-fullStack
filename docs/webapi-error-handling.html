    🚨 Error Handling

    Global exception handling, ProblemDetails standard, custom exceptions, and returning consistent error responses from your API.

    Intermediate Web API
    ℹ️

    The goal

    A production API should never leak stack traces to clients, always return a consistent JSON error format, log all errors server-side, and return appropriate HTTP status codes. One global error handler is better than try-catch in every controller.

    ProblemDetails Standard (RFC 7807)

    ASP.NET Core uses the ProblemDetails format for error responses. It's an industry standard your clients can reliably parse.

    Error Response Format 
    // 404 Not Found
{
  "type":   "https://tools.ietf.org/html/rfc7231#section-6.5.4",
  "title":  "Not Found",
  "status": 404,
  "detail": "Product with ID 99 was not found.",
  "traceId": "00-abc123-def456-00"
}

// 400 Validation Error
{
  "type":   "https://tools.ietf.org/html/rfc7231#section-6.5.1",
  "title":  "One or more validation errors occurred.",
  "status": 400,
  "errors": {
    "Name":  ["The Name field is required."],
    "Price": ["Price must be between 0.01 and 999999."]
  }
}

    Global Exception Handler

    Register a global handler that catches unhandled exceptions anywhere in the pipeline and returns a safe error response.

    Middleware/GlobalExceptionHandler.cs 
    using Microsoft.AspNetCore.Diagnostics;
using Microsoft.AspNetCore.Mvc;

public class GlobalExceptionHandler : IExceptionHandler
{
    private readonly ILogger<GlobalExceptionHandler> _logger;

    public GlobalExceptionHandler(ILogger<GlobalExceptionHandler> logger)
    {
        _logger = logger;
    }

    public async ValueTask<bool> TryHandleAsync(
        HttpContext context,
        Exception exception,
        CancellationToken cancellationToken)
    {
        // Log every unhandled exception
        _logger.LogError(exception,
            "Unhandled exception: {Message} | Path: {Path}",
            exception.Message,
            context.Request.Path);

        // Map exception type to HTTP status code
        var (statusCode, title) = exception switch
        {
            NotFoundException     => (404, "Not Found"),
            ValidationException   => (400, "Validation Error"),
            UnauthorizedAccessException => (401, "Unauthorized"),
            _ => (500, "Internal Server Error")
        };

        var problemDetails = new ProblemDetails
        {
            Status = statusCode,
            Title  = title,
            Detail = exception.Message,
            Instance = context.Request.Path
        };

        context.Response.StatusCode  = statusCode;
        context.Response.ContentType = "application/problem+json";
        await context.Response.WriteAsJsonAsync(problemDetails, cancellationToken);

        return true; // Exception handled — stop propagation
    }
}
    Program.cs — Register Handler 
    // Register the exception handler service
builder.Services.AddExceptionHandler<GlobalExceptionHandler>();
builder.Services.AddProblemDetails();

// Add to middleware pipeline (must be FIRST)
app.UseExceptionHandler();

    Custom Exception Classes

    Define custom exceptions for domain-specific errors. The global handler maps them to appropriate status codes.

    Exceptions/DomainExceptions.cs 
    // 404 — Resource not found
public class NotFoundException : Exception
{
    public NotFoundException(string resourceType, object id)
        : base($"{resourceType} with ID {id} was not found.") { }
}

// 400 — Invalid input / business rule violation
public class ValidationException : Exception
{
    public ValidationException(string message) : base(message) { }
}

// 409 — Conflict (duplicate resource, etc.)
public class ConflictException : Exception
{
    public ConflictException(string message) : base(message) { }
}

// Usage in ProductService:
public async Task<Product> GetByIdAsync(int id)
{
    var product = await _context.Products.FindAsync(id);
    if (product == null)
        throw new NotFoundException(nameof(Product), id); // → 404
    return product;
}

    Error Handling in Services

    With a global handler, controllers don't need try-catch. Services can still catch specific exceptions when they need to perform cleanup or retry logic.

    Clean Controller (no try-catch needed) 
    // Controller is clean — global handler catches any exception
[HttpGet("{id:int}")]
public async Task<ActionResult<Product>> GetById(int id)
{
    var product = await _service.GetByIdAsync(id);
    return Ok(product);
    // If product not found, service throws NotFoundException
    // Global handler catches it and returns 404 ProblemDetails
}

// Service handles specific DB exceptions
public async Task<Product> CreateAsync(CreateProductRequest request)
{
    try
    {
        var product = MapToEntity(request);
        _context.Products.Add(product);
        await _context.SaveChangesAsync();
        return product;
    }
    catch (DbUpdateException ex) when (ex.InnerException?.Message.Contains("UNIQUE") == true)
    {
        throw new ConflictException($"A product named '{request.Name}' already exists.");
    }
    // All other exceptions propagate to GlobalExceptionHandler
}
    💡

    Interview tip

    Use a global exception handler (IExceptionHandler in .NET 8) instead of try-catch in every controller. It centralizes error logging, maps exceptions to status codes, and ensures consistent ProblemDetails responses. Never expose raw exception messages or stack traces to clients in production.
