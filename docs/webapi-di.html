    💉 Dependency Injection

    ASP.NET Core has a built-in IoC container. Understand service lifetimes, constructor injection, and how to register and resolve services.

    Intermediate Web API
    ℹ️

    What is Dependency Injection?

    DI is a design pattern where a class receives its dependencies from the outside rather than creating them itself. ASP.NET Core includes a built-in IoC (Inversion of Control) container that creates and manages object lifetimes automatically.

    Service Lifetimes

    This is the most-asked DI interview topic. You must know all three lifetimes and when to use each.

    LifetimeWhen createdWhen destroyedUse for
    Singleton First request / app startup App shutdown Config, caching, logging, HttpClient
    Scoped Each HTTP request End of HTTP request DbContext, business services (most common)
    Transient Every injection point When injected object is disposed Lightweight stateless utilities

    Registering Services

    Services are registered in Program.cs using builder.Services.

    Program.cs — Service Registration 
    // ── Scoped (one per HTTP request) ────────────────────────────
// DbContext is always Scoped — it tracks changes per request
builder.Services.AddDbContext<ProductDbContext>(options =>
    options.UseSqlServer(connectionString));

// Business service — scoped because it uses DbContext
builder.Services.AddScoped<IProductService, ProductService>();

// ── Singleton (one for entire app lifetime) ──────────────────
// Caching service — safe as singleton (thread-safe)
builder.Services.AddMemoryCache();  // registers IMemoryCache as singleton
builder.Services.AddSingleton<IConfiguration>(builder.Configuration);

// ── Transient (new every injection) ─────────────────────────
// Rarely needed. Useful for stateless utility classes.
builder.Services.AddTransient<IGuidGenerator, GuidGenerator>();

// ── Register concrete type without interface ──────────────────
builder.Services.AddScoped<ProductService>();

// ── Register with factory ─────────────────────────────────────
builder.Services.AddScoped<IProductService>(sp =>
    new ProductService(
        sp.GetRequiredService<ProductDbContext>(),
        sp.GetRequiredService<IMemoryCache>()
    ));

    Constructor Injection

    The most common pattern. The DI container automatically resolves constructor parameters when it creates the class.

    Controller Constructor Injection 
    public class ProductsController : ControllerBase
{
    // DI container fills these from the registered services
    private readonly IProductService _service;
    private readonly ILogger<ProductsController> _logger;
    private readonly IMemoryCache _cache;

    public ProductsController(
        IProductService service,
        ILogger<ProductsController> logger,
        IMemoryCache cache)
    {
        _service = service;
        _logger  = logger;
        _cache   = cache;
    }
}

// Service with its own injected dependencies
public class ProductService : IProductService
{
    private readonly ProductDbContext _context;
    private readonly IMemoryCache _cache;
    private readonly ILogger<ProductService> _logger;

    public ProductService(
        ProductDbContext context,
        IMemoryCache cache,
        ILogger<ProductService> logger)
    {
        _context = context;
        _cache   = cache;
        _logger  = logger;
    }
}

    Why Program to Interfaces

    Registering as IProductService (interface) rather than ProductService (concrete class) makes code testable — you can swap implementations during tests.

    Services/IProductService.cs 
    public interface IProductService
{
    Task<IEnumerable<Product>> GetAllAsync();
    Task<Product?> GetByIdAsync(int id);
    Task<IEnumerable<Product>> GetByCategoryAsync(string category);
    Task<Product> CreateAsync(CreateProductRequest request);
    Task<Product?> UpdateAsync(int id, UpdateProductRequest request);
    Task<Product?> UpdateStockAsync(int id, int newStock);
    Task<bool> DeleteAsync(int id);
}

// In unit tests, swap real service for a mock:
// var mockService = new Mock<IProductService>();
// mockService.Setup(s => s.GetAllAsync()).ReturnsAsync(fakeProducts);
// controller = new ProductsController(mockService.Object, ...);

    Captive Dependency Warning

    Never inject a shorter-lived service into a longer-lived one. This is a common mistake that causes bugs.

    ⚠️

    Never inject Scoped into Singleton

    If a Singleton holds a reference to a Scoped service, the scoped service lives forever (captured by the singleton). For DbContext in a singleton, use IServiceScopeFactory to create a scope manually instead.

    Correct: Use IServiceScopeFactory in Singleton 
    // WRONG — captive dependency: DbContext is scoped, BackgroundService is singleton
public class MyBackgroundService : BackgroundService
{
    private readonly ProductDbContext _context; // ❌ singleton captures scoped
    public MyBackgroundService(ProductDbContext context) { _context = context; }
}

// CORRECT — create scope on demand
public class MyBackgroundService : BackgroundService
{
    private readonly IServiceScopeFactory _scopeFactory;
    public MyBackgroundService(IServiceScopeFactory scopeFactory) { _scopeFactory = scopeFactory; }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        using var scope   = _scopeFactory.CreateScope();  // New scope per operation
        var context       = scope.ServiceProvider.GetRequiredService<ProductDbContext>();
        // use context safely
    }
}
    💡

    Interview cheat sheet

    • Transient = new every time. Use for lightweight, stateless utilities.
    • Scoped = new per request. Use for DbContext, business services.
    • Singleton = one instance ever. Use for caching, config, HttpClientFactory.
    • Never inject Scoped → Singleton (captive dependency).
