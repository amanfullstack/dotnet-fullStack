    🎮 Controllers & REST API

    How to write ASP.NET Core controllers — routing, HTTP methods, action results, model binding, and RESTful design.

    Beginner Web API
    ℹ️

    What is a Controller?

    A controller is a class that handles incoming HTTP requests. It maps URLs to C# methods (actions), reads input from the request (route params, query string, body), calls services, and returns an HTTP response with a status code and optional body.

    Controller Anatomy

    Every Web API controller inherits from ControllerBase and is decorated with two key attributes.

    Controllers/ProductsController.cs 
    [ApiController]               // Enables automatic model validation + binding
[Route("api/[controller]")]   // Sets base route: /api/products
public class ProductsController : ControllerBase
{
    private readonly IProductService _service;
    private readonly ILogger<ProductsController> _logger;

    // Constructor injection — services provided by DI container
    public ProductsController(IProductService service, ILogger<ProductsController> logger)
    {
        _service = service;
        _logger  = logger;
    }
}
    [ApiController]Automatic 400 on validation errors, binding source inference, problem details responses
    ControllerBaseBase class for API controllers (no View support). Use Controller if you need Views (MVC).
    [Route][controller] resolves to the class name minus "Controller" — "ProductsController" → "products"

    HTTP Methods (CRUD Mapping)

    REST maps HTTP verbs to CRUD operations. Each action method in the ProductsController follows this convention.

    Full CRUD Controller 
    // GET /api/products — Read all
[HttpGet]
public async Task<ActionResult<IEnumerable<Product>>> GetAll()
{
    var products = await _service.GetAllAsync();
    return Ok(products);
}

// GET /api/products/5 — Read one
[HttpGet("{id:int}")]
public async Task<ActionResult<Product>> GetById(int id)
{
    var product = await _service.GetByIdAsync(id);
    if (product == null) return NotFound();
    return Ok(product);
}

// GET /api/products/category/Electronics — Filter
[HttpGet("category/{category}")]
public async Task<ActionResult<IEnumerable<Product>>> GetByCategory(string category)
{
    var products = await _service.GetByCategoryAsync(category);
    return Ok(products);
}

// POST /api/products — Create
[HttpPost]
public async Task<ActionResult<Product>> Create([FromBody] CreateProductRequest request)
{
    var product = await _service.CreateAsync(request);
    // 201 Created with Location header pointing to the new resource
    return CreatedAtAction(nameof(GetById), new { id = product.Id }, product);
}

// PUT /api/products/5 — Full update
[HttpPut("{id:int}")]
public async Task<ActionResult<Product>> Update(int id, [FromBody] UpdateProductRequest request)
{
    var product = await _service.UpdateAsync(id, request);
    if (product == null) return NotFound();
    return Ok(product);
}

// PATCH /api/products/5/stock — Partial update
[HttpPatch("{id:int}/stock")]
public async Task<ActionResult<Product>> UpdateStock(int id, [FromBody] int newStock)
{
    var product = await _service.UpdateStockAsync(id, newStock);
    if (product == null) return NotFound();
    return Ok(product);
}

// DELETE /api/products/5 — Soft delete
[HttpDelete("{id:int}")]
public async Task<IActionResult> Delete(int id)
{
    var success = await _service.DeleteAsync(id);
    if (!success) return NotFound();
    return NoContent(); // 204 — success but no body
}

    Action Results

    Returning the right status code is part of good REST design. ASP.NET Core provides helper methods on ControllerBase.

    MethodStatus CodeWhen to use
    Ok(data)200Successful GET, PUT, PATCH
    Created(url, data)201Resource created via POST
    CreatedAtAction(...)201POST with Location header
    NoContent()204DELETE, PUT with no response body
    BadRequest(errors)400Validation failure, bad input
    Unauthorized()401Missing/invalid auth token
    Forbidden()403Authenticated but not authorized
    NotFound()404Resource does not exist
    Conflict()409Duplicate resource, optimistic concurrency
    StatusCode(500, msg)500Unexpected server error

    Model Binding

    ASP.NET Core reads values from different parts of the request based on binding source attributes.

    Binding Sources 
    // [FromRoute]   — URL segment: /api/products/{id}
public IActionResult Get([FromRoute] int id) { }

// [FromQuery]   — Query string: /api/products?page=2&size=10
public IActionResult GetPaged([FromQuery] int page = 1, [FromQuery] int size = 10) { }

// [FromBody]    — Request body (JSON)
public IActionResult Create([FromBody] CreateProductRequest request) { }

// [FromHeader]  — HTTP header value
public IActionResult Get([FromHeader(Name = "X-Api-Key")] string apiKey) { }

// [FromForm]    — multipart/form-data (file uploads)
public IActionResult Upload([FromForm] IFormFile file) { }

// [ApiController] infers binding automatically:
//   Simple types (int, string) → [FromRoute] or [FromQuery]
//   Complex types              → [FromBody]

    Route Constraints

    Route constraints restrict which values can match a route segment. They prevent invalid inputs before they reach your action.

    Route Constraints 
    [HttpGet("{id:int}")]           // only matches integers
[HttpGet("{id:int:min(1)}")]    // integer ≥ 1
[HttpGet("{name:alpha}")]       // only letters
[HttpGet("{id:guid}")]          // only GUIDs
[HttpGet("{date:datetime}")]    // only valid datetime strings

// Example: prevents /api/products/abc from routing to GetById
[HttpGet("{id:int}")]
public async Task<IActionResult> GetById(int id) { ... }
    💡

    Interview tip: ActionResult vs IActionResult

    ActionResult<T> is preferred because it clearly documents the success return type in Swagger and allows implicit conversion from T. IActionResult works too but provides no type information to OpenAPI tooling.
