    📦 Models & DTOs

    Entity models define the database schema. DTOs define what the API accepts and returns. Keeping them separate is essential.

    Beginner Web API
    ℹ️

    Model vs DTO

    A Model (entity) maps directly to a database table — it's what EF Core works with. A DTO (Data Transfer Object) is what the API exposes to clients. They're kept separate so you can change the DB schema without breaking the API contract, and to avoid over-posting vulnerabilities.

    Entity Model

    The Product entity maps to the Products table. Data annotations configure column constraints.

    Models/Product.cs 
    using System.ComponentModel.DataAnnotations;
using System.ComponentModel.DataAnnotations.Schema;

public class Product
{
    [Key]                          // Primary key (auto-increment)
    public int Id { get; set; }

    [Required]
    [StringLength(200)]
    public string Name { get; set; } = string.Empty;

    [StringLength(1000)]
    public string? Description { get; set; }

    [Required]
    [Column(TypeName = "decimal(18,2)")]
    [Range(0.01, 999999)]
    public decimal Price { get; set; }

    [Required]
    [StringLength(100)]
    public string Category { get; set; } = string.Empty;

    [Range(0, int.MaxValue)]
    public int StockQuantity { get; set; }

    public bool IsActive { get; set; } = true;    // Soft delete flag
    public DateTime CreatedAt { get; set; } = DateTime.UtcNow;
    public DateTime? UpdatedAt { get; set; }
}

    Data Transfer Objects (DTOs)

    DTOs are separate classes for API input/output. They prevent internal fields (like IsActive, CreatedAt) from being set by clients.

    Models/DTOs/CreateProductRequest.cs 
    public class CreateProductRequest
{
    [Required(ErrorMessage = "Name is required")]
    [StringLength(200, MinimumLength = 2)]
    public string Name { get; set; } = string.Empty;

    [StringLength(1000)]
    public string? Description { get; set; }

    [Required]
    [Range(0.01, 999999, ErrorMessage = "Price must be between 0.01 and 999999")]
    public decimal Price { get; set; }

    [Required]
    [StringLength(100)]
    public string Category { get; set; } = string.Empty;

    [Range(0, int.MaxValue)]
    public int StockQuantity { get; set; } = 0;
}
    Models/DTOs/UpdateProductRequest.cs 
    public class UpdateProductRequest
{
    [Required]
    [StringLength(200, MinimumLength = 2)]
    public string Name { get; set; } = string.Empty;

    [StringLength(1000)]
    public string? Description { get; set; }

    [Required]
    [Range(0.01, 999999)]
    public decimal Price { get; set; }

    [Required]
    [StringLength(100)]
    public string Category { get; set; } = string.Empty;

    [Range(0, int.MaxValue)]
    public int StockQuantity { get; set; }
}

    Manual Mapping (No AutoMapper)

    In the service layer, map DTOs to entities manually. This is explicit and doesn't require extra dependencies for simple cases.

    Services/ProductService.cs — Create mapping 
    public async Task<Product> CreateAsync(CreateProductRequest request)
{
    // Map DTO → Entity
    var product = new Product
    {
        Name          = request.Name,
        Description   = request.Description,
        Price         = request.Price,
        Category      = request.Category,
        StockQuantity = request.StockQuantity,
        IsActive      = true,
        CreatedAt     = DateTime.UtcNow
    };

    _context.Products.Add(product);
    await _context.SaveChangesAsync();
    return product;
}

public async Task<Product?> UpdateAsync(int id, UpdateProductRequest request)
{
    var product = await _context.Products.FindAsync(id);
    if (product == null) return null;

    // Map DTO fields onto existing entity
    product.Name          = request.Name;
    product.Description   = request.Description;
    product.Price         = request.Price;
    product.Category      = request.Category;
    product.StockQuantity = request.StockQuantity;
    product.UpdatedAt     = DateTime.UtcNow;

    await _context.SaveChangesAsync();
    return product;
}

    Automatic Validation with [ApiController]

    The [ApiController] attribute automatically validates the model and returns a 400 response if validation fails — no manual ModelState.IsValid check needed.

    Automatic 400 response 
    // POST /api/products with invalid body
// Response: 400 Bad Request
{
  "type": "https://tools.ietf.org/html/rfc7231#section-6.5.1",
  "title": "One or more validation errors occurred.",
  "status": 400,
  "errors": {
    "Name":  ["Name is required"],
    "Price": ["Price must be between 0.01 and 999999"]
  }
}
    💡

    Interview tip: Why DTOs?

    Three reasons: (1) Security — prevent over-posting (client sets fields they shouldn't). (2) Decoupling — API contract can evolve independently from DB schema. (3) Projection — return only the fields the client needs, improving performance.
