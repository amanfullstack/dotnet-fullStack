📦 Components & Templates

Master Angular components, property binding, and template syntax.

What Are Angular Components?

Components are the building blocks of Angular applications. They combine TypeScript logic, HTML templates, and CSS styling into reusable units. Every Angular app is a tree of components, starting from the root component.

A component is a TypeScript class decorated with @Component that controls a piece of the screen (a view).

Why Components Matter

  • Reusability: Build components once, use in many places
  • Modularity: Break UI into manageable pieces
  • Maintainability: Easier to understand and update
  • Testability: Test components in isolation
  • Encapsulation: Components manage their own state and style

Key Concepts

🏗️ Component Anatomy

A component has three parts:

  • Class: TypeScript code with logic (component.ts)
  • Template: HTML markup (component.html)
  • Style: CSS styling (component.css)

🔗 Property Binding

Pass data from component to template:

  • [property]="value" - One-way binding
  • {{expression}} - Interpolation (same as binding)
  • Used for: displaying data, setting attributes, applying classes

⚡ Event Binding

Respond to user interactions:

  • (click)="handler()" - Click events
  • (keyup)="handler($event)" - Keyboard events
  • (change)="handler()" - Form changes
  • $event - Browser event object

🔄 Two-Way Binding

Synchronize data between template and component:

<input [(ngModel)]="name" />
  • Requires FormsModule import
  • Updates component property when user types
  • Updates template when property changes

♻️ Lifecycle Hooks

Component lifecycle events:

  • OnInit - After component created
  • OnChanges - When input properties change
  • OnDestroy - Before component destroyed
  • Use for initialization, cleanup, responding to changes

Code Examples

Example 1: Basic Component with Property & Event Binding

import { Component } from '@angular/core';

@Component({
  selector: 'app-product',
  templateUrl: './product.component.html',
  styleUrls: ['./product.component.css']
})
export class ProductComponent {
  // Component properties (data)
  productName = 'Laptop';
  price = 999.99;
  inStock = true;
  quantity = 1;

  // Methods (event handlers)
  addToCart() {
    console.log(`Added ${this.quantity} ${this.productName} to cart`);
    alert('Product added!');
  }

  decreaseQuantity() {
    if (this.quantity > 1) {
      this.quantity--;
    }
  }

  increaseQuantity() {
    this.quantity++;
  }
}

<!-- product.component.html -->
<div class="product-card">
  <h2>{{productName}}</h2>
  <p class="price">Price: ${{price}}</p>

  <!-- Show/hide based on condition -->
  <p *ngIf="inStock" class="in-stock">In Stock</p>
  <p *ngIf="!inStock" class="out-of-stock">Out of Stock</p>

  <!-- Quantity selector -->
  <div class="quantity-selector">
    <button (click)="decreaseQuantity()">-</button>
    <span>{{quantity}}</span>
    <button (click)="increaseQuantity()">+</button>
  </div>

  <!-- Add to cart button -->
  <button
    (click)="addToCart()"
    [disabled]="!inStock"
    [class.btn-primary]="inStock"
    [class.btn-disabled]="!inStock">
    Add to Cart
  </button>
</div>

Example 2: Component with Lifecycle Hooks

import { Component, OnInit, OnDestroy, Input, OnChanges, SimpleChanges } from '@angular/core';

@Component({
  selector: 'app-product-list',
  templateUrl: './product-list.component.html'
})
export class ProductListComponent implements OnInit, OnDestroy, OnChanges {
  @Input() categoryId: number = 0;
  products: any[] = [];
  loading = false;

  ngOnInit() {
    // Called once when component initializes
    console.log('ProductListComponent initialized');
    this.loadProducts();
  }

  ngOnChanges(changes: SimpleChanges) {
    // Called when @Input properties change
    if (changes['categoryId']) {
      console.log('Category changed to:', changes['categoryId'].currentValue);
      this.loadProducts();
    }
  }

  ngOnDestroy() {
    // Called when component is destroyed
    console.log('ProductListComponent destroyed');
    // Clean up subscriptions, timers, etc.
  }

  loadProducts() {
    this.loading = true;
    // Simulate API call
    setTimeout(() => {
      this.products = [
        { id: 1, name: 'Product 1', price: 100 },
        { id: 2, name: 'Product 2', price: 200 }
      ];
      this.loading = false;
    }, 1000);
  }
}

Example 3: Two-Way Binding & Form Handling

import { Component } from '@angular/core';

@Component({
  selector: 'app-search',
  templateUrl: './search.component.html'
})
export class SearchComponent {
  searchTerm: string = '';
  results: string[] = [];

  onSearch() {
    if (this.searchTerm.trim()) {
      // Simulate search
      this.results = [
        `Result for "${this.searchTerm}" 1`,
        `Result for "${this.searchTerm}" 2`,
        `Result for "${this.searchTerm}" 3`
      ];
    } else {
      this.results = [];
    }
  }

  clearSearch() {
    this.searchTerm = '';
    this.results = [];
  }
}

<!-- search.component.html -->
<div class="search-container">
  <div class="search-input">
    <!-- Two-way binding with ngModel -->
    <input
      [(ngModel)]="searchTerm"
      placeholder="Search products..."
      (keyup.enter)="onSearch()">
    <button (click)="onSearch()">Search</button>
    <button (click)="clearSearch()">Clear</button>
  </div>

  <!-- Display search term -->
  <p *ngIf="searchTerm">Searching for: {{searchTerm}}</p>

  <!-- Display results -->
  <div class="results" *ngIf="results.length > 0">
    <ul>
      <li *ngFor="let result of results">{{result}}</li>
    </ul>
  </div>

  <!-- Show message if no results -->
  <p *ngIf="searchTerm && results.length === 0">No results found</p>
</div>

✅ Best Practices

1. Keep Components Focused

Why: Single responsibility principle makes code easier to test and maintain

❌ Bad: ProductListComponent handles listing, filtering, sorting, AND API calls

✅ Good: ProductListComponent only displays, use service for API calls

2. Use @Input for Parent-to-Child Communication

Why: Makes component reusable with different data

@Component({selector: 'app-product-card'})
export class ProductCardComponent {
  @Input() product: any; // Parent passes data
}

3. Use @Output for Child-to-Parent Events

Why: Child notifies parent of important events

@Component({selector: 'app-product-card'})
export class ProductCardComponent {
  @Output() addedToCart = new EventEmitter<any>();

  onAddClick() {
    this.addedToCart.emit(this.product); // Notify parent
  }
}

4. Always Unsubscribe from Observables

Why: Prevents memory leaks when component is destroyed

export class DataComponent implements OnInit, OnDestroy {
  subscription: Subscription;

  ngOnInit() {
    this.subscription = this.service.getData().subscribe(data => {
      // Handle data
    });
  }

  ngOnDestroy() {
    this.subscription.unsubscribe(); // Clean up
  }
}

5. Use Async Pipe for Observable Subscriptions

Why: Automatically unsubscribes, cleaner code

<!-- Instead of manual subscription -->
<div>{{data | async}}</div>

🎓 Interview Q&A

Q1: What's the difference between property binding and interpolation?

Answer:

  • Interpolation: {{expression}} - Only for text content, less powerful
  • Property binding: [property]="value" - Can set attributes, classes, properties
<!-- Interpolation - only for text -->
<p>Price: {{price}}</p>

<!-- Property binding - more powerful -->
<img [src]="imageUrl" />
<button [disabled]="isLoading">Submit</button>

Q2: What happens if you don't unsubscribe from observables?

Answer: Memory leaks occur. Subscriptions keep running even after component is destroyed, causing:

  • Unnecessary API calls
  • Increased memory usage
  • Old data being processed

Always unsubscribe in ngOnDestroy or use async pipe.

Q3: What's the order of lifecycle hooks?

Answer: Components go through this sequence:

  1. Constructor - Creates the component instance
  2. OnChanges - When input properties change
  3. OnInit - After first property binding
  4. DoCheck - Custom change detection
  5. AfterContentInit - After content projected
  6. AfterViewInit - After view initialized
  7. OnDestroy - Before component destroyed
