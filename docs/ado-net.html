    💾 ADO.NET

    Raw database access with SqlConnection, SqlCommand, and SqlDataReader — maximum control and performance.

    Intermediate Data Access
    ℹ️

    What is ADO.NET?

    ADO.NET is low-level database access built into the .NET framework. There is no abstraction layer — you write SQL directly and read results row by row. It is faster than EF Core for bulk operations and complex queries, but requires significantly more code. ADO.NET gives you complete control over every SQL statement that runs.

    Core ADO.NET Classes

    ADO.NET is built around a small set of classes in the System.Data.SqlClient (or Microsoft.Data.SqlClient) namespace. Understanding what each class does is the foundation for everything else.

    • SqlConnection Represents a connection to SQL Server. Must be opened before use and closed (or disposed) after. Closing returns the connection to the connection pool — not discarded.
    • SqlCommand Represents a SQL statement or stored procedure to execute against the database. Linked to a SqlConnection and optionally a SqlTransaction.
    • SqlDataReader A forward-only, read-only cursor over query results. Reads one row at a time with while (reader.Read()). The most memory-efficient way to read large result sets.
    • SqlParameter Represents a parameterized input/output value for a SqlCommand. Always use parameters instead of string concatenation to prevent SQL injection.
    • SqlTransaction Wraps multiple SQL statements in a single atomic unit. Either all statements commit or all roll back. Created from an open SqlConnection.
    • SqlDataAdapter Bridges a SqlCommand and a DataSet (disconnected, in-memory table). Useful for batch operations and desktop applications; less common in modern Web APIs.

    Closed Connection Pattern (Web APIs)

    In Web APIs, you open a connection, execute your query, read all results, then immediately close. The using statement guarantees the connection is returned to the pool even if an exception occurs.

    GetAll with SqlDataReader

    C# Repositories/ProductRepository.cs 
    public async Task<List<Product>> GetAllAsync()
{
    var products = new List<Product>();
    var connectionString = _configuration.GetConnectionString("DefaultConnection");

    // 'using' ensures connection is disposed (returned to pool) after the block
    using var connection = new SqlConnection(connectionString);
    await connection.OpenAsync();   // Grab a connection from the pool

    using var command = new SqlCommand(
        "SELECT Id, Name, Price, Category, StockQuantity, IsActive, CreatedAt " +
        "FROM Products WHERE IsActive = 1 ORDER BY Name",
        connection);

    // ExecuteReaderAsync returns a forward-only cursor over the result set
    using var reader = await command.ExecuteReaderAsync();

    while (await reader.ReadAsync())   // Advance to the next row
    {
        products.Add(new Product
        {
            Id            = reader.GetInt32("Id"),
            Name          = reader.GetString("Name"),
            Price         = reader.GetDecimal("Price"),
            Category      = reader.GetString("Category"),
            StockQuantity = reader.GetInt32("StockQuantity"),
            IsActive      = reader.GetBoolean("IsActive"),
            CreatedAt     = reader.GetDateTime("CreatedAt")
        });
    }

    return products;
    // Connection automatically returned to pool here (end of 'using' block)
}

    GetById with Parameterized Query

    C# Repositories/ProductRepository.cs 
    public async Task<Product?> GetByIdAsync(int id)
{
    var connectionString = _configuration.GetConnectionString("DefaultConnection");

    using var connection = new SqlConnection(connectionString);
    await connection.OpenAsync();

    using var command = new SqlCommand(
        "SELECT Id, Name, Price, Category, StockQuantity, Description " +
        "FROM Products WHERE Id = @Id AND IsActive = 1",
        connection);

    // @Id is a named parameter — value is bound separately, never concatenated
    command.Parameters.AddWithValue("@Id", id);

    using var reader = await command.ExecuteReaderAsync();

    if (await reader.ReadAsync())   // Check if a row was returned
    {
        return new Product
        {
            Id            = reader.GetInt32("Id"),
            Name          = reader.GetString("Name"),
            Price         = reader.GetDecimal("Price"),
            Category      = reader.GetString("Category"),
            StockQuantity = reader.GetInt32("StockQuantity"),
            Description   = reader.IsDBNull("Description") ? null : reader.GetString("Description")
        };
    }

    return null;   // No row found for this Id
}
    ⚠️

    Always use parameterized queries

    Never concatenate user input directly into SQL strings. Use @ParameterName placeholders and bind values with Parameters.AddWithValue(). Parameterized queries are also faster — SQL Server can cache and reuse the execution plan.

    CRUD with ADO.NET

    Each operation uses a different SqlCommand execution method: ExecuteReaderAsync for SELECT, ExecuteScalarAsync for INSERT (to retrieve the new ID), and ExecuteNonQueryAsync for UPDATE and DELETE.

    INSERT — Create a New Product

    C# Repositories/ProductRepository.cs 
    public async Task<int> CreateAsync(Product product)
{
    var connectionString = _configuration.GetConnectionString("DefaultConnection");

    using var connection = new SqlConnection(connectionString);
    await connection.OpenAsync();

    // SCOPE_IDENTITY() returns the auto-generated Id for the just-inserted row
    // It is scoped to the current connection — safe in concurrent environments
    using var command = new SqlCommand(@"
        INSERT INTO Products (Name, Price, Category, StockQuantity, Description, IsActive, CreatedAt)
        VALUES (@Name, @Price, @Category, @StockQuantity, @Description, 1, GETUTCDATE());
        SELECT SCOPE_IDENTITY();", connection);

    command.Parameters.AddWithValue("@Name",          product.Name);
    command.Parameters.AddWithValue("@Price",         product.Price);
    command.Parameters.AddWithValue("@Category",      product.Category);
    command.Parameters.AddWithValue("@StockQuantity", product.StockQuantity);
    command.Parameters.AddWithValue("@Description",
        (object?)product.Description ?? DBNull.Value);  // Handle nullable columns

    // ExecuteScalarAsync returns the first column of the first row (the new Id)
    var result = await command.ExecuteScalarAsync();
    return Convert.ToInt32(result);
}

    SELECT with Filtering

    C# Repositories/ProductRepository.cs 
    public async Task<List<Product>> GetByCategoryAsync(string category, decimal? maxPrice = null)
{
    var connectionString = _configuration.GetConnectionString("DefaultConnection");

    using var connection = new SqlConnection(connectionString);
    await connection.OpenAsync();

    // Build query conditionally — still parameterized, never concatenated
    var sql = "SELECT Id, Name, Price, Category FROM Products WHERE IsActive = 1 AND Category = @Category";
    if (maxPrice.HasValue)
        sql += " AND Price <= @MaxPrice";
    sql += " ORDER BY Price ASC";

    using var command = new SqlCommand(sql, connection);
    command.Parameters.AddWithValue("@Category", category);
    if (maxPrice.HasValue)
        command.Parameters.AddWithValue("@MaxPrice", maxPrice.Value);

    var products = new List<Product>();
    using var reader = await command.ExecuteReaderAsync();
    while (await reader.ReadAsync())
    {
        products.Add(new Product
        {
            Id       = reader.GetInt32("Id"),
            Name     = reader.GetString("Name"),
            Price    = reader.GetDecimal("Price"),
            Category = reader.GetString("Category")
        });
    }
    return products;
}

    UPDATE — Modify an Existing Product

    C# Repositories/ProductRepository.cs 
    public async Task<bool> UpdateAsync(int id, Product product)
{
    var connectionString = _configuration.GetConnectionString("DefaultConnection");

    using var connection = new SqlConnection(connectionString);
    await connection.OpenAsync();

    // @@ROWCOUNT returns the number of rows affected by the last statement
    using var command = new SqlCommand(@"
        UPDATE Products
        SET Name          = @Name,
            Price         = @Price,
            Category      = @Category,
            StockQuantity = @StockQuantity,
            UpdatedAt     = GETUTCDATE()
        WHERE Id = @Id AND IsActive = 1;
        SELECT @@ROWCOUNT;", connection);

    command.Parameters.AddWithValue("@Id",            id);
    command.Parameters.AddWithValue("@Name",          product.Name);
    command.Parameters.AddWithValue("@Price",         product.Price);
    command.Parameters.AddWithValue("@Category",      product.Category);
    command.Parameters.AddWithValue("@StockQuantity", product.StockQuantity);

    // ExecuteScalarAsync returns the row count from SELECT @@ROWCOUNT
    var rowsAffected = Convert.ToInt32(await command.ExecuteScalarAsync());
    return rowsAffected > 0;   // true = product found and updated
}

    DELETE — Soft Delete

    C# Repositories/ProductRepository.cs 
    public async Task<bool> DeleteAsync(int id)
{
    var connectionString = _configuration.GetConnectionString("DefaultConnection");

    using var connection = new SqlConnection(connectionString);
    await connection.OpenAsync();

    // Soft delete — UPDATE to IsActive = 0 instead of DELETE FROM
    // The row stays in the database and can be recovered or audited later
    using var command = new SqlCommand(@"
        UPDATE Products
        SET IsActive  = 0,
            UpdatedAt = GETUTCDATE()
        WHERE Id = @Id AND IsActive = 1", connection);

    command.Parameters.AddWithValue("@Id", id);

    // ExecuteNonQueryAsync returns rows affected directly — no SELECT needed
    var rowsAffected = await command.ExecuteNonQueryAsync();
    return rowsAffected > 0;
}

    Parameterized Queries — SQL Injection Prevention

    SQL injection is one of the most common and dangerous web vulnerabilities. An attacker manipulates a query by injecting SQL syntax through user input. Parameterized queries completely prevent this because the value is never interpreted as SQL.

    The Wrong Way (String Concatenation)

    C# DANGEROUS — Do Not Use 
    // DANGEROUS — SQL injection vulnerability
// If name = "'; DROP TABLE Products;--"
// The query becomes: SELECT * FROM Products WHERE Name = ''; DROP TABLE Products;--'
// The attacker just deleted your table.

string name = Request.Query["name"];   // User-controlled input
string sql  = "SELECT * FROM Products WHERE Name = '" + name + "'";

using var command = new SqlCommand(sql, connection);
using var reader  = await command.ExecuteReaderAsync();   // NEVER do this!

    The Right Way (Parameterized)

    C# Repositories/ProductRepository.cs 
    // SAFE — parameterized query
// The value is transmitted separately from the SQL text.
// SQL Server treats @Name as a literal string value, never as SQL syntax.
// Input "'; DROP TABLE Products;--" is safely stored or returned as plain text.

string name = Request.Query["name"];

using var command = new SqlCommand(
    "SELECT * FROM Products WHERE Name = @Name", connection);

// Option 1: AddWithValue — quick, type inferred from the value
command.Parameters.AddWithValue("@Name", name);

// Option 2: Add with explicit type — preferred for production code
// Avoids implicit type conversion issues with SQL Server
command.Parameters.Add("@Name", SqlDbType.NVarChar, 200).Value = name;

using var reader = await command.ExecuteReaderAsync();  // Safe!
    🚨

    NEVER concatenate user input into SQL strings

    String concatenation with user-supplied values is an unconditional SQL injection vulnerability. Always use @ParameterName placeholders and bind values through Parameters.AddWithValue() or Parameters.Add(). This applies to search fields, IDs, category names, and every other user-controlled value.

    Transactions

    A transaction groups multiple SQL statements into a single atomic unit. Either all statements succeed and commit to the database, or any failure causes all statements to roll back — leaving the database exactly as it was before.

    ℹ️

    When to use transactions

    Use transactions when two or more SQL statements must succeed or fail together. Classic examples: transferring money (debit one account, credit another), placing an order (insert order + deduct inventory), and bulk data imports where partial writes would leave inconsistent data.

    C# Repositories/ProductRepository.cs 
    // Transfer stock from one product to another — both UPDATE must succeed or neither applies
public async Task<bool> TransferStockAsync(int fromProductId, int toProductId, int quantity)
{
    var connectionString = _configuration.GetConnectionString("DefaultConnection");

    // Transactions require the "Open Connection" pattern — keep it open for both commands
    using var connection = new SqlConnection(connectionString);
    await connection.OpenAsync();

    // Begin a transaction on the open connection
    using var transaction = await connection.BeginTransactionAsync();

    try
    {
        // STEP 1: Deduct stock from the source product
        using var debitCmd = new SqlCommand(
            "UPDATE Products SET StockQuantity = StockQuantity - @Qty WHERE Id = @FromId",
            connection, (SqlTransaction)transaction);

        debitCmd.Parameters.AddWithValue("@Qty",    quantity);
        debitCmd.Parameters.AddWithValue("@FromId", fromProductId);
        await debitCmd.ExecuteNonQueryAsync();

        // STEP 2: Add stock to the destination product
        using var creditCmd = new SqlCommand(
            "UPDATE Products SET StockQuantity = StockQuantity + @Qty WHERE Id = @ToId",
            connection, (SqlTransaction)transaction);

        creditCmd.Parameters.AddWithValue("@Qty",   quantity);
        creditCmd.Parameters.AddWithValue("@ToId",  toProductId);
        await creditCmd.ExecuteNonQueryAsync();

        // Both steps succeeded — write to the database
        await transaction.CommitAsync();
        return true;
    }
    catch (Exception ex)
    {
        // Something went wrong — undo all changes made in this transaction
        await transaction.RollbackAsync();
        _logger.LogError(ex, "Stock transfer failed, transaction rolled back.");
        return false;
    }
    // Connection returns to pool here — both commands and the transaction are disposed
}

    Connection Pooling

    Opening a database connection is expensive (TCP handshake, authentication, session setup). .NET automatically pools connections so they can be reused. When you call connection.Open(), you are borrowing a connection from the pool, not creating a new one. When you Dispose() the connection, it returns to the pool.

    text appsettings.json — Connection String 
    {
  "ConnectionStrings": {
    "DefaultConnection": "Server=(localdb)\\mssqllocaldb;Database=ProductCatalogDB;Trusted_Connection=true;Max Pool Size=100;Min Pool Size=5;Connection Timeout=30;"
  }
}

// Key connection pool options:
// Max Pool Size=100   — maximum simultaneous connections (default: 100)
// Min Pool Size=5     — connections kept alive (always ready, no wait time)
// Connection Timeout  — seconds to wait if pool is exhausted (throws if exceeded)
    C# Best Practice — using statement 
    // CORRECT — using statement returns connection to pool immediately after work
using (var connection = new SqlConnection(connectionString))
{
    await connection.OpenAsync();    // Borrows from pool
    // ... execute queries ...
}   // connection.Dispose() called here — returns to pool, DOES NOT close TCP socket

// Also correct — using declaration (C# 8+)
using var connection = new SqlConnection(connectionString);
await connection.OpenAsync();
// ... queries ...
// Disposed at end of enclosing scope

// WRONG — holding connection open longer than needed
var connection = new SqlConnection(connectionString);
await connection.OpenAsync();
// ... long business logic, external API calls, waiting ...
// Connection is held the entire time — blocks pool slots unnecessarily
// Other requests starve waiting for a free connection
    • Open connection Borrows a connection from the pool (or creates a new one if pool is empty and under limit)
    • Close / Dispose Returns the connection to the pool — the TCP socket stays open for the next borrower
    • Pool exhausted Request waits up to Connection Timeout seconds, then throws InvalidOperationException
    • Pooling disabled Add Pooling=false to connection string — each open/close creates and destroys a real TCP connection (not recommended)

    EF Core vs ADO.NET — When to Use What

    EF Core and ADO.NET are complementary tools. Most production applications use both: EF Core for the majority of CRUD operations and ADO.NET for the queries that need raw SQL performance or cannot be expressed cleanly in LINQ.

    Criteria EF Core ADO.NET
    Setup time Moderate — models, context, migrations Minimal — just a connection string
    Performance Good — overhead of change tracking & LINQ translation Fastest — zero abstraction overhead
    SQL control Limited — EF generates the SQL Complete — you write every statement
    Complex queries Possible via raw SQL or complex LINQ (verbose) Natural — write any SQL you need
    Schema migrations Built-in — code-first, version-controlled Manual SQL scripts required
    Transactions Supported via BeginTransactionAsync() First-class SqlTransaction support
    Amount of code Less — LINQ replaces SQL boilerplate More — read loops, parameter binding, mapping
    Bulk operations Slow — one INSERT per entity by default Fast — write optimized bulk INSERT / MERGE
    Reporting queries Awkward — complex aggregations are verbose in LINQ Natural — arbitrary GROUP BY, window functions
    ✅

    ProductCatalogAPI uses both — EF Core and ADO.NET

    Standard CRUD (create, update, soft-delete, get by ID) uses EF Core for brevity and type safety. Complex reporting queries, bulk stock operations, and performance-critical endpoints use ADO.NET for raw SQL control. You do not have to choose — mix them in the same service class by injecting both ProductDbContext and IConfiguration.
