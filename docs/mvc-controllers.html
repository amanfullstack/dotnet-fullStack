🎮 Controllers & Actions

Master MVC controllers, action methods, and handling HTTP requests.

What Are Controllers?

In ASP.NET MVC, a controller is a class that handles user requests, processes business logic, and returns responses. Think of it as the traffic controller of your web application—it receives requests, decides what to do, and sends back the appropriate response.

Controllers contain action methods (or just "actions")—public methods that respond to specific URLs. When a user visits a URL, the router finds the matching controller and action, executes it, and returns a result (usually a View with HTML, JSON, or a redirect).

Why Controllers Matter

Controllers are the bridge between user requests and business logic. Without them:

  • No organization: You wouldn't know where to put your code
  • No routing: URLs wouldn't map to specific functionality
  • No security: You couldn't authenticate/authorize users per action
  • Hard to test: Logic would be scattered everywhere

Real example: In ProductCatalog MVC, the ProductController handles all product-related requests: listing products, showing details, editing, and deleting.

Key Concepts

🎯 Action Methods

Public methods in a controller that respond to HTTP requests. Each action can:

  • Accept parameters from the URL, query string, or request body
  • Return different types: ViewResult, JsonResult, RedirectResult
  • Be decorated with HTTP attributes: [HttpGet], [HttpPost], [HttpPut], [HttpDelete]

🛣️ Routing

How URLs map to controllers and actions. The default route pattern is:

/[Controller]/[Action]/[id]

Examples:

  • /Product/Index → ProductController.Index()
  • /Product/Details/5 → ProductController.Details(5)
  • /Home/About → HomeController.About()

📨 HTTP Verbs

Controllers respond to different HTTP methods:

  • GET: Retrieve data (no side effects)
  • POST: Create or update data
  • PUT: Replace entire resource
  • DELETE: Remove data
  • PATCH: Partial update

🔐 Action Filters

Attributes that run before or after an action executes:

  • [Authorize] - Require login
  • [HttpGet] - Only respond to GET requests
  • [ValidateAntiForgeryToken] - Prevent CSRF attacks
  • [ServiceFilter] - Custom filter logic

Code Examples

Example 1: Basic Controller with GET and POST Actions

using Microsoft.AspNetCore.Mvc;
using YourApp.Services;
using YourApp.Models;

[Route("[controller]")]
public class ProductController : Controller
{
    private readonly IProductService _productService;

    // Dependency Injection - service injected via constructor
    public ProductController(IProductService productService)
    {
        _productService = productService;
    }

    // GET: /Product or /Product/Index
    // Returns a list of all products
    [HttpGet]
    public async Task<IActionResult> Index()
    {
        try
        {
            var products = await _productService.GetAllProductsAsync();
            return View(products); // Return View with model
        }
        catch (Exception ex)
        {
            TempData["Error"] = "Error loading products";
            return RedirectToAction("Error", "Home");
        }
    }

    // GET: /Product/Details/5
    // id parameter comes from URL, model binding extracts it
    [HttpGet("{id}")]
    public async Task<IActionResult> Details(int id)
    {
        if (id <= 0)
            return BadRequest("Invalid product ID");

        var product = await _productService.GetProductByIdAsync(id);

        if (product == null)
            return NotFound("Product not found");

        return View(product);
    }

    // POST: /Product/Create
    // Accepts form data, validates it, saves to database
    [HttpPost]
    [ValidateAntiForgeryToken] // CSRF protection
    public async Task<IActionResult> Create([FromForm] ProductViewModel model)
    {
        // ModelState checks if validation passed
        if (!ModelState.IsValid)
        {
            return View(model); // Return form with errors
        }

        try
        {
            await _productService.CreateProductAsync(model);
            TempData["Success"] = "Product created successfully";
            return RedirectToAction(nameof(Index));
        }
        catch (Exception ex)
        {
            ModelState.AddModelError("", "Error creating product: " + ex.Message);
            return View(model);
        }
    }
}

Example 2: Controller with Different Return Types

public class ProductController : Controller
{
    // Returns HTML (ViewResult)
    public IActionResult Index()
    {
        var products = new List<Product> { /* ... */ };
        return View(products); // Renders Index.cshtml
    }

    // Returns JSON (for AJAX calls, mobile apps, etc.)
    [HttpGet("api/search")]
    public IActionResult SearchJson(string term)
    {
        var results = _service.Search(term);
        return Json(results); // Returns JSON data
    }

    // Returns redirect
    [HttpPost]
    public IActionResult Delete(int id)
    {
        _service.DeleteProduct(id);
        return RedirectToAction(nameof(Index));
    }

    // Returns file download
    [HttpGet("{id}/export")]
    public async Task<IActionResult> ExportProductAsync(int id)
    {
        var data = await _service.ExportProductAsync(id);
        return File(data, "application/pdf", "product.pdf");
    }

    // Returns error response
    [HttpGet("{id}")]
    public IActionResult Details(int id)
    {
        var product = _service.GetProduct(id);
        if (product == null)
            return NotFound(); // HTTP 404

        if (!User.IsInRole("Admin"))
            return Forbid(); // HTTP 403

        return Ok(product); // HTTP 200
    }
}

Example 3: Advanced Routing and Model Binding

// Custom route with constraints
[Route("products/{year:int}/{month:int}")]
public IActionResult GetByDate(int year, int month)
{
    // Only matches URLs like /products/2025/04
    var products = _service.GetProductsByDate(year, month);
    return View(products);
}

// Binding from different sources
public IActionResult Search(
    [FromQuery] string term,           // From query string: ?term=value
    [FromRoute] int categoryId,        // From URL route
    [FromBody] FilterModel filters,    // From JSON body (POST)
    [FromHeader] string authorization) // From HTTP header
{
    // Use these parameters...
    return View();
}

// Async action - doesn't block thread while database works
public async Task<IActionResult> ListAsync()
{
    var products = await _service.GetAllAsync(); // Waits without blocking
    return View(products);
}

✅ Best Practices

1. Use Async/Await for I/O Operations

Why: Prevents blocking threads, improves scalability

❌ Bad:

public IActionResult GetProduct(int id)
{
    var product = _service.GetProduct(id); // Blocks thread
    return View(product);
}

✅ Good:

public async Task<IActionResult> GetProduct(int id)
{
    var product = await _service.GetProductAsync(id); // Non-blocking
    return View(product);
}

2. Use Dependency Injection

Why: Makes code testable, loosely coupled, and maintainable

❌ Bad:

public class ProductController : Controller
{
    public IActionResult Index()
    {
        var service = new ProductService(); // Tightly coupled
        var products = service.GetAll();
        return View(products);
    }
}

✅ Good:

public class ProductController : Controller
{
    private readonly IProductService _service;

    public ProductController(IProductService service)
    {
        _service = service; // Injected, loosely coupled
    }

    public async Task<IActionResult> Index()
    {
        var products = await _service.GetAllAsync();
        return View(products);
    }
}

3. Validate Input and Handle Errors

Why: Prevents bad data, improves user experience

public async Task<IActionResult> Create([FromForm] ProductViewModel model)
{
    // Check if model passes validation
    if (!ModelState.IsValid)
        return View(model); // Show errors

    try
    {
        await _service.CreateAsync(model);
        return RedirectToAction(nameof(Index));
    }
    catch (Exception ex)
    {
        // Log error, show user-friendly message
        ModelState.AddModelError("", "Failed to create product");
        return View(model);
    }
}

4. Use Authorization Attributes

Why: Protects sensitive actions from unauthorized access

[Authorize] // Any authenticated user
public IActionResult MyProfile() { }

[Authorize(Roles = "Admin")] // Only admins
public IActionResult AdminPanel() { }

[Authorize(Policy = "CanEditProducts")] // Custom policy
public IActionResult Edit(int id) { }

5. Common Mistakes to Avoid

  • Not validating input: Always check ModelState.IsValid
  • Blocking operations: Use async/await for DB/API calls
  • Hard-coded values: Use dependency injection instead
  • No error handling: Always wrap business logic in try-catch
  • Forgetting CSRF protection: Add [ValidateAntiForgeryToken] to POST actions
  • Exposing sensitive data: Don't return passwords or keys in View

🎓 Interview Q&A

Q1: What's the difference between a Controller action returning View() vs Json()?

Answer:

  • View(): Returns HTML rendered from a .cshtml file. Used for traditional web pages. The browser displays the HTML page.
  • Json(): Returns JSON data. Used for APIs, AJAX calls, or mobile apps. The client (JavaScript, mobile app) receives data and processes it.
// Returns HTML page
public IActionResult Index() => View(products);

// Returns JSON data
[HttpGet("api/products")]
public IActionResult GetJson() => Json(products);

Q2: What is model binding and how does it work?

Answer: Model binding automatically maps HTTP request data (URL, query string, form data, JSON body) to action method parameters. ASP.NET Core does this for you:

// URL: /product/details/5
[HttpGet("{id}")]
public IActionResult Details(int id) // id=5, automatically from URL
{
}

// POST form data
[HttpPost]
public IActionResult Create(ProductViewModel model) // Entire form mapped to model
{
}

// Query string: ?name=Phone&price=99
[HttpGet]
public IActionResult Search(string name, decimal price)
{
}

Q3: Why should actions be async and what happens if they're not?

Answer:

  • Async actions: Don't block threads. One thread can handle many requests while waiting for database/API responses. Better scalability.
  • Sync actions: Block the thread for the entire duration. If your app has 100 threads and all are blocked, new requests fail. Wastes resources.

Impact: Async actions let your server handle 10x more concurrent users with the same hardware.

// Sync - blocks thread for 2 seconds
public IActionResult GetProduct(int id)
{
    var data = _db.Products.Find(id); // Thread blocked here
    return View(data);
}

// Async - thread is freed up
public async Task<IActionResult> GetProductAsync(int id)
{
    var data = await _db.Products.FindAsync(id); // Thread is free
    return View(data);
}
