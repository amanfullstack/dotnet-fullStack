🎨 Micro Frontends / Module Federation

Break your giant React/Angular app into independent micro apps. Each team deploys their own app simultaneously without affecting others.

🏬

Real-World Analogy: Shopping Mall

❌ Monolithic Frontend: One big store. Want to renovate shoes section? Entire store closes.

✅ Micro Frontends: Shopping mall with independent stores. Shoes store renovates, others stay open. Customers navigate freely between stores.

The Monolithic Frontend Problem

One Giant React App Issues:

  • ❌ 5 teams share same codebase (merge conflicts)
  • ❌ Bug in Products tab crashes entire app
  • ❌ Deploy one fix = redeploy everything (risky)
  • ❌ Bundle size grows (50MB+), slow load times
  • ❌ Teams can't use different React versions/libraries
  • ❌ Long feedback loop (code → test → deploy)

Key Concepts

1️⃣ Module Federation (Webpack 5)

Share code at runtime (not build time):

// webpack.config.js
module.exports = {
  output: {
    publicPath: 'http://localhost:3000/'
  },
  plugins: [
    new ModuleFederationPlugin({
      name: 'products',              // App name
      filename: 'remoteEntry.js',    // Entry point
      exposes: {
        './ProductCatalog': './src/components/ProductCatalog',
        './ProductDetail': './src/components/ProductDetail'
      },
      shared: {
        react: { singleton: true },
        'react-dom': { singleton: true }
      }
    })
  ]
};

// Share React between apps = avoid loading twice!
// Products app and Cart app use same React instance

How it works:

  1. Products app (micro) built → outputs remoteEntry.js + components
  2. Products app deployed to http://micro1.com/remoteEntry.js
  3. Host app loads remoteEntry.js at runtime
  4. Host imports ProductCatalog from remote app
  5. Both share same React instance (avoid duplication)

Method 2: Event Bus (More Decoupled)

// shared-event-bus.js (simple pub/sub)
const eventBus = {
  subscribers: {},

  subscribe(event, callback) {
    if (!this.subscribers[event]) {
      this.subscribers[event] = [];
    }
    this.subscribers[event].push(callback);
  },

  publish(event, data) {
    if (this.subscribers[event]) {
      this.subscribers[event].forEach(cb => cb(data));
    }
  }
};

export default eventBus;

Best Practices

✅ Shared React Singleton

Ensure all micro apps load React once. Prevents version conflicts and bundle bloat.

✅ Clear Boundaries

Each micro app owns specific features. Products team ≠ Cart team.

✅ Event-Based Communication

Micro apps publish events, subscribe to events. Loosely coupled.

✅ Independent Deployments

Each micro app has own CI/CD pipeline. Deploy Products without touching Cart.

✅ Separate Versioning

Micro apps can use different library versions independently.

✅ Error Boundaries

One micro app crashes ≠ host app crashes. Use React Error Boundaries.

Common Mistakes

⚠️

❌ Mistake: Tight Coupling Between Micro Apps

If Products app directly imports Cart app code, they're not independent! Use event bus or shared state instead.

⚠️

❌ Mistake: Loading Shared Libraries Multiple Times

If React/Redux loaded 3 times = bundle bloat, memory waste. Configure singleton: true in shared config.

⚠️

❌ Mistake: No Version Compatibility

One micro app uses React 17, another uses 18 = conflicts. Define minimum versions in shared config.

When to Use Micro Frontends

✅ Use when:

  • Large team (5+ developers)
  • Multiple independent features (Products, Cart, Checkout)
  • Want independent deployments
  • Need different tech stacks (one team React, one Vue)
  • Bundle size is problem (code splitting not enough)

❌ Don't use when:

  • Small app, small team
  • Tightly coupled features (need constant sync)
  • Complex shared state management
  • Real-time data sync critical
  • Learning React (adds too much complexity)

Interview Q&A

Q1: How is Micro Frontends different from Lazy Loading?

A: Similar concept but different:

Lazy Loading: One app, split code by routes. ProductCatalog loads when /products visited.

Micro Frontends: Multiple independent apps. Product app is separate codebase, deployment, team!

Analogy: Lazy loading = one bookstore with sections. Micro frontends = separate bookstores that share mall.

Q2: What happens if a micro app crashes?

A: Only that micro app fails, not entire host!

  • Product micro app crashes → /products shows error
  • Host app still works → /home, /cart, /checkout fine
  • Use Suspense + Error Boundary for graceful fallback
// Error boundary wrapper

Q3: How do you manage shared state between micro apps?

A: Three approaches:

1. Redux Store (Simple): Host app provides Redux store, all micro apps access it. Problem: tight coupling.

2. Event Bus (Better): Publish/subscribe pattern. Micro apps emit events, others listen. Loosely coupled.

3. External API: All micro apps call backend API for data. No shared state needed! Most scalable.

Recommendation: Use Event Bus for UI state, API calls for data state.
