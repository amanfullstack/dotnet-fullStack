    🗄️ Entity Framework Core

    ORM for .NET — map C# classes to database tables, write CRUD with LINQ, and manage schema with migrations.

    Intermediate Data Access
    ℹ️

    What is EF Core?

    EF Core is an ORM (Object-Relational Mapper). It lets you work with a database using C# objects instead of raw SQL. You define classes (models), and EF Core handles generating SQL queries, tracking changes, and keeping your schema in sync with your code through migrations.

    Code-First Approach (Recommended)

    In the code-first workflow, you write C# classes first and EF Core generates the database schema from them. This keeps your schema version-controlled alongside your code.

    1. 1
      Define your model class

      Create a plain C# class with properties that map to database columns. Use data annotations or Fluent API to configure constraints.

    2. 2
      Create a DbContext

      Inherit from DbContext and add a DbSet<T> property for each entity. Configure the model inside OnModelCreating.

    3. 3
      Add a migration

      Run dotnet ef migrations add InitialCreate to generate a migration file that describes the schema change.

    4. 4
      Apply the migration

      Run dotnet ef database update to execute the migration SQL against your database. The database is created if it does not exist.

    Entity Model with Data Annotations

    C# Models/Product.cs 
    using System.ComponentModel.DataAnnotations;
using System.ComponentModel.DataAnnotations.Schema;

public class Product
{
    // Primary key — EF Core auto-configures Id as PK by convention
    public int Id { get; set; }

    [Required]
    [MaxLength(200)]
    public string Name { get; set; } = string.Empty;

    [MaxLength(1000)]
    public string? Description { get; set; }

    // decimal(18,2) maps to SQL Server money-safe type
    [Column(TypeName = "decimal(18,2)")]
    public decimal Price { get; set; }

    public int StockQuantity { get; set; }

    [Required]
    [MaxLength(100)]
    public string Category { get; set; } = string.Empty;

    // Soft-delete flag — never remove rows, just hide them
    public bool IsActive { get; set; } = true;

    public DateTime CreatedAt { get; set; } = DateTime.UtcNow;

    public DateTime? UpdatedAt { get; set; }
}

    DbContext Configuration

    C# Data/ProductDbContext.cs 
    using Microsoft.EntityFrameworkCore;

public class ProductDbContext : DbContext
{
    // Constructor receives options from DI (connection string, provider)
    public ProductDbContext(DbContextOptions<ProductDbContext> options)
        : base(options) { }

    // DbSet exposes Products table as a queryable collection
    public DbSet<Product> Products { get; set; }

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        base.OnModelCreating(modelBuilder);

        modelBuilder.Entity<Product>(entity =>
        {
            entity.HasKey(e => e.Id);

            entity.Property(e => e.Name)
                  .IsRequired()
                  .HasMaxLength(200);

            entity.Property(e => e.Price)
                  .HasColumnType("decimal(18,2)");

            // Global query filter — all queries automatically add WHERE IsActive = 1
            entity.HasQueryFilter(p => p.IsActive);

            // Index on Category for fast category filtering
            entity.HasIndex(e => e.Category);
        });
    }
}

    Migration Commands

    bash Terminal 
    # Create a new migration (generates a file in /Migrations)
dotnet ef migrations add InitialCreate

# Apply pending migrations to the database (creates DB if needed)
dotnet ef database update

# View migration history in the database
dotnet ef migrations list
    ✅

    Code-first keeps schema in version control

    Migration files are plain C# code you commit to Git. Every schema change is tracked, reversible, and reviewable in pull requests — the same way you track code changes.

    CRUD Operations

    EF Core wraps your SQL in async LINQ methods. Always use the Async variants (ToListAsync, FindAsync, SaveChangesAsync) in ASP.NET Core to avoid thread starvation.

    Create

    C# Services/ProductService.cs 
    public async Task<Product> CreateAsync(CreateProductDto dto)
{
    var product = new Product
    {
        Name          = dto.Name,
        Price         = dto.Price,
        Category      = dto.Category,
        StockQuantity = dto.StockQuantity,
        Description   = dto.Description,
        IsActive      = true,
        CreatedAt     = DateTime.UtcNow
    };

    // EF Core marks entity as "Added" state
    await _context.Products.AddAsync(product);

    // Generates and executes INSERT INTO Products (...)
    await _context.SaveChangesAsync();

    // product.Id is now populated with the database-assigned ID
    return product;
}

    Read

    C# Services/ProductService.cs 
    // GET ALL — read-only, no change tracking overhead
public async Task<List<Product>> GetAllAsync()
{
    return await _context.Products
        .AsNoTracking()          // Skip change tracking for read-only
        .OrderBy(p => p.Name)
        .ToListAsync();
}

// GET BY ID — FindAsync checks the identity map cache first
public async Task<Product?> GetByIdAsync(int id)
{
    // Best choice for PK lookups — checks DbContext cache first
    return await _context.Products.FindAsync(id);
}

// GET FILTERED — composed LINQ translated to WHERE clause
public async Task<List<Product>> GetByCategoryAsync(string category)
{
    return await _context.Products
        .AsNoTracking()
        .Where(p => p.Category == category)
        .OrderBy(p => p.Price)
        .ToListAsync();
        // SQL: SELECT ... FROM Products WHERE IsActive=1 AND Category=@p0
        //      ORDER BY Price
}

    Update

    C# Services/ProductService.cs 
    public async Task<Product?> UpdateAsync(int id, UpdateProductDto dto)
{
    // FindAsync uses tracked entity if already loaded
    var product = await _context.Products.FindAsync(id);
    if (product == null) return null;

    // Modify properties — EF Core's change tracker detects these
    product.Name          = dto.Name;
    product.Price         = dto.Price;
    product.Category      = dto.Category;
    product.StockQuantity = dto.StockQuantity;
    product.UpdatedAt     = DateTime.UtcNow;

    // EF Core generates: UPDATE Products SET Name=@p0, Price=@p1 ... WHERE Id=@id
    // Only changed columns are included in the UPDATE statement
    await _context.SaveChangesAsync();

    return product;
}

    Delete (Soft vs Hard)

    C# Services/ProductService.cs 
    // SOFT DELETE (recommended) — set IsActive = false, row stays in DB
public async Task<bool> DeleteAsync(int id)
{
    var product = await _context.Products.FindAsync(id);
    if (product == null) return false;

    product.IsActive  = false;           // Hide from all queries (global filter)
    product.UpdatedAt = DateTime.UtcNow;

    await _context.SaveChangesAsync();   // UPDATE Products SET IsActive=0 WHERE Id=@id
    return true;
}

// HARD DELETE — permanently removes the row from the database
public async Task<bool> HardDeleteAsync(int id)
{
    // IgnoreQueryFilters bypasses the IsActive global filter
    var product = await _context.Products
        .IgnoreQueryFilters()
        .FirstOrDefaultAsync(p => p.Id == id);

    if (product == null) return false;

    _context.Products.Remove(product);   // Marks entity as "Deleted" state
    await _context.SaveChangesAsync();   // DELETE FROM Products WHERE Id=@id
    return true;
}
    ⚠️

    Always use async methods in ASP.NET Core

    Use ToListAsync(), FindAsync(), SaveChangesAsync() — never the synchronous versions. Synchronous database calls block the thread pool and reduce throughput under load.

    Change Tracking

    EF Core automatically tracks the state of every entity it loads. When you call SaveChangesAsync(), it generates the exact SQL needed to sync those changes to the database.

    State Meaning SQL Generated
    Added New entity, not yet in DB INSERT INTO ...
    Modified Existing entity with changed properties UPDATE ... SET ...
    Deleted Marked for removal DELETE FROM ...
    Unchanged Loaded but not modified No SQL
    Detached Not tracked by this context No SQL
    C# Services/ProductService.cs 
    // WITH TRACKING (default) — EF Core watches for changes
var product = await _context.Products.FindAsync(id);
product.Price = 799.99m;
await _context.SaveChangesAsync(); // EF detects the change automatically — no explicit Update() call needed

// WITHOUT TRACKING — read-only queries, faster and lower memory
var products = await _context.Products
    .AsNoTracking()        // Tells EF Core: "I won't modify these, skip the tracking overhead"
    .ToListAsync();

// Check entity state manually (useful for debugging)
var entry = _context.Entry(product);
Console.WriteLine(entry.State); // EntityState.Unchanged / Modified / Added / Deleted
    Criteria With Tracking (default) AsNoTracking()
    Speed Slightly slower (bookkeeping overhead) Faster (no bookkeeping)
    Memory Higher (entities cached in context) Lower (no cache)
    Can SaveChanges? Yes — changes auto-detected No — must re-attach or use Attach()
    Best for Create / Update / Delete operations GET endpoints, reports, read-only APIs

    Soft Delete Pattern

    Instead of removing rows, set an IsActive flag to false. A global query filter in OnModelCreating ensures soft-deleted records are automatically excluded from every query without any extra Where() clauses in your service code.

    C# Data/ProductDbContext.cs 
    protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    // Global query filter — automatically appended to EVERY query on Products
    // Every LINQ query now implicitly adds: WHERE IsActive = 1
    modelBuilder.Entity<Product>()
        .HasQueryFilter(p => p.IsActive);

    // These queries are now equivalent:
    //   context.Products.ToListAsync()
    //   context.Products.Where(p => p.IsActive).ToListAsync()

    // To bypass the filter when you need soft-deleted records:
    //   context.Products.IgnoreQueryFilters().ToListAsync()
    //   (useful for admin panels, audit logs, recovery tools)
}
    ✅

    Soft delete preserves data history

    Soft-deleted records are never lost. You can recover them, audit who deleted what and when, and maintain referential integrity. Hard deletes can cause foreign key violations and make audit trails impossible to reconstruct.

    Relationships

    EF Core understands navigation properties and uses them to build JOINs. Define the relationship in your model classes; EF Core infers the foreign key by convention or you can configure it explicitly.

    One-to-Many: Category → Products

    C# Models/Category.cs & Product.cs 
    public class Category
{
    public int Id { get; set; }
    public string Name { get; set; } = string.Empty;

    // Navigation property — collection side (one category has many products)
    public ICollection<Product> Products { get; set; } = new List<Product>();
}

public class Product
{
    public int Id { get; set; }
    public string Name { get; set; } = string.Empty;

    // Foreign key — EF Core matches this to Category.Id by convention
    public int CategoryId { get; set; }

    // Navigation property — reference side (product belongs to one category)
    public Category Category { get; set; } = null!;
}

    Eager Loading with Include and ThenInclude

    C# Services/ProductService.cs 
    // Eager loading — fetch products AND their category in one SQL JOIN
var products = await _context.Products
    .Include(p => p.Category)               // JOIN Categories ON CategoryId = Id
    .AsNoTracking()
    .ToListAsync();

// ThenInclude — load nested relationships (Category -> ParentCategory)
var products = await _context.Products
    .Include(p => p.Category)
        .ThenInclude(c => c.ParentCategory) // JOIN on the related entity's navigation
    .AsNoTracking()
    .ToListAsync();

// Filtered Include — only load active sub-entities (EF Core 5+)
var categories = await _context.Categories
    .Include(c => c.Products.Where(p => p.Price > 100))
    .AsNoTracking()
    .ToListAsync();
    Loading Strategy How Best For
    Eager Loading Include() / ThenInclude() Known upfront — avoids N+1 queries
    Explicit Loading context.Entry(e).Collection(...).LoadAsync() Conditional — only load when needed
    Lazy Loading Proxy navigation (requires config) Avoid in APIs — causes N+1 query problems

    Migrations

    Migrations are versioned, incremental schema change files. Each migration has an Up() method (apply) and a Down() method (revert). EF Core tracks which migrations have been applied in a __EFMigrationsHistory table.

    • migrations add Name Generates a new migration file in /Migrations based on the diff between your models and the last snapshot
    • database update Applies all pending migrations to the database; creates the database if it does not exist
    • database update Name Applies or reverts migrations up to (or back to) a specific migration name
    • migrations remove Removes the last migration file — only works if it has not been applied to the database
    • database drop Drops the entire database (destructive — use only in development)
    • migrations list Lists all migrations and shows which ones have been applied

    Generated Migration File Structure

    C# Migrations/20240101000000_InitialCreate.cs 
    public partial class InitialCreate : Migration
{
    // Up() — applied when running 'database update'
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.CreateTable(
            name: "Products",
            columns: table => new
            {
                Id = table.Column<int>(nullable: false)
                    .Annotation("SqlServer:Identity", "1, 1"),
                Name = table.Column<string>(maxLength: 200, nullable: false),
                Price = table.Column<decimal>(type: "decimal(18,2)", nullable: false),
                Category = table.Column<string>(maxLength: 100, nullable: false),
                IsActive = table.Column<bool>(nullable: false, defaultValue: true),
                CreatedAt = table.Column<DateTime>(nullable: false)
            },
            constraints: table =>
            {
                table.PrimaryKey("PK_Products", x => x.Id);
            });

        migrationBuilder.CreateIndex(
            name: "IX_Products_Category",
            table: "Products",
            column: "Category");
    }

    // Down() — applied when reverting with 'database update PreviousMigration'
    protected override void Down(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.DropTable(name: "Products");
    }
}
    ⚠️

    Never edit generated migration files manually

    Manually editing migration files desynchronizes them from the model snapshot (Migrations/ProductDbContextModelSnapshot.cs) and breaks future migrations. If you need to change a migration, remove it with dotnet ef migrations remove and regenerate it.

    Performance Tips

    EF Core is fast by default, but a few common patterns cause unnecessary overhead. The following table contrasts mistakes with the recommended approaches.

    Scenario Common Mistake Best Practice
    Read-only lists ToListAsync() with tracking on .AsNoTracking().ToListAsync()
    Single PK lookup FirstOrDefaultAsync(p => p.Id == id) FindAsync(id) — checks identity cache first
    Existence check .Count() > 0 — fetches count of all rows .AnyAsync(p => ...) — stops at first match
    Partial data Load entire entity, use 2 fields .Select(p => new { p.Id, p.Name })
    Related data Lazy loading in a loop (N+1) .Include() once before the loop
    Hot path queries Standard LINQ query re-compiled each call EF.CompileAsyncQuery() for compiled queries
    C# Services/ProductService.cs 
    // Select only needed columns — reduces data transfer and memory allocation
var summaries = await _context.Products
    .AsNoTracking()
    .Where(p => p.Category == category)
    .Select(p => new ProductSummaryDto        // Project to DTO — EF translates to SELECT Id, Name, Price
    {
        Id       = p.Id,
        Name     = p.Name,
        Price    = p.Price
    })
    .ToListAsync();

// AnyAsync — stops scanning after the first matching row
bool exists = await _context.Products
    .AnyAsync(p => p.Name == name);

// Compiled query — query plan is compiled once and reused on every call
// Best for frequently called queries (e.g., GetById called hundreds of times/second)
private static readonly Func<ProductDbContext, int, Task<Product?>> GetByIdQuery =
    EF.CompileAsyncQuery(
        (ProductDbContext ctx, int id) => ctx.Products.FirstOrDefault(p => p.Id == id)
    );

// Usage — reuses compiled query plan
var product = await GetByIdQuery(_context, productId);
