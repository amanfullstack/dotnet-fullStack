💼 Services & DI

Understand Angular services, dependency injection, and provider configuration.

What Are Services & DI?

Services are singleton classes that contain reusable business logic, API calls, or data access code. They're injected into components via Dependency Injection (DI), which automatically creates and provides service instances.

DI is a design pattern that handles object creation, reducing tight coupling and improving testability.

Why Services & DI Matter

  • Code reuse: One service shared across many components
  • Separation of concerns: Components display, services handle logic
  • Testability: Mock services for unit tests
  • Single responsibility: Each service does one thing well
  • Maintainability: Change logic once, applies everywhere

Key Concepts

💉 Dependency Injection

Framework automatically provides dependencies to components:

  • Services registered in module providers or component
  • Constructor injection - service passed to constructor
  • Singleton by default - same instance across app
  • Hierarchy: Root, Module, Component level providers

🏢 Service Structure

Services are TypeScript classes decorated with @Injectable():

@Injectable({providedIn: 'root'})
export class DataService {
  // Provide globally
}

📡 HTTP Services

Services typically wrap HttpClient for API calls:

  • Inject HttpClient in service constructor
  • Methods return Observables
  • Components subscribe to get data
  • Centralize API logic in one place

🔄 Providers & Scope

Control when/where services are created:

  • providedIn: 'root' - App-wide singleton
  • providedIn: ModuleClass - Module scope
  • providers: [Service] - Component scope (new instance)

Code Examples

Example 1: Basic Service with HTTP

import { Injectable } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { Observable } from 'rxjs';

@Injectable({
  providedIn: 'root' // Available app-wide
})
export class ProductService {
  private apiUrl = 'http://api.example.com/products';

  constructor(private http: HttpClient) {} // DI in constructor

  getProducts(): Observable<any[]> {
    return this.http.get<any[]>(this.apiUrl);
  }

  getProductById(id: number): Observable<any> {
    return this.http.get<any>(`${this.apiUrl}/${id}`);
  }

  createProduct(product: any): Observable<any> {
    return this.http.post<any>(this.apiUrl, product);
  }

  updateProduct(id: number, product: any): Observable<any> {
    return this.http.put<any>(`${this.apiUrl}/${id}`, product);
  }

  deleteProduct(id: number): Observable<void> {
    return this.http.delete<void>(`${this.apiUrl}/${id}`);
  }
}

Example 2: Using Service in Component

import { Component, OnInit } from '@angular/core';
import { ProductService } from './services/product.service';

@Component({
  selector: 'app-product-list',
  templateUrl: './product-list.component.html'
})
export class ProductListComponent implements OnInit {
  products: any[] = [];
  loading = true;
  error: string | null = null;

  constructor(private productService: ProductService) {} // Inject service

  ngOnInit() {
    this.loadProducts();
  }

  loadProducts() {
    this.loading = true;
    this.productService.getProducts().subscribe(
      (data) => {
        this.products = data;
        this.loading = false;
      },
      (error) => {
        this.error = 'Failed to load products';
        this.loading = false;
      }
    );
  }

  deleteProduct(id: number) {
    this.productService.deleteProduct(id).subscribe(
      () => {
        this.products = this.products.filter(p => p.id !== id);
      },
      (error) => console.error('Delete failed', error)
    );
  }
}

Example 3: Shared State Service

import { Injectable } from '@angular/core';
import { BehaviorSubject, Observable } from 'rxjs';

@Injectable({providedIn: 'root'})
export class ShoppingCartService {
  private cartItems = new BehaviorSubject<any[]>([]);
  cart$ = this.cartItems.asObservable();

  addItem(item: any) {
    const current = this.cartItems.value;
    this.cartItems.next([...current, item]);
  }

  removeItem(id: number) {
    const updated = this.cartItems.value.filter(item => item.id !== id);
    this.cartItems.next(updated);
  }

  getTotal(): number {
    return this.cartItems.value.reduce((sum, item) => sum + item.price, 0);
  }
}

// Use in component
export class CartComponent {
  cartItems$ = this.cart.cart$;

  constructor(private cart: ShoppingCartService) {}

  onRemove(id: number) {
    this.cart.removeItem(id);
  }
}

✅ Best Practices

1. Use providedIn: 'root' for Global Services

Why: Tree-shakeable, only included if used, modern approach

@Injectable({providedIn: 'root'})
export class MyService { }

2. Inject Services in Constructor

Why: Clear dependencies, easier to test

constructor(
  private http: HttpClient,
  private logger: LoggerService
) { }

3. Return Observables, Let Components Subscribe

Why: Services stay generic, components control subscription

// Service - returns observable
getData(): Observable<Data> {
  return this.http.get<Data>(url);
}

// Component subscribes
this.service.getData().subscribe(data => { });

4. Use BehaviorSubject for Shared State

Why: Multiple components can access same state

See Example 3 above for cart service pattern.

5. Handle Errors in Services

Why: Provide meaningful error feedback

getProducts(): Observable<any[]> {
  return this.http.get<any[]>(this.apiUrl).pipe(
    catchError(error => {
      console.error('API error:', error);
      return throwError(() => new Error('Failed to fetch products'));
    })
  );
}

🎓 Interview Q&A

Q1: What's the difference between providedIn and providers array?

Answer:

  • providedIn: 'root' - App-wide singleton, created once, tree-shakeable
  • providers: [Service] in component - New instance per component, never shared
  • providedIn: ModuleClass - Module scope singleton

Q2: How do you pass data between sibling components?

Answer: Use a shared service with BehaviorSubject or Subject:

// Service
@Injectable()
export class DataService {
  private data$ = new BehaviorSubject<string>('');
  data = this.data$.asObservable();

  updateData(value: string) {
    this.data$.next(value);
  }
}

// Component A - sends data
this.service.updateData('Hello');

// Component B - receives data
this.service.data.subscribe(value => console.log(value));

Q3: What's a service provider and when would you create one?

Answer: Provider configures how DI creates service instances. Use custom providers for:

  • Different implementations based on environment
  • Configuration-based setup
  • Factory functions
// Use mock service in testing
providers: [
  { provide: ProductService, useClass: MockProductService }
]
