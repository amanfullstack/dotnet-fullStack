    🗄️ Data Access: EF Core & ADO.NET

    How ProductCatalogAPI reads and writes data — EF Core with LINQ, raw ADO.NET with SqlCommand, and when to use each.

    Intermediate Data Access

    EF Core vs ADO.NET vs GraphQL

    The ProductCatalogAPI uses three data access strategies. Each has its place:

    FeatureEF Core 8ADO.NETGraphQL
    Abstraction levelHigh — ORMLow — raw SQLQuery language over API
    Code styleLINQ, strongly typedSQL strings + readersSchema + resolvers
    PerformanceGood (cached queries)Best (direct ADO)Flexible over-fetching fix
    Migrations✅ Built-in❌ Manual SQL scriptsN/A (uses EF/ADO under hood)
    Best forCRUD, most operationsBulk ops, stored procs, perf-criticalComplex queries from clients

    EF Core 8 in ProductCatalogAPI

    ℹ️

    What is EF Core?

    Entity Framework Core is an Object-Relational Mapper (ORM). It lets you work with database rows as C# objects and write database queries using LINQ instead of raw SQL. EF Core then translates LINQ to SQL behind the scenes.

    DbContext Setup

    The DbContext is your gateway to the database. It holds DbSet<T> properties — one per table.

    C# — ProductDbContext.cs
    public class ProductDbContext : DbContext
{
    // ← Each DbSet maps to a database table
    public DbSet<Product> Products { get; set; }

    public ProductDbContext(DbContextOptions<ProductDbContext> options)
        : base(options) { }

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        // Global query filter — automatically excludes soft-deleted rows
        // Any query on Products will add WHERE IsActive = 1 automatically
        modelBuilder.Entity<Product>()
            .HasQueryFilter(p => p.IsActive);

        // Composite index for faster category filtering
        modelBuilder.Entity<Product>()
            .HasIndex(p => p.Category);

        // Precise decimal storage (money-safe)
        modelBuilder.Entity<Product>()
            .Property(p => p.Price)
            .HasColumnType("decimal(18,2)");
    }
}

    Registering DbContext in Program.cs

    C# — Program.cs
    // Register EF Core with SQL Server
builder.Services.AddDbContext<ProductDbContext>(options =>
    options.UseSqlServer(
        builder.Configuration.GetConnectionString("DefaultConnection")
    )
);

// Auto-create database on startup (dev only — use migrations in prod)
using var scope = app.Services.CreateScope();
var db = scope.ServiceProvider.GetRequiredService<ProductDbContext>();
db.Database.EnsureCreated();

    CRUD with LINQ

    EF Core translates LINQ method chains directly to SQL. The translation happens at runtime when you call ToListAsync() or FirstOrDefaultAsync().

    C# — ProductService.cs (EF Core CRUD)
    // READ ALL — translates to: SELECT * FROM Products WHERE IsActive = 1
var products = await _context.Products
    .AsNoTracking()           // faster reads — no change tracking needed
    .OrderBy(p => p.Name)
    .ToListAsync();

// READ ONE — SELECT TOP 1 * FROM Products WHERE Id = @id AND IsActive = 1
var product = await _context.Products
    .FirstOrDefaultAsync(p => p.Id == id);

if (product is null)
    throw new NotFoundException($"Product {id} not found");

// READ by CATEGORY — WHERE Category = @category AND IsActive = 1
var filtered = await _context.Products
    .Where(p => p.Category == category)
    .AsNoTracking()
    .ToListAsync();

// CREATE
var product = new Product
{
    Name = request.Name,
    Price = request.Price,
    Category = request.Category,
    Stock = request.Stock,
    IsActive = true,
    CreatedAt = DateTime.UtcNow
};
_context.Products.Add(product);
await _context.SaveChangesAsync();   // ← INSERT INTO Products ...

// UPDATE — EF Core tracks the entity and generates UPDATE on SaveChanges
product.Name = request.Name;
product.Price = request.Price;
product.UpdatedAt = DateTime.UtcNow;
await _context.SaveChangesAsync();   // ← UPDATE Products SET ... WHERE Id = @id

// SOFT DELETE — set IsActive = false instead of DELETE
product.IsActive = false;
product.UpdatedAt = DateTime.UtcNow;
await _context.SaveChangesAsync();   // ← UPDATE Products SET IsActive=0 WHERE Id = @id

    Migrations

    Migrations are version-controlled snapshots of your database schema. Run them from the terminal:

    Terminal — EF Core CLI
    # Create a new migration after model changes
dotnet ef migrations add InitialCreate

# Apply pending migrations to database
dotnet ef database update

# Revert to a previous migration
dotnet ef database update PreviousMigrationName

# View migration history
dotnet ef migrations list

# Remove last migration (before applying to DB)
dotnet ef migrations remove
    ⚠️

    EnsureCreated vs Migrations

    EnsureCreated() creates the schema from scratch — it skips migrations entirely. Use it only in development/prototyping. In production always use dotnet ef database update to apply migrations incrementally without data loss.

    ADO.NET in ProductCatalogAPI

    ℹ️

    What is ADO.NET?

    ADO.NET is the low-level .NET data access technology. You write SQL strings directly and use SqlConnection, SqlCommand, and SqlDataReader to execute them. No ORM magic — full control, maximum performance.

    SqlConnection & SqlCommand

    C# — ADO.NET Setup
    // Inject connection string via IConfiguration
public class ProductAdoService
{
    private readonly string _connStr;

    public ProductAdoService(IConfiguration config)
    {
        _connStr = config.GetConnectionString("DefaultConnection")!;
    }
}

    Reading Data with SqlDataReader

    C# — SELECT with SqlDataReader
    public async Task<List<Product>> GetAllAsync()
{
    var products = new List<Product>();

    // 'using' ensures connection is closed/disposed even if exception is thrown
    await using var conn = new SqlConnection(_connStr);
    await conn.OpenAsync();

    const string sql = "SELECT Id, Name, Price, Category, Stock FROM Products WHERE IsActive = 1";

    await using var cmd = new SqlCommand(sql, conn);
    await using var reader = await cmd.ExecuteReaderAsync();

    while (await reader.ReadAsync())
    {
        products.Add(new Product
        {
            Id       = reader.GetInt32(0),
            Name     = reader.GetString(1),
            Price    = reader.GetDecimal(2),
            Category = reader.GetString(3),
            Stock    = reader.GetInt32(4)
        });
    }

    return products;
}

    Parameterized Queries — Prevent SQL Injection

    C# — Parameterized INSERT / SELECT
    // ✅ CORRECT — parameterized query (safe from SQL injection)
public async Task<Product?> GetByIdAsync(int id)
{
    await using var conn = new SqlConnection(_connStr);
    await conn.OpenAsync();

    // @id is a parameter — NEVER use string concatenation
    const string sql = "SELECT * FROM Products WHERE Id = @id AND IsActive = 1";

    await using var cmd = new SqlCommand(sql, conn);
    cmd.Parameters.AddWithValue("@id", id);     // ← safe substitution

    await using var reader = await cmd.ExecuteReaderAsync();

    if (await reader.ReadAsync())
    {
        return new Product { Id = reader.GetInt32(0), Name = reader.GetString(1) };
    }
    return null;
}

// ❌ WRONG — SQL injection vulnerability
// var sql = $"SELECT * FROM Products WHERE Id = {id}";  // NEVER do this!

    ExecuteNonQuery & ExecuteScalar

    C# — INSERT, UPDATE, DELETE, Scalar
    // ExecuteNonQuery — for INSERT / UPDATE / DELETE (returns rows affected)
public async Task<int> CreateAsync(CreateProductRequest req)
{
    await using var conn = new SqlConnection(_connStr);
    await conn.OpenAsync();

    const string sql = @"
        INSERT INTO Products (Name, Price, Category, Stock, IsActive, CreatedAt)
        VALUES (@name, @price, @category, @stock, 1, @created);
        SELECT SCOPE_IDENTITY();";   // ← return new Id

    await using var cmd = new SqlCommand(sql, conn);
    cmd.Parameters.AddWithValue("@name",     req.Name);
    cmd.Parameters.AddWithValue("@price",    req.Price);
    cmd.Parameters.AddWithValue("@category", req.Category);
    cmd.Parameters.AddWithValue("@stock",    req.Stock);
    cmd.Parameters.AddWithValue("@created",  DateTime.UtcNow);

    // ExecuteScalar — returns single value (the new ID from SCOPE_IDENTITY)
    var result = await cmd.ExecuteScalarAsync();
    return Convert.ToInt32(result);
}

// Count with ExecuteScalar
public async Task<int> CountAsync()
{
    await using var conn = new SqlConnection(_connStr);
    await conn.OpenAsync();
    await using var cmd = new SqlCommand("SELECT COUNT(*) FROM Products WHERE IsActive=1", conn);
    return (int)(await cmd.ExecuteScalarAsync())!;
}

// Soft delete with ExecuteNonQuery
public async Task DeleteAsync(int id)
{
    await using var conn = new SqlConnection(_connStr);
    await conn.OpenAsync();
    await using var cmd = new SqlCommand(
        "UPDATE Products SET IsActive=0 WHERE Id=@id", conn);
    cmd.Parameters.AddWithValue("@id", id);
    int rowsAffected = await cmd.ExecuteNonQueryAsync();
    if (rowsAffected == 0) throw new NotFoundException($"Product {id} not found");
}

    Transactions in ADO.NET

    C# — SqlTransaction
    // Wrap multiple operations in a transaction — all succeed or all fail
public async Task TransferStockAsync(int fromId, int toId, int qty)
{
    await using var conn = new SqlConnection(_connStr);
    await conn.OpenAsync();

    await using var tx = conn.BeginTransaction();  // ← start transaction
    try
    {
        await using var decreaseCmd = new SqlCommand(
            "UPDATE Products SET Stock = Stock - @qty WHERE Id = @id", conn, tx);
        decreaseCmd.Parameters.AddWithValue("@qty", qty);
        decreaseCmd.Parameters.AddWithValue("@id",  fromId);
        await decreaseCmd.ExecuteNonQueryAsync();

        await using var increaseCmd = new SqlCommand(
            "UPDATE Products SET Stock = Stock + @qty WHERE Id = @id", conn, tx);
        increaseCmd.Parameters.AddWithValue("@qty", qty);
        increaseCmd.Parameters.AddWithValue("@id",  toId);
        await increaseCmd.ExecuteNonQueryAsync();

        await tx.CommitAsync();   // ← both succeeded → commit
    }
    catch
    {
        await tx.RollbackAsync(); // ← any failure → rollback both
        throw;
    }
}

    When to Use Each Approach

    ScenarioUseReason
    Standard CRUD (most cases)EF CoreLess code, auto-migrations, change tracking
    Complex joins / reporting queriesEF Core + LINQFluent API composes well
    Bulk insert/update (10k+ rows)ADO.NETNo change tracking overhead; use SqlBulkCopy
    Stored procedures with complex logicADO.NETDirect SqlCommand with CommandType.StoredProcedure
    Flexible client-driven queriesGraphQLClient specifies exactly what fields it needs
    Legacy database (no migrations possible)ADO.NETFull control over raw SQL
    Performance-critical hot pathDapper or ADO.NETMinimal overhead vs EF Core
    💡

    Interview Tips — Data Access

    • Q: What is the difference between EF Core and ADO.NET? — EF Core is an ORM (abstract, LINQ, auto-SQL). ADO.NET is raw (SQL strings, readers, full control).
    • Q: What is AsNoTracking()? — Disables EF Core's change-tracking for read-only queries. Faster because EF doesn't keep an in-memory snapshot of returned entities.
    • Q: How do you prevent SQL injection in ADO.NET? — Always use parameterized queries (cmd.Parameters.AddWithValue). Never concatenate user input into SQL strings.
    • Q: What is a soft delete? — Setting IsActive = false instead of running DELETE. Data is preserved for auditing; EF Core's HasQueryFilter automatically filters it out.
    • Q: When would you use a transaction? — When multiple operations must succeed or fail together (e.g., debit one account, credit another). Use SqlTransaction in ADO.NET or SaveChangesAsync() wraps everything in a transaction automatically in EF Core.
