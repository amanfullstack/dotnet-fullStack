🔌 Microservices Architecture

Break your massive backend into small, independent services. Scale, deploy, and fix individually without breaking everything.

🏪

Real-World Analogy: Restaurant vs One Chef

❌ Monolith (One Chef does everything): One person cooks, plates, cleans, serves. If chef gets sick = restaurant closes.

✅ Microservices (Specialized team): Grill chef cooks, plating chef arranges, cleaner cleans. Each person specializes. One person sick ≠ restaurant closes.

The Monolith Problem

Single Large App Issues:

  • ❌ One bug = entire app crashes
  • ❌ Deploy small fix = redeploy EVERYTHING (risky)
  • ❌ Scaling: Need more traffic? Scale entire app (even parts you don't need)
  • ❌ Tech debt: Huge codebase, hard to understand
  • ❌ Team bottleneck: 5 teams waiting on same codebase

Key Concepts

1️⃣ Service Boundaries (Dividing the Monolith)

Question: Where do I split? Answer: By business domain!

Each service = one business capability:

Product Service        → Manage products, search, details
Order Service          → Create orders, track status
Payment Service        → Process payments, invoices
Inventory Service      → Stock levels, reservations
User/Auth Service      → User accounts, login, profiles
Notification Service   → Send emails, SMS, push notifications

Why by domain? Each domain has different requirements (Product = read-heavy, Payment = needs transactions)

2️⃣ Database Per Service Pattern

❌ WRONG: All services share one database = tightly coupled!

✅ RIGHT: Each service owns its database

-- Product Service Database
CREATE TABLE products (id, name, price, category);

-- Order Service Database
CREATE TABLE orders (id, userId, totalAmount, status);

-- Payment Service Database
CREATE TABLE payments (id, orderId, amount, status);

-- Each team manages their own DB!
-- Can use different databases if needed
-- Product: SQL (structured)
-- Payments: NoSQL (document-based for transactions)

If services need data? Call another service's API (not the database directly)

3️⃣ Service Communication Patterns

Synchronous (REST/gRPC) - Wait for Answer

Order Service → (HTTP Call) → Inventory Service
"Do we have stock?"
← (Response) ← "Yes! 5 items left"

✅ Pros: Simple, immediate feedback
❌ Cons: If Inventory Service slow = Order Service slow
         If Inventory down = Order Service fails

Asynchronous (Message Queue) - Fire and Forget

Order Service → (Publish Event) → Message Queue
"Order created!"
                                  ↓
                          Inventory Service
                          (processes when ready)

✅ Pros: Services independent, no waiting
❌ Cons: Eventual consistency (data not updated instantly)

4️⃣ API Gateway Pattern

Single entry point for clients:

Client (Mobile App, Web)
        ↓
    ┌─────────────────────────┐
    │   API Gateway           │
    │ - Routes /products      │
    │   to Product Service    │
    │ - Routes /orders        │
    │   to Order Service      │
    │ - Handles auth          │
    │ - Rate limiting         │
    └─────────────────────────┘
        ↓  ↓  ↓  ↓
    (All Services)

✅ Clients don't know service addresses
✅ Can change service locations without client changes
✅ Single point for security, logging

Code Example: REST Service Call

// Order Service calling Inventory Service
public class OrderService
{
    private readonly HttpClient _httpClient;

    public OrderService(HttpClient httpClient)
    {
        _httpClient = httpClient;
    }

    public async Task CreateOrder(Order order)
    {
        // Call Inventory Service to check stock
        var response = await _httpClient.GetAsync(
            "http://inventory-service:5004/api/inventory/" + order.ProductId
        );

        if (!response.IsSuccessStatusCode)
        {
            throw new Exception("Product out of stock!");
        }

        var inventory = await response.Content.ReadAsAsync();

        if (inventory.QuantityAvailable < order.Quantity)
        {
            throw new Exception("Not enough stock!");
        }

        // Create order
        order.Status = "Confirmed";
        // Save to database...

        // Notify Inventory Service (could be async/event)
        await _httpClient.PutAsync(
            $"http://inventory-service:5004/api/inventory/{order.ProductId}/reserve",
            new StringContent(JsonConvert.SerializeObject(
                new { quantity = order.Quantity }
            ))
        );
    }
}

// Program.cs - Dependency Injection
builder.Services.AddHttpClient();

Best Practices

✅ Use API Gateway

Clients talk to gateway, not individual services. Easier to manage, secure.

✅ Database Per Service

Each service owns its data. No shared databases!

✅ Versioning APIs

Use /v1, /v2 for backward compatibility. Services evolve independently.

✅ Circuit Breaker Pattern

If service slow/down, fail fast instead of waiting forever.

✅ Centralized Logging

All services log to one place. Easy debugging across services.

✅ Health Checks

Each service has /health endpoint. Gateway knows who's alive.

Common Mistakes

⚠️

❌ Mistake: Too Many Tiny Services

Breaking everything into 50 services = complexity nightmare. Complexity overhead becomes worse than monolith!

⚠️

❌ Mistake: Shared Database

Services should NOT share database. Defeats the purpose (loose coupling). Access data via APIs only.

⚠️

❌ Mistake: All Synchronous Calls

Service A calls Service B calls Service C = chain failures. Use async/events for non-critical operations.

When to Use Microservices?

✅ Use when:

  • Multiple teams working on same product
  • Different parts need different scaling
  • Want to deploy independently
  • Teams prefer different tech stacks
  • System is complex, large codebase

❌ Don't use when:

  • Small team, simple app
  • Everything is tightly coupled
  • Real-time consistency critical
  • Network is unreliable (distributed system risks)

Interview Q&A

Q1: When should you move from monolith to microservices?

A: When:

  • Teams are blocked (waiting on same code)
  • Scaling one feature means scaling everything
  • Deployment is risky (entire app to update one service)
  • Different services have different requirements (tech, scaling, reliability)

Not just because it's cool! Microservices add complexity.

Q2: How do you handle transactions across services?

A: Two approaches:

Saga Pattern (Distributed Transactions):

Order Service:
1. Create order (status = PENDING)
2. Call Payment Service → "Pay $100"
3. If payment succeeds: Call Inventory → "Reserve product"
4. If anything fails: Rollback all changes

Challenge: Each service manages its own transaction
Solution: Coordinating saga or choreography pattern

Q3: What's a circuit breaker?

A: Pattern to handle service failures gracefully

States:
1. CLOSED (normal)     → Requests flow normally
2. OPEN (circuit broken)→ Service is failing, STOP sending requests
3. HALF_OPEN (testing)  → Try one request, see if service recovered

Example: Inventory Service down
  Order Service detects failures
  → Stops calling Inventory Service (OPEN)
  → Returns "Out of stock" to prevent cascading failures
  → After timeout, tries again (HALF_OPEN)
  → If works: back to normal (CLOSED)
