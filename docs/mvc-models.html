📦 Models & Validation

Understand MVC models, data annotations, and validation strategies.

What Are Models & ViewModels?

In MVC, models represent your data. The "M" in MVC. A ViewModel is a model specifically designed for the View—it contains only the data the view needs, formatted the way the view expects.

Key difference: Don't pass your database entity directly to views. Create a ViewModel instead. This keeps your database schema private and lets views work with the exact data structure they need.

Why Models Matter

  • Validation: Catch bad data before it reaches the database
  • Type safety: Know exactly what data you're working with
  • Separation: Database entities stay hidden from views
  • Formatting: Display data in the format users expect (formatted dates, currency)
  • Security: Don't expose sensitive fields to clients

Key Concepts

📋 ViewModels

Custom classes designed for specific views. Only include data the view needs:

// Bad: Using database entity in view
public IActionResult Index()
{
    var products = _db.Products.ToList(); // Exposes internal structure
    return View(products);
}

// Good: Using ViewModel
public IActionResult Index()
{
    var products = _db.Products
        .Select(p => new ProductViewModel
        {
            Id = p.Id,
            Name = p.Name,
            Price = p.Price
            // Sensitive fields like DatabasePassword are NOT here
        })
        .ToList();
    return View(products);
}

✅ Data Annotations

Attributes that specify validation rules and display hints:

  • [Required] - Field cannot be empty
  • [StringLength(50)] - Max 50 characters
  • [EmailAddress] - Must be valid email
  • [Range(0, 1000)] - Between 0 and 1000
  • [Display(Name = "Product Name")] - Label for UI
  • [RegularExpression(@"^\d{3}-\d{4}$")] - Match pattern

🔍 ModelState

ModelState tracks validation errors. Check it in POST actions:

if (!ModelState.IsValid)
{
    // Some validation failed, show form with errors
    return View(model);
}
// All validation passed, proceed

🔄 Model Binding

Automatic mapping of form/URL data to model properties. Works with nested objects:

  • Simple properties: Name → form field name="Name"
  • Nested objects: Address.City → form field name="Address.City"
  • Collections: Items[0].Name → indexed form fields

Code Examples

Example 1: ViewModel with Validation Attributes

using System.ComponentModel.DataAnnotations;

public class ProductViewModel
{
    public int Id { get; set; }

    [Required(ErrorMessage = "Product name is required")]
    [StringLength(100, MinimumLength = 3,
        ErrorMessage = "Product name must be 3-100 characters")]
    [Display(Name = "Product Name")]
    public string Name { get; set; }

    [Required]
    [Range(0.01, 999999,
        ErrorMessage = "Price must be between $0.01 and $999,999")]
    [Display(Name = "Price")]
    [DataType(DataType.Currency)]
    public decimal Price { get; set; }

    [StringLength(500)]
    [Display(Name = "Description")]
    public string Description { get; set; }

    [Required]
    [Range(1, int.MaxValue, ErrorMessage = "Please select a category")]
    [Display(Name = "Category")]
    public int CategoryId { get; set; }

    [EmailAddress(ErrorMessage = "Invalid email address")]
    [Display(Name = "Supplier Email")]
    public string SupplierEmail { get; set; }

    [Phone]
    [Display(Name = "Supplier Phone")]
    public string SupplierPhone { get; set; }

    [Url]
    [Display(Name = "Website")]
    public string Website { get; set; }

    [Range(0, 5)]
    [Display(Name = "Rating")]
    public int Rating { get; set; }

    [Display(Name = "Created Date")]
    [DisplayFormat(DataFormatString = "{0:yyyy-MM-dd}",
        ApplyFormatInEditMode = true)]
    public DateTime CreatedDate { get; set; }
}

Example 2: Custom Validation

// Custom validation attribute
[AttributeUsage(AttributeTargets.Property)]
public class PriceGreaterThanAttribute : ValidationAttribute
{
    private readonly decimal _minPrice;

    public PriceGreaterThanAttribute(double minPrice)
    {
        _minPrice = (decimal)minPrice;
    }

    protected override ValidationResult IsValid(
        object value, ValidationContext context)
    {
        if (value == null)
            return ValidationResult.Success;

        if (decimal.TryParse(value.ToString(), out decimal price))
        {
            if (price > _minPrice)
                return ValidationResult.Success;
        }

        return new ValidationResult(
            $"Price must be greater than ${_minPrice}");
    }
}

// Usage in ViewModel
public class ProductViewModel
{
    [Required]
    [PriceGreaterThan(9.99)]
    public decimal Price { get; set; }
}

Example 3: Validating in Controller

[HttpPost]
public async Task<IActionResult> Create(ProductViewModel model)
{
    // ModelState automatically filled by model binding
    // All validation attributes are checked

    if (!ModelState.IsValid)
    {
        // Show form with validation errors highlighted
        // ValidationMessageFor in view shows these errors
        return View(model);
    }

    try
    {
        // Map ViewModel to database entity
        var product = new Product
        {
            Name = model.Name,
            Price = model.Price,
            Description = model.Description,
            CategoryId = model.CategoryId
        };

        // Save to database
        await _context.Products.AddAsync(product);
        await _context.SaveChangesAsync();

        TempData["Success"] = "Product created successfully!";
        return RedirectToAction(nameof(Index));
    }
    catch (Exception ex)
    {
        ModelState.AddModelError("",
            "Error saving product: " + ex.Message);
        return View(model);
    }
}

✅ Best Practices

1. Always Use ViewModels, Never Pass Entities

Why: Hides database structure, improves security, makes schema changes easier

❌ Bad:

public IActionResult Details(int id)
{
    var product = _context.Products.Find(id);
    return View(product); // Exposes all fields including internal data
}

✅ Good:

public IActionResult Details(int id)
{
    var product = _context.Products.Where(p => p.Id == id)
        .Select(p => new ProductViewModel
        {
            Id = p.Id,
            Name = p.Name,
            Price = p.Price // Only what view needs
        })
        .FirstOrDefault();
    return View(product);
}

2. Validate on Both Client and Server

Why: Client validation improves UX, server validation protects against cheating

Client: HTML5 + jQuery Validation (fast feedback)

Server: ModelState.IsValid (security)

Both use the same validation attributes!

3. Use [Display] for User-Friendly Labels

Why: Automatic label generation, easy localization

public class UserViewModel
{
    [Display(Name = "Email Address")] // Shows "Email Address" in forms
    public string Email { get; set; }

    [Display(Name = "First Name")]
    public string FirstName { get; set; }
}

4. Format Display Data with [DisplayFormat]

Why: Shows data in user-friendly format without changing the actual value

public class ProductViewModel
{
    [DisplayFormat(DataFormatString = "{0:C}", ApplyFormatInEditMode = false)]
    public decimal Price { get; set; } // Shows as $99.99

    [DisplayFormat(DataFormatString = "{0:yyyy-MM-dd}", ApplyFormatInEditMode = true)]
    public DateTime CreatedDate { get; set; } // Shows as 2025-04-08
}

5. Keep ViewModels Lean

Why: Only include what the view needs, easier to understand and maintain

❌ Bad: ViewModel with 50 properties for a simple form

✅ Good: Separate ViewModels for different views

// ProductListViewModel - for listing
public class ProductListViewModel
{
    public int Id { get; set; }
    public string Name { get; set; }
    public decimal Price { get; set; }
}

// ProductDetailViewModel - for details page
public class ProductDetailViewModel : ProductListViewModel
{
    public string Description { get; set; }
    public int CategoryId { get; set; }
    public DateTime CreatedDate { get; set; }
}

// ProductEditViewModel - for forms
public class ProductEditViewModel
{
    public int Id { get; set; }
    [Required]
    public string Name { get; set; }
    // Only fields that can be edited
}

🎓 Interview Q&A

Q1: Why use ViewModels instead of passing database entities directly?

Answer:

  • Security: Don't expose internal fields (passwords, IDs)
  • Flexibility: Combine multiple entities into one model
  • Separation: Database changes don't break UI
  • Formatting: Format data for display without changing entity
// Entity has sensitive data
public class User
{
    public int Id { get; set; }
    public string Email { get; set; }
    public string PasswordHash { get; set; } // Don't expose this!
    public string InternalNotes { get; set; } // Confidential
}

// ViewModel shows only needed fields
public class UserViewModel
{
    public int Id { get; set; }
    public string Email { get; set; }
    // PasswordHash and InternalNotes not included
}

Q2: How does ModelState.IsValid work and what triggers validation?

Answer:

  • Triggered automatically: When model binding happens in POST action parameters
  • Checks: All [Required], [StringLength], [Email], etc. attributes
  • IsValid: Returns true only if ALL validation passes
  • Errors: Stored in ModelState dictionary with field names
[HttpPost]
public IActionResult Create(ProductViewModel model)
{
    // ModelState automatically populated during model binding
    // Each validation attribute is checked

    if (!ModelState.IsValid)
    {
        // Show form with errors
        // View displays validation messages automatically
        return View(model);
    }
    // Continue...
}

Q3: What's the difference between client-side and server-side validation?

Answer:

  • Client-side: Runs in browser using HTML5 + JavaScript. Fast, better UX, user sees errors immediately.
  • Server-side: Runs on server after form submission. Slower but NECESSARY—users can bypass client validation.
  • Both needed: Use same validation attributes, ASP.NET generates client validators automatically.

The HTML5 attributes are auto-generated from your data annotations!
