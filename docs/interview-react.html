    ⚛️ React Fundamentals

    Modern JavaScript frontend library — components, hooks, and state management for Product Catalog UI

    Frontend Interview Prep 9 Concepts
    ⚛️

    Why React matters

    React is declarative — you describe what the UI should look like, and React handles the updates. This is fundamentally different from jQuery's imperative "find this element and change it" approach. React is component-based, meaning you compose UIs from reusable pieces. One-way data flow (props down, events up) makes debugging straightforward — if state is wrong, you know the source is above the component. React's Virtual DOM efficiently updates only the parts of the real DOM that changed, making even complex UIs fast.

    React Concepts Quick Reference

    Concept Analogy Implementation
    Components Reusable building blocks (like LEGO bricks) export function ProductList() { return <div>...</div> }
    JSX HTML-like syntax that lives in JavaScript <button onClick={handleClick}>Click me</button>
    Props Parameters passed from parent to child (immutable) <ProductCard product={product} />
    State Component's internal memory (mutable, triggers re-render) const [count, setCount] = useState(0)
    Virtual DOM Lightweight copy of real DOM that React uses to calculate changes React compares before/after, applies minimal updates to actual DOM
    Context API Avoids prop drilling; shares state across components without passing through every level const ProductContext = createContext(); <ProductContext.Provider value={state}>
    Hooks Functions that let you use React features in functional components useState, useEffect, useContext, useReducer
    Effect Hook Component lifecycle moments (mount, update, unmount) useEffect(() => { /* runs on mount/update */ }, [dependency])
    Keys in Lists Unique identifier to preserve component state across re-renders .map(product => <Item key={product.id} ... />)

    Interview Questions on React

    Q1. What problem does React solve that jQuery does not? ▼

    jQuery's approach (imperative): find DOM elements and directly manipulate them. $('#products').html(data) works fine for small pages, but as your app grows, it becomes hard to track which parts of the code changed which DOM elements. You end up with spaghetti logic and silent bugs.

    React's approach (declarative): describe your UI as a function of state. When state changes, React re-renders the component automatically. You do not manually update the DOM — React does it for you, efficiently.

    The Virtual DOM advantage: React maintains a lightweight in-memory representation of the DOM. When state changes, React renders to the Virtual DOM, compares it to the previous version (diffing), and updates only the parts of the real DOM that actually changed. This is far faster than jQuery's "find and replace" approach, especially with complex UIs.

    Concrete example — Product Catalog UI: jQuery: manually fetch products, loop through them, build HTML string, find #products element, call .html(htmlString). If a product updates, you re-fetch all products and re-render the whole list. React: <ProductList products={products} />. When a product updates, React re-renders only that product's row. The rest of the page stays untouched.

    Scalability: jQuery works for 100 lines of DOM manipulation. React scales to 100,000 lines. Large teams can work on different components independently because React enforces clear boundaries (props in, events out).

    Q2. Explain JSX syntax — what is it, and how does it compile? ▼

    JSX stands for JavaScript XML. It lets you write HTML-like syntax directly in JavaScript. When you write <button>Click</button> in a React component, it is not actual HTML — it is JSX that the compiler translates to JavaScript function calls.

    Before compilation (JSX you write):

    jsx 
    function ProductCard() {
  return (
    <div className="card">
      <h3>Laptop</h3>
      <p>Price: $999</p>
      <button onClick={() => alert('Added to cart')}>
        Add to Cart
      </button>
    </div>
  );
}

    After compilation (React.createElement calls):

    javascript 
    function ProductCard() {
  return React.createElement(
    'div',
    { className: 'card' },
    React.createElement('h3', null, 'Laptop'),
    React.createElement('p', null, 'Price: $999'),
    React.createElement(
      'button',
      { onClick: () => alert('Added to cart') },
      'Add to Cart'
    )
  );
}

    Key JSX rules:

    • className not class (class is a reserved word in JavaScript)
    • onClick, onChange use camelCase
    • Embed JavaScript expressions with curly braces: {productPrice}
    • Custom components must start with uppercase: <ProductList /> (JSX knows it is a React component, not a DOM element)
    • Self-closing tags are OK: <input type="text" />

    Why JSX? It is more readable than nested React.createElement calls. A component tree with 20 elements becomes a nightmare to read as function calls. JSX is familiar to anyone who knows HTML, making React approachable.

    Q3. What is the difference between Props and State? ▼

    Props = parameters to a component. Passed down from parent to child, immutable. If you need different UI, the parent changes the prop and re-renders the child.

    State = component's internal memory. Mutable (via setState or setters). When state changes, the component re-renders automatically.

    PropsState
    ImmutabilityRead-only (child cannot change)Mutable (use setState/setters)
    SourcePassed from parentLocal to component
    Triggers re-render?Yes, if parent re-renders with new propYes, always
    Use caseConfigure child behavior; pass data downTrack user input, toggles, timers
    Example<ProductCard product={product} />const [liked, setLiked] = useState(false)

    Example — ProductCard component:

    jsx 
    // Props (immutable) — the product is passed down
function ProductCard({ product }) {
  // State (mutable) — track local UI state
  const [liked, setLiked] = useState(false);

  return (
    <div className="card">
      <h3>{product.name}</h3> {/* prop */}
      <p>${product.price}</p> {/* prop */}
      <button onClick={() => setLiked(!liked)}>
        {liked ? '❤️ Liked' : '🤍 Like'}
      </button>
    </div>
  );
}

// Parent passes props
function ProductList() {
  const [products, setProducts] = useState([...]);
  return (
    <div>
      {products.map(product =>
        <ProductCard key={product.id} product={product} />
      )}
    </div>
  );
}

    Key principle: state should live in the component that needs to use it, or in a common parent if multiple components need it. Props flow down; events bubble up.

    Q4. What is prop drilling and how does the Context API solve it? ▼

    Prop drilling = passing props through many levels of components that do not need them, just to get data to a deeply nested child. For example, if ProductList needs to pass user theme preference to a ProductCard 5 levels deep, you would thread it through every intermediate component.

    The problem: refactoring becomes painful (add a new intermediate component, and you have to update 20 prop signatures), and the code is noisy with unrelated props.

    Context API solution: create a Context at the top level, wrap your component tree with a Provider, and deeply nested components can consume the context without intermediate drilling.

    jsx 
    // Step 1: Create ProductContext
import { createContext, useState, useContext } from 'react';

const ProductContext = createContext();

// ProductProvider — manages shared state
export function ProductProvider({ children }) {
  const [products, setProducts] = useState([]);
  const [loading, setLoading] = useState(false);

  return (
    <ProductContext.Provider value={{ products, setProducts, loading, setLoading }}>
      {children}
    </ProductContext.Provider>
  );
}

// Custom hook for convenience
export function useProducts() {
  return useContext(ProductContext);
}

// ─────────────────────────────────────────────
// Step 2: Wrap your app with Provider (App.js or main.jsx)
function App() {
  return (
    <ProductProvider>
      <Navbar />
      <ProductListPage />
      <Footer />
    </ProductProvider>
  );
}

// ─────────────────────────────────────────────
// Step 3: Any component can consume context — no drilling
function ProductListPage() {
  const { products, loading } = useProducts();

  return (
    <div>
      {loading ? <Spinner /> : <ProductList />}
    </div>
  );
}

function ProductList() {
  const { products } = useProducts(); // access WITHOUT prop drilling

  return (
    <div>
      {products.map(p => <ProductItem key={p.id} product={p} />)}
    </div>
  );
}

function ProductItem({ product }) {
  // ProductItem receives product as prop (reasonable level),
  // but if it needed global state, it could call useProducts() directly
  return <div>{product.name}</div>;
}

    When to use Context: theme (dark/light), language, authenticated user, global notifications. NOT for data that changes frequently (like every keystroke) — Context causes all consumers to re-render, which can hurt performance.

    Q5. Explain the useState hook — how does it manage local component state? ▼

    useState lets functional components store and update local state. Before hooks, you needed class components for state. Now, any functional component can have state.

    Syntax: const [state, setState] = useState(initialValue)

    jsx 
    import { useState } from 'react';

function ProductList() {
  // Declare state: [current value, function to update it]
  const [products, setProducts] = useState([]);
  const [searchTerm, setSearchTerm] = useState('');
  const [selectedCategory, setSelectedCategory] = useState('All');

  // When user types in search
  function handleSearch(e) {
    setSearchTerm(e.target.value); // triggers re-render
  }

  // Filter products based on search and category
  const filtered = products.filter(p =>
    p.name.includes(searchTerm) &&
    (selectedCategory === 'All' || p.category === selectedCategory)
  );

  return (
    <div>
      <input
        type="text"
        placeholder="Search products..."
        value={searchTerm}
        onChange={handleSearch}
      />

      <select value={selectedCategory} onChange={(e) => setSelectedCategory(e.target.value)}>
        <option>All</option>
        <option>Electronics</option>
        <option>Clothing</option>
      </select>

      <ul>
        {filtered.map(product =>
          <li key={product.id}>{product.name} - ${product.price}</li>
        )}
      </ul>
    </div>
  );
}

// When you call setSearchTerm('laptop'):
// 1. React updates the state (searchTerm is now 'laptop')
// 2. React re-renders the entire component
// 3. filtered is recalculated
// 4. Virtual DOM is compared to previous render
// 5. Only changed DOM nodes are updated (the list)

    Key rules:

    • Call hooks at the top level of a functional component, not inside loops or conditionals
    • Never call setState directly (e.g., products = []) — always use the setter
    • setState is asynchronous — you cannot rely on the new value immediately after calling it
    • React batches multiple setState calls in an event handler into one re-render for performance
    Q6. Explain the useEffect hook — what are side effects and how do dependencies work? ▼

    useEffect lets you perform side effects in functional components. Side effects = anything that reaches outside the component (fetch data, update DOM, subscribe to events, set timers, etc.).

    Syntax: useEffect(() => { /* effect code */ }, [dependencies])

    When it runs:

    • No dependency array: runs after every render (usually bad — infinite loops)
    • [] (empty array): runs once after initial mount (perfect for fetch on load)
    • [dependency]: runs when the dependency changes
    jsx 
    import { useEffect, useState } from 'react';

function ProductList() {
  const [products, setProducts] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);
  const [category, setCategory] = useState('All');

  // Fetch products when component mounts (run once)
  useEffect(() => {
    console.log('Component mounted — fetching products');

    async function fetchProducts() {
      try {
        setLoading(true);
        const url = category === 'All'
          ? 'http://localhost:5000/api/products'
          : `http://localhost:5000/api/products/category/${category}`;

        const response = await fetch(url);
        const data = await response.json();
        setProducts(data);
        setError(null);
      } catch (err) {
        setError(err.message);
      } finally {
        setLoading(false);
      }
    }

    fetchProducts();
  }, [category]); // re-run when category changes

  // Cleanup: unsubscribe from events or cancel timers
  useEffect(() => {
    console.log('Component mounted or updated');

    const handleKeyPress = (e) => {
      if (e.key === 'Escape') setProducts([]);
    };

    window.addEventListener('keydown', handleKeyPress);

    // Cleanup function: runs when component unmounts or before re-running effect
    return () => {
      console.log('Cleaning up event listener');
      window.removeEventListener('keydown', handleKeyPress);
    };
  }, []); // empty array = cleanup runs on unmount only

  if (loading) return <div>Loading...</div>;
  if (error) return <div>Error: {error}</div>;

  return (
    <div>
      <select value={category} onChange={(e) => setCategory(e.target.value)}>
        <option>All</option>
        <option>Electronics</option>
        <option>Clothing</option>
      </select>
      <ul>
        {products.map(p => <li key={p.id}>{p.name}</li>)}
      </ul>
    </div>
  );
}

    Dependency array mistakes: If you omit dependencies, the effect runs every render (infinite loops if it calls setState). If you pass stale dependencies, the effect uses old values. List every value from the component scope that is used inside the effect.

    Q7. How does the useContext hook work and how does it simplify sharing state? ▼

    useContext subscribes a component to a Context. When Context value changes, all components using that context re-render automatically. It eliminates prop drilling and centralizes state management.

    jsx 
    import { createContext, useContext, useState } from 'react';

// Step 1: Create ProductContext
const ProductContext = createContext();

// Step 2: Provider component (manages state)
export function ProductProvider({ children }) {
  const [cart, setCart] = useState([]);
  const [theme, setTheme] = useState('light');

  function addToCart(product) {
    setCart([...cart, product]);
  }

  function removeFromCart(productId) {
    setCart(cart.filter(p => p.id !== productId));
  }

  return (
    <ProductContext.Provider value={{ cart, addToCart, removeFromCart, theme, setTheme }}>
      {children}
    </ProductContext.Provider>
  );
}

// Step 3: Custom hook for cleaner code
export function useProductContext() {
  const context = useContext(ProductContext);
  if (!context) throw new Error('useProductContext must be used within ProductProvider');
  return context;
}

// ─────────────────────────────────────────────
// Step 4: Any component can consume context
function CartBadge() {
  const { cart } = useProductContext(); // no prop drilling!
  return <span>Cart ({cart.length})</span>;
}

function ProductCard({ product }) {
  const { addToCart } = useProductContext();

  return (
    <div className="card">
      <h3>{product.name}</h3>
      <button onClick={() => addToCart(product)}>
        Add to Cart
      </button>
    </div>
  );
}

// ─────────────────────────────────────────────
// Step 5: Wrap your app
function App() {
  return (
    <ProductProvider>
      <Navbar /> {/* CartBadge inside Navbar sees cart automatically */}
      <ProductList /> {/* ProductCard can add to cart */}
    </ProductProvider>
  );
}

    Performance caveat: when Context value changes, all components consuming it re-render — even if they only use one part of the context value. For large apps, split context into smaller pieces (UserContext, ThemeContext, CartContext) so changes to one do not trigger unnecessary re-renders of the whole tree.

    Q8. What are the different patterns for conditional rendering in JSX? ▼

    React supports several ways to conditionally render content:

    jsx 
    function ProductPage() {
  const [loading, setLoading] = useState(true);
  const [product, setProduct] = useState(null);

  // ═══════════════════════════════════════════════
  // Pattern 1: if/else outside JSX (most readable)
  // ═══════════════════════════════════════════════
  if (loading) {
    return <div>Loading...</div>;
  }

  if (!product) {
    return <div>Product not found</div>;
  }

  // ═══════════════════════════════════════════════
  // Pattern 2: Ternary operator (inline)
  // ═══════════════════════════════════════════════
  return (
    <div>
      <h1>{product.name}</h1>
      {product.inStock ? (
        <button>Buy Now</button>
      ) : (
        <button disabled>Out of Stock</button>
      )}

      {/* ═══════════════════════════════════════════════ */}
      {/* Pattern 3: Logical AND operator (simple true case) */}
      {/* ═══════════════════════════════════════════════ */}
      {product.discount && (
        <span className="badge">{product.discount}% OFF!</span>
      )}

      {/* ═══════════════════════════════════════════════ */}
      {/* Pattern 4: switch/case (multiple branches) */}
      {/* ═══════════════════════════════════════════════ */}
      {(() => {
        switch (product.category) {
          case 'Electronics':
            return <div>📱 Tech specs: {product.specs}</div>;
          case 'Clothing':
            return <div>👕 Size: {product.size}</div>;
          case 'Books':
            return <div>📖 Author: {product.author}</div>;
          default:
            return <div>Category unknown</div>;
        }
      })()}

      {/* ═══════════════════════════════════════════════ */}
      {/* Pattern 5: Early returns in a helper function */}
      {/* ═══════════════════════════════════════════════ */}
      {renderRating(product.rating)}
    </div>
  );
}

// Helper function with early returns (very readable)
function renderRating(rating) {
  if (!rating) return null;
  if (rating < 3) return <span>⭐ Poor</span>;
  if (rating < 4) return <span>⭐⭐⭐ Good</span>;
  return <span>⭐⭐⭐⭐⭐ Excellent</span>;
}

    Best practices:

    • Use if/else outside JSX for major branches (loading, error, not found)
    • Use ternary for simple true/false choices
    • Use && for rendering when a single condition is true
    • Avoid deep nesting (multiple ternaries) — extract to a helper function
    • Never use && with falsy non-boolean values; 0 && content renders 0
    Q9. Why do lists need keys and what happens without them? ▼

    Keys help React identify which items have changed, been added, or removed. Without keys, React uses the array index as the key, which is problematic when the list is filtered, sorted, or has items inserted/removed.

    Problem without keys (or with index keys):

    jsx 
    // BAD: No key or key={index}
function ProductList() {
  const [products, setProducts] = useState([
    { id: 1, name: 'Laptop', liked: false },
    { id: 2, name: 'Phone', liked: false },
    { id: 3, name: 'Tablet', liked: false },
  ]);

  return (
    <ul>
      {products.map((product, index) => (
        <ProductItem
          key={index} {/* ❌ BAD */}
          product={product}
        />
      ))}
    </ul>
  );
}

// Scenario: User likes product 1, then filters by name "Laptop"
// Without proper keys:
// 1. [{ id: 1, liked: true }, { id: 2, liked: false }, { id: 3, liked: false }]
// 2. Filter leaves: [{ id: 1, liked: true }]
// 3. React uses index key: the first <ProductItem key={0} /> is still there
// 4. BUT the component state inside ProductItem (internal liked toggle) persists
// 5. User sees the wrong product as liked!

// ═════════════════════════════════════════════════════
// GOOD: Use unique, stable ID
function ProductList() {
  const [products, setProducts] = useState([
    { id: 1, name: 'Laptop' },
    { id: 2, name: 'Phone' },
    { id: 3, name: 'Tablet' },
  ]);

  return (
    <ul>
      {products.map(product => (
        <ProductItem
          key={product.id} {/* ✅ GOOD */}
          product={product}
        />
      ))}
    </ul>
  );
}

    With correct keys: React knows "this component represents product with id 1" regardless of where it appears in the list. If you filter, delete, or reorder items, React correctly maintains component state for each item.

    Key characteristics:

    • Unique within the list — no duplicates (the product ID, UUID, etc.)
    • Stable — does not change between renders (not index, not random numbers)
    • Not object.key() result — that changes on every render

    When to use index: only if the list is static, non-filterable, and will never have insertions/deletions. In practice, always use a unique ID from your data.

    React Hooks Reference Quick Table

    Hook Purpose Signature Common Use
    useState Manage local component state const [state, setState] = useState(initial) Form inputs, toggles, counters
    useEffect Side effects (fetch, timers, subscriptions) useEffect(() => {}, [deps]) Fetch data on mount, subscribe to events
    useContext Access Context value without drilling const value = useContext(SomeContext) Global state, theme, user auth
    useReducer Complex state logic (alternative to useState) const [state, dispatch] = useReducer(reducer, init) Multiple related state updates, undo/redo
    useCallback Memoize a function (prevent re-creation) const callback = useCallback(fn, [deps]) Pass callbacks to optimized child components
    useMemo Memoize a computed value (expensive calculations) const value = useMemo(() => expensiveFn(), [deps]) Filter/sort large lists, complex calculations
    useRef Access DOM directly or store mutable value const ref = useRef(initial) Focus input, access video player, store previous state
    useLayoutEffect Effect that runs BEFORE paint (rarely needed) useLayoutEffect(() => {}, [deps]) Measure DOM size before rendering, avoid flashing
    useId Generate unique IDs (form labels, ARIA) const id = useId() <label htmlFor={id}> matching
    ✅

    Custom Hooks Pattern

    You can create your own hooks by extracting state logic into reusable functions. Example: function useFetch(url) { const [data, setData] = useState(null); useEffect(() => { fetch(url).then(r => r.json()).then(setData); }, [url]); return data; }. Custom hooks start with use and can call other hooks. This makes code reusable across components.
