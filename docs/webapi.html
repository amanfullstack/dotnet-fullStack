    🌐 ASP.NET Core Web API

    Build RESTful HTTP APIs with ASP.NET Core — controllers, routing, DI, middleware, caching, and more.

    Intermediate Web API
    ℹ️

    What is ASP.NET Core Web API?

    ASP.NET Core Web API is a framework for building HTTP services that can be consumed by browsers, mobile apps, or other services. It follows REST conventions, uses attribute routing, and integrates tightly with dependency injection, middleware, and EF Core for database access.

    What We Built: ProductCatalogAPI

    All examples in this section are drawn from a real ProductCatalogAPI project — a full CRUD API with EF Core, caching, soft delete, and proper architecture. The project is available in the ProductCatalogAPI-WebAPI-EFCore/ folder.

    Framework.NET 8.0 / ASP.NET Core 8
    DatabaseSQL Server LocalDB via EF Core 8
    ArchitectureN-Tier: Controllers → Services → DbContext
    Endpoints7 REST endpoints (GET, POST, PUT, DELETE, PATCH)
    Data AccessEF Core 8 · ADO.NET · GraphQL (HotChocolate)
    CachingIn-Memory + HTTP Response Caching

    Quick Start

    Get the API running locally in under 2 minutes.

    1. 1
      Navigate to the project
      bash 
      cd d:\dotnet-fullStack\ProductCatalogAPI-WebAPI-EFCore
    2. 2
      Restore packages and run
      bash 
      dotnet run

      EF Core will auto-create the database on first run via EnsureCreated().

    3. 3
      Open Swagger UI

      Navigate to http://localhost:5000/swagger to explore and test all endpoints interactively.

    API Endpoints

    The ProductCatalogAPI exposes 7 REST endpoints:

    Method Route Description Response
    GET/api/productsGet all active products200 + List
    GET/api/products/{id}Get product by ID200 / 404
    GET/api/products/category/{cat}Filter by category200 + List
    POST/api/productsCreate new product201 Created
    PUT/api/products/{id}Full update200 / 404
    PATCH/api/products/{id}/stockUpdate stock only200 / 404
    DELETE/api/products/{id}Soft delete (IsActive=false)204 / 404

    Architecture at a Glance

    The API follows a clean N-Tier pattern to keep concerns separated and code testable.

    Request Flow 
    HTTP Request
    ↓
Middleware Pipeline  (Auth, Logging, Error Handling, CORS)
    ↓
ProductsController   (Route matching, model binding, validation)
    ↓
ProductService       (Business logic, caching, orchestration)
    ↓
ProductDbContext     (EF Core → SQL → Database)
    ↓
HTTP Response        (Serialized JSON + status code)
