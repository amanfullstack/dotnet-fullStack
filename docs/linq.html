    🔗 LINQ & Filtering

    Language Integrated Query — filter, sort, project, and aggregate data with elegant C# syntax.

    Intermediate Data Access
    ℹ️

    What is LINQ?

    LINQ (Language Integrated Query) is a set of C# language features and extension methods in System.Linq that let you query any IEnumerable<T> or IQueryable<T> source using the same consistent syntax — whether that source is a list, an array, an XML document, or a SQL database via EF Core. LINQ comes in two syntaxes: method syntax (also called fluent or lambda syntax) and query syntax (SQL-like). Both compile to the same IL.

    Method Syntax vs Query Syntax

    Both syntaxes are equivalent — they produce the same compiled output. Method syntax is the dominant style in modern C# codebases and is the one you will see in EF Core repositories. Query syntax is sometimes more readable for complex multi-join scenarios.

    C# — Same query, two syntaxes 
    var products = new List<Product>
{
    new Product { Id = 1, Name = "Laptop",   Price = 999.99m, Category = "Electronics" },
    new Product { Id = 2, Name = "Mouse",    Price = 29.99m,  Category = "Electronics" },
    new Product { Id = 3, Name = "Desk",     Price = 349.00m, Category = "Furniture" },
    new Product { Id = 4, Name = "Keyboard", Price = 79.99m,  Category = "Electronics" },
};

// ─── Method Syntax (preferred in modern C#) ───────────────────────────────
var methodResult = products
    .Where(p => p.Category == "Electronics" && p.Price < 500)
    .OrderBy(p => p.Price)
    .Select(p => new { p.Name, p.Price });

// ─── Query Syntax (SQL-like) ──────────────────────────────────────────────
var queryResult =
    from p in products
    where p.Category == "Electronics" && p.Price < 500
    orderby p.Price
    select new { p.Name, p.Price };

// Both produce:
// { Name = "Mouse",    Price = 29.99  }
// { Name = "Keyboard", Price = 79.99  }

// Query syntax MUST end with a select or group clause.
// Method syntax can be chained arbitrarily without a terminal keyword.
    💡

    Method syntax is preferred in modern C#

    Method syntax integrates naturally with the rest of C# (lambda expressions, var, IDE auto-complete). Some LINQ operators — Count, Skip, Take, Distinct, Zip — have no query-syntax equivalent and must always be written in method syntax. Use query syntax only if it significantly improves readability (typically for complex multi-source joins).

    Filtering with Where

    Where is the core filtering operator. It accepts a predicate (a lambda returning bool) and returns every element for which the predicate is true. Multiple Where calls can be chained — they are combined with AND logic and, when used with EF Core, merged into a single SQL WHERE clause.

    C# — Filtering patterns 
    // 1. Single condition
var expensive = products.Where(p => p.Price > 500);

// 2. AND — both conditions must be true
var cheapElectronics = products
    .Where(p => p.Category == "Electronics" && p.Price < 100);

// 3. OR — either condition can be true
var budgetOrFurniture = products
    .Where(p => p.Price < 50 || p.Category == "Furniture");

// 4. String methods — all translate to SQL when used with EF Core
var namedLap = products.Where(p => p.Name.Contains("Lap"));          // LIKE '%Lap%'
var startK   = products.Where(p => p.Name.StartsWith("Key"));        // LIKE 'Key%'
var endrd    = products.Where(p => p.Name.EndsWith("rd"));           // LIKE '%rd'
var toLower  = products.Where(p => p.Name.ToLower() == "laptop");    // LOWER(Name) = 'laptop'

// 5. Null checks
var hasDesc  = products.Where(p => p.Description != null);
var noDesc   = products.Where(p => p.Description == null);

// 6. Chained Where — results in a single SQL WHERE with AND
var result = products
    .Where(p => p.IsActive)
    .Where(p => p.Category == "Electronics")   // AND IsActive AND Category = 'Electronics'
    .Where(p => p.Price < 200);               // AND Price < 200

// 7. Dynamic conditional filtering (build query step by step)
IQueryable<Product> query = _context.Products.AsQueryable();

if (!string.IsNullOrEmpty(searchTerm))
    query = query.Where(p => p.Name.Contains(searchTerm));

if (minPrice.HasValue)
    query = query.Where(p => p.Price >= minPrice.Value);

if (maxPrice.HasValue)
    query = query.Where(p => p.Price <= maxPrice.Value);

// Only ONE SQL query is executed, with all conditions combined
var filtered = await query.ToListAsync();

    Select — Projection

    Select transforms each element into a new shape. It is the LINQ equivalent of SQL's SELECT column list. Use it to pick specific properties, compute new values, or map entities to DTOs. When used with EF Core, projection limits the columns in the generated SQL — a significant performance win.

    C# — Select (projection) 
    // 1. Select specific properties — SQL: SELECT Name, Price FROM Products
var names = products.Select(p => p.Name);                 // IEnumerable<string>
var namePrices = products.Select(p => new { p.Name, p.Price });  // anonymous type

// 2. Map to a DTO (recommended over anonymous types for API responses)
var dtos = products
    .Where(p => p.IsActive)
    .Select(p => new ProductDto
    {
        Id          = p.Id,
        Name        = p.Name,
        Price       = p.Price,
        DisplayName = $"{p.Name} ({p.Category})"   // computed value
    })
    .ToList();

// 3. Select with index (in-memory only — not translateable to SQL)
var indexed = products
    .Select((p, index) => new { Index = index + 1, p.Name });

// 4. SelectMany — flatten nested collections into a single sequence
var orders = new List<Order>
{
    new Order { Id = 1, Items = new[] { "Laptop", "Mouse" } },
    new Order { Id = 2, Items = new[] { "Desk" } },
    new Order { Id = 3, Items = new[] { "Keyboard", "Webcam" } }
};

// Without SelectMany you get IEnumerable<string[]>
IEnumerable<string[]> nested = orders.Select(o => o.Items);  // 3 arrays

// With SelectMany you get IEnumerable<string> — all items in one flat list
IEnumerable<string> allItems = orders.SelectMany(o => o.Items);
// [ "Laptop", "Mouse", "Desk", "Keyboard", "Webcam" ]

// 5. SelectMany with result selector — correlate parent and child
var orderItems = orders.SelectMany(
    collectionSelector: o => o.Items,
    resultSelector:     (order, item) => new { order.Id, Item = item }
);
// { Id=1, Item="Laptop" }, { Id=1, Item="Mouse" }, { Id=2, Item="Desk" }, ...

    OrderBy / ThenBy — Sorting

    LINQ provides four primary sorting operators: OrderBy, OrderByDescending, ThenBy, and ThenByDescending. With EF Core these translate directly to SQL ORDER BY clauses.

    C# — Sorting 
    // 1. Ascending (default)
var byPrice = products.OrderBy(p => p.Price);
// SQL: ORDER BY Price ASC

// 2. Descending
var mostExpensive = products.OrderByDescending(p => p.Price);
// SQL: ORDER BY Price DESC

// 3. Multiple sort criteria with ThenBy
var sortedProducts = products
    .OrderBy(p => p.Category)                  // primary sort: category A-Z
    .ThenByDescending(p => p.Price)            // secondary: highest price first
    .ThenBy(p => p.Name);                      // tertiary: name A-Z within same price
// SQL: ORDER BY Category ASC, Price DESC, Name ASC

// 4. Sort by computed value
var byNameLength = products.OrderBy(p => p.Name.Length);

// 5. Conditional sorting (dynamic — build OrderBy at runtime)
IOrderedQueryable<Product> sorted = sortField switch
{
    "name"  => _context.Products.OrderBy(p => p.Name),
    "price" => _context.Products.OrderBy(p => p.Price),
    _       => _context.Products.OrderBy(p => p.Id)
};

if (descending)
    sorted = sorted.OrderByDescending(p => p.Id);   // re-order

// 6. Getting top N results (sort + take)
var top5 = products
    .OrderByDescending(p => p.Price)
    .Take(5)
    .ToList();

// 7. Do NOT use OrderBy before Where — the DB optimiser handles ordering last
// ❌ sub-optimal: order first, then filter
var bad  = products.OrderBy(p => p.Price).Where(p => p.Price > 100);
// ✅ better: filter first, then order
var good = products.Where(p => p.Price > 100).OrderBy(p => p.Price);

    GroupBy and Aggregation

    GroupBy partitions elements into groups based on a key selector. Each group is an IGrouping<TKey, TElement> — it has a Key property and is itself an IEnumerable<TElement> of the grouped items. Aggregation methods (Count, Sum, Average, Min, Max) are usually applied after grouping.

    C# — GroupBy and aggregation 
    // 1. Simple grouping — group products by category
var grouped = products.GroupBy(p => p.Category);

foreach (IGrouping<string, Product> group in grouped)
{
    Console.WriteLine($"Category: {group.Key}  ({group.Count()} items)");
    foreach (Product p in group)
        Console.WriteLine($"  - {p.Name}: {p.Price:C}");
}

// 2. GroupBy with a projection — project each group into a summary object
var categorySummaries = products
    .GroupBy(p => p.Category)
    .Select(g => new
    {
        Category     = g.Key,
        Count        = g.Count(),
        AveragePrice = g.Average(p => p.Price),
        TotalValue   = g.Sum(p => p.Price),
        MinPrice     = g.Min(p => p.Price),
        MaxPrice     = g.Max(p => p.Price),
        Names        = g.Select(p => p.Name).ToList()
    })
    .OrderByDescending(x => x.TotalValue)
    .ToList();

// Result example:
// { Category="Electronics", Count=3, AveragePrice=369.99, TotalValue=1109.97, ... }
// { Category="Furniture",   Count=1, AveragePrice=349.00, TotalValue=349.00,  ... }

// 3. GroupBy with EF Core — generates GROUP BY in SQL
var dbSummary = await _context.Products
    .GroupBy(p => p.CategoryId)
    .Select(g => new
    {
        CategoryId = g.Key,
        Count      = g.Count(),
        Average    = g.Average(p => p.Price)
    })
    .ToListAsync();
// SQL: SELECT CategoryId, COUNT(*), AVG(Price)
//      FROM Products
//      GROUP BY CategoryId

// 4. HAVING equivalent — filter groups after aggregation
var bigCategories = products
    .GroupBy(p => p.Category)
    .Where(g => g.Count() >= 2)              // HAVING COUNT(*) >= 2
    .Select(g => g.Key)
    .ToList();

// 5. Standalone aggregation (no grouping needed)
decimal totalValue  = products.Sum(p => p.Price);          // 1458.97
decimal avgPrice    = products.Average(p => p.Price);      // 364.74
Product cheapest    = products.MinBy(p => p.Price)!;       // Mouse  (C# 10+)
Product mostCostly  = products.MaxBy(p => p.Price)!;       // Laptop (C# 10+)

    Join Operations

    LINQ provides explicit join operators, but in EF Core you rarely need them because navigation properties let you traverse relationships naturally. Understanding when to use each approach is important for both interview discussions and performance optimisation.

    C# — Inner Join 
    var categories = new List<Category>
{
    new Category { Id = 1, Name = "Electronics" },
    new Category { Id = 2, Name = "Furniture" },
    new Category { Id = 3, Name = "Stationery" }  // no products in this category
};

var products = new List<Product>
{
    new Product { Id = 1, Name = "Laptop",   CategoryId = 1, Price = 999.99m },
    new Product { Id = 2, Name = "Mouse",    CategoryId = 1, Price = 29.99m  },
    new Product { Id = 3, Name = "Desk",     CategoryId = 2, Price = 349.00m },
    new Product { Id = 4, Name = "Keyboard", CategoryId = 1, Price = 79.99m  }
};

// ─── Inner Join (Join) ────────────────────────────────────────────────────
// Returns only rows that have a matching record in BOTH collections.
// "Stationery" is excluded because no products have CategoryId = 3.
var innerJoin = products.Join(
    inner:            categories,
    outerKeySelector: p => p.CategoryId,
    innerKeySelector: c => c.Id,
    resultSelector:   (p, c) => new { p.Name, c.Name, p.Price }
);
// Equivalent method syntax with navigation property in EF Core:
// _context.Products.Select(p => new { p.Name, p.Category.Name, p.Price })
    C# — Left Join (GroupJoin + SelectMany) 
    // ─── Left Join (GroupJoin + SelectMany + DefaultIfEmpty) ─────────────────
// Returns ALL categories, even if they have no products.
// Products that match appear as normal; unmatched categories get null product.
var leftJoin = categories
    .GroupJoin(
        inner:            products,
        outerKeySelector: c => c.Id,
        innerKeySelector: p => p.CategoryId,
        resultSelector:   (c, prods) => new { Category = c, Products = prods }
    )
    .SelectMany(
        collectionSelector: x => x.Products.DefaultIfEmpty(),  // null if no products
        resultSelector:     (x, p) => new
        {
            CategoryName = x.Category.Name,
            ProductName  = p?.Name ?? "(none)",
            Price        = p?.Price
        }
    );
// Result includes: "Stationery" / "(none)" / null — the unmatched category

// ─── EF Core — use Include() for navigation properties (simpler!) ─────────
// Rather than explicit joins, EF Core with navigation properties is cleaner:
var withCategory = await _context.Products
    .Include(p => p.Category)             // adds INNER JOIN in SQL automatically
    .Select(p => new
    {
        ProductName  = p.Name,
        CategoryName = p.Category.Name,
        p.Price
    })
    .ToListAsync();

// Use explicit Join only when no navigation property exists or for
// cross-context queries involving two separate DbContexts.
    💡

    Navigation properties vs explicit joins in EF Core

    If your entities have navigation properties configured (e.g., Product.Category), prefer .Include() or dot-navigation in Select. EF Core generates the correct JOIN automatically. Use explicit .Join() only when working with collections that lack a relational key or when joining across contexts.

    Deferred vs Immediate Execution

    This is one of the most important LINQ concepts for performance — and a top interview question. LINQ queries are lazy by default: the query is defined but not executed until you actually iterate over the results.

    Execution Type Operators When it runs
    Deferred (lazy) Where, Select, OrderBy, GroupBy, Skip, Take, SelectMany Not until iterated
    Immediate ToList(), ToArray(), ToDictionary(), Count(), First(), Single(), Sum(), Any(), All(), Max(), Min() Immediately when called
    C# — Deferred vs immediate execution 
    var numbers = new List<int> { 1, 2, 3, 4, 5 };

// ─── Deferred execution — query is defined but NOT executed yet ───────────
IEnumerable<int> query = numbers.Where(n => n > 2);
// At this point: zero iterations have happened. "query" is just a description.

// ─── Deferred — query executes NOW when foreach iterates ─────────────────
foreach (int n in query)           // ← execution starts here
{
    Console.WriteLine(n);          // 3, 4, 5
}

// ─── Deferred — query executes again on second iteration ─────────────────
Console.WriteLine(query.Count()); // query runs AGAIN → 3

// ─── Source modification affects deferred queries ────────────────────────
numbers.Add(6);
Console.WriteLine(query.Count()); // 4 — now includes 6 (query re-evaluates!)

// ─── Immediate execution — snapshot taken right now ─────────────────────
List<int> snapshot = numbers.Where(n => n > 2).ToList();
numbers.Add(7);
Console.WriteLine(snapshot.Count); // still 4 — list is a fixed snapshot

// ─── EF Core: deferred = no SQL yet; immediate = SQL fires ───────────────
// This is just defining the query (IQueryable) — no database round-trip:
IQueryable<Product> dbQuery = _context.Products
    .Where(p => p.Price > 100)
    .OrderBy(p => p.Name);

// Adding more conditions is still free (no DB call):
if (onlyActive)
    dbQuery = dbQuery.Where(p => p.IsActive);

// SQL executes HERE — one round-trip, all conditions in the WHERE clause:
List<Product> results = await dbQuery.ToListAsync();

// ─── Multiple enumerations — a common performance mistake ────────────────
IQueryable<Product> q = _context.Products.Where(p => p.IsActive);

int  total   = await q.CountAsync();    // ← SQL round-trip 1
List<Product> page = await q            // ← SQL round-trip 2
    .Skip(0).Take(10).ToListAsync();

// ✅ Better: capture in one list if the set is small
// Or use a single projection that returns both in one query.
    ⚠️

    Common mistake: modifying the source after defining a deferred query

    Because deferred queries re-evaluate the source on each iteration, modifying a list after defining a LINQ query changes the results of future iterations. This is usually unexpected. When you need a fixed snapshot, always call .ToList() immediately after the query definition.

    Essential LINQ Methods Reference

    • First() Returns the first element matching the predicate. Throws InvalidOperationException if none found. Use when you are certain at least one exists.
    • FirstOrDefault() Returns the first match or default(T) (null for reference types) if none found. The safe alternative to First().
    • Single() Returns the one and only element matching the predicate. Throws if zero OR more than one element matches. Use to assert uniqueness.
    • SingleOrDefault() Returns the match or null; throws only if MORE than one element matches. Use when zero is acceptable but duplicates indicate a bug.
    • Any() Returns true if at least one element matches the predicate (or if the sequence is non-empty when called without a predicate). More efficient than Count() > 0.
    • All() Returns true if every element in the sequence satisfies the predicate. Short-circuits on the first failure.
    • Count() Returns the number of elements matching the optional predicate. For EF Core: generates SELECT COUNT(*). Avoid using Count() > 0 as an existence check — use Any() instead.
    • Sum() / Average() Aggregates a numeric property across all elements. For EF Core: SELECT SUM(…) / SELECT AVG(…).
    • Min() / Max() Returns the minimum or maximum value of a property. Use MinBy() / MaxBy() (C# 10) to return the entire object with the min/max key.
    • Distinct() Removes duplicate elements from the sequence. Works on the full object (requires Equals / GetHashCode) or use DistinctBy(p => p.CategoryId) (C# 10).
    • Take(n) Returns the first n elements. Essential for pagination. EF Core generates FETCH NEXT n ROWS / LIMIT n.
    • Skip(n) Skips the first n elements and returns the rest. Used with Take for page-based pagination: Skip(page * size).Take(size).
    • ToList() Materialises the query into a List<T>. Triggers immediate execution. Use at the end of a query chain.
    • ToDictionary() Materialises into a Dictionary<TKey, TValue>. Throws on duplicate keys. Useful for building fast-lookup maps: .ToDictionary(p => p.Id).
    • Contains() Checks whether the sequence contains a specific value. With EF Core and a collection of values it generates an IN (…) clause: .Where(p => ids.Contains(p.Id)).

    Performance Tips

    LINQ is expressive but can silently hide expensive operations. These guidelines will keep your LINQ code fast in both in-memory and database-backed scenarios.

    C# — LINQ performance patterns 
    // ─── 1. Use IQueryable with EF Core, not IEnumerable ─────────────────────
// ❌ Bad — loads ALL products, then filters in memory
IEnumerable<Product> bad = _context.Products;
var result = bad.Where(p => p.Price > 100).ToList();

// ✅ Good — filter is sent to the database
IQueryable<Product> good = _context.Products.Where(p => p.Price > 100);
var result2 = good.ToList();


// ─── 2. Select only the columns you need ─────────────────────────────────
// ❌ Bad — SELECT * (transfers all 20 columns across network)
var allColumns = await _context.Products
    .Where(p => p.IsActive)
    .ToListAsync();

// ✅ Good — SELECT Id, Name, Price (only 3 columns)
var projected = await _context.Products
    .Where(p => p.IsActive)
    .Select(p => new ProductDto { Id = p.Id, Name = p.Name, Price = p.Price })
    .ToListAsync();


// ─── 3. Use Any() instead of Count() > 0 ────────────────────────────────
// ❌ Bad — COUNT(*) scans all matching rows
bool existsBad = _context.Products.Where(p => p.IsActive).Count() > 0;

// ✅ Good — EXISTS query stops at the first match
bool existsGood = await _context.Products.AnyAsync(p => p.IsActive);


// ─── 4. Use First() not ToList().First() ─────────────────────────────────
// ❌ Bad — loads ALL matching products, then picks the first in memory
Product bad2 = _context.Products.Where(p => p.Name == "Laptop").ToList().First();

// ✅ Good — SELECT TOP 1 / LIMIT 1 in the SQL
Product good2 = await _context.Products.FirstOrDefaultAsync(p => p.Name == "Laptop")!;


// ─── 5. Avoid N+1 queries — use Include() instead of lazy loading ─────────
// ❌ Bad — 1 query for products + 1 query per product for category
var productsBad = await _context.Products.ToListAsync();
foreach (var p in productsBad)
{
    Console.WriteLine(p.Category.Name);  // triggers a new SQL query per product!
}

// ✅ Good — single JOIN query loads both products and categories
var productsGood = await _context.Products
    .Include(p => p.Category)            // LEFT JOIN Categories
    .ToListAsync();


// ─── 6. Pagination — always use Skip + Take on IQueryable ────────────────
int page     = 2;
int pageSize = 20;
var paged = await _context.Products
    .Where(p => p.IsActive)
    .OrderBy(p => p.Id)              // ORDER BY is required before Skip/Take
    .Skip((page - 1) * pageSize)
    .Take(pageSize)
    .ToListAsync();
// SQL: ... ORDER BY Id OFFSET 20 ROWS FETCH NEXT 20 ROWS ONLY


// ─── 7. Compiled queries for hot paths ────────────────────────────────────
// EF Core compiles queries automatically (query plan cache). For the very
// hottest paths you can use EF.CompileAsyncQuery for max speed:
private static readonly Func<AppDbContext, int, Task<Product?>> GetByIdQuery =
    EF.CompileAsyncQuery((AppDbContext ctx, int id) =>
        ctx.Products.FirstOrDefault(p => p.Id == id));

// Usage (skips expression tree build every time):
Product? product = await GetByIdQuery(_context, 42);
    ✅

    LINQ performance checklist

    Keep these rules in mind when writing LINQ for EF Core: (1) Stay on IQueryable until you call ToList(); (2) Project with Select to avoid SELECT *; (3) Replace Count() > 0 with Any(); (4) Use FirstOrDefault directly instead of ToList().First(); (5) Avoid accessing navigation properties in a loop without Include; (6) Always OrderBy before Skip/Take.
