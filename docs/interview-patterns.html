    🎨 Design Patterns

    Master 9 essential patterns using ProductCatalogAPI as the consistent real-world scenario. Learn when to use each pattern and why.

    Advanced Architecture 9 Patterns
    💡

    One Scenario, All Patterns

    Design Patterns are reusable solutions to common programming problems. The Gang of Four (GoF) identified 23 patterns; we focus on the 9 most essential. Throughout this guide, ProductCatalogAPI demonstrates every pattern in action — same scenario, different solutions for different problems.

    Quick Category Overview

    Design patterns fall into three categories. Here's how each tackles problems in ProductCatalogAPI:

    Pattern Category Problem It Solves Our Example
    Singleton Creational Objects that must exist only once DB connection, Config, Logger
    Factory Creational Create objects without exposing how Different ProductRepository types
    Builder Creational Build complex objects step-by-step Complex filter queries
    Adapter Structural Use incompatible code together Wrap third-party payment API
    Decorator Structural Add behavior dynamically Add caching layer to repository
    Facade Structural Simplify complex subsystem Hide EF Core + ADO.NET behind service
    Observer Behavioral Notify multiple listeners of change Price changes notify email/cache/audit
    Strategy Behavioral Switch algorithms at runtime Discount types, payment methods, sorting
    Command Behavioral Encapsulate request as object Undo/redo product operations, queued jobs

    Creational Patterns

    Creational patterns focus on object creation — how to create objects flexibly and efficiently without tight coupling.

    Singleton Pattern

    Definition: Ensures only ONE instance of a class exists throughout the application lifetime. Used for objects that must be shared globally — database connections, configuration, loggers.

    Real-World Analogy: There's only ONE CEO of a company. No matter who asks, they get the same person. If you tried creating a "second CEO," it would just return the existing one.

    Problem (BAD)

    When you don't enforce a single instance, resource waste and inconsistency follow:

    csharp 
    // BAD: No guarantee of single instance
// Every time needed, creates a new DatabaseConnection
// Different parts of app use different "connections" → connection leaks, resource waste
public class DatabaseConnection
{
    public string ConnectionString { get; set; } = "Server=localhost";
    public void Connect() => Console.WriteLine("Connected");
}

public class Program
{
    static void Main()
    {
        var db1 = new DatabaseConnection(); // Instance 1
        var db2 = new DatabaseConnection(); // Instance 2 — different object!
        // db1 and db2 are different connections — resource waste!
    }
}

    Solution (GOOD)

    Enforce a single instance using Lazy or thread-safe locking:

    csharp 
    // GOOD: Singleton ensures exactly ONE instance
// Thread-safe using Lazy (easiest in C#)
public class DatabaseConnection
{
    private static readonly Lazy _lazy =
        new(() => new DatabaseConnection());

    public static DatabaseConnection GetInstance() => _lazy.Value;

    private DatabaseConnection() { } // Private constructor — can't be `new`'d outside

    public string ConnectionString { get; set; } = "Server=localhost";
    public void Connect() => Console.WriteLine("Connected to " + ConnectionString);
}

// Usage
var db1 = DatabaseConnection.GetInstance(); // Instance 1
var db2 = DatabaseConnection.GetInstance(); // Same instance!
Console.WriteLine(ReferenceEquals(db1, db2)); // true ✓

    Best in .NET with DI (cleaner):

    csharp 
    // Program.cs — .NET DI makes Singleton pattern trivial
builder.Services.AddSingleton();

// Controller — DI gives same instance every time
public class ProductsController(DatabaseConnection db)
{
    public void UpdateProduct(int id)
    {
        db.Connect(); // Always the same DatabaseConnection instance!
    }
}
    ❓

    Interview Q: "When would Singleton cause problems?"

    Answer: If the singleton holds mutable state and multiple threads modify it, you get race conditions. Also, singletons make unit testing hard — you can't easily create isolated test instances. .NET DI with AddSingleton avoids these by using the container to manage lifecycle.

    Factory Pattern

    Definition: Creates objects without forcing the caller to know HOW they're created or WHICH concrete class is being instantiated. The caller just asks for an interface — the factory decides the implementation.

    Real-World Analogy: Going to a car factory and asking for "a sedan" — you don't care if it's built from blue parts or red parts. The factory encapsulates those details and gives you a working sedan.

    Problem (BAD)

    Hard-coded object creation couples code to concrete types:

    csharp 
    // BAD: Controller creates explicit concrete types
// Tightly coupled — if you add PostgresProductRepository, update every caller!
public class ProductsController
{
    [HttpGet]
    public async Task GetAll()
    {
        // Hard way: caller must know about SqlProductRepository
        IProductRepository repo = new SqlProductRepository(); // Tight coupling!
        var products = await repo.GetAllAsync();
        return Ok(products);
    }
}

// Adding a new repo type? Must find and change every place!
public class CacheService
{
    public void CacheProducts()
    {
        IProductRepository repo = new SqlProductRepository(); // Change here too!
        // ...
    }
}

    Solution (GOOD)

    Use a factory to encapsulate the decision of which type to create:

    csharp 
    // GOOD: Factory encapsulates which type to create
public interface IProductRepository
{
    Task> GetAllAsync();
}

public class SqlProductRepository : IProductRepository
{
    public async Task> GetAllAsync()
    {
        // Real SQL logic
        return await _context.Products.ToListAsync();
    }
}

public class InMemoryProductRepository : IProductRepository
{
    private readonly List _products = new()
    {
        new Product { Id = 1, Name = "Laptop", Price = 999 },
        new Product { Id = 2, Name = "Mouse", Price = 25 }
    };

    public Task> GetAllAsync() =>
        Task.FromResult(_products.AsEnumerable());
}

// FACTORY: Single place that decides which repo type to create
public class ProductRepositoryFactory
{
    public static IProductRepository CreateRepository(string environment)
    {
        return environment.ToLower() switch
        {
            "production" => new SqlProductRepository(),
            "testing" => new InMemoryProductRepository(),
            _ => throw new ArgumentException("Unknown environment")
        };
    }
}

// Usage: caller only knows about the interface
var repo = ProductRepositoryFactory.CreateRepository("production"); // Factory decides!
var products = await repo.GetAllAsync(); // Doesn't care which type
    ❓

    Interview Q: "Why use Factory instead of just `new` in the controller?"

    Answer: Factories decouple creation from usage. If you hard-code `new SqlProductRepository()` in 50 places and later need to switch to PostgreSQL, you change 50 files and risk bugs. With a factory (or DI), you change one place. This is called 'Dependency Inversion' — the caller should depend on interfaces, not concrete types.

    Builder Pattern

    Definition: Constructs complex objects step-by-step instead of requiring a giant constructor with many parameters. Each step adds one piece, and you call Build() at the end. Great for objects with many optional fields.

    Real-World Analogy: Building a pizza. You start with dough, then add sauce, then toppings one by one. At the end, you get the finished pizza. You don't pass 20 parameters to PizzaMaker constructor — you build it step-by-step.

    Problem (BAD)

    Constructor with too many optional parameters — confusing and hard to maintain:

    csharp 
    // BAD: Constructor nightmare — which params are required? optional?
public class ProductFilter
{
    public int? MinPrice { get; set; }
    public int? MaxPrice { get; set; }
    public string? Category { get; set; }
    public bool InStockOnly { get; set; }
    public int Page { get; set; } = 1;
    public int PageSize { get; set; } = 10;
}

// Caller has NO IDEA what parameters mean
var filter = new ProductFilter
{
    MinPrice = null,
    MaxPrice = 500,
    Category = "Electronics",
    InStockOnly = false,
    Page = 1,
    PageSize = 10
};
// What does each field mean? Read the class... (poor UX)

    Solution (GOOD)

    Use the Builder pattern — step-by-step, self-documenting:

    csharp 
    // GOOD: Builder pattern — self-documenting
public class ProductFilterBuilder
{
    private ProductFilter _filter = new();

    public ProductFilterBuilder WithMinPrice(int price)
    {
        _filter.MinPrice = price;
        return this; // Return self for chaining
    }

    public ProductFilterBuilder WithMaxPrice(int price)
    {
        _filter.MaxPrice = price;
        return this;
    }

    public ProductFilterBuilder WithCategory(string category)
    {
        _filter.Category = category;
        return this;
    }

    public ProductFilterBuilder InStockOnly()
    {
        _filter.InStockOnly = true;
        return this;
    }

    public ProductFilterBuilder WithPage(int page, int pageSize = 10)
    {
        _filter.Page = page;
        _filter.PageSize = pageSize;
        return this;
    }

    public ProductFilter Build() => _filter; // Return the built object
}

// Usage — fluent, readable, self-documenting
var filter = new ProductFilterBuilder()
    .WithMinPrice(100)
    .WithMaxPrice(500)
    .WithCategory("Electronics")
    .InStockOnly()
    .WithPage(1, 20)
    .Build();
    ❓

    Interview Q: "Is Builder only for objects with many fields?"

    Answer: No, but that's the most common case. Builder shines when: (1) many optional fields, (2) construction has complex logic, (3) you want immutability (build step-by-step, then lock the object). In .NET 8+, record types with property initialization (`new Product { Name = "", Price = 0 }`) often replace builders for simple cases.

    Structural Patterns

    Structural patterns focus on object composition — how to combine objects and classes to form larger structures while keeping them flexible and efficient.

    Adapter Pattern

    Definition: Converts the interface of an object into another interface the client expects. Like an electrical adapter — European and American outlets are incompatible, but an adapter makes them work together.

    Real-World Analogy: Your laptop charger is 220V. You're in a country with 110V outlets. An adapter converts the voltage so your charger works. Without the adapter, they're incompatible.

    Problem (BAD)

    Third-party APIs have incompatible interfaces — code must handle multiple interface types:

    csharp 
    // Third-party payment API uses a completely different interface
public class StripePaymentService
{
    public StripeResponse Charge(StripeChargeRequest request) { /* ... */ }
}

// But your app expects this interface
public interface IPaymentProcessor
{
    void ProcessPayment(PaymentData data);
}

// Now controller must handle TWO different interfaces — code smell!
public class CheckoutController
{
    private IPaymentProcessor _processor; // Your interface
    private StripePaymentService _stripe; // Third-party interface

    public async Task Checkout()
    {
        // Must convert between the two manually
        var stripeRequest = new StripeChargeRequest { /* ... */ };
        _stripe.Charge(stripeRequest); // Uses Stripe directly
        // Inconsistent — hard to swap payment providers!
    }
}

    Solution (GOOD)

    Use an adapter to convert Stripe to your IPaymentProcessor interface:

    csharp 
    // GOOD: Adapter pattern — convert Stripe to IPaymentProcessor
public class StripePaymentAdapter : IPaymentProcessor
{
    private readonly StripePaymentService _stripe;
    private string? _lastTransactionId;

    public StripePaymentAdapter(StripePaymentService stripe) => _stripe = stripe;

    // Implement the interface, translate to Stripe calls
    public void ProcessPayment(PaymentData data)
    {
        var stripeRequest = new StripeChargeRequest
        {
            Amount = (int)(data.Amount * 100), // Stripe uses cents
            Currency = "USD",
            Description = data.Description
        };

        var response = _stripe.Charge(stripeRequest);
        _lastTransactionId = response.Id;
    }
}

// Controller uses ONE interface — adapters hidden
public class CheckoutController
{
    private readonly IPaymentProcessor _paymentProcessor;

    public CheckoutController(IPaymentProcessor paymentProcessor)
        => _paymentProcessor = paymentProcessor;

    public async Task Checkout(OrderRequest order)
    {
        var data = new PaymentData { Amount = order.Total };
        _paymentProcessor.ProcessPayment(data);
        return Ok("Payment processed");
    }
}
    ❓

    Interview Q: "When is Adapter useful in a real project?"

    Answer: Whenever you use third-party libraries (payment gateways, email providers, cloud services), wrap them in adapters. If you later switch providers (Stripe → PayPal), you only touch the adapter. The rest of your codebase uses IPaymentProcessor unchanged.

    Decorator Pattern

    Definition: Wraps an object to add behavior BEFORE or AFTER delegating to the original. Like a decorator — you take a plain room and add layers (paint, furniture, lights) without changing the room itself.

    Real-World Analogy: Coffee. Start with black coffee. Add a decorator: "whip cream." Then another: "caramel sauce." Each decorator wraps the previous, building up behavior. The core coffee is unchanged — only wrappers add behavior.

    Problem (BAD)

    Mixing cross-cutting concerns into the core class violates the Open/Closed principle:

    csharp 
    // BAD: Original class modified to add caching directly
// Violates Open/Closed principle — changes core business logic
public class ProductRepository
{
    private readonly DbContext _context;
    private readonly IMemoryCache _cache; // Added for caching

    public async Task> GetAllAsync()
    {
        // Mixed concerns: data access + caching logic
        if (_cache.TryGetValue("products:all", out var cached))
            return (IEnumerable)cached;

        var products = await _context.Products.ToListAsync();
        _cache.Set("products:all", products, TimeSpan.FromMinutes(5));
        return products;
    }
}

// Later: add logging? Add retry? Keep modifying GetAllAsync!
// Violates SRP — now handles data + cache + logging

    Solution (GOOD)

    Use decorators to wrap the repository without modifying it:

    csharp 
    // GOOD: Decorator wraps the repository, adding caching WITHOUT modifying it
public class CachedProductRepository : IProductRepository
{
    private readonly IProductRepository _innerRepo;  // Decorated object
    private readonly IMemoryCache _cache;

    public CachedProductRepository(IProductRepository innerRepo, IMemoryCache cache)
    {
        _innerRepo = innerRepo;
        _cache = cache;
    }

    public async Task> GetAllAsync()
    {
        if (_cache.TryGetValue("products:all", out IEnumerable cached))
            return cached; // ✅ HIT

        var products = await _innerRepo.GetAllAsync(); // Delegate
        _cache.Set("products:all", products, TimeSpan.FromMinutes(5));
        return products; // ✅ MISS, cached, returned
    }
}

// Decorator for Logging — another layer!
public class LoggedProductRepository : IProductRepository
{
    private readonly IProductRepository _innerRepo;
    private readonly ILogger _logger;

    public LoggedProductRepository(IProductRepository innerRepo,
        ILogger logger)
    {
        _innerRepo = innerRepo;
        _logger = logger;
    }

    public async Task> GetAllAsync()
    {
        _logger.LogInformation("GetAll called");
        var result = await _innerRepo.GetAllAsync(); // Delegate
        _logger.LogInformation("GetAll returned {Count} products", result.Count());
        return result;
    }
}

// Stack decorators — Logged(Cached(SqlRepository))
// Now GetAll is automatically cached AND logged!
    ❓

    Interview Q: "How is Decorator different from Inheritance?"

    Answer: Inheritance is static — DecoratedRepository extends Repository at compile-time. Decorator is dynamic — you wrap at runtime. If you need to add 5 behaviors (cache, logging, retry, compression, validation), decorators let you mix-and-match. Inheritance would require 2^5 = 32 subclass combinations. Decorators = cleaner, more flexible.

    Facade Pattern

    Definition: Provides a simple, unified interface to a complex subsystem. The subsystem is complicated inside, but users see only one simple door.

    Real-World Analogy: Bank teller. Behind the scenes, handling payments involves checking credit, updating ledgers, sending confirmations, updating inventory. You ask the teller one question, they handle all the complexity.

    Problem (BAD)

    Controller must orchestrate complex subsystems directly — mixing concerns:

    csharp 
    // BAD: Controller handles too much complexity
public class ProductsController
{
    private readonly ProductDbContext _context;
    private readonly IMemoryCache _cache;
    private readonly IEmailService _emailService;
    private readonly ILogger _logger;

    [HttpPost]
    public async Task CreateProduct(CreateProductRequest req)
    {
        // SO MUCH complexity in the controller!

        // Validation
        if (req.Price <= 0) return BadRequest("Price must be > 0");

        // Check duplicate
        var exists = await _context.Products.AnyAsync(p => p.Name == req.Name);
        if (exists) return Conflict("Product already exists");

        // Create
        var product = new Product { Name = req.Name, Price = req.Price };
        _context.Products.Add(product);
        await _context.SaveChangesAsync();

        // Invalidate cache
        _cache.Remove("products:all");

        // Send notification
        await _emailService.SendProductCreatedAsync(product);

        // Log
        _logger.LogInformation("Product created: {Name}", product.Name);

        return CreatedAtAction(nameof(GetById), new { id = product.Id }, product);
    }
}

    Solution (GOOD)

    Use a Facade to hide complexity behind one simple interface:

    csharp 
    // GOOD: Facade hides complexity — one simple service interface
public interface IProductFacade
{
    Task CreateProductAsync(CreateProductRequest req);
    Task> GetAllProductsAsync();
}

// Facade orchestrates the subsystem internally
public class ProductFacade : IProductFacade
{
    private readonly ProductDbContext _context;
    private readonly IMemoryCache _cache;
    private readonly IEmailService _emailService;
    private readonly ILogger _logger;

    public ProductFacade(ProductDbContext context, IMemoryCache cache,
        IEmailService emailService, ILogger logger)
    {
        _context = context;
        _cache = cache;
        _emailService = emailService;
        _logger = logger;
    }

    public async Task CreateProductAsync(CreateProductRequest req)
    {
        // All complexity hidden behind this one method
        if (req.Price <= 0) throw new ValidationException("Price must be > 0");
        if (await _context.Products.AnyAsync(p => p.Name == req.Name))
            throw new ConflictException("Product already exists");

        var product = new Product { Name = req.Name, Price = req.Price };
        _context.Products.Add(product);
        await _context.SaveChangesAsync();

        _cache.Remove("products:all");
        await _emailService.SendProductCreatedAsync(product);
        _logger.LogInformation("Product created: {Name}", product.Name);

        return new ProductResponse(product);
    }

    public async Task> GetAllProductsAsync()
    {
        if (_cache.TryGetValue("products:all", out IEnumerable cached))
            return cached!;

        var products = await _context.Products.AsNoTracking().ToListAsync();
        var responses = products.Select(p => new ProductResponse(p));
        _cache.Set("products:all", responses, TimeSpan.FromMinutes(5));
        return responses;
    }
}

// Controller becomes CLEAN — just calls the facade
public class ProductsController
{
    private readonly IProductFacade _productFacade;

    [HttpPost]
    public async Task CreateProduct(CreateProductRequest req)
    {
        var result = await _productFacade.CreateProductAsync(req);
        return CreatedAtAction(nameof(GetById), new { id = result.Id }, result);
    }
}
    ❓

    Interview Q: "What's the difference between Facade and Adapter?"

    Answer: Adapter converts incompatible interfaces (Stripe → IPaymentProcessor). Facade simplifies a complex interface (EF + Cache + Email + Logging unified into one ProductService). Adapter is about compatibility; Facade is about simplification.

    Behavioral Patterns

    Behavioral patterns focus on communication between objects — how objects interact, distribute responsibility, and maintain flexible communication patterns.

    Observer Pattern

    Definition: Defines a one-to-many relationship where many observers listen to one subject. When the subject changes, all observers are notified automatically.

    Real-World Analogy: Magazine subscribers. Magazine (subject) publishes a new issue, and all subscribers (observers) automatically get notified. Subscribers register with the magazine; they don't poll asking "is there a new issue?"

    Problem (BAD)

    ProductService directly calls every listener — tight coupling:

    csharp 
    // BAD: ProductService knows about Email, Cache, Audit services
// Adding a new listener requires modifying ProductService!
public class ProductService
{
    private readonly ProductDbContext _context;
    private readonly IEmailService _emailService;
    private readonly ICacheService _cacheService;
    private readonly IAuditService _auditService;

    public async Task UpdatePriceAsync(int productId, decimal newPrice)
    {
        var product = await _context.Products.FindAsync(productId);
        product.Price = newPrice;
        await _context.SaveChangesAsync();

        // Directly calling all listeners — tight coupling!
        await _emailService.NotifyPriceChangeAsync(product);
        await _cacheService.InvalidateProductAsync(product.Id);
        await _auditService.LogPriceChangeAsync(product.Id, newPrice);

        // Adding a new listener (SMS notifications)? Must modify this method!
    }
}

    Solution (GOOD)

    Use Observer pattern — ProductService doesn't know about listeners:

    csharp 
    // GOOD: Observer pattern — ProductService doesn't know about listeners
public interface IProductObserver
{
    Task OnProductPriceChangedAsync(Product product, decimal oldPrice);
}

// Each observer implements the interface
public class EmailNotificationObserver : IProductObserver
{
    private readonly IEmailService _emailService;
    public EmailNotificationObserver(IEmailService emailService) => _emailService = emailService;

    public async Task OnProductPriceChangedAsync(Product product, decimal oldPrice)
        => await _emailService.NotifyPriceChangeAsync(product);
}

public class CacheInvalidationObserver : IProductObserver
{
    private readonly ICacheService _cacheService;
    public CacheInvalidationObserver(ICacheService cacheService) => _cacheService = cacheService;

    public async Task OnProductPriceChangedAsync(Product product, decimal oldPrice)
        => await _cacheService.InvalidateProductAsync(product.Id);
}

// Subject broadcasts to all observers
public class ProductService
{
    private readonly ProductDbContext _context;
    private readonly List _observers = new();

    public void Subscribe(IProductObserver observer) => _observers.Add(observer);

    public async Task UpdatePriceAsync(int productId, decimal newPrice)
    {
        var product = await _context.Products.FindAsync(productId);
        var oldPrice = product.Price;
        product.Price = newPrice;
        await _context.SaveChangesAsync();

        // Notify all subscribers
        foreach (var observer in _observers)
            await observer.OnProductPriceChangedAsync(product, oldPrice);

        // Adding SmsObserver? Just Subscribe() it! ProductService unchanged!
    }
}

// Program.cs registration
var emailObserver = new EmailNotificationObserver(emailService);
var cacheObserver = new CacheInvalidationObserver(cacheService);
var productService = new ProductService();
productService.Subscribe(emailObserver);
productService.Subscribe(cacheObserver);
// Done! Multiple observers, no coupling
    ❓

    Interview Q: "How does Observer relate to events in C#?"

    Answer: C# events (EventHandler, Action) are essentially the Observer pattern built-in. `public event EventHandler PriceChanged;` is Observer. When you `PriceChanged?.Invoke()`, you're notifying all subscribers. The formal Observer pattern is more explicit — useful for understanding the concept.

    Strategy Pattern

    Definition: Defines a family of interchangeable algorithms, encapsulates each one, and lets the caller choose which to use at runtime.

    Real-World Analogy: Burger King. You order a burger, then choose: "Mayo or mustard?" Each sauce is a strategy — same burger, different strategy applied.

    Problem (BAD)

    Business logic has if/else for different algorithms:

    csharp 
    // BAD: ProductService has if/else for discount types
// Adding a new discount type requires modifying the method!
public class ProductService
{
    public decimal CalculateFinalPrice(Product product, string discountType)
    {
        decimal discount = discountType switch
        {
            "seasonal" => product.Price * 0.10m,
            "clearance" => product.Price * 0.50m,
            "member" => product.Price * 0.15m,
            // Adding "senior"? Modify this switch!
            _ => 0
        };

        return product.Price - discount;
    }
}

    Solution (GOOD)

    Each discount is its own strategy class:

    csharp 
    // GOOD: Strategy pattern — each discount is its own class
public interface IDiscountStrategy
{
    decimal CalculateDiscount(Product product);
    string Name { get; }
}

public class SeasonalDiscountStrategy : IDiscountStrategy
{
    public string Name => "Seasonal";
    public decimal CalculateDiscount(Product product) => product.Price * 0.10m;
}

public class ClearanceDiscountStrategy : IDiscountStrategy
{
    public string Name => "Clearance";
    public decimal CalculateDiscount(Product product) => product.Price * 0.50m;
}

public class MemberDiscountStrategy : IDiscountStrategy
{
    public string Name => "Member";
    public decimal CalculateDiscount(Product product) => product.Price * 0.15m;
}

// Service uses any strategy
public class ProductService
{
    public decimal CalculateFinalPrice(Product product, IDiscountStrategy discount)
    {
        var discountAmount = discount.CalculateDiscount(product);
        return product.Price - discountAmount;
    }
}

// Usage
var finalPrice = _productService.CalculateFinalPrice(product, new MemberDiscountStrategy());
// Adding SeniorDiscountStrategy? Just create the class! ProductService unchanged!
    ❓

    Interview Q: "Strategy vs if/else — when do you use Strategy?"

    Answer: If/else works for 2-3 options. Strategy shines when: (1) many options (5+), (2) options change frequently, (3) options are complex logic, (4) you want to add new options without modifying existing code. Strategy = Open/Closed Principle in action.

    Command Pattern

    Definition: Encapsulates a request as a COMMAND OBJECT, allowing you to queue commands, undo/redo, and execute commands at different times.

    Real-World Analogy: Restaurant waiter. Customer orders food (command). Waiter writes it down, passes it to chef (queue). Chef processes commands one by one. Waiter can cancel orders (undo) or retry the same order (redo/replay).

    Problem (BAD)

    Business logic mixed into controller — no way to queue, log, undo, or replay:

    csharp 
    // BAD: Business logic mixed into controller
// No way to: undo operation, queue async, log command history, replay
public class ProductsController
{
    [HttpPost]
    public async Task CreateProduct(CreateProductRequest req)
    {
        var product = new Product { Name = req.Name, Price = req.Price };
        await _productService.CreateAsync(product);
        return Ok(product);
    }

    [HttpPut("{id}")]
    public async Task UpdateProduct(int id, UpdateProductRequest req)
    {
        var product = await _productService.GetByIdAsync(id);
        product.Price = req.Price;
        await _productService.UpdateAsync(product);
        return Ok(product);
    }

    // No way to: undo operation, queue async, log command history, replay
}

    Solution (GOOD)

    Encapsulate operations as Command objects:

    csharp 
    // GOOD: Command pattern — encapsulate operations as objects
public interface ICommand
{
    Task ExecuteAsync();
    Task UndoAsync();
    string Description { get; }
}

public class CreateProductCommand : ICommand
{
    private readonly Product _product;
    private readonly ProductService _productService;

    public string Description => $"Create product: {_product.Name}";

    public CreateProductCommand(Product product, ProductService productService)
    {
        _product = product;
        _productService = productService;
    }

    public async Task ExecuteAsync()
        => await _productService.CreateAsync(_product);

    public async Task UndoAsync()
        => await _productService.DeleteAsync(_product.Id);
}

public class UpdateProductCommand : ICommand
{
    private readonly int _productId;
    private readonly decimal _newPrice;
    private readonly decimal _oldPrice;
    private readonly ProductService _productService;

    public string Description => $"Update product {_productId} price to {_newPrice}";

    public UpdateProductCommand(int productId, decimal newPrice, decimal oldPrice,
        ProductService productService)
    {
        _productId = productId;
        _newPrice = newPrice;
        _oldPrice = oldPrice;
        _productService = productService;
    }

    public async Task ExecuteAsync()
    {
        var product = await _productService.GetByIdAsync(_productId);
        product.Price = _newPrice;
        await _productService.UpdateAsync(product);
    }

    public async Task UndoAsync()
    {
        var product = await _productService.GetByIdAsync(_productId);
        product.Price = _oldPrice;
        await _productService.UpdateAsync(product);
    }
}

// Command invoker — queues and executes commands
public class CommandInvoker
{
    private readonly Stack _commandHistory = new();

    public async Task ExecuteAsync(ICommand command)
    {
        await command.ExecuteAsync();
        _commandHistory.Push(command); // Remember for undo
    }

    public async Task UndoAsync()
    {
        if (_commandHistory.Count > 0)
        {
            var command = _commandHistory.Pop();
            await command.UndoAsync();
        }
    }
}

// Controller uses commands
public class ProductsController
{
    private readonly CommandInvoker _invoker;
    private readonly ProductService _productService;

    [HttpPost]
    public async Task CreateProduct(CreateProductRequest req)
    {
        var product = new Product { Name = req.Name, Price = req.Price };
        var command = new CreateProductCommand(product, _productService);
        await _invoker.ExecuteAsync(command);
        return Ok(product);
    }

    [HttpPost("undo")]
    public async Task UndoLastOperation()
    {
        await _invoker.UndoAsync();
        return Ok("Last operation undone");
    }
}

    Interview Q: "When is Command pattern overkill?"

    Answer: If you just need to execute simple operations, Command adds ceremony. But if you need: undo/redo, command history, async queuing, transaction-like rollback, or replay — Command is invaluable. It transforms random method calls into first-class objects you can queue, persist, and reason about.

    Which Pattern When? — Decision Table

    Confused about which pattern to use? This table covers all 9 patterns with their problems and real-world ProductCatalogAPI examples:

    Pattern When You Have This Problem... Use This Pattern ProductCatalogAPI Example
    Singleton Objects being created many times, should exist only once Enforce single instance via Lazy<T> DatabaseConnection, Logger, Config
    Factory Caller needs object but shouldn't know HOW it's created Create a static method that returns interface, hides implementation ProductRepositoryFactory → SqlRepo, InMemoryRepo, PostgresRepo
    Builder Object has many optional parameters or complex construction Create a builder class, return this for chaining, Build() at end ProductFilterBuilder → WithMinPrice → WithMaxPrice → Build()
    Adapter Need to use third-party library with incompatible interface Create adapter class implementing your interface, delegates to third-party StripePaymentAdapter → IPaymentProcessor, adapts StripeAPI
    Decorator Need to add behavior to object WITHOUT modifying it Create decorator class implementing same interface, wraps original CachedProductRepository wraps SqlProductRepository, adds caching
    Facade Client sees complex subsystem, needs simplified entry point Create facade class exposing one simple interface, orchestrates subsystem ProductFacade → hides EF + Cache + Email + Logging
    Observer One object needs to notify many others of state change Create observer interface, subject manages list, calls all on change ProductService notifies EmailObserver, CacheObserver, AuditObserver
    Strategy Need to switch between algorithms at runtime Create strategy interface, multiple implementations, use any at runtime IDiscountStrategy → SeasonalDiscount, ClearanceDiscount, MemberDiscount
    Command Need to queue, undo/redo, or persist operations Create command class implementing ICommand, invoke via invoker CreateProductCommand, UpdateProductCommand with undo support

    Common Interview Questions

    1. What's the difference between Factory and Abstract Factory patterns?

    Answer:

    Factory: Creates ONE type of object. Example: `ProductRepositoryFactory.CreateRepository(env)` returns an IProductRepository (could be SQL or InMemory).

    Abstract Factory: Creates FAMILIES of related objects. Example: UIFactory that creates both Button and TextField for either Windows or Mac themes. Rarely needed in .NET Web API work.

    In interviews: Be prepared to explain Factory is "simpler, more common." Abstract Factory is "for object families that vary together."

    2. Can you give an example of combining multiple patterns together?

    Answer: Yes! Real ProductCatalogAPI architecture uses multiple patterns:

    Factory + Dependency Injection: Program.cs uses factory logic to create the right IProductRepository based on environment.

    Decorator + Observer: CachedProductRepository (Decorator) wraps SqlProductRepository; when prices change, Observer pattern notifies multiple listeners (cache invalidation, email, audit).

    Facade + Strategy: ProductService (Facade) hides EF + cache + email. Uses IDiscountStrategy at runtime to apply different discount algorithms.

    In interviews: Mention that patterns work together. Real code doesn't use just ONE pattern — they compose.

    3. What's the difference between Decorator and Proxy patterns?

    Answer:

    Decorator: Adds BEHAVIOR to the object. Example: CachedProductRepository adds caching. You ALWAYS want to add the behavior.

    Proxy: Controls ACCESS to the object. Example: SecuredProductRepository only allows requests from admin users. You CONDITIONALLY allow access.

    Code looks similar: Both wrap the object. But intent is different — Decorator enhances, Proxy controls.

    In interviews: "Decorator adds behavior, Proxy controls access. Same implementation pattern, different purpose."

    4. Why are design patterns important in interviews?

    Answer:

    Patterns show you understand:

    • OOP principles — Abstraction, composition, loose coupling
    • Clean code — Maintainable, testable, scalable
    • Problem-solving — Recognizing common problems and applying proven solutions
    • Communication — Patterns are a shared vocabulary (senior devs speak in patterns)

    In interviews: Don't memorize definitions. Instead, explain when you'd use a pattern and why it solves a real problem.

    5. What's the relationship between patterns and SOLID principles?

    Answer:

    SOLID Principles: Guidelines for writing GOOD code (Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion).

    Design Patterns: Proven SOLUTIONS that follow SOLID.

    Relationship: Patterns are IMPLEMENTATIONS of SOLID. Example:

    • Decorator follows Open/Closed — "open for extension (decorators), closed for modification (repository)"
    • Observer follows Open/Closed — add new observers without modifying subject
    • Strategy follows Dependency Inversion — depend on IDiscountStrategy interface, not concrete discount classes

    In interviews: "SOLID is theory, patterns are practice. I follow SOLID by using appropriate patterns."
