    💎 C# & OOP Fundamentals

    Object-Oriented Programming concepts used throughout ProductCatalogAPI — classes, interfaces, inheritance, polymorphism, records, async/await, and SOLID principles.

    Intermediate C# 12

    The Four Pillars of OOP

    PillarDefinitionExample in ProductCatalogAPI
    EncapsulationHide internal state; expose via properties/methodsProduct class — private fields, public properties, IsActive never set directly from outside
    AbstractionExpose only what callers need; hide complexityIProductService interface — controller only knows the contract, not the implementation
    InheritanceDerive from a base class to reuse and extend behaviourCustom exceptions inherit from Exception: class NotFoundException : Exception
    PolymorphismSame method name behaves differently depending on typeMiddleware InvokeAsync override; controller action methods sharing route via HTTP verb

    Classes & Objects

    C# — Product Entity Class
    // Class = blueprint; Object = instance created with 'new'
public class Product
{
    // Properties (encapsulate the backing field automatically)
    public int     Id        { get; set; }
    public string  Name      { get; set; } = string.Empty;  // init-safe default
    public decimal Price     { get; set; }
    public string  Category  { get; set; } = string.Empty;
    public int     Stock     { get; set; }
    public bool    IsActive  { get; set; } = true;

    // Computed property — no setter, derived from other data
    public bool IsInStock => Stock > 0;

    // Read-only property — set only via constructor
    public DateTime CreatedAt { get; init; } = DateTime.UtcNow;
    public DateTime? UpdatedAt { get; set; }
}

// Creating an instance (object)
var product = new Product
{
    Name     = "Laptop",
    Price    = 999.99m,
    Category = "Electronics",
    Stock    = 50
};

Console.WriteLine(product.IsInStock); // true

    Encapsulation

    Keep state private; control access and validation through properties and methods.

    C# — Encapsulation with Validation
    public class Product
{
    private decimal _price;   // ← private backing field

    // Property with validation in setter
    public decimal Price
    {
        get => _price;
        set
        {
            if (value < 0) throw new ArgumentException("Price cannot be negative");
            _price = value;
        }
    }

    private int _stock;
    public int Stock
    {
        get => _stock;
        set => _stock = value < 0 ? 0 : value;  // clamp to 0
    }
}

    Interfaces & Abstraction

    Interfaces define a contract — a list of methods and properties a class must implement. They enable Dependency Injection and make code testable.

    C# — IProductService Interface
    // Interface = contract (no implementation, just signatures)
public interface IProductService
{
    Task<IEnumerable<Product>> GetAllAsync();
    Task<Product?>             GetByIdAsync(int id);
    Task<Product>              CreateAsync(CreateProductRequest request);
    Task<Product>              UpdateAsync(int id, UpdateProductRequest request);
    Task                       DeleteAsync(int id);
    Task<Product>              UpdateStockAsync(int id, int stock);
}

// Concrete implementation
public class ProductService : IProductService
{
    private readonly ProductDbContext _context;

    public ProductService(ProductDbContext context)   // ← constructor injection
    {
        _context = context;
    }

    public async Task<IEnumerable<Product>> GetAllAsync()
    {
        return await _context.Products.AsNoTracking().ToListAsync();
    }
    // ... all other methods
}

// Controller depends on the INTERFACE — not the concrete class
// This is abstraction + dependency inversion in action
[ApiController]
public class ProductsController : ControllerBase
{
    private readonly IProductService _service;  // ← interface type

    public ProductsController(IProductService service)
    {
        _service = service;  // DI injects ProductService at runtime
    }
}

    Inheritance

    Derive a class from a base class to reuse and extend behaviour. Used extensively for custom exceptions in the API.

    C# — Custom Exception Hierarchy
    // Base custom exception
public class AppException : Exception
{
    public int StatusCode { get; }

    public AppException(string message, int statusCode = 500)
        : base(message)          // ← calls base class (Exception) constructor
    {
        StatusCode = statusCode;
    }
}

// Derived exceptions — inherit AppException which inherits Exception
public class NotFoundException : AppException
{
    public NotFoundException(string message)
        : base(message, 404) { }   // always 404
}

public class ConflictException : AppException
{
    public ConflictException(string message)
        : base(message, 409) { }
}

public class ValidationException : AppException
{
    public ValidationException(string message)
        : base(message, 400) { }
}

// Usage in service
public async Task<Product> GetByIdAsync(int id)
{
    var product = await _context.Products.FindAsync(id);
    if (product is null)
        throw new NotFoundException($"Product with id {id} was not found");  // 404
    return product;
}

    Polymorphism

    C# — Virtual / Override
    // Base class with virtual method
public class BaseMiddleware
{
    public virtual string GetName() => "Base";
    public virtual Task InvokeAsync(HttpContext ctx) => Task.CompletedTask;
}

// Derived class OVERRIDES the method — polymorphism
public class LoggingMiddleware : BaseMiddleware
{
    public override string GetName() => "Logging";   // ← different behaviour

    public override async Task InvokeAsync(HttpContext ctx)
    {
        Console.WriteLine($"[{DateTime.UtcNow}] {ctx.Request.Method} {ctx.Request.Path}");
        await base.InvokeAsync(ctx);  // ← optionally call base
    }
}

// Runtime polymorphism — method called depends on actual object type
BaseMiddleware mw = new LoggingMiddleware();
Console.WriteLine(mw.GetName());  // "Logging" — not "Base"

// Interface polymorphism — used everywhere in the API
IProductService service = new ProductService(context);
// Could also be: IProductService service = new CachedProductService(context);
// The controller doesn't care — it calls the same interface methods

    Records & DTOs (C# 9+)

    Records are immutable reference types — perfect for DTOs because they represent data shapes, not objects with behaviour. They get Equals, GetHashCode, and ToString for free.

    C# — Record DTOs
    // Positional record — concise syntax for DTOs
public record CreateProductRequest(
    [Required][MaxLength(200)] string Name,
    [Range(0.01, double.MaxValue)] decimal Price,
    [Required] string Category,
    [Range(0, int.MaxValue)] int Stock
);

public record UpdateProductRequest(
    [Required][MaxLength(200)] string Name,
    [Range(0.01, double.MaxValue)] decimal Price,
    [Required] string Category,
    [Range(0, int.MaxValue)] int Stock
);

// Records support 'with' expressions for non-destructive mutation
var original = new CreateProductRequest("Laptop", 999m, "Electronics", 10);
var cheaper  = original with { Price = 799m };  // new record, original unchanged

// Class-style record with explicit properties
public record ProductResponse
{
    public int     Id       { get; init; }
    public string  Name     { get; init; } = string.Empty;
    public decimal Price    { get; init; }
    public string  Category { get; init; } = string.Empty;
    public bool    IsInStock { get; init; }
}

    Async / Await

    All API operations are async so the server thread is not blocked waiting for database or I/O calls. This allows the same thread pool to serve thousands of concurrent requests.

    C# — Async Patterns in the API
    // ✅ Correct async all the way down
public async Task<Product> GetByIdAsync(int id)
{
    // 'await' suspends this method — thread returns to pool
    // When DB responds, execution resumes
    var product = await _context.Products.FindAsync(id);

    if (product is null)
        throw new NotFoundException($"Product {id} not found");

    return product;
}

// ✅ Run multiple async operations in parallel with Task.WhenAll
public async Task<DashboardData> GetDashboardAsync()
{
    var productsTask = _context.Products.CountAsync();
    var categoriesTask = _context.Products.Select(p => p.Category).Distinct().CountAsync();

    await Task.WhenAll(productsTask, categoriesTask);  // ← runs both at once

    return new DashboardData
    {
        TotalProducts   = productsTask.Result,
        TotalCategories = categoriesTask.Result
    };
}

// ❌ Avoid .Result or .Wait() — causes deadlocks in ASP.NET Core
// var product = _context.Products.FindAsync(id).Result;  // NEVER do this

    Generics

    C# — Generics Used in the API
    // Generic list — strongly typed, no boxing
List<Product> products = new List<Product>();

// Generic DbSet<T> — EF Core uses generics for each table
DbSet<Product> Products { get; set; }

// Generic Task<T> — async return type
Task<Product>           GetByIdAsync(int id);
Task<IEnumerable<Product>> GetAllAsync();

// Generic cache method
bool found = _cache.TryGetValue<List<Product>>("all_products", out var cached);

// Generic constraint — T must implement an interface
public class Repository<T> where T : class
{
    private readonly DbSet<T> _set;

    public Repository(ProductDbContext ctx)
    {
        _set = ctx.Set<T>();
    }

    public async Task<List<T>> GetAllAsync()
        => await _set.AsNoTracking().ToListAsync();
}

    SOLID Principles in ProductCatalogAPI

    PrincipleRuleApplied in API
    S — Single Responsibility One class = one reason to change ProductsController only handles HTTP. ProductService only handles business logic. ProductDbContext only handles data access.
    O — Open/Closed Open for extension, closed for modification Add a new exception handler by creating a new class implementing IExceptionHandler — no changes to existing code needed.
    L — Liskov Substitution Derived class must be substitutable for base class NotFoundException can be used anywhere AppException is expected. The global error handler works without knowing the specific subtype.
    I — Interface Segregation Prefer small specific interfaces IProductService only contains product operations. Don't create one giant IService with everything.
    D — Dependency Inversion Depend on abstractions, not concretions Controller depends on IProductService, not ProductService. Registered in Program.cs: services.AddScoped<IProductService, ProductService>()

    C# Patterns Used in the API

    C# — Modern C# Syntax Quick Reference
    // Null-conditional operator — safe navigation
string? category = product?.Category;
int? id = product?.Id;

// Null-coalescing operator — default if null
string name = product?.Name ?? "Unknown";

// Null-coalescing assignment — assign if null
product.Name ??= "Default Product";

// Pattern matching — cleaner than is + cast
if (ex is NotFoundException notFound)
    return NotFound(notFound.Message);

// Switch expression (C# 8+)
return statusCode switch
{
    404 => NotFound(detail),
    400 => BadRequest(detail),
    409 => Conflict(detail),
    _   => StatusCode(500, detail)  // default
};

// Collection expressions (C# 12)
List<string> categories = ["Electronics", "Clothing", "Food"];
int[] ids = [1, 2, 3, 4, 5];

// Primary constructor (C# 12) — cleaner service classes
public class ProductService(ProductDbContext context, ILogger<ProductService> logger)
    : IProductService
{
    // context and logger are directly usable — no assignment needed
    public async Task<IEnumerable<Product>> GetAllAsync()
    {
        logger.LogInformation("Fetching all products");
        return await context.Products.AsNoTracking().ToListAsync();
    }
}

    Interview Tips — C# & OOP

    • Q: What is the difference between a class and an interface? — A class is a blueprint with implementation. An interface is a contract with only signatures. A class can implement multiple interfaces but inherit only one class.
    • Q: What is the difference between abstract class and interface? — Abstract class can have implementation (concrete methods), constructors, and state. Interface cannot (before C# 8 default implementations). Use abstract class when sharing code; interface when defining a contract.
    • Q: What is a record vs a class? — Record is immutable by default, value-equality by default, and great for DTOs. Class is mutable by default and uses reference equality.
    • Q: What are value types vs reference types? — Value types (int, decimal, bool, struct) live on the stack; copying creates a new value. Reference types (class, string, arrays) live on the heap; copying copies the reference.
    • Q: Why use interfaces for Dependency Injection? — Allows swapping implementations without changing callers. Makes unit testing easy (mock the interface). Follows Dependency Inversion Principle (SOLID-D).
    • Q: What happens if you call .Result on a Task in ASP.NET Core? — It deadlocks. ASP.NET Core has a synchronization context. Calling .Result blocks the current thread waiting for the task, but the task needs the same thread to complete. Use await always.
