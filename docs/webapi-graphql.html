    🌐 GraphQL & HotChocolate

    Build flexible APIs with GraphQL using the HotChocolate library for .NET. Queries, mutations, subscriptions, and when to use GraphQL vs REST.

    Advanced Web API
    ℹ️

    What is GraphQL?

    GraphQL is a query language for APIs. Unlike REST (multiple endpoints per resource), GraphQL has a single endpoint. Clients specify exactly what data they need in the request — no over-fetching, no under-fetching. Facebook created it in 2012 and open-sourced it in 2015.

    REST vs GraphQL

    AspectRESTGraphQL
    EndpointsMultiple (one per resource)Single (/graphql)
    Data fetchingFixed response shapeClient picks fields it needs
    Over-fetchingCommonEliminated
    Under-fetchingMultiple requests neededOne request for nested data
    Versioningv1, v2 routes neededEvolve schema, no versioning
    CachingHTTP cache (easy)Per-query cache (harder)
    Learning curveLowMedium
    Best forSimple CRUD APIsComplex, client-driven data needs

    HotChocolate Setup

    HotChocolate is the leading GraphQL server for .NET. Install and configure it alongside your existing ASP.NET Core app.

    Install NuGet Packages 
    dotnet add package HotChocolate.AspNetCore
dotnet add package HotChocolate.Data.EntityFramework
    Program.cs — Register GraphQL 
    builder.Services
    .AddGraphQLServer()
    .AddQueryType<Query>()
    .AddMutationType<Mutation>()
    .AddSubscriptionType<Subscription>()
    .AddFiltering()    // Enables [UseFiltering] on resolvers
    .AddSorting()      // Enables [UseSorting]  on resolvers
    .AddProjections(); // Enables [UseProjection] for EF Core

// Map GraphQL endpoint (use alongside REST — they can coexist)
app.MapGraphQL(); // Available at /graphql

    Queries (Read Data)

    A Query class defines all readable operations. Each method becomes a GraphQL field clients can query.

    GraphQL/Query.cs 
    public class Query
{
    // Get all products — with built-in filtering, sorting, pagination
    [UseDbContext(typeof(ProductDbContext))]
    [UseProjection]
    [UseFiltering]
    [UseSorting]
    public IQueryable<Product> GetProducts([ScopedService] ProductDbContext context)
        => context.Products;

    // Get single product by ID
    [UseDbContext(typeof(ProductDbContext))]
    public async Task<Product?> GetProductById(
        int id,
        [ScopedService] ProductDbContext context)
        => await context.Products.FindAsync(id);
}

    Client GraphQL Query

    Client sends to POST /graphql 
    # Request only the fields you need — no over-fetching
query {
  products(
    where: { category: { eq: "Electronics" } }
    order: { price: ASC }
  ) {
    id
    name
    price
    category
  }
}

# Nested query — get product with related data in one request
query {
  productById(id: 1) {
    id
    name
    price
    reviews {
      rating
      comment
    }
  }
}

    Mutations (Write Data)

    Mutations are GraphQL's equivalent of POST, PUT, DELETE. They change data and return the updated state.

    GraphQL/Mutation.cs 
    public class Mutation
{
    public async Task<Product> CreateProduct(
        CreateProductInput input,
        [ScopedService] ProductDbContext context)
    {
        var product = new Product
        {
            Name          = input.Name,
            Price         = input.Price,
            Category      = input.Category,
            StockQuantity = input.StockQuantity,
            IsActive      = true,
            CreatedAt     = DateTime.UtcNow
        };

        context.Products.Add(product);
        await context.SaveChangesAsync();
        return product;
    }

    public async Task<Product?> DeleteProduct(
        int id,
        [ScopedService] ProductDbContext context)
    {
        var product = await context.Products.FindAsync(id);
        if (product == null) return null;

        product.IsActive = false; // Soft delete
        await context.SaveChangesAsync();
        return product;
    }
}

// Input type (DTO for GraphQL)
public record CreateProductInput(
    string Name,
    decimal Price,
    string Category,
    int StockQuantity
);

    Client Mutation

    GraphQL Mutation Request 
    mutation {
  createProduct(input: {
    name: "Laptop Pro"
    price: 1299.99
    category: "Electronics"
    stockQuantity: 50
  }) {
    id
    name
    price
    createdAt
  }
}

    Subscriptions (Real-Time)

    Subscriptions push data to clients over WebSocket when something changes — ideal for live dashboards, notifications, and stock updates.

    GraphQL/Subscription.cs 
    public class Subscription
{
    // Client subscribes to this — receives event whenever a product is created
    [Subscribe]
    [Topic("ProductCreated")]
    public Product OnProductCreated([EventMessage] Product product) => product;
}

// In Mutation.cs — publish the event after creating a product
public async Task<Product> CreateProduct(
    CreateProductInput input,
    [ScopedService] ProductDbContext context,
    [Service] ITopicEventSender eventSender)
{
    var product = ...; // Create product
    await context.SaveChangesAsync();

    // Notify all subscribers
    await eventSender.SendAsync("ProductCreated", product);
    return product;
}

// Program.cs — add WebSocket support for subscriptions
app.UseWebSockets();
app.MapGraphQL();
    💡

    When to use GraphQL vs REST

    • Use REST for simple CRUD APIs, public APIs, when HTTP caching is important.
    • Use GraphQL when clients have diverse data needs, deep nested fetching, or you want to avoid over/under-fetching.
    • Both can coexist in the same ASP.NET Core app — REST controllers + MapGraphQL().
