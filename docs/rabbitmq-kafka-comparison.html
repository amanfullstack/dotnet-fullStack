⚔️ RabbitMQ vs Kafka

Both are message brokers but solve different problems. Learn when to use each, and how to use both together.

🎯

The Core Difference

RabbitMQ: "Please deliver this message to someone. Once delivered, forget it."

Kafka: "Remember all messages forever. Anyone can replay them anytime."

Side-by-Side Comparison

Feature RabbitMQ 🐰 Kafka 📡
Message Retention Deletes after delivery Keeps for days/years
Throughput 10k-100k msgs/sec 1M+ msgs/sec
Latency Very low (ms) Low (50ms+)
Ordering Per queue Per partition
Replay No (messages deleted) Yes (from any point)
Consumer Groups Complex (exchange/binding) Built-in, elegant
Multiple Subscribers Fanout exchanges (complex) Consumer groups (simple)
Setup Complexity Simple, single node Requires cluster (3+ nodes)
Use Case Task queues, RPC calls Streaming, audit logs, analytics
Data Loss Risk Higher (deletes messages) Lower (stores everything)
Learning Curve Moderate (exchanges confusing) Steep (distributed systems)

Visual Comparison

🐰 RabbitMQ

Delivery Model: Pull & Delete

Message → Queue → Consumer → ✓ Deleted

Good for: Job dispatch, task queues

Like a taxi: deliver passenger, taxi forgets

VS

📡 Kafka

Delivery Model: Store & Replay

Event → Topic → Stored Forever → Multiple Consumers

Good for: Event streaming, audit trail

Like a library: book stays, many readers borrow

RabbitMQ: When to Use

✅ Perfect for:

  • Task Queues: "Email to send" → Worker processes → Done
  • Job Scheduling: Background jobs, cron-like work
  • RPC Calls: Service A asks Service B, waits for response
  • Simple Messaging: One-off messages, not historical data
  • Low Latency Critical: Need instant delivery (ms level)
  • Single Machine: Small deployments, one RabbitMQ instance

Example Use Cases:

1. Email Queue:
   Web app: "Send welcome email to john@example.com"
   → RabbitMQ holds message
   → Email worker picks up → Sends → Message deleted
   (History not needed)

2. Image Processing:
   Upload image → Queue job → Worker resizes → Done
   (Old jobs don't matter)

3. Task Scheduling:
   "Clean old files every hour"
   → RabbitMQ holds job → Worker executes
   (No need to replay yesterday's cleanup)

Kafka: When to Use

✅ Perfect for:

  • Event Streaming: Track all user actions, orders, events
  • Audit Logs: Regulatory compliance, trace everything
  • Analytics: Historical data analysis, reporting
  • High Throughput: Millions of events/second
  • Multiple Subscribers: Same event triggers multiple actions
  • Replay Needed: "Reprocess all orders from yesterday"

Example Use Cases:

1. Order Events:
   Order created → Kafka stores forever
   → Email service reads → Sends confirmation
   → Inventory service reads → Reserves stock
   → Analytics service reads → Records metrics
   (All need same event, different processing)

2. Compliance Audit Trail:
   Every transaction → Kafka stores 7 years
   → Auditors can replay any point in time
   (History is critical)

3. Real-time Analytics:
   Website clicks → Kafka streams → Analytics engine
   → Dashboard updates → Reports generated
   (Need volume + history + distribution)

Decision Matrix

Choose RabbitMQ if:
✅ Throughput < 100k msgs/sec
✅ Don't need message history
✅ One-off tasks/jobs
✅ Low latency critical (< 10ms)
✅ Simple pub/sub pattern
✅ Small, single-machine deployment

Choose Kafka if:
✅ Throughput > 1M msgs/sec
✅ Need message history/replay
✅ Multiple subscribers same event
✅ Real-time streaming/analytics
✅ Audit trail required
✅ Distributed cluster (scalability)

Hybrid Approach: Using Both

Some companies use BOTH for different purposes:

Architecture Example: E-commerce Platform

                    Kafka (Event Stream)
                    │
                    ├─ Stores all order events (forever)
                    ├─ Email service reads → sends emails
                    ├─ Analytics service reads → generates reports
                    └─ Inventory service reads → updates stock
                        (Multiple subscribers, same stream)

                    RabbitMQ (Task Queue)
                    │
                    ├─ Send welcome email (background job)
                    ├─ Generate PDF invoice (one-off task)
                    ├─ Upload image to S3 (async work)
                    └─ Clean temp files (scheduled task)
                        (One-off tasks, deleted after completion)

Real Flow:
User places order
  ↓
API publishes "OrderCreated" to Kafka
  ↓
  ├→ Kafka: Multiple services read (streaming)
  │   Inventory updates stock
  │   Analytics records event
  │
  └→ RabbitMQ: Queue one-off task
      "Send email" job → Worker processes → Job deleted

Why both?

  • Kafka: Official system of truth for all events
  • RabbitMQ: Work queue for background jobs
  • Clear separation: Events vs Tasks
  • Scalability: Each optimized for its job

Migration Path

Start with RabbitMQ, grow to Kafka:

PHASE 1: Small Startup
├─ Only RabbitMQ
├─ Task queues for jobs
├─ Simple pub/sub for notifications
└─ Works fine for low volume

PHASE 2: Growing Traffic
├─ RabbitMQ struggling (throughput)
├─ Add Kafka for high-volume events
├─ RabbitMQ for low-latency tasks
└─ Both systems coexist

PHASE 3: Enterprise
├─ Kafka clusters in multiple regions
├─ RabbitMQ for critical low-latency work
├─ Event sourcing with Kafka
├─ Analytics on Kafka streams
└─ Unified observability

Don't switch EVERYTHING to Kafka! RabbitMQ is still better for some use cases (low latency, simple jobs).

Quick Comparison Code

RabbitMQ - Simple Email Job

// Send email = one-off job, deleted after
var factory = new ConnectionFactory() { HostName = "localhost" };
using (var connection = factory.CreateConnection())
using (var channel = connection.CreateModel())
{
    channel.QueueDeclare(queue: "email_queue");

    // Publish job
    var emailJob = JsonSerializer.Serialize(new { to = "user@example.com", subject = "Welcome!" });
    channel.BasicPublish("", "email_queue", null, Encoding.UTF8.GetBytes(emailJob));

    // Consumer
    var consumer = new EventingBasicConsumer(channel);
    consumer.Received += (model, ea) =>
    {
        var message = Encoding.UTF8.GetString(ea.Body.ToArray());
        // Send email...
        channel.BasicAck(ea.DeliveryTag, false); // Message deleted after ACK
    };
}

Kafka - Permanent Event Stream

// Publish order event = stored forever for replay
using (var producer = new ProducerBuilder(config).Build())
{
    var orderEvent = JsonSerializer.Serialize(new { orderId = 123, amount = 99.99 });
    await producer.ProduceAsync("orders", new Message
    {
        Key = "order-123",
        Value = orderEvent  // Stored permanently
    });
}

// Multiple subscribers, same event
// Email service: read for confirmation
// Inventory: read for stock update
// Analytics: read for metrics
// Audit: read for compliance trail

Interview Q&A

Q1: Should we use RabbitMQ or Kafka?

A: Depends on requirements:

  • Only job queues? → RabbitMQ (simpler, lighter)
  • High-volume events? → Kafka
  • Need history? → Kafka
  • Multiple subscribers? → Kafka (cleaner)
  • Low latency critical? → RabbitMQ
  • Both? → Use both for different purposes!

Honest answer: "Start with RabbitMQ. If you hit its limits (throughput, need replay), add Kafka. Many companies use both."

Q2: What happens if RabbitMQ crashes with messages in queue?

A: Messages might be lost (unless persistent)!

  • RabbitMQ stores messages in memory by default
  • Crash → messages gone
  • Solution: Enable message persistence (disk storage)
  • Slight performance hit but messages survived crashes

Kafka: All messages on disk by default. Crash = no data loss.

Q3: Can I use RabbitMQ for event sourcing?

A: Technically yes, but not recommended.

Event Sourcing needs:

  • Messages stored forever (RabbitMQ deletes)
  • Ability to replay from any point (RabbitMQ doesn't support)
  • Append-only log (RabbitMQ doesn't guarantee)

Better choice: Kafka or database event log

Kafka + Event Sourcing = Perfect fit (store events, replay for state reconstruction)
