    🏗️ Architecture & Project Setup

    How to structure an ASP.NET Core Web API project using N-Tier architecture — Controllers, Services, DbContext, and Program.cs.

    Beginner Web API
    ℹ️

    Why Architecture Matters

    A clean architecture separates concerns so each layer has one job. This makes code easier to test, maintain, and extend. The standard ASP.NET Core pattern uses Controllers for routing, Services for business logic, and a DbContext for data access.

    Project Structure

    A well-organized Web API project follows a layered folder structure. Each folder has a single responsibility.

    Project Structure 
    ProductCatalogAPI/
├── Controllers/
│   └── ProductsController.cs    ← HTTP routing & request handling
├── Services/
│   ├── IProductService.cs       ← Interface (contract)
│   └── ProductService.cs        ← Business logic + caching
├── Data/
│   └── ProductDbContext.cs      ← EF Core DbContext
├── Models/
│   ├── Product.cs               ← Entity (maps to DB table)
│   └── DTOs/
│       ├── CreateProductRequest.cs
│       └── UpdateProductRequest.cs
├── Migrations/                  ← Auto-generated by EF Core
├── Program.cs                   ← App bootstrap & DI registration
└── ProductCatalogAPI.csproj     ← Project file with NuGet packages

    N-Tier Architecture

    Each request flows through three layers. Each layer only calls the one below it — controllers never touch the database directly.

    Layer 1: ControllerReceives HTTP request, validates input, calls service, returns HTTP response
    Layer 2: ServiceBusiness logic, caching, orchestration, calls DbContext
    Layer 3: DbContextEF Core — translates LINQ to SQL, manages connections & transactions
    Flow 
    HTTP Request → ProductsController → IProductService → ProductDbContext → SQL Server
HTTP Response ←               ←                   ←                  ←

    Program.cs — App Bootstrap

    Program.cs is the entry point. It configures services (dependency injection container) and the middleware pipeline. Everything is registered here before the app starts.

    Program.cs 
    var builder = WebApplication.CreateBuilder(args);

// ── SERVICES (DI Container) ──────────────────────────────────
builder.Services.AddControllers();
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen();

// Database — EF Core with SQL Server
builder.Services.AddDbContext<ProductDbContext>(options =>
    options.UseSqlServer(builder.Configuration.GetConnectionString("DefaultConnection")));

// Business logic service
builder.Services.AddScoped<IProductService, ProductService>();

// In-memory caching
builder.Services.AddMemoryCache();

// Response caching middleware
builder.Services.AddResponseCaching();

// CORS — allow React (3000) and Angular (4200)
builder.Services.AddCors(options =>
{
    options.AddPolicy("AllowFrontend", policy =>
        policy.WithOrigins("http://localhost:3000", "http://localhost:4200")
              .AllowAnyHeader()
              .AllowAnyMethod());
});

var app = builder.Build();

// ── MIDDLEWARE PIPELINE (order matters!) ─────────────────────
if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI();
}

app.UseHttpsRedirection();
app.UseCors("AllowFrontend");
app.UseResponseCaching();
app.UseAuthorization();
app.MapControllers();

// Auto-create database on startup
using (var scope = app.Services.CreateScope())
{
    var db = scope.ServiceProvider.GetRequiredService<ProductDbContext>();
    db.Database.EnsureCreated();
}

app.Run();

    DbContext Configuration

    The DbContext is the bridge between your C# code and the database. It holds DbSet<T> properties for each entity and configuration in OnModelCreating.

    Data/ProductDbContext.cs 
    public class ProductDbContext : DbContext
{
    public ProductDbContext(DbContextOptions<ProductDbContext> options)
        : base(options) { }

    public DbSet<Product> Products { get; set; }

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        // Soft delete — only return IsActive=true by default
        modelBuilder.Entity<Product>()
            .HasQueryFilter(p => p.IsActive);

        // Index for performance
        modelBuilder.Entity<Product>()
            .HasIndex(p => p.Category);

        // Decimal precision
        modelBuilder.Entity<Product>()
            .Property(p => p.Price)
            .HasColumnType("decimal(18,2)");
    }
}

    Connection String

    The connection string lives in appsettings.json and is injected into AddDbContext.

    appsettings.json 
    {
  "ConnectionStrings": {
    "DefaultConnection": "Server=(localdb)\\mssqllocaldb;Database=ProductCatalogDB;Trusted_Connection=True;"
  },
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft.EntityFrameworkCore.Database.Command": "Warning"
    }
  }
}
    ⚠️

    Never commit secrets

    Use appsettings.Development.json or environment variables for real connection strings and API keys. The .gitignore template excludes appsettings.*.json by default.

    Interview Tips

    💡

    Common questions on this topic

    • What is the difference between AddSingleton, AddScoped, and AddTransient? — Singleton: one instance forever. Scoped: one per HTTP request. Transient: new instance every time.
    • Why separate controllers from services? — Controllers handle HTTP concerns; services handle business logic. This makes services testable without HTTP context.
    • Where does middleware run relative to controllers? — Middleware runs before routing reaches a controller. Each middleware component can short-circuit the pipeline.
