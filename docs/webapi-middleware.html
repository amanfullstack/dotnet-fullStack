    ⚙️ Middleware Pipeline

    Every HTTP request passes through a chain of middleware components. Learn how to use built-in middleware and write your own.

    Intermediate Web API
    ℹ️

    What is Middleware?

    Middleware is software assembled into an application pipeline to handle requests and responses. Each component can process the request, pass it to the next component, and then process the response on the way back. Think of it as a chain of responsibility.

    Pipeline Order (Matters!)

    The order middleware is registered in Program.cs determines the order it executes. This is critical — wrong order causes bugs.

    Correct Middleware Order in Program.cs 
    var app = builder.Build();

// ① Exception handling — must be FIRST to catch errors from any middleware below
app.UseExceptionHandler("/error");
// or in development:
app.UseDeveloperExceptionPage();

// ② HTTPS redirection — early, before anything that needs secure connection
app.UseHttpsRedirection();

// ③ Static files — short-circuits before routing (no auth check needed)
app.UseStaticFiles();

// ④ Routing — maps URL to route endpoint (before Auth)
app.UseRouting();

// ⑤ CORS — must come before UseAuthorization for preflight to work
app.UseCors("AllowFrontend");

// ⑥ Authentication — verifies identity (who you are)
app.UseAuthentication();

// ⑦ Authorization — verifies permissions (what you can do)
app.UseAuthorization();

// ⑧ Response caching — cache responses
app.UseResponseCaching();

// ⑨ Custom middleware
app.UseMiddleware<RequestLoggingMiddleware>();
app.UseMiddleware<TimingMiddleware>();

// ⑩ Controllers — terminal middleware (handles request, returns response)
app.MapControllers();

    Writing Custom Middleware

    Two approaches: inline with app.Use() or a dedicated class.

    Inline Middleware

    Inline with app.Use() 
    // app.Use() — passes to next middleware
app.Use(async (context, next) =>
{
    // Code runs BEFORE next middleware (request phase)
    Console.WriteLine($"→ {context.Request.Method} {context.Request.Path}");

    await next(context);   // Call next middleware in pipeline

    // Code runs AFTER next middleware (response phase)
    Console.WriteLine($"← {context.Response.StatusCode}");
});

// app.Run() — terminal, does NOT call next (short-circuits pipeline)
app.Run(async context =>
{
    await context.Response.WriteAsync("Hello from terminal middleware");
});

// app.Map() — branch pipeline based on path
app.Map("/health", healthApp =>
{
    healthApp.Run(async context =>
    {
        await context.Response.WriteAsync("Healthy");
    });
});

    Class-Based Middleware

    Middleware/RequestLoggingMiddleware.cs 
    public class RequestLoggingMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<RequestLoggingMiddleware> _logger;

    // RequestDelegate is injected by the framework — it's the next middleware
    public RequestLoggingMiddleware(RequestDelegate next, ILogger<RequestLoggingMiddleware> logger)
    {
        _next   = next;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        var stopwatch = Stopwatch.StartNew();

        _logger.LogInformation(
            "→ {Method} {Path}",
            context.Request.Method,
            context.Request.Path);

        await _next(context);   // Call next middleware

        stopwatch.Stop();
        _logger.LogInformation(
            "← {StatusCode} in {ElapsedMs}ms",
            context.Response.StatusCode,
            stopwatch.ElapsedMilliseconds);
    }
}

// Register in Program.cs:
// app.UseMiddleware<RequestLoggingMiddleware>();

    IMiddleware Interface

    IMiddleware is a factory-activated alternative. The class is resolved from DI (supports scoped services), unlike the convention-based approach.

    IMiddleware — DI-friendly 
    public class ApiKeyMiddleware : IMiddleware
{
    // Constructor can inject Scoped or Transient services
    private readonly IApiKeyService _apiKeyService;

    public ApiKeyMiddleware(IApiKeyService apiKeyService)
    {
        _apiKeyService = apiKeyService;
    }

    public async Task InvokeAsync(HttpContext context, RequestDelegate next)
    {
        if (!context.Request.Headers.TryGetValue("X-Api-Key", out var key))
        {
            context.Response.StatusCode = 401;
            await context.Response.WriteAsync("Missing API Key");
            return; // Short-circuit — do NOT call next
        }

        if (!await _apiKeyService.IsValidAsync(key!))
        {
            context.Response.StatusCode = 403;
            await context.Response.WriteAsync("Invalid API Key");
            return;
        }

        await next(context); // Valid key — continue pipeline
    }
}

// Register in Program.cs:
// builder.Services.AddScoped<ApiKeyMiddleware>();  // must register as service
// app.UseMiddleware<ApiKeyMiddleware>();
    💡

    Use/Run/Map summary

    • Use — process request AND pass to next. Most middleware uses this.
    • Run — terminal. Handles request, does NOT call next. Short-circuits pipeline.
    • Map — branch pipeline based on path prefix (/api, /health, etc.).
