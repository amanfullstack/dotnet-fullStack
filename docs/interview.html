    Web API & Architecture

    Q1. What is ASP.NET Core Web API? How does it differ from MVC? ▼

    ASP.NET Core Web API is a framework for building HTTP services that return data (JSON/XML) instead of HTML. MVC returns rendered HTML views for browsers. Web API targets API consumers — mobile apps, SPAs, other backend services.

    Key differences:

    • Web API uses [ApiController]; MVC uses [Controller]
    • Web API returns ActionResult<T> with data; MVC returns views via IActionResult
    • Web API supports content negotiation (JSON/XML); MVC returns HTML
    • [ApiController] automatically validates models and returns 400 on failure
    • Web API is stateless; MVC can use session state
    Q2. Explain the N-Tier architecture in a real project. ▼

    N-Tier separates concerns into distinct layers. In ProductCatalogAPI:

    • Controllers — handle HTTP requests, validate input, call services, return HTTP responses
    • Services — contain business logic, caching, orchestration; call DbContext or repositories
    • Data (DbContext) — all database communication via EF Core or ADO.NET
    • Models — Entity models (DB) and DTOs (API contract) are separate classes

    Benefits: each layer is independently testable, changes don't cascade across layers, and implementations can be swapped (e.g., SQL Server → PostgreSQL).

    Q3. What HTTP status codes do you return and when? ▼
    • 200 OK — successful GET, PUT, PATCH with response body
    • 201 Created — successful POST; include Location header pointing to new resource
    • 204 No Content — successful DELETE or PUT with no body
    • 400 Bad Request — validation failure, invalid input
    • 401 Unauthorized — not authenticated (no/invalid token)
    • 403 Forbidden — authenticated but not authorized for the action
    • 404 Not Found — resource does not exist
    • 409 Conflict — duplicate resource or concurrency conflict
    • 500 Internal Server Error — unhandled exception
    Q4. What is model binding? What sources does it use? ▼

    Model binding automatically maps HTTP request data to action method parameters. ASP.NET Core reads from multiple sources:

    • [FromRoute] — URL path segment (e.g., /products/{id})
    • [FromQuery] — query string (e.g., ?category=Electronics)
    • [FromBody] — JSON request body (POST/PUT); requires Content-Type: application/json
    • [FromHeader] — HTTP request header
    • [FromForm] — multipart form data

    With [ApiController], complex types on POST/PUT default to [FromBody] and simple types default to [FromQuery].

    Q5. What are DTOs and why use them instead of entities directly? ▼

    DTOs (Data Transfer Objects) are API-facing classes separate from database entity models. Benefits:

    • Security — hide sensitive fields (passwords, internal fields) from responses
    • Shape — return only fields clients need; reduce payload size
    • Versioning — change entity structure without breaking the API contract
    • Validation — put request validation attributes on DTOs, not entities
    • Separation — DB concerns in entities, API concerns in DTOs

    Common pattern: CreateProductRequest for POST body, ProductResponse for all responses, UpdateProductRequest for PUT.

    Q6. What is IActionResult vs ActionResult<T>? ▼

    IActionResult is a non-generic interface — it can return any result type (Ok, NotFound, BadRequest) but Swagger and the compiler have no information about the success response type.

    ActionResult<T> is the generic version. It explicitly declares the success return type, enabling Swagger to infer response schemas and giving you compile-time type safety. You can still return NotFound() alongside a typed result.

    Best practice: Use ActionResult<T> for better Swagger documentation and IDE support. Use IActionResult only when the return type truly varies beyond success/failure.

    Q7. How does input validation work with [ApiController]? ▼

    With [ApiController], if a model fails validation, ASP.NET Core automatically returns 400 Bad Request with a ValidationProblemDetails JSON response before the action method even runs. You don't need to check ModelState.IsValid manually.

    Add data annotations to DTOs: [Required], [StringLength(100, MinimumLength=3)], [Range(0, 9999)], [EmailAddress], [RegularExpression(@"^\d{5}$")].

    For complex cross-property validation, implement IValidatableObject or use FluentValidation library.

    Q8. What is content negotiation? ▼

    Content negotiation is the process where client and server agree on the response format. The client sends an Accept header (e.g., Accept: application/json or application/xml) and ASP.NET Core chooses a matching formatter.

    Default: JSON. To add XML support: builder.Services.AddControllers().AddXmlSerializerFormatters(). If no matching formatter exists, returns 406 Not Acceptable.

    Dependency Injection & Middleware

    Q9. What is Dependency Injection and why is it important? ▼

    Dependency Injection (DI) is a pattern where a class receives dependencies from an external container instead of creating them itself. In ASP.NET Core, the built-in DI container manages service lifetimes and injects them via constructor parameters.

    Why it matters:

    • Testability — inject mock implementations in unit tests
    • Loose coupling — depend on interfaces, not concrete classes
    • Single Responsibility — classes don't manage their own dependencies
    • Lifetime management — the container handles creation, sharing, and disposal

    Register: builder.Services.AddScoped<IProductService, ProductService>(). Inject: public ProductsController(IProductService service).

    Q10. What are the 3 DI lifetimes? When do you use each? ▼
    • Singleton — one instance for the entire app lifetime. Shared by all requests. Use for: stateless services, caches, configuration helpers.
      builder.Services.AddSingleton<IMyService, MyService>()
    • Scoped — one instance per HTTP request. All components within the same request share the same instance. Use for: DbContext, business services.
      builder.Services.AddScoped<IProductService, ProductService>()
    • Transient — new instance every time it's requested from the container. Most lightweight. Use for: lightweight, stateless services.
      builder.Services.AddTransient<IEmailSender, EmailSender>()

    Captive dependency trap: Injecting a Scoped service into a Singleton causes the Scoped service to live as long as the Singleton (forever), leaking request data. .NET throws InvalidOperationException at startup in Development mode.

    Q11. What is middleware? How does it work? ▼

    Middleware is software assembled into a pipeline that handles HTTP requests and responses. Each middleware component can inspect/modify the request, pass it to the next middleware (via await _next(context)), and then inspect/modify the response on the way back.

    Pipeline order in Program.cs:

    csharp 
    app.UseMiddleware<GlobalErrorHandlingMiddleware>(); // 1st — catch all errors
app.UseMiddleware<RequestLoggingMiddleware>();       // 2nd — log requests
app.UseMiddleware<PerformanceMiddleware>();          // 3rd — timing
app.UseResponseCaching();                           // 4th — cache headers
app.UseCors("AllowFrontend");
app.UseAuthentication();
app.UseAuthorization();
app.MapControllers();
    Q12. Why does middleware ORDER matter? Give a real example. ▼

    Each middleware wraps the next one. Wrong order = bugs and security holes.

    Real example — Authentication before Authorization: UseAuthentication must come before UseAuthorization. Authentication sets HttpContext.User from the token/cookie. Authorization then checks if that user has permission. If order is reversed, HttpContext.User is empty and all authorization checks fail.

    Error handling must be first: UseExceptionHandler must be the first middleware so it can catch exceptions from all subsequent middleware. If it's last, exceptions from routing/auth won't be caught.

    Q13. app.Use() vs app.Run() vs app.Map() — what's the difference? ▼
    • app.Use() — adds middleware that calls next() to continue the pipeline. Can process both request (before await next()) and response (after).
    • app.Run() — terminal middleware. Never calls next(). Useful for a final catch-all handler. Nothing after it in the same branch executes.
    • app.Map("/api", branch => { ... }) — branches the pipeline based on URL path. Each branch has its own middleware. Useful for sub-apps or admin sections.

    Entity Framework Core

    Q14. What is EF Core and what problem does it solve? ▼

    EF Core is an ORM (Object-Relational Mapper) for .NET. It solves the impedance mismatch between OOP and relational databases — you define C# classes, and EF Core generates SQL, creates tables, and maps rows back to objects automatically.

    Without EF Core: write raw SQL for every operation, manually map result rows to objects, handle schema changes manually, write parameterization for every query. EF Core eliminates this boilerplate while providing LINQ queries, migrations, change tracking, and relationship management.

    Q15. Code-First vs Database-First — when do you use each? ▼

    Code-First: Define C# entity classes → run dotnet ef migrations add → EF Core generates the database. Code is the source of truth. Best for new projects — schema changes are in version control as migration files.

    Database-First: Have an existing database → run dotnet ef dbcontext scaffold → EF Core generates entity classes and DbContext. Database is the source of truth. Best for existing legacy databases you can't modify.

    Recommendation: Code-First for new projects — migrations enable team collaboration, rollbacks, and CI/CD pipeline automation.

    Q16. How does EF Core change tracking work? What is AsNoTracking? ▼

    When you load entities with EF Core, the DbContext records their original state as a snapshot. When you call SaveChanges(), EF Core compares current state to the snapshot and generates INSERT/UPDATE/DELETE SQL for changed entities.

    Entity states: Added, Modified, Deleted, Unchanged, Detached.

    AsNoTracking() tells EF Core to skip the snapshot — entities are loaded but not tracked. This is ~30% faster for read-only queries because EF skips the overhead of tracking state. Use it for all GET endpoints that don't modify data:

    csharp 
    // With tracking (needed for UPDATE/DELETE)
var product = await _context.Products.FindAsync(id);

// Without tracking (faster, for read-only)
var products = await _context.Products.AsNoTracking().ToListAsync();
    Q17. What is a migration? Walk me through the commands. ▼

    A migration is a versioned C# file representing a schema change. EF Core compares your current entity model to the last migration snapshot and generates the diff.

    • dotnet ef migrations add InitialCreate — creates a migration file with Up() and Down() methods
    • dotnet ef database update — applies pending migrations to the database
    • dotnet ef migrations remove — removes the last unapplied migration
    • dotnet ef database update PreviousMigration — rolls back to a specific migration
    • dotnet ef database drop — drops the entire database

    Never manually edit migration files. If you need to change a migration, remove it and recreate it.

    Q18. What is soft delete? How do you implement it in EF Core? ▼

    Soft delete marks records as inactive instead of physically deleting rows. Benefits: preserves history, enables recovery, maintains referential integrity.

    csharp 
    // 1. Add IsActive flag to entity
public class Product {
    public bool IsActive { get; set; } = true;
}

// 2. Global query filter in DbContext.OnModelCreating
modelBuilder.Entity<Product>()
    .HasQueryFilter(p => p.IsActive);

// 3. "Delete" method sets IsActive = false
product.IsActive = false;
await _context.SaveChangesAsync();

// All queries now automatically exclude IsActive=false
// Use .IgnoreQueryFilters() for admin screens that show deleted items
    Q19. What is the N+1 problem? How do you fix it? ▼

    N+1 occurs when you load N entities and then trigger 1 additional query per entity for related data — N+1 total queries instead of 2.

    csharp 
    // BAD: N+1 problem (101 queries for 100 products)
var products = await _context.Products.ToListAsync(); // 1 query
foreach(var p in products) {
    var category = p.Category; // 100 lazy load queries!
}

// GOOD: Eager loading with Include (2 queries or 1 JOIN)
var products = await _context.Products
    .Include(p => p.Category)
    .ToListAsync();

    ADO.NET

    Q20. What is ADO.NET and when do you choose it over EF Core? ▼

    ADO.NET is .NET's low-level data access layer. You write raw SQL statements and manually map results to objects. No abstraction, maximum control.

    Choose ADO.NET when:

    • Complex reporting queries with many JOINs/subqueries EF Core can't express well
    • Bulk operations — inserting/updating thousands of rows (SqlBulkCopy)
    • Maximum performance for critical hot paths
    • Legacy databases with stored procedures

    Best practice (used in ProductCatalogAPI): use both — EF Core for standard CRUD, ADO.NET for complex queries and bulk operations.

    Q21. What are parameterized queries? Why are they critical? ▼

    Parameterized queries use placeholders instead of string concatenation for values. The driver sends SQL and values separately, so user input can never be interpreted as SQL code.

    csharp 
    // DANGEROUS — SQL Injection vulnerability!
string sql = "SELECT * FROM Products WHERE Name = '" + userInput + "'";
// Input: '; DROP TABLE Products; -- → deletes your database!

// SAFE — Parameterized query
string sql = "SELECT * FROM Products WHERE Name = @Name";
command.Parameters.AddWithValue("@Name", userInput);
// Input is treated as a literal value, not SQL code

    Always use parameterized queries for ANY user-provided input. This is the #1 defense against SQL injection — OWASP's top web vulnerability.

    Q22. How do you implement a transaction in ADO.NET? ▼

    Use SqlTransaction to wrap multiple operations in an atomic unit — all succeed or all roll back.

    csharp 
    using var connection = new SqlConnection(_connectionString);
await connection.OpenAsync();
using var transaction = connection.BeginTransaction();
try {
    // Debit account
    var cmd1 = new SqlCommand("UPDATE Accounts SET Balance -= @Amount WHERE Id=@From", connection, transaction);
    cmd1.Parameters.AddWithValue("@Amount", amount);
    cmd1.Parameters.AddWithValue("@From", fromId);
    await cmd1.ExecuteNonQueryAsync();

    // Credit account
    var cmd2 = new SqlCommand("UPDATE Accounts SET Balance += @Amount WHERE Id=@To", connection, transaction);
    cmd2.Parameters.AddWithValue("@Amount", amount);
    cmd2.Parameters.AddWithValue("@To", toId);
    await cmd2.ExecuteNonQueryAsync();

    await transaction.CommitAsync();
} catch {
    await transaction.RollbackAsync();
    throw;
}
    Q23. What is connection pooling and how does it work? ▼

    Connection pooling reuses database connections instead of creating a new TCP connection for every query. Opening a new DB connection costs ~100-200ms. The pool maintains pre-created connections that are leased and returned.

    In .NET it's automatic — enabled by default. When you call connection.Close() or a using block ends, the connection returns to the pool (not physically closed). The pool is per connection string. Always use using to promptly return connections. Configure pool size in the connection string: Min Pool Size=5;Max Pool Size=100;

    LINQ & Collections

    Q24. What is LINQ? What are the two syntax styles? ▼

    LINQ (Language Integrated Query) lets you query any IEnumerable<T> or IQueryable<T> — in-memory collections, databases, XML — with consistent C# syntax.

    csharp 
    // Method syntax (preferred — more flexible for chaining)
var results = products
    .Where(p => p.Price > 100 && p.IsActive)
    .OrderBy(p => p.Name)
    .Select(p => new { p.Id, p.Name, p.Price })
    .ToList();

// Query syntax (SQL-like, sometimes more readable for joins)
var results = from p in products
              where p.Price > 100 && p.IsActive
              orderby p.Name
              select new { p.Id, p.Name, p.Price };

    Both compile to identical IL. Method syntax is more flexible for complex operations.

    Q25. What is deferred execution? Why does it matter? ▼

    Deferred execution means a LINQ query is defined but NOT executed until you iterate over it. Where(), Select(), OrderBy() build a query definition. Actual execution happens when you call ToList(), First(), Count(), or enter a foreach.

    Why it matters with EF Core: You can compose queries conditionally — add .Where() clauses based on parameters — and EF Core generates a single optimized SQL query when you finally materialize. Without deferred execution, each filter would be a separate DB round trip.

    csharp 
    // Composing a query — no DB call yet
var query = _context.Products.Where(p => p.IsActive);
if (!string.IsNullOrEmpty(category))
    query = query.Where(p => p.Category == category);
if (minPrice.HasValue)
    query = query.Where(p => p.Price >= minPrice.Value);

// Single DB call happens HERE
var results = await query.ToListAsync();
    Q26. IEnumerable vs IQueryable — what's the difference? ▼

    IEnumerable<T> — executes in memory (C# process). Filter happens after all data is loaded. Use for in-memory collections.

    IQueryable<T> — builds an expression tree. EF Core translates it to SQL. Filter happens in the database — only matching rows returned. Use with EF Core/LINQ to SQL.

    csharp 
    // BAD: Loads ALL products into memory, then filters in C#
var expensive = _context.Products
    .AsEnumerable()            // forces in-memory execution!
    .Where(p => p.Price > 100)
    .ToList();

// GOOD: SQL WHERE clause filters in database
var expensive = _context.Products
    .Where(p => p.Price > 100)   // translates to SQL WHERE
    .ToList();
    Q27. What collection types exist in C# and when do you use each? ▼
    • List<T> — ordered, index access, duplicates allowed. General-purpose default.
    • Dictionary<TKey,TValue> — O(1) key lookup. Use when finding by unique key frequently.
    • HashSet<T> — unique elements, O(1) Contains. Use for deduplication or set operations.
    • Queue<T> — FIFO. Use for job/message queues, BFS.
    • Stack<T> — LIFO. Use for undo/redo, DFS, parsing expressions.
    • SortedDictionary<TKey,TValue> — sorted by key, O(log n). Use when you need ordered enumeration.
    • ConcurrentDictionary<TKey,TValue> — thread-safe. Use in multi-threaded code.
    • IEnumerable<T> — read-only lazy sequence. Use as parameter/return type for flexibility.

    Caching & Performance

    Q28. What caching strategies did you implement? Explain each. ▼

    ProductCatalogAPI implements 5 caching strategies:

    • IMemoryCache — in-process RAM cache. ~1ms. Single server. Invalidated on data changes. Best for frequently-accessed lookup data.
    • Response Caching — HTTP Cache-Control headers via [ResponseCache(Duration=300)]. Browser/proxy caches response. Zero server load on cache hit.
    • IDistributedCache — shared cache interface (Redis, SQL Server). ~5ms. Works across multiple servers.
    • Redis — high-performance distributed cache. Survives app restarts. Used in production for multi-server apps.
    • HybridCache (.NET 9) — checks memory first → then Redis → then DB. Best of all worlds.
    Q29. How do you implement IMemoryCache? Show cache invalidation. ▼
    csharp 
    // Registration in Program.cs
builder.Services.AddMemoryCache();

// Usage in service
public async Task<IEnumerable<Product>> GetAllAsync()
{
    const string key = "products:all";
    if (_cache.TryGetValue(key, out IEnumerable<Product> cached))
        return cached; // ✓ HIT — returned in ~1ms

    var products = await _context.Products.AsNoTracking().ToListAsync();
    _cache.Set(key, products, TimeSpan.FromMinutes(5)); // TTL: 5 min
    return products; // ✗ MISS — fetched from DB (~150ms)
}

// Invalidation on data change
public async Task CreateProductAsync(CreateProductRequest req)
{
    // ... create product ...
    _cache.Remove("products:all"); // Stale cache cleared
}

    Async/Await & General

    Q31. Why is async/await critical in Web API applications? ▼

    ASP.NET Core handles requests on a thread pool (typically ~250 threads). Without async, a thread blocked waiting for a DB query can't handle other requests — under load, threads exhaust and the app becomes unresponsive.

    With async/await, when you await _context.ToListAsync(), the current thread is returned to the pool while waiting for the database. When the DB responds, any available thread picks up the work. This means one server can handle thousands of concurrent requests instead of ~250.

    Rule: use async/await for all I/O (DB, HTTP, file). Never use .Result or .Wait() — they block a thread and can deadlock.

    Q32. What is CORS and how do you configure it? ▼

    CORS (Cross-Origin Resource Sharing) is a browser security mechanism that blocks web pages from calling APIs on a different domain. When your React app on localhost:3000 calls the API on localhost:5000, the browser requires the API to explicitly allow this origin.

    csharp 
    // Program.cs
builder.Services.AddCors(options => options.AddPolicy("AllowFrontend",
    policy => policy
        .WithOrigins("http://localhost:3000", "http://localhost:4200")
        .AllowAnyMethod()
        .AllowAnyHeader()));

// In pipeline (before MapControllers)
app.UseCors("AllowFrontend");

    In production: specify exact origins only. Never use .AllowAnyOrigin() in production.

    Q33. What is GraphQL vs REST — when do you use each? ▼

    REST: Multiple endpoints, each returns a fixed structure. Simple, well-known, HTTP cache-friendly. Best for: most APIs, microservices, public APIs.

    GraphQL: Single endpoint, client specifies exactly what fields they need. Eliminates over-fetching (unused fields) and under-fetching (needing multiple requests). Best for: mobile apps (minimize bandwidth), dashboards fetching varied data, APIs consumed by many different clients needing different data shapes.

    In .NET, use HotChocolate for GraphQL: define Query/Mutation types, register with AddGraphQLServer(), expose at /graphql with Banana Cake Pop playground.

    Q34. How do you handle global error handling in a Web API? ▼

    Best practice: custom global exception middleware as the first middleware in the pipeline. It wraps all subsequent middleware in a try-catch, catches any unhandled exception, logs it, and returns a standardized ProblemDetails response.

    csharp 
    public class GlobalErrorHandlingMiddleware(RequestDelegate next, ILogger<GlobalErrorHandlingMiddleware> logger)
{
    public async Task InvokeAsync(HttpContext context)
    {
        try {
            await next(context);
        } catch (Exception ex) {
            logger.LogError(ex, "Unhandled exception for {Path}", context.Request.Path);
            context.Response.StatusCode = 500;
            context.Response.ContentType = "application/json";
            await context.Response.WriteAsJsonAsync(new {
                Status = 500,
                Message = "An internal error occurred."
            });
        }
    }
}
    Q35. What is the Options pattern and how do you use IConfiguration? ▼

    The Options pattern binds configuration sections to strongly-typed C# classes, giving compile-time safety and IntelliSense instead of magic strings.

    csharp 
    // appsettings.json
{
  "CacheSettings": {
    "DefaultTtlMinutes": 5,
    "ProductsKey": "products:all"
  }
}

// Typed settings class
public class CacheSettings {
    public int DefaultTtlMinutes { get; set; }
    public string ProductsKey { get; set; } = "";
}

// Program.cs registration
builder.Services.Configure<CacheSettings>(
    builder.Configuration.GetSection("CacheSettings"));

// Inject and use
public ProductService(IOptions<CacheSettings> options) {
    var ttl = options.Value.DefaultTtlMinutes;
}
