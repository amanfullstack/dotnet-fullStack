    Web API & Architecture

    Q1. What is ASP.NET Core Web API? How does it differ from MVC? ▼

    ASP.NET Core Web API is a framework for building HTTP services that return data (JSON/XML) instead of HTML. MVC returns rendered HTML views for browsers. Web API targets API consumers — mobile apps, SPAs, other backend services.

    Key differences:

    • Web API uses [ApiController]; MVC uses [Controller]
    • Web API returns ActionResult<T> with data; MVC returns views via IActionResult
    • Web API supports content negotiation (JSON/XML); MVC returns HTML
    • [ApiController] automatically validates models and returns 400 on failure
    • Web API is stateless; MVC can use session state
    Q2. Explain the N-Tier architecture in a real project. ▼

    N-Tier separates concerns into distinct layers. In ProductCatalogAPI:

    • Controllers — handle HTTP requests, validate input, call services, return HTTP responses
    • Services — contain business logic, caching, orchestration; call DbContext or repositories
    • Data (DbContext) — all database communication via EF Core or ADO.NET
    • Models — Entity models (DB) and DTOs (API contract) are separate classes

    Benefits: each layer is independently testable, changes don't cascade across layers, and implementations can be swapped (e.g., SQL Server → PostgreSQL).

    Q3. What HTTP status codes do you return and when? ▼
    • 200 OK — successful GET, PUT, PATCH with response body
    • 201 Created — successful POST; include Location header pointing to new resource
    • 204 No Content — successful DELETE or PUT with no body
    • 400 Bad Request — validation failure, invalid input
    • 401 Unauthorized — not authenticated (no/invalid token)
    • 403 Forbidden — authenticated but not authorized for the action
    • 404 Not Found — resource does not exist
    • 409 Conflict — duplicate resource or concurrency conflict
    • 500 Internal Server Error — unhandled exception
    Q4. What is model binding? What sources does it use? ▼

    Model binding automatically maps HTTP request data to action method parameters. ASP.NET Core reads from multiple sources:

    • [FromRoute] — URL path segment (e.g., /products/{id})
    • [FromQuery] — query string (e.g., ?category=Electronics)
    • [FromBody] — JSON request body (POST/PUT); requires Content-Type: application/json
    • [FromHeader] — HTTP request header
    • [FromForm] — multipart form data

    With [ApiController], complex types on POST/PUT default to [FromBody] and simple types default to [FromQuery].

    Q5. What are DTOs and why use them instead of entities directly? ▼

    DTOs (Data Transfer Objects) are API-facing classes separate from database entity models. Benefits:

    • Security — hide sensitive fields (passwords, internal fields) from responses
    • Shape — return only fields clients need; reduce payload size
    • Versioning — change entity structure without breaking the API contract
    • Validation — put request validation attributes on DTOs, not entities
    • Separation — DB concerns in entities, API concerns in DTOs

    Common pattern: CreateProductRequest for POST body, ProductResponse for all responses, UpdateProductRequest for PUT.

    Q6. What is IActionResult vs ActionResult<T>? ▼

    IActionResult is a non-generic interface — it can return any result type (Ok, NotFound, BadRequest) but Swagger and the compiler have no information about the success response type.

    ActionResult<T> is the generic version. It explicitly declares the success return type, enabling Swagger to infer response schemas and giving you compile-time type safety. You can still return NotFound() alongside a typed result.

    Best practice: Use ActionResult<T> for better Swagger documentation and IDE support. Use IActionResult only when the return type truly varies beyond success/failure.

    Q7. How does input validation work with [ApiController]? ▼

    With [ApiController], if a model fails validation, ASP.NET Core automatically returns 400 Bad Request with a ValidationProblemDetails JSON response before the action method even runs. You don't need to check ModelState.IsValid manually.

    Add data annotations to DTOs: [Required], [StringLength(100, MinimumLength=3)], [Range(0, 9999)], [EmailAddress], [RegularExpression(@"^\d{5}$")].

    For complex cross-property validation, implement IValidatableObject or use FluentValidation library.

    Q8. What is content negotiation? ▼

    Content negotiation is the process where client and server agree on the response format. The client sends an Accept header (e.g., Accept: application/json or application/xml) and ASP.NET Core chooses a matching formatter.

    Default: JSON. To add XML support: builder.Services.AddControllers().AddXmlSerializerFormatters(). If no matching formatter exists, returns 406 Not Acceptable.

    Dependency Injection & Middleware

    Q9. What is Dependency Injection and why is it important? ▼

    Dependency Injection (DI) is a pattern where a class receives dependencies from an external container instead of creating them itself. In ASP.NET Core, the built-in DI container manages service lifetimes and injects them via constructor parameters.

    Why it matters:

    • Testability — inject mock implementations in unit tests
    • Loose coupling — depend on interfaces, not concrete classes
    • Single Responsibility — classes don't manage their own dependencies
    • Lifetime management — the container handles creation, sharing, and disposal

    Register: builder.Services.AddScoped<IProductService, ProductService>(). Inject: public ProductsController(IProductService service).

    Q10. What are the 3 DI lifetimes? When do you use each? ▼
    • Singleton — one instance for the entire app lifetime. Shared by all requests. Use for: stateless services, caches, configuration helpers.
      builder.Services.AddSingleton<IMyService, MyService>()
    • Scoped — one instance per HTTP request. All components within the same request share the same instance. Use for: DbContext, business services.
      builder.Services.AddScoped<IProductService, ProductService>()
    • Transient — new instance every time it's requested from the container. Most lightweight. Use for: lightweight, stateless services.
      builder.Services.AddTransient<IEmailSender, EmailSender>()

    Captive dependency trap: Injecting a Scoped service into a Singleton causes the Scoped service to live as long as the Singleton (forever), leaking request data. .NET throws InvalidOperationException at startup in Development mode.

    Q11. What is middleware? How does it work? ▼

    Middleware is software assembled into a pipeline that handles HTTP requests and responses. Each middleware component can inspect/modify the request, pass it to the next middleware (via await _next(context)), and then inspect/modify the response on the way back.

    Pipeline order in Program.cs:

    csharp 
    app.UseMiddleware<GlobalErrorHandlingMiddleware>(); // 1st — catch all errors
app.UseMiddleware<RequestLoggingMiddleware>();       // 2nd — log requests
app.UseMiddleware<PerformanceMiddleware>();          // 3rd — timing
app.UseResponseCaching();                           // 4th — cache headers
app.UseCors("AllowFrontend");
app.UseAuthentication();
app.UseAuthorization();
app.MapControllers();
    Q12. Why does middleware ORDER matter? Give a real example. ▼

    Each middleware wraps the next one. Wrong order = bugs and security holes.

    Real example — Authentication before Authorization: UseAuthentication must come before UseAuthorization. Authentication sets HttpContext.User from the token/cookie. Authorization then checks if that user has permission. If order is reversed, HttpContext.User is empty and all authorization checks fail.

    Error handling must be first: UseExceptionHandler must be the first middleware so it can catch exceptions from all subsequent middleware. If it's last, exceptions from routing/auth won't be caught.

    Q13. app.Use() vs app.Run() vs app.Map() — what's the difference? ▼
    • app.Use() — adds middleware that calls next() to continue the pipeline. Can process both request (before await next()) and response (after).
    • app.Run() — terminal middleware. Never calls next(). Useful for a final catch-all handler. Nothing after it in the same branch executes.
    • app.Map("/api", branch => { ... }) — branches the pipeline based on URL path. Each branch has its own middleware. Useful for sub-apps or admin sections.

    Entity Framework Core

    Q14. What is EF Core and what problem does it solve? ▼

    EF Core is an ORM (Object-Relational Mapper) for .NET. It solves the impedance mismatch between OOP and relational databases — you define C# classes, and EF Core generates SQL, creates tables, and maps rows back to objects automatically.

    Without EF Core: write raw SQL for every operation, manually map result rows to objects, handle schema changes manually, write parameterization for every query. EF Core eliminates this boilerplate while providing LINQ queries, migrations, change tracking, and relationship management.

    Q15. Code-First vs Database-First — when do you use each? ▼

    Code-First: Define C# entity classes → run dotnet ef migrations add → EF Core generates the database. Code is the source of truth. Best for new projects — schema changes are in version control as migration files.

    Database-First: Have an existing database → run dotnet ef dbcontext scaffold → EF Core generates entity classes and DbContext. Database is the source of truth. Best for existing legacy databases you can't modify.

    Recommendation: Code-First for new projects — migrations enable team collaboration, rollbacks, and CI/CD pipeline automation.

    Q16. How does EF Core change tracking work? What is AsNoTracking? ▼

    When you load entities with EF Core, the DbContext records their original state as a snapshot. When you call SaveChanges(), EF Core compares current state to the snapshot and generates INSERT/UPDATE/DELETE SQL for changed entities.

    Entity states: Added, Modified, Deleted, Unchanged, Detached.

    AsNoTracking() tells EF Core to skip the snapshot — entities are loaded but not tracked. This is ~30% faster for read-only queries because EF skips the overhead of tracking state. Use it for all GET endpoints that don't modify data:

    csharp 
    // With tracking (needed for UPDATE/DELETE)
var product = await _context.Products.FindAsync(id);

// Without tracking (faster, for read-only)
var products = await _context.Products.AsNoTracking().ToListAsync();
    Q17. What is a migration? Walk me through the commands. ▼

    A migration is a versioned C# file representing a schema change. EF Core compares your current entity model to the last migration snapshot and generates the diff.

    • dotnet ef migrations add InitialCreate — creates a migration file with Up() and Down() methods
    • dotnet ef database update — applies pending migrations to the database
    • dotnet ef migrations remove — removes the last unapplied migration
    • dotnet ef database update PreviousMigration — rolls back to a specific migration
    • dotnet ef database drop — drops the entire database

    Never manually edit migration files. If you need to change a migration, remove it and recreate it.

    Q18. What is soft delete? How do you implement it in EF Core? ▼

    Soft delete marks records as inactive instead of physically deleting rows. Benefits: preserves history, enables recovery, maintains referential integrity.

    csharp 
    // 1. Add IsActive flag to entity
public class Product {
    public bool IsActive { get; set; } = true;
}

// 2. Global query filter in DbContext.OnModelCreating
modelBuilder.Entity<Product>()
    .HasQueryFilter(p => p.IsActive);

// 3. "Delete" method sets IsActive = false
product.IsActive = false;
await _context.SaveChangesAsync();

// All queries now automatically exclude IsActive=false
// Use .IgnoreQueryFilters() for admin screens that show deleted items
    Q19. What is the N+1 problem? How do you fix it? ▼

    N+1 occurs when you load N entities and then trigger 1 additional query per entity for related data — N+1 total queries instead of 2.

    csharp 
    // BAD: N+1 problem (101 queries for 100 products)
var products = await _context.Products.ToListAsync(); // 1 query
foreach(var p in products) {
    var category = p.Category; // 100 lazy load queries!
}

// GOOD: Eager loading with Include (2 queries or 1 JOIN)
var products = await _context.Products
    .Include(p => p.Category)
    .ToListAsync();

    ADO.NET

    Q20. What is ADO.NET and when do you choose it over EF Core? ▼

    ADO.NET is .NET's low-level data access layer. You write raw SQL statements and manually map results to objects. No abstraction, maximum control.

    Choose ADO.NET when:

    • Complex reporting queries with many JOINs/subqueries EF Core can't express well
    • Bulk operations — inserting/updating thousands of rows (SqlBulkCopy)
    • Maximum performance for critical hot paths
    • Legacy databases with stored procedures

    Best practice (used in ProductCatalogAPI): use both — EF Core for standard CRUD, ADO.NET for complex queries and bulk operations.

    Q21. What are parameterized queries? Why are they critical? ▼

    Parameterized queries use placeholders instead of string concatenation for values. The driver sends SQL and values separately, so user input can never be interpreted as SQL code.

    csharp 
    // DANGEROUS — SQL Injection vulnerability!
string sql = "SELECT * FROM Products WHERE Name = '" + userInput + "'";
// Input: '; DROP TABLE Products; -- → deletes your database!

// SAFE — Parameterized query
string sql = "SELECT * FROM Products WHERE Name = @Name";
command.Parameters.AddWithValue("@Name", userInput);
// Input is treated as a literal value, not SQL code

    Always use parameterized queries for ANY user-provided input. This is the #1 defense against SQL injection — OWASP's top web vulnerability.

    Q22. How do you implement a transaction in ADO.NET? ▼

    Use SqlTransaction to wrap multiple operations in an atomic unit — all succeed or all roll back.

    csharp 
    using var connection = new SqlConnection(_connectionString);
await connection.OpenAsync();
using var transaction = connection.BeginTransaction();
try {
    // Debit account
    var cmd1 = new SqlCommand("UPDATE Accounts SET Balance -= @Amount WHERE Id=@From", connection, transaction);
    cmd1.Parameters.AddWithValue("@Amount", amount);
    cmd1.Parameters.AddWithValue("@From", fromId);
    await cmd1.ExecuteNonQueryAsync();

    // Credit account
    var cmd2 = new SqlCommand("UPDATE Accounts SET Balance += @Amount WHERE Id=@To", connection, transaction);
    cmd2.Parameters.AddWithValue("@Amount", amount);
    cmd2.Parameters.AddWithValue("@To", toId);
    await cmd2.ExecuteNonQueryAsync();

    await transaction.CommitAsync();
} catch {
    await transaction.RollbackAsync();
    throw;
}
    Q23. What is connection pooling and how does it work? ▼

    Connection pooling reuses database connections instead of creating a new TCP connection for every query. Opening a new DB connection costs ~100-200ms. The pool maintains pre-created connections that are leased and returned.

    In .NET it's automatic — enabled by default. When you call connection.Close() or a using block ends, the connection returns to the pool (not physically closed). The pool is per connection string. Always use using to promptly return connections. Configure pool size in the connection string: Min Pool Size=5;Max Pool Size=100;

    LINQ & Collections

    Q24. What is LINQ? What are the two syntax styles? ▼

    LINQ (Language Integrated Query) lets you query any IEnumerable<T> or IQueryable<T> — in-memory collections, databases, XML — with consistent C# syntax.

    csharp 
    // Method syntax (preferred — more flexible for chaining)
var results = products
    .Where(p => p.Price > 100 && p.IsActive)
    .OrderBy(p => p.Name)
    .Select(p => new { p.Id, p.Name, p.Price })
    .ToList();

// Query syntax (SQL-like, sometimes more readable for joins)
var results = from p in products
              where p.Price > 100 && p.IsActive
              orderby p.Name
              select new { p.Id, p.Name, p.Price };

    Both compile to identical IL. Method syntax is more flexible for complex operations.

    Q25. What is deferred execution? Why does it matter? ▼

    Deferred execution means a LINQ query is defined but NOT executed until you iterate over it. Where(), Select(), OrderBy() build a query definition. Actual execution happens when you call ToList(), First(), Count(), or enter a foreach.

    Why it matters with EF Core: You can compose queries conditionally — add .Where() clauses based on parameters — and EF Core generates a single optimized SQL query when you finally materialize. Without deferred execution, each filter would be a separate DB round trip.

    csharp 
    // Composing a query — no DB call yet
var query = _context.Products.Where(p => p.IsActive);
if (!string.IsNullOrEmpty(category))
    query = query.Where(p => p.Category == category);
if (minPrice.HasValue)
    query = query.Where(p => p.Price >= minPrice.Value);

// Single DB call happens HERE
var results = await query.ToListAsync();
    Q26. IEnumerable vs IQueryable — what's the difference? ▼

    IEnumerable<T> — executes in memory (C# process). Filter happens after all data is loaded. Use for in-memory collections.

    IQueryable<T> — builds an expression tree. EF Core translates it to SQL. Filter happens in the database — only matching rows returned. Use with EF Core/LINQ to SQL.

    csharp 
    // BAD: Loads ALL products into memory, then filters in C#
var expensive = _context.Products
    .AsEnumerable()            // forces in-memory execution!
    .Where(p => p.Price > 100)
    .ToList();

// GOOD: SQL WHERE clause filters in database
var expensive = _context.Products
    .Where(p => p.Price > 100)   // translates to SQL WHERE
    .ToList();
    Q27. What collection types exist in C# and when do you use each? ▼
    • List<T> — ordered, index access, duplicates allowed. General-purpose default.
    • Dictionary<TKey,TValue> — O(1) key lookup. Use when finding by unique key frequently.
    • HashSet<T> — unique elements, O(1) Contains. Use for deduplication or set operations.
    • Queue<T> — FIFO. Use for job/message queues, BFS.
    • Stack<T> — LIFO. Use for undo/redo, DFS, parsing expressions.
    • SortedDictionary<TKey,TValue> — sorted by key, O(log n). Use when you need ordered enumeration.
    • ConcurrentDictionary<TKey,TValue> — thread-safe. Use in multi-threaded code.
    • IEnumerable<T> — read-only lazy sequence. Use as parameter/return type for flexibility.

    Caching & Performance

    Q28. What caching strategies did you implement? Explain each. ▼

    ProductCatalogAPI implements 5 caching strategies:

    • IMemoryCache — in-process RAM cache. ~1ms. Single server. Invalidated on data changes. Best for frequently-accessed lookup data.
    • Response Caching — HTTP Cache-Control headers via [ResponseCache(Duration=300)]. Browser/proxy caches response. Zero server load on cache hit.
    • IDistributedCache — shared cache interface (Redis, SQL Server). ~5ms. Works across multiple servers.
    • Redis — high-performance distributed cache. Survives app restarts. Used in production for multi-server apps.
    • HybridCache (.NET 9) — checks memory first → then Redis → then DB. Best of all worlds.
    Q29. How do you implement IMemoryCache? Show cache invalidation. ▼
    csharp 
    // Registration in Program.cs
builder.Services.AddMemoryCache();

// Usage in service
public async Task<IEnumerable<Product>> GetAllAsync()
{
    const string key = "products:all";
    if (_cache.TryGetValue(key, out IEnumerable<Product> cached))
        return cached; // ✓ HIT — returned in ~1ms

    var products = await _context.Products.AsNoTracking().ToListAsync();
    _cache.Set(key, products, TimeSpan.FromMinutes(5)); // TTL: 5 min
    return products; // ✗ MISS — fetched from DB (~150ms)
}

// Invalidation on data change
public async Task CreateProductAsync(CreateProductRequest req)
{
    // ... create product ...
    _cache.Remove("products:all"); // Stale cache cleared
}

    Benchmark from ProductCatalogAPI — GetAll endpoint with 100 sequential requests:

    ScenarioTotal TimeDB Queries

    Improvement60x faster99% reduction

    In production with hundreds of requests per minute for the same product list, this eliminates essentially all database load for read operations.

    Async/Await & General

    Q31. Why is async/await critical in Web API applications? ▼

    ASP.NET Core handles requests on a thread pool (typically ~250 threads). Without async, a thread blocked waiting for a DB query can't handle other requests — under load, threads exhaust and the app becomes unresponsive.

    With async/await, when you await _context.ToListAsync(), the current thread is returned to the pool while waiting for the database. When the DB responds, any available thread picks up the work. This means one server can handle thousands of concurrent requests instead of ~250.

    Rule: use async/await for all I/O (DB, HTTP, file). Never use .Result or .Wait() — they block a thread and can deadlock.

    Q32. What is CORS and how do you configure it? ▼

    CORS (Cross-Origin Resource Sharing) is a browser security mechanism that blocks web pages from calling APIs on a different domain. When your React app on localhost:3000 calls the API on localhost:5000, the browser requires the API to explicitly allow this origin.

    csharp 
    // Program.cs
builder.Services.AddCors(options => options.AddPolicy("AllowFrontend",
    policy => policy
        .WithOrigins("http://localhost:3000", "http://localhost:4200")
        .AllowAnyMethod()
        .AllowAnyHeader()));

// In pipeline (before MapControllers)
app.UseCors("AllowFrontend");

    In production: specify exact origins only. Never use .AllowAnyOrigin() in production.

    Q33. What is GraphQL vs REST — when do you use each? ▼

    REST: Multiple endpoints, each returns a fixed structure. Simple, well-known, HTTP cache-friendly. Best for: most APIs, microservices, public APIs.

    GraphQL: Single endpoint, client specifies exactly what fields they need. Eliminates over-fetching (unused fields) and under-fetching (needing multiple requests). Best for: mobile apps (minimize bandwidth), dashboards fetching varied data, APIs consumed by many different clients needing different data shapes.

    In .NET, use HotChocolate for GraphQL: define Query/Mutation types, register with AddGraphQLServer(), expose at /graphql with Banana Cake Pop playground.

    Q34. How do you handle global error handling in a Web API? ▼

    Best practice: custom global exception middleware as the first middleware in the pipeline. It wraps all subsequent middleware in a try-catch, catches any unhandled exception, logs it, and returns a standardized ProblemDetails response.

    csharp 
    public class GlobalErrorHandlingMiddleware(RequestDelegate next, ILogger<GlobalErrorHandlingMiddleware> logger)
{
    public async Task InvokeAsync(HttpContext context)
    {
        try {
            await next(context);
        } catch (Exception ex) {
            logger.LogError(ex, "Unhandled exception for {Path}", context.Request.Path);
            context.Response.StatusCode = 500;
            context.Response.ContentType = "application/json";
            await context.Response.WriteAsJsonAsync(new {
                Status = 500,
                Message = "An internal error occurred."
            });
        }
    }
}
    Q35. What is the Options pattern and how do you use IConfiguration? ▼

    The Options pattern binds configuration sections to strongly-typed C# classes, giving compile-time safety and IntelliSense instead of magic strings.

    csharp 
    // appsettings.json
{
  "CacheSettings": {
    "DefaultTtlMinutes": 5,
    "ProductsKey": "products:all"
  }
}

// Typed settings class
public class CacheSettings {
    public int DefaultTtlMinutes { get; set; }
    public string ProductsKey { get; set; } = "";
}

// Program.cs registration
builder.Services.Configure<CacheSettings>(
    builder.Configuration.GetSection("CacheSettings"));

// Inject and use
public ProductService(IOptions<CacheSettings> options) {
    var ttl = options.Value.DefaultTtlMinutes;
}

    C# & OOP

    Q36. What are the 4 pillars of OOP? ▼
    • Encapsulation: bundle data and methods together, hide internal details, expose only necessary interface.
    • Inheritance: derive classes from base classes to reuse code and establish hierarchies.
    • Polymorphism: same method name, different implementations based on object type (overriding) or parameters (overloading).
    • Abstraction: simplify complexity by defining essential features and hiding implementation details. Use abstract classes/interfaces.
    Q37. What is encapsulation? ▼

    Encapsulation hides internal details and prevents invalid states. Expose only what's necessary via public properties with validation.

    csharp
    // GOOD: property with validation
public class Product {
    private decimal _price;
    public decimal Price {
        get => _price;
        set => _price = value < 0 ? throw new ArgumentException("Cannot be negative") : value;
    }
}

    Benefits: prevents invalid states, allows logic changes later without breaking API.

    Q38. What is inheritance? ▼

    Inheritance allows a class (child) to inherit properties and methods from another class (parent), promoting code reuse and establishing hierarchies. C# supports single inheritance but multiple interface implementation.

    Q39. What is polymorphism? ▼

    Polymorphism = "many forms" = same method name, different behavior. Compile-time (overloading): same name, different parameters. Runtime (overriding): child class overrides parent's virtual method.

    Q40. What is abstraction? ▼

    Abstraction simplifies complexity by hiding implementation and exposing only essential features. Use abstract classes and interfaces. Example: IPaymentProcessor defines Process(amount). CreditCardProcessor and PayPalProcessor implement it. Calling code doesn't know HOW, only WHAT.

    Q41. What is a constructor? Types? ▼
    • Default: no parameters. Provided automatically by C# if not defined.
    • Parameterized: accepts arguments to initialize fields.
    • Copy: accepts another instance to clone it.
    • Static: runs once when class loads (static fields initialization).
    Q42. Destructor vs Finalize? ▼

    Both clean up resources. Destructor (~): syntax sugar for Finalize. Called by garbage collector (unpredictable timing). Best practice: use IDisposable.Dispose() instead for immediate cleanup. Destructors/Finalize are fallback for unmanaged resources if Dispose isn't called.

    Q43. What are generics? ▼

    Generics enable type-safe code reuse. Define methods/classes that work with any type. Example: Repository<T> works with any entity. Benefits: type-safe at compile-time, no boxing/unboxing, IntelliSense support.

    Q44. What are constraints in generics? ▼
    • where T : class — reference type only
    • where T : struct — value type only
    • where T : IComparable — must implement interface
    • where T : new() — must have parameterless constructor
    Q45. What is ref vs out? ▼
    • ref: pass by reference. Method can read AND modify original variable. Must be initialized before call.
    • out: pass by reference for OUTPUT. Method must assign before returning. Doesn't need to be initialized before call.
    Q46. What is is vs as operators? ▼
    • is: returns bool. Check if object is of type. Safe, no exceptions.
    • as: returns object (or null). Cast safely. Returns null if cast fails.
    Q47. What are extension methods? ▼

    Add new methods to existing classes without inheritance. First parameter uses this keyword. Heavily used in LINQ. Must be in static class. Limited to adding methods, not state/properties.

    Q48. What is POCO? ▼

    POCO = Plain Old CLR Object. Simple class with no dependencies, no special base class. Just properties. Entity models are POCOs. DTOs are POCOs. Benefits: testable, serializable, reusable across layers.

    Q49. What are attributes? Examples? ▼

    Attributes provide metadata to code. Framework reads them at runtime. [Required] for validation, [Authorize] for security, [Table] for EF Core configuration.

    Q50. What is a delegate? ▼

    Delegate is a type-safe function pointer. Represents a method signature. Enables passing methods as parameters (callbacks, events). Events use delegates. Func<T, TResult> and Action<T> are built-in delegates.

    ASP.NET MVC

    Q51. What is ASP.NET MVC? How does it differ from ASP.NET Core? ▼

    ASP.NET MVC: older framework (.NET Framework 4.7+). Monolithic, server-renders HTML, runs only on Windows. ASP.NET Core: modern rewrite. Cross-platform (Windows/Linux/macOS), lightweight, can render HTML (MVC) OR return JSON (Web API) from same code. The future.

    Q52. Explain MVC execution flow: Request → Model → Controller → View → Response ▼

    1) Browser sends request to /products/details/5 2) Routing maps URL to ProductsController.Details(id=5) 3) Controller queries ProductService (hits Model/DB) 4) Controller returns View("Details", product) with product data 5) View engine (Razor) renders HTML using product data 6) HTML returned to browser. Difference from Web API: Web API stops at step 3 and returns JSON.

    Q53. What is routing? Attribute vs Convention? ▼

    Convention-based: {controller}/{action}/{id}. Defined in RouteConfig. Implicit, automatic. Attribute-based: [Route("api/[controller]")] on controller/action. Explicit, flexible. Modern apps prefer attribute routing.

    Q54. What are MVC Filters? Types? ▼
    1. Authorization: check permissions before controller runs. [Authorize], [AllowAnonymous].
    2. Resource: run after authorization, can short-circuit.
    3. Action: OnActionExecuting (before action) and OnActionExecuted (after). Can modify model.
    4. Result: OnResultExecuting (before view renders) and OnResultExecuted (after). Modify view data.
    5. Exception: catch unhandled exceptions. Convert to error pages.

    Order: Authorization → Resource → Action → Result → View

    Q55. What is ViewBag, ViewData, and TempData? ▼
    • ViewBag: dynamic object. Pass data from controller to view. Lost after response.
    • ViewData: dictionary. Same purpose as ViewBag, slightly stricter.
    • TempData: persists one request-response cycle. Survives redirect. Use for validation messages after POST→redirect.

    Best practice: Prefer strongly-typed ViewModels over ViewBag/ViewData.

    Q56. What are layout pages and sections? ▼

    Layout pages (_Layout.cshtml) define master template shared by multiple views. Sections (@section) allow child views to define content for named areas in the layout. Example: @section Scripts in child view injects JavaScript into parent layout's @RenderSection("Scripts").

    Q57. What are partial views? When do you use them? ▼

    Partial views are reusable view fragments. Similar to components in React. Use @Html.PartialAsync() to include. Benefits: DRY principle, reuse across multiple pages, easier testing, cleaner code organization.

    Q58. How does model validation work in MVC? ▼

    Data annotations on model properties ([Required], [StringLength], [Range]) define validation rules. During model binding, ASP.NET Core validates the model. Access validation results via ModelState.IsValid. Display validation messages in views with @Html.ValidationSummary() and @Html.ValidationMessageFor().

    Q59. What is Razor syntax (@)? ▼

    Razor is view engine syntax mixing HTML and C#. @ prefix indicates C# code. Examples: @Model.Name, @for (var i = 0; i < 10; i++) { }, @if (condition) { }. Use @@ to escape @ literal in output.

    Q60. What is bundling and minification? ▼

    Bundling: combine multiple CSS/JS files into one. Reduces HTTP requests. Minification: remove unnecessary characters (spaces, comments) from files. Reduces file size. Modern apps use webpack/esbuild instead.

    Q61. What is TempData and how does it differ from ViewBag? ▼

    TempData persists across a redirect (unlike ViewBag which is lost after response). Useful for POST-Redirect-GET pattern: POST processes form → sets TempData → redirects to GET → GET displays success message from TempData. Implemented using session or cookie.

    Q62. Authentication vs Authorization in MVC? ▼

    Authentication: WHO are you? Verify identity. Authorization: WHAT can you do? Check permissions. Use [Authorize] for whole action, [AllowAnonymous] for specific. User.IsInRole("Admin") checks role-based authorization.

    Q63. What are action results? ▼

    Action results represent controller action response. View() renders view, RedirectToAction() redirects, Json() returns JSON, Content() returns custom content, StatusCode() returns HTTP status.

    Q64. What is model binding? ▼

    Model binding automatically maps HTTP request data to action method parameters. From route, query string, form data, headers. Use [Bind] attribute to include/exclude properties. Use [BindRequired] to require a property.

    Q65. What is HTML encoding? Why important? ▼

    HTML encoding converts special characters (< > & ") to HTML entities (< > & "). Prevents XSS attacks. @Model.UserInput automatically HTML-encodes. @Html.Raw() disables encoding (use carefully with trusted input only).

    SQL Server

    Q66. What is SQL? DDL vs DML? ▼

    SQL: language for querying and manipulating databases. DDL (Data Definition): CREATE, ALTER, DROP — structure. DML (Data Manipulation): SELECT, INSERT, UPDATE, DELETE — data.

    Q67. What are types of JOINs? ▼
    • INNER JOIN — only rows matching in BOTH tables.
    • LEFT JOIN — all rows from left table + matching from right.
    • RIGHT JOIN — all rows from right table + matching from left.
    • FULL OUTER JOIN — all rows from both tables.
    Q68. How do you find 2nd highest salary? ▼

    Using window functions: SELECT DISTINCT Salary FROM Employees ORDER BY Salary DESC OFFSET 1 ROW FETCH NEXT 1 ROW ONLY; Or using subquery: SELECT MAX(Salary) FROM Employees WHERE Salary < (SELECT MAX(Salary) FROM Employees);

    Q69. What is GROUP BY vs HAVING? ▼

    GROUP BY: groups rows by column(s). HAVING: filters groups (like WHERE for GROUP BY). Example: SELECT Department, COUNT(*) FROM Employees GROUP BY Department HAVING COUNT(*) > 5;

    Q70. What are subqueries? ▼

    Query within a query. SELECT * FROM Products WHERE Price > (SELECT AVG(Price) FROM Products); Can return single value, row, or table. Slower than JOINs. Use sparingly.

    Q71. What are window functions? ▼

    Functions that operate on a set of rows within a "window." Examples: ROW_NUMBER(), RANK(), DENSE_RANK(), LAG(), LEAD(). Useful for analytics, rankings, cumulative totals without GROUP BY.

    Q72. ROW_NUMBER vs RANK vs DENSE_RANK? ▼
    • ROW_NUMBER(): unique sequential number (1,2,3,4).
    • RANK(): with gaps on ties (1,2,2,4).
    • DENSE_RANK(): no gaps on ties (1,2,2,3).
    Q73. What are transactions? ACID? ▼

    Transactions ensure data integrity. Atomicity: all or nothing. Consistency: valid state before/after. Isolation: concurrent transactions don't interfere. Durability: committed data persists. BEGIN TRANSACTION, COMMIT, ROLLBACK.

    Q74. What are indexes? Clustered vs Non-Clustered? ▼
    • Clustered: sorts table physically. One per table. Primary key uses clustered index.
    • Non-Clustered: separate index structure pointing to data. Multiple per table.

    CREATE INDEX idx_category ON Products(CategoryId);

    Q75. What is normalization? 1NF, 2NF, 3NF? ▼
    • 1NF: atomic values only. No repeating groups.
    • 2NF: non-key columns depend on entire primary key.
    • 3NF: non-key columns depend ONLY on primary key, not other non-key columns.

    Most real-world databases aim for 3NF.

    Q76. DELETE vs TRUNCATE? ▼
    • DELETE: removes rows one by one. Slow. Can have WHERE clause. Triggers fire. Can rollback.
    • TRUNCATE: removes all rows at once. Fast. No WHERE. Resets identity. Minimal logging.
    Q77. What is a stored procedure? ▼

    Pre-compiled SQL code saved on server. Called by name with parameters. Benefits: pre-compiled (fast), encapsulation, reuse. Cons: database-specific, hard to version control, harder to test. Use for complex reporting and bulk operations.

    Q78. Stored Procedure vs Function? ▼
    • Procedure: doesn't return value (optional OUTPUT parameters). Performs actions. EXECUTE proc.
    • Function: MUST return value. SELECT dbo.Func(@param).
    Q79. What is DISTINCT? ▼

    Removes duplicates from result set. SELECT DISTINCT CategoryId FROM Products; Returns unique category IDs only. Use sparingly on large datasets — can be slow.

    Q80. What is a database view? ▼

    Virtual table based on SQL query result. Doesn't store data. Useful for: simplified complex queries, security (expose only needed columns), abstraction. CREATE VIEW v_ActiveProducts AS SELECT * FROM Products WHERE IsActive = 1;

    React & Redux

    Q81. What is React? ▼

    JavaScript library for building user interfaces with components. Declarative (describe what UI SHOULD look like), component-based, one-way data flow. Virtual DOM efficiently updates only what changed. Scales to complex UIs.

    Q82. What is JSX? ▼

    JavaScript mixed with HTML-like syntax. Compiles to React.createElement() calls. Allows writing UI declaratively. Variables in {} are evaluated. Example: <h2>{product.name}</h2>

    Q83. What is Virtual DOM? ▼

    In-memory representation of real DOM. When state changes, React creates new virtual tree, diffs it against old tree, updates only changed nodes in real DOM. Much faster than directly updating DOM.

    Q84. Components: Functional vs Class? ▼
    • Functional: JavaScript functions returning JSX. Modern approach. Use hooks for state/lifecycle.
    • Class: ES6 classes extending React.Component. Older approach. Use this.state and lifecycle methods.

    Hooks replaced class components. Functional + hooks is now the standard.

    Q85. Props vs State? ▼
    • Props: read-only data passed from parent to child. Cannot modify. Like function parameters.
    • State: mutable data owned by component. Changing state triggers re-render. useState hook manages it.

    Props are immutable (parent has control). State is local (component manages it).

    Q86. What is useState hook? ▼

    Returns [state, setState]. const [count, setCount] = useState(0); Changes state, triggers re-render. Initial value can be function for expensive computations.

    Q87. What is useEffect hook? ▼

    Run side effects (fetch data, subscribe, timers). useEffect(() => { ... }, []); Empty dependency array runs once on mount. Runs after render. Return function for cleanup (unsubscribe).

    Q88. What is useContext hook? ▼

    Access context values without prop drilling. const value = useContext(MyContext); Avoids passing props through many intermediate components.

    Q89. What is useRef hook? ▼

    Returns mutable object persisting across renders. ref.current accesses value. Used for: DOM access, storing values without triggering re-render, focusing input.

    Q90. What are React lifecycle methods? ▼

    Mounting: constructor, componentDidMount. Updating: componentDidUpdate. Unmounting: componentWillUnmount. Modern: use useEffect instead (replaces all lifecycle methods).

    Q91. What is componentDidMount? ▼

    Runs after component mounts. Equivalent to useEffect with empty dependency array. Use for: API calls, DOM setup, timers. Runs only once.

    Q92. What is useMemo hook? ▼

    Memoizes expensive computations. const memoized = useMemo(() => expensiveCalculation(), [deps]); Only recalculates when dependencies change. Performance optimization.

    Q93. What is useCallback hook? ▼

    Memoizes function. const memoized = useCallback(() => { ... }, [deps]); Returns same function reference if dependencies unchanged. Pass stable callbacks to child components.

    Q94. What is React.memo? ▼

    Prevents re-render if props unchanged. const MyComponent = React.memo(Component); Performance optimization. Use sparingly — profiling before optimizing.

    Q95. Prop drilling and how to avoid? ▼

    Prop drilling = passing props through many intermediate components. Solutions: Context API (useContext), Redux state management, component composition.

    Q96. What is Redux? ▼

    State management library. Centralized store for application state. Predictable, debuggable. Actions dispatch, reducers update state, selectors access state. Overkill for small apps.

    Q97. Redux flow? ▼

    1) Component dispatches action 2) Reducer processes action and updates state 3) Store notifies subscribers 4) Component re-renders with new state. Actions are plain objects {type, payload}. Reducers are pure functions.

    Q98. Actions, reducers, dispatch? ▼

    Action: plain object {type, payload}. Reducer: pure function (state, action) => newState. Dispatch: send action to store. dispatch(addProduct(product));

    Q99. What is React Router? ▼

    Client-side routing library. BrowserRouter wraps app. Routes define paths, Route component renders based on match. Link/NavLink for navigation. Enable SPA routing without page refresh.

    Q100. React performance optimization? ▼

    Code splitting (lazy loading), memoization (useMemo, useCallback, React.memo), virtualization for long lists, profiling with React DevTools, reducing re-renders.

    Angular

    Q101. What is Angular? ▼

    Full framework for building web apps. TypeScript-based, opinionated, includes routing, HTTP, forms, testing. Two-way data binding. Includes all you need — no picking libraries.

    Q102. Angular vs React? ▼

    React: library, minimal, flexible, one-way data flow. Angular: full framework, opinionated, includes everything, two-way binding. Choose React for flexibility, Angular for structure.

    Q103. How to inject service in Angular? ▼

    @Injectable() decorator on service. Constructor parameter in component. Angular DI container provides it. constructor(private productService: ProductService) { }

    Q104. What is dependency injection? ▼

    Container creates and injects services into components automatically. Benefits: loose coupling, testability (inject mocks), reusability. Services are singletons by default.

    Q105. What are Observables? ▼

    Event streams emitting values over time. Lazy (don't execute until subscribed). HTTP methods return Observables. Subscribe to get values. Use operators (map, filter) to transform.

    Q106. RxJS operators? ▼
    • map: transform each value
    • filter: keep values matching condition
    • switchMap: flatten nested Observables
    • debounceTime: wait before emitting
    • takeUntil: unsubscribe when notifier emits
    Q107. What are custom directives? ▼

    Reusable DOM manipulators. @Directive decorator. Access DOM via ElementRef, Renderer2. Example: appHighlight directive. Use in templates: <p appHighlight>

    Q108. What is @Input/@Output? ▼
    • @Input: receives data from parent. Pass via property binding <app-child [product]="product">
    • @Output: emits events to parent. EventEmitter. (onDelete)="delete($event)"
    Q109. What is two-way binding? ▼

    [(ngModel)] combines property binding and event binding. Change input = update component state = update display. Used in forms. Syntax sugar for [ngModel] and (ngModelChange).

    Q110. Component lifecycle in Angular? ▼

    OnInit (initialize), OnChanges (input change), OnDestroy (cleanup). ngOnInit for initialization. ngOnDestroy for unsubscribe, cleanup. Implement OnInit interface.

    JavaScript & Event Loop

    Q111. What is JavaScript? ▼

    Dynamic, interpreted language. Originally for browsers, now everywhere (Node.js). Dynamically typed, prototypical inheritance, functional programming support.

    Q112. What is the event loop? ▼

    Mechanism running JavaScript in browser. Call stack executes synchronous code. Async callbacks go to web APIs, then to callback queue when ready. Event loop moves callback to call stack when stack empty. Single-threaded but non-blocking.

    Q113. What are Promises? ▼

    Object representing eventual completion/failure of async operation. States: pending, fulfilled, rejected. then() chains success, catch() handles error. Prevents callback hell.

    Q114. What is async/await? ▼

    Syntactic sugar over Promises. async function returns Promise. await pauses execution until Promise resolves. Looks synchronous, is actually async. Much cleaner than Promises.

    Q115. What is callback hell? ▼

    Nested callbacks becoming hard to read. Solved by: Promises (then chains), async/await. Write clean, readable async code.

    Q116. setTimeout and microtask queue? ▼

    setTimeout goes to macrotask queue. Promise.then() goes to microtask queue. Microtasks run before next macrotask. Microtask queue always empty before checking macrotask queue.

    Q117. Object comparison? ▼

    Objects compared by reference, not value. {} === {} is false. Same reference === true. For value comparison, compare properties or use JSON.stringify().

    Q118. What are closures? ▼

    Function with access to outer function's variables. Inner function "closes over" outer variables. Useful for data privacy, callbacks. var counter = (function() { let count = 0; return () => ++count; })()

    Q119. What is 'this' binding? ▼

    this refers to calling context. Regular function: window/undefined. Arrow function: parent's this. Method: object. Use .call(), .apply(), .bind() to control.

    Q120. What is hoisting? ▼

    Declarations moved to top of scope. var hoisted (undefined initially). let/const hoisted (temporal dead zone). Functions fully hoisted. Define before using to avoid bugs.

    Security & Authentication

    Q121. Authentication vs Authorization? ▼

    Authentication: WHO are you? Verify identity with credentials. Authorization: WHAT can you do? Check permissions. Flow: Login → authenticate → JWT token → every request includes token → verify token → check permissions → execute.

    Q122. What is JWT? ▼

    JSON Web Token. Stateless token with header (algorithm), payload (claims like user ID), signature (server-signed). Three parts separated by dots.

    Q123. How JWT works? ▼

    1) User logs in 2) Server creates JWT with user ID, role, exp time 3) Returns JWT to client 4) Client includes JWT in every request header 5) Server verifies signature and expiry 6) Grant/deny access.

    Q124. JWT vs Session? ▼
    • JWT: stateless, scalable, no server storage needed.
    • Session: stateful, server stores session data, cookie-based.

    JWT better for distributed systems, APIs, microservices.

    Q125. What is CORS? ▼

    Cross-Origin Resource Sharing. Browser blocks requests to different domains (security). Whitelisted origins can call API. Implemented via response headers: Access-Control-Allow-Origin.

    Q126. What is CSRF? ▼

    Cross-Site Request Forgery. Attacker tricks you into making unwanted requests on authenticated site. Prevention: CSRF tokens. Server generates unique token, includes in forms, verifies on submission. Attacker can't know token.

    Q127. SQL Injection prevention? ▼

    Use parameterized queries ALWAYS. Never concatenate SQL with user input. Input: '; DROP TABLE Products; -- → Safe with parameters, dangerous without. OWASP top web vulnerability.

    Q128. XSS prevention? ▼

    HTML encode user input. <script> becomes <script>. React/Angular automatically encode. Use @Html.Encode() in .NET. @Html.Raw() only with trusted input.

    Q129. HTTPS vs HTTP? ▼
    • HTTP: unencrypted, vulnerable to man-in-middle.
    • HTTPS: encrypted (TLS), certificate-based, secure.

    Always use HTTPS in production, especially for sensitive data.

    Q130. Password hashing? ▼

    Never store plain passwords. Hash with salt (bcrypt, PBKDF2, Argon2). Hash is one-way. On login, hash input and compare. Prevent rainbow table attacks.

    Azure & DevOps

    Q131. What is Azure? ▼

    Microsoft's cloud platform. Services: compute (VMs, App Services), databases (SQL, Cosmos), messaging (Service Bus), analytics (Power BI). IaaS, PaaS, SaaS options.

    Q132. What are Azure Functions? ▼

    Serverless functions. Pay per execution, not infrastructure. Triggered by events (HTTP, timer, queue). Fast scaling. Use for background jobs, webhooks, microservices.

    Q133. What are Azure App Services? ▼

    PaaS for hosting web apps, APIs, mobile backends. Built-in scaling, monitoring, deployment. Linux or Windows. No server management. Ideal for Web API, ASP.NET Core apps.

    Q134. What are deployment slots? ▼

    Separate deployment environments (staging, production). Deploy to staging first, test, then swap to production. Zero downtime deployments. Quick rollback if issues.

    Q135. What is CI/CD pipeline? ▼

    CI (Continuous Integration): code pushed → automatic build/test. CD (Continuous Deployment): passing tests → auto deploy to production. Automate: build, test, deploy.

    Q136. What are GitHub Actions? ▼

    CI/CD automation in GitHub. Workflows defined in YAML. Trigger on push/PR. Run tests, build, deploy. No external tools needed. Integrated with GitHub.

    Q137. What are Azure Logic Apps? ▼

    Serverless workflow orchestration. Connect services without code. Drag-drop workflow builder. Integrations with 200+ services. Scheduled workflows, event-driven.

    Q138. What is Terraform? ▼

    Infrastructure as Code tool. Define infrastructure in HCL. Version control infrastructure. Reproducible, automated deployments. Works with Azure, AWS, GCP.

    Q139. What is Infrastructure as Code? ▼

    Define infrastructure using code (Terraform, ARM templates). Version controlled. Reproducible, automated. Fast provisioning, disaster recovery, scalability.

    Q140. What is Azure Kubernetes Service (AKS)? ▼

    Managed Kubernetes cluster. Container orchestration. Auto-scaling, self-healing, rolling updates. DevOps-friendly. Use for microservices, containerized applications at scale.
