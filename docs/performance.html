    EF Core vs ADO.NET Query Performance

    Both EF Core and ADO.NET are excellent data access tools — the right choice depends on the operation. ADO.NET wins on raw speed for pure data retrieval, while EF Core wins on developer productivity, change tracking, and migrations.

    Operation EF Core ADO.NET Winner
    Simple SELECT by ID ~5ms ~2ms ADO.NET (2x faster)
    Complex JOIN query ~25ms ~8ms ADO.NET (3x faster)
    INSERT with validation ~8ms ~5ms ADO.NET (1.5x faster)
    CRUD with change tracking ~10ms N/A EF Core (built-in feature)
    Startup / setup time High (model building) None ADO.NET
    Developer time (CRUD) Low (auto-generated) High (manual SQL) EF Core
    💡

    Practical Guidance

    For most CRUD operations, EF Core is fast enough. Use ADO.NET for bulk operations and complex reporting queries where you need full control over the SQL and maximum throughput.

    Caching Impact (Real Numbers from ProductCatalogAPI)

    Adding IMemoryCache to the GetAll endpoint in ProductCatalogAPI produced one of the most dramatic single-change performance improvements possible — without touching the database schema or query logic at all.

    Scenario Time DB Queries Notes
    100 requests, no cache ~15 seconds 100 queries Every request hits the database
    100 requests, memory cache ~150ms 1 query 99x improvement!
    Cache hit ~1ms 0 queries Served entirely from RAM
    Cache miss ~150ms 1 query DB query + stores result in cache
    C# — IMemoryCache Setup 
    // Program.cs — register the cache service
builder.Services.AddMemoryCache();

// ProductService.cs — inject and use
public class ProductService
{
    private readonly IMemoryCache _cache;
    private const string ALL_PRODUCTS_KEY = "all_products";

    public ProductService(AppDbContext context, IMemoryCache cache)
    {
        _cache = cache;
        // ...
    }

    public async Task<List<Product>> GetAllAsync()
    {
        // Try cache first
        if (_cache.TryGetValue(ALL_PRODUCTS_KEY, out List<Product>? cached))
        {
            Console.WriteLine("✓ CACHE HIT — GetAll");
            return cached!;
        }

        // Cache miss: query the database
        Console.WriteLine("✗ CACHE MISS — GetAll");
        var products = await _context.Products
            .Where(p => p.IsActive)
            .ToListAsync();

        // Store in cache for 5 minutes
        _cache.Set(ALL_PRODUCTS_KEY, products, TimeSpan.FromMinutes(5));
        return products;
    }
}
    ✅

    Measured Result

    Implementing IMemoryCache in GetAll reduced 100 sequential requests from 15 seconds to 150ms — a 100x improvement — with a single service change and zero schema modifications.

    All 5 Caching Strategies

    There is no single caching solution — the right approach depends on your architecture, traffic pattern, and whether you run on one server or many.

    A) IMemoryCache — In-Process Memory

    • When to useSingle-server apps, session data, frequently read reference data
    • Response speed~1ms (RAM read)
    • Survives restartNo — cache is cleared on app restart
    • Multi-serverNo — each server has its own independent cache
    C# — IMemoryCache GetOrCreate 
    // Inject
private readonly IMemoryCache _cache;

// GetOrCreate pattern — atomic check + set
var result = await _cache.GetOrCreateAsync("products_all", async entry =>
{
    entry.AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5);
    entry.SlidingExpiration = TimeSpan.FromMinutes(2);
    return await _context.Products.Where(p => p.IsActive).ToListAsync();
});

    B) Response Caching — HTTP Cache Headers

    • When to usePublic GET endpoints that rarely change (e.g., product listings, categories)
    • Response speed0ms — browser serves from its local cache without a network request
    • Survives restartYes — stored in the browser or CDN
    • Multi-serverYes — lives in the client, not the server
    C# — ResponseCache Attribute 
    // Register in Program.cs
builder.Services.AddResponseCaching();
app.UseResponseCaching(); // must come before UseCors

// Apply on controller actions
[HttpGet]
[ResponseCache(Duration = 300)] // Cache for 5 minutes
public async Task<ActionResult<IEnumerable<ProductDto>>> GetProducts() { ... }

// Disable caching for mutation endpoints
[HttpPost]
[ResponseCache(NoStore = true, Location = ResponseCacheLocation.None)]
public async Task<ActionResult<ProductDto>> CreateProduct(...) { ... }

    C) IDistributedCache — External Cache Store

    • When to useMulti-server apps, data that must survive app restarts, shared session state
    • Response speed~5ms (Redis network roundtrip on same LAN)
    • Survives restartYes — stored in external Redis or SQL Server
    • Multi-serverYes — all servers share the same cache store
    C# — IDistributedCache Pattern 
    // Inject
private readonly IDistributedCache _cache;

// Must serialize/deserialize manually
var json = await _cache.GetStringAsync("products_all");
if (json != null)
    return JsonSerializer.Deserialize<List<Product>>(json);

var products = await _context.Products.ToListAsync();
await _cache.SetStringAsync("products_all",
    JsonSerializer.Serialize(products),
    new DistributedCacheEntryOptions
    {
        AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5)
    });

    D) Redis — Production Distributed Cache

    • When to useProduction apps, sharing cache across multiple server instances, session data that must not be lost
    • Response speed~5ms over local network; sub-millisecond on same host
    • Multi-serverYes — the definitive solution for horizontal scaling
    JSON — appsettings.json (Redis connection) 
    {
  "ConnectionStrings": {
    "Redis": "localhost:6379,abortConnect=false"
  }
}
    C# — Register Redis in Program.cs 
    // Install: dotnet add package Microsoft.Extensions.Caching.StackExchangeRedis
builder.Services.AddStackExchangeRedisCache(options =>
{
    options.Configuration = builder.Configuration.GetConnectionString("Redis");
    options.InstanceName = "ProductCatalog_";
});

    E) HybridCache (.NET 9) — Best of Both Worlds

    • When to use.NET 9+ apps that need both speed (in-memory) and resilience (distributed)
    • How it worksFirst checks in-process memory → then distributed cache (Redis) → then database
    • Multi-serverYes — distributed layer keeps all instances consistent
    C# — HybridCache (.NET 9) 
    // Program.cs
builder.Services.AddHybridCache(options =>
{
    options.DefaultEntryOptions = new HybridCacheEntryOptions
    {
        Expiration = TimeSpan.FromMinutes(5),
        LocalCacheExpiration = TimeSpan.FromMinutes(1) // L1 in-process TTL
    };
});

// Service usage — no manual serialize/deserialize needed
public async Task<List<Product>> GetAllAsync()
{
    return await _hybridCache.GetOrCreateAsync(
        "products_all",
        async token => await _context.Products
            .Where(p => p.IsActive)
            .ToListAsync(token)
    );
}

    The N+1 Query Problem

    The N+1 problem is what happens when code loads a list of entities (1 query) and then makes an additional database call for each entity (N queries) — usually inside a loop. Loading 100 products and then hitting the database once per product for its category data results in 101 database queries instead of 1 or 2.

    ⚠️

    N+1 is one of the most common performance bugs in EF Core apps

    It is silent — the app looks correct, but scales terribly. A page that loads fine with 10 rows may take 30 seconds with 1,000 rows because EF Core is firing 1,000 separate queries.

    C# — BAD: N+1 Pattern 
    // BAD — 1 query for products, then 1 query PER product for its category
var products = await _context.Products.ToListAsync(); // Query 1

foreach (var product in products)
{
    // EF Core fires a NEW query for each product — N additional queries!
    var category = await _context.Categories
        .FirstOrDefaultAsync(c => c.Id == product.CategoryId);

    Console.WriteLine($"{product.Name} — {category?.Name}");
}
// 100 products = 101 queries. 1000 products = 1001 queries.
    C# — GOOD: Use Include() to Eager Load 
    // GOOD — single JOIN query loads products AND categories together
var products = await _context.Products
    .Include(p => p.Category)          // JOIN in one query
    .Where(p => p.IsActive)
    .ToListAsync();

foreach (var product in products)
{
    // Category is already in memory — no extra query fired
    Console.WriteLine($"{product.Name} — {product.Category?.Name}");
}
// 100 products = 1 query. 1000 products = still 1 query.

// ALTERNATIVE — load categories separately, then join in memory
var products    = await _context.Products.ToListAsync();                    // Query 1
var categories  = await _context.Categories.ToDictionaryAsync(c => c.Id);  // Query 2

foreach (var product in products)
{
    var categoryName = categories.TryGetValue(product.CategoryId, out var cat)
        ? cat.Name : "Unknown";
    Console.WriteLine($"{product.Name} — {categoryName}");
}
// 2 queries total, regardless of product count.

    LINQ Performance Tips

    LINQ is translated into SQL by EF Core — but only if you stay in IQueryable mode. Calling .ToList() too early pulls all rows into memory and causes subsequent filtering to run in C# rather than in SQL. These patterns make the difference between a query that runs in the database and one that overloads the application server.

    Slow Pattern Fast Pattern Why
    .ToList().Where(...) .Where(...).ToList() Filter in the database (SQL WHERE), not in C# memory
    if (.Count() > 0) if (.Any()) Any() short-circuits at the first match; Count() scans the full set
    .First() (throws on empty) .FirstOrDefault() Choose based on expectation; FirstOrDefault avoids unhandled exceptions
    SELECT * (all columns) .Select(p => new { p.Id, p.Name }) Fetch only the columns you need — reduces network I/O and mapping overhead
    Repeated .Single() in a loop .ToDictionary() then O(1) lookup Dictionary lookup is O(1); scanning a list per iteration is O(n²) total

    Decision Matrix: Which Technology to Use

    Use this table as a quick-reference when designing a new feature or reviewing an existing one.

    Scenario Recommended Why
    Simple CRUD API EF Core Fast development, automatic migrations, clean LINQ queries
    Reporting / analytics queries ADO.NET Raw SQL gives full control; optimal performance for complex JOINs and aggregates
    High-traffic GET endpoint IMemoryCache 100x speed boost; serves repeated reads entirely from RAM
    Multiple server instances Redis All servers share the same cache; no stale data across instances
    Public, rarely-changing content Response Cache Browser and CDN cache eliminates server roundtrip entirely
    Operations requiring transactions Either (EF Core preferred) EF Core handles transactions automatically; ADO.NET requires manual SqlTransaction
    Bulk INSERT (1,000+ rows) ADO.NET / SqlBulkCopy EF Core AddRange is slow at scale; SqlBulkCopy inserts millions of rows per second

    Production Optimization Checklist

    Run through this list before deploying a data-heavy .NET API to production.

    • 1
      Use AsNoTracking() for read-only queries

      Disables EF Core's change-tracking overhead when you only need to read data, not update it. Can reduce memory and CPU usage significantly on large result sets.

    • 2
      Add IMemoryCache for frequently accessed data

      Reference data (categories, settings, lookup tables) is perfect for in-memory caching with a 5–30 minute TTL.

    • 3
      Use parameterized queries in ADO.NET to prevent SQL injection

      Always use SqlParameter — never string concatenation. Parameterized queries also allow the database engine to cache and reuse query execution plans.

    • 4
      Select only needed columns, not SELECT *

      Use .Select(p => new ProductDto { ... }) in EF Core or explicit column lists in ADO.NET SQL to reduce network bandwidth and deserialization cost.

    • 5
      Use Include() to avoid N+1 queries

      Always eager-load related data you know you will use. Profile your queries with EF Core logging to catch unexpected lazy loading.

    • 6
      Add database indexes on filtered columns

      Columns used in WHERE, ORDER BY, or JOIN clauses should have indexes. A missing index can turn a 2ms query into a 2-second full table scan.

    • 7
      Use Any() instead of Count() > 0

      Any() translates to SELECT TOP 1 or EXISTS in SQL; Count() scans the entire result set to return a number you then compare against zero.

    • 8
      Implement pagination for large datasets

      Never return unbounded result sets. Use .Skip(page * size).Take(size) in EF Core. Return total count separately so the client can render page navigation.

    • 9
      Use async/await throughout to free thread pool

      Async I/O frees the thread pool to handle other requests while waiting on the database. A synchronous thread blocked on a 150ms query cannot serve any other request during that time.

    • 10
      Monitor slow queries with EF Core logging

      Enable EnableSensitiveDataLogging() and LogTo(Console.WriteLine, LogLevel.Information) during development to see every generated SQL statement and its execution time.

    C# — Enable EF Core Query Logging 
    // Program.cs — log all EF Core SQL in development
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlServer(connectionString)
           .LogTo(Console.WriteLine, LogLevel.Information)
           .EnableSensitiveDataLogging() // shows parameter values
           .EnableDetailedErrors()
);
