    🖥️ ASP.NET MVC

    Server-side rendering with HTML forms — understand when MVC differs from Web API

    Interview Prep ASP.NET Architecture Intermediate
    💡

    What is MVC?

    MVC (Model-View-Controller) is a server-side architectural pattern that separates concerns into three layers. Controllers handle incoming HTTP requests and orchestrate business logic. Views render HTML markup (Razor templates in ASP.NET). Models hold data and validation rules. Unlike Web APIs that return JSON, MVC applications render complete HTML pages on the server and send them to the browser. This guide covers ASP.NET Core MVC using the Product Catalog admin panel scenario — the same business logic you've already learned, but now rendered as server-side forms instead of JSON.

    MVC Component Reference

    MVC Component Responsibility Technology / File Pattern
    Model Data structure, validation rules, business logic C# class (e.g., Product.cs), ViewModel for views
    View Render HTML to the browser, display model data Razor template (e.g., Products/Index.cshtml), @Model, @Html
    Controller Handle HTTP requests, call services, pass data to views C# class inheriting Controller, action methods returning IActionResult
    Routing Map URLs to controller actions Route attributes [Route(...)], convention-based routing in Program.cs
    Filters Cross-cutting concerns (auth, validation, logging, error handling) [Authorize], [ValidateAntiForgeryToken], custom IActionFilter
    View Engine Parse Razor syntax and generate HTML ASP.NET Core Razor (built-in), supports @if, @foreach, @Model, @Html
    ViewModel Data transfer object shaped for the view (may combine multiple models) Separate C# class (e.g., ProductEditViewModel) with @model directive in view

    Interview Questions on ASP.NET MVC

    Q47. What is the difference between ASP.NET MVC and ASP.NET Core? Why is Core better? ▼

    ASP.NET MVC (the .NET Framework version) is a mature, full-featured framework released in 2009. It runs only on Windows and depends on the .NET Framework runtime. It is powerful but heavyweight, with many built-in abstractions that can hide complexity.

    ASP.NET Core (released in 2016, now just ".NET") is a complete rewrite designed for the cloud era. Key improvements:

    • Cross-platform — runs on Linux, macOS, and Windows; deploy to any container environment
    • Lightweight — smaller footprint, faster startup time, uses only what you need
    • Open-source — full source available on GitHub, community contributions, transparency
    • Unified platform — Web API and MVC share the same underlying stack (both inherit from ControllerBase)
    • Built-in DI — dependency injection in the core, not an add-on
    • Modern middleware pipeline — simpler, more predictable request handling
    • Better performance — benchmarks show 3-10x faster request handling than .NET Framework MVC
    • First-class async/await — all APIs are async by default, no sync-over-async pitfalls

    In interviews: you should always assume ASP.NET Core unless explicitly told otherwise. The .NET Framework version is legacy. Say "ASP.NET Core MVC is the modern choice for new projects because it's cross-platform, lightweight, and has unified Web API + MVC architecture."

    Q48. Describe the MVC execution pipeline from HTTP request to response. ▼

    The flow is: Request → Routing → Controller instantiation → Action execution → Service call → Model population → View rendering → Response HTML.

    Step 1: Browser sends HTTP Request GET /Products/Edit/42 Host: localhost:5000 Step 2: Routing Module matches URL to action Route: /Products/{action}/{id} Matched Controller: ProductsController Matched Action: Edit(42) Step 3: Create controller instance (via DI container) new ProductsController(IProductService service, ...) Step 4: Filters execute (Authorization filter) [Authorize] — check if user is logged in Step 5: Bind route/query data to action parameters id = 42 Step 6: Model validation (if model binding involved) Check [Required], [Range], custom [ValidationAttribute] Step 7: Call controller action var result = await controller.Edit(42); Step 8: Action executes, calls business logic var product = await _service.GetProductAsync(42); Step 9: Action returns result (e.g., View(...) or IActionResult) return View(new ProductEditViewModel { ... }); Step 10: View Engine (Razor) renders HTML Parse @Model, @foreach, @Html.TextBox, etc. Generate HTML string Step 11: Action Result executes Send HTML as response body with Status 200 OK Step 12: Send response to browser HTTP/1.1 200 OK Content-Type: text/html <!DOCTYPE html> <body> ... </body> </html>

    In code: A simple Product edit flow looks like this:

    csharp 
    // ProductsController.cs
[Route("[controller]")]
public class ProductsController : Controller
{
    private readonly IProductService _service;
    public ProductsController(IProductService service) => _service = service;

    // GET /Products/Edit/42
    [HttpGet("Edit/{id}")]
    public async Task<IActionResult> Edit(int id)   // Step 5: id bound from URL
    {
        var product = await _service.GetProductAsync(id);  // Step 8: call service
        if (product is null) return NotFound();

        var viewModel = new ProductEditViewModel     // Step 9: populate ViewModel
        {
            Id = product.Id,
            Name = product.Name,
            Price = product.Price,
            Category = product.Category
        };

        return View(viewModel);  // Step 9: return View with model
                                 // Step 10: Razor renders <form>, <input>, etc.
    }
}

    The key insight: MVC renders HTML on the server. The browser receives a complete HTML page, not JSON. Every form submission cycles through this pipeline again.

    Q49. What are MVC Filters? Name the 5 types and explain their execution order. ▼

    Filters are middleware-like components that run at specific points in the request/response lifecycle. They allow cross-cutting concerns (authentication, logging, validation, error handling) without polluting controller actions. There are five types, and they execute in a specific order.

    EXECUTION ORDER: → Request arrives → 1️⃣ AUTHORIZATION FILTER (first, guards entry) Implementations: [Authorize], [AllowAnonymous] Runs: before action, can short-circuit → 2️⃣ RESOURCE FILTER (second, guards resource access) Implementations: Custom IResourceFilter Runs: after auth, before model binding Use case: caching, short-circuit expensive operations → 3️⃣ ACTION FILTER (third, wraps action) Implementations: [ActionName], custom IActionFilter Runs: before AND after action execution Use case: logging inputs/outputs, timing → 4️⃣ RESULT FILTER (fourth, wraps result execution) Implementations: Custom IResultFilter Runs: before AND after result.ExecuteAsync() Use case: modify response, logging → 5️⃣ EXCEPTION FILTER (catches unhandled exceptions) Implementations: Custom IExceptionFilter Runs: if any filter/action throws Use case: global error handling, logging → Response sent to browser

    Code example — logging with Action Filter:

    csharp 
    // Custom Action Filter for logging
public class TimingActionFilter : IActionFilter
{
    private readonly ILogger<TimingActionFilter> _logger;
    public TimingActionFilter(ILogger<TimingActionFilter> logger) => _logger = logger;

    // Runs BEFORE the action executes
    public void OnActionExecuting(ActionExecutingContext context)
    {
        context.HttpContext.Items["StartTime"] = DateTime.UtcNow;
        _logger.LogInformation("Action {Action} starting", context.ActionDescriptor.DisplayName);
    }

    // Runs AFTER the action executes
    public void OnActionExecuted(ActionExecutedContext context)
    {
        var startTime = (DateTime)context.HttpContext.Items["StartTime"];
        var elapsed = (DateTime.UtcNow - startTime).TotalMilliseconds;
        _logger.LogInformation("Action {Action} completed in {Elapsed}ms",
            context.ActionDescriptor.DisplayName, elapsed);
    }
}

// Register in Program.cs
builder.Services.AddControllers(options =>
{
    options.Filters.Add<TimingActionFilter>();  // global filter
});

// Or use attribute on specific action
[ServiceFilter(typeof(TimingActionFilter))]
public async Task<IActionResult> Edit(int id) { /* ... */ }

    Authorization Filter example — product ownership check:

    csharp 
    // Custom Authorization Filter — ensure user owns product before edit
public class ProductOwnerAuthorizationFilter : IAsyncAuthorizationFilter
{
    private readonly IProductService _service;

    public ProductOwnerAuthorizationFilter(IProductService service) => _service = service;

    public async Task OnAuthorizationAsync(AuthorizationFilterContext context)
    {
        var userId = context.HttpContext.User.FindFirst(ClaimTypes.NameIdentifier)?.Value;
        var productIdStr = context.RouteData.Values["id"]?.ToString();

        if (!int.TryParse(productIdStr, out var productId))
        {
            context.Result = new BadRequestResult();
            return;
        }

        var product = await _service.GetProductAsync(productId);
        if (product?.OwnerId != userId)
        {
            context.Result = new ForbidResult();  // Stop here, return 403
        }
    }
}

// Use on action
[ServiceFilter(typeof(ProductOwnerAuthorizationFilter))]
public async Task<IActionResult> Delete(int id)
{
    await _service.DeleteProductAsync(id);
    return RedirectToAction(nameof(Index));
}

    Key point: The five filter types have a strict execution order. Understand that order so you can place cross-cutting logic at the right stage in the pipeline.

    Q50. What are ViewBag, ViewData, and TempData? When would you use each? ▼

    All three are ways to pass data from a controller action to a view (or persist it across redirects). The differences are subtle but important.

    Mechanism Scope Type Best for
    ViewBag Current action → current view only Dynamic (no compile-time type checking) Small, ad-hoc data to the view (page title, flags)
    ViewData Current action → current view + child views Dictionary<string, object> (cast needed) Legacy code, shared data across layout and partials
    TempData Current action → next action (survives redirect) Dictionary<string, object> (cast needed) One-time messages across redirects (error alerts, success notifications)

    ViewBag example — pass ad-hoc data to view:

    csharp 
    // ProductsController.cs
public async Task<IActionResult> Index()
{
    var products = await _service.GetAllProductsAsync();

    ViewBag.PageTitle = "Product Catalog";           // Pass via ViewBag
    ViewBag.TotalCount = products.Count();
    ViewBag.IsAdmin = User.IsInRole("Admin");

    return View(products);
}
    html 
    <!-- Views/Products/Index.cshtml -->
<h1>@ViewBag.PageTitle</h1>

<p>Total Products: <strong>@ViewBag.TotalCount</strong></p>

@if ((bool)ViewBag.IsAdmin)
{
    <a href="@Url.Action("Create")">+ New Product</a>
}

    ViewData example — shared with layout/partials:

    csharp 
    // ProductsController.cs
public async Task<IActionResult> Edit(int id)
{
    var product = await _service.GetProductAsync(id);
    ViewData["Title"] = $"Edit Product: {product.Name}";  // Accessible in layout
    ViewData["BreadcrumbItems"] = new[] { "Products", product.Name };

    return View(product);
}
    html 
    <!-- Views/Shared/_Layout.cshtml -->
<title>@ViewData["Title"] - My App</title>

<!-- Breadcrumb partial -->
@if (ViewData["BreadcrumbItems"] is string[] items)
{
    <div class="breadcrumb">
        @foreach (var item in items)
        {
            <span>@item</span>
        }
    </div>
}

    TempData example — success message after form submission:

    csharp 
    // ProductsController.cs
[HttpPost]
public async Task<IActionResult> Create(CreateProductViewModel vm)
{
    if (!ModelState.IsValid)
        return View(vm);

    var product = new Product { Name = vm.Name, Price = vm.Price };
    await _service.CreateProductAsync(product);

    // Store message in TempData — survives the redirect
    TempData["SuccessMessage"] = $"Product '{product.Name}' created successfully!";

    return RedirectToAction(nameof(Index));  // TempData persists to the next request
}
    html 
    <!-- Views/Products/Index.cshtml -->
@if (TempData["SuccessMessage"] is string message)
{
    <div class="alert alert-success alert-dismissible">
        @message
        <button type="button" class="btn-close"></button>
    </div>
}

    Interview tip: "ViewBag and ViewData pass data to the current view. ViewBag is cleaner syntax (dynamic); ViewData requires casting. TempData persists across a redirect, which is why I use it for success/error messages after form posts. Avoid TempData for large data — it's stored in the session, which doesn't scale."

    Q51. What are strongly-typed views and ViewModels? Why create a separate ViewModel class? ▼

    A strongly-typed view declares its model using the @model directive at the top of the .cshtml file. This gives you IntelliSense in the editor and compile-time type checking — instead of casting ViewData["Product"], you write @Model.Name directly.

    A ViewModel is a dedicated C# class shaped exactly for a specific view — it may combine data from multiple domain models, include validation metadata, and exclude sensitive fields. Never pass a database entity directly to a view; instead, project it into a ViewModel.

    Why separate ViewModel from domain model?

    • View-specific concerns: A view might need a dropdown list of categories, a "IsEditing" boolean, and a collection of related items — none of which are part of the domain model.
    • Security: The domain Product entity might have an internal cost field — you don't want that exposed to the view. A ViewModel includes only what the UI needs.
    • Flexibility: Changing the view's layout doesn't force changes to the domain model. The ViewModel is a contract between controller and view; the domain model is internal to the business layer.
    • Testing: You can test the controller's mapping logic (entity → ViewModel) in isolation from the view rendering.

    Domain model (entity):

    csharp 
    // Models/Product.cs — from the database
public class Product
{
    public int Id { get; set; }
    public string Name { get; set; } = string.Empty;
    public decimal Price { get; set; }
    public decimal Cost { get; set; }              // ❌ internal only, hide from view
    public string Category { get; set; } = string.Empty;
    public int StockQuantity { get; set; }
    public bool IsActive { get; set; }
    public DateTime CreatedAt { get; set; }

    public decimal Profit => Price - Cost;         // ❌ not for display
}

    ViewModel for edit action:

    csharp 
    // ViewModels/ProductEditViewModel.cs — shaped for the view
public class ProductEditViewModel
{
    public int Id { get; set; }

    [Required(ErrorMessage = "Product name is required")]
    [StringLength(100)]
    public string Name { get; set; } = string.Empty;

    [Required]
    [Range(0.01, 999999)]
    public decimal Price { get; set; }

    [Required]
    public string Category { get; set; } = string.Empty;

    [Range(0, 10000)]
    public int StockQuantity { get; set; }

    // ✅ For the view dropdowns and UI logic
    public List<string> AvailableCategories { get; set; } = new();
    public bool CanEdit { get; set; }               // view needs to know if user can edit
    public string? LastModifiedBy { get; set; }
}

// ViewModel for the index/list view
public class ProductListItemViewModel
{
    public int Id { get; set; }
    public string Name { get; set; } = string.Empty;
    public decimal Price { get; set; }
    public string Category { get; set; } = string.Empty;
    public int Stock { get; set; }
    public string StatusBadge { get; set; } = string.Empty;  // "In Stock", "Low Stock", "Out"
}

    Controller mapping domain model to ViewModel:

    csharp 
    // ProductsController.cs
[HttpGet("Edit/{id}")]
public async Task<IActionResult> Edit(int id)
{
    var product = await _service.GetProductAsync(id);
    if (product is null) return NotFound();

    // Map domain model → ViewModel
    var vm = new ProductEditViewModel
    {
        Id = product.Id,
        Name = product.Name,
        Price = product.Price,
        Category = product.Category,
        StockQuantity = product.StockQuantity,
        AvailableCategories = await _service.GetCategoriesAsync(),  // only for view
        CanEdit = User.IsInRole("Admin"),                           // only for view
        LastModifiedBy = product.LastModifiedBy
    };

    // Cost is NOT copied — it stays internal
    return View(vm);
}

    Key benefits: You get IntelliSense on @Model.Name, validation metadata is tied to the view's needs (not the domain model's internal rules), and the view is isolated from database concerns. The domain model and view can evolve independently.

    Q52. What are HTML Helpers? When should you use @Html.TextBox, @Html.DropDownList, @Html.ValidationMessage? ▼

    HTML Helpers are methods that generate HTML markup. They bind to your ViewModel properties, auto-populate values, generate proper name attributes for model binding, and include validation attributes. They improve productivity and reduce typos.

    Common HTML Helpers:

    Helper Generates Use case
    @Html.TextBox("name", value) <input type="text" name="name" value="..." /> Simple text input
    @Html.TextBoxFor(m => m.ProductName) <input type="text" name="ProductName" value="..." /> Strongly-typed; uses ViewModel property
    @Html.DropDownList("category", items) <select name="category"><option>...</select> Select list
    @Html.DropDownListFor(m => m.Category, items) <select name="Category">...</select> Strongly-typed; binds to ViewModel property
    @Html.CheckBox("isActive") <input type="checkbox" name="isActive" /> Boolean toggle
    @Html.ValidationMessage("Name") <span class="field-validation-error">...</span> Display validation errors for a property
    @Html.ValidationSummary() <div><ul><li>errors</li></ul></div> Display all validation errors at once

    Example — Product edit form with helpers:

    html 
    <!-- Views/Products/Edit.cshtml -->
@model ProductEditViewModel

<form method="post" asp-action="Update">
    @Html.ValidationSummary()  <!-- Show all errors at top -->

    <div class="form-group">
        @Html.LabelFor(m => m.Name)
        @Html.TextBoxFor(m => m.Name, new { @class = "form-control" })
        @Html.ValidationMessageFor(m => m.Name)
    </div>

    <div class="form-group">
        @Html.LabelFor(m => m.Price)
        @Html.TextBoxFor(m => m.Price, new { @class = "form-control", type = "number", step = "0.01" })
        @Html.ValidationMessageFor(m => m.Price)
    </div>

    <div class="form-group">
        @Html.LabelFor(m => m.Category)
        @Html.DropDownListFor(m => m.Category,
            new SelectList(Model.AvailableCategories),
            "-- Select Category --",
            new { @class = "form-control" })
        @Html.ValidationMessageFor(m => m.Category)
    </div>

    <div class="form-check">
        @Html.CheckBoxFor(m => m.IsActive, new { @class = "form-check-input" })
        @Html.LabelFor(m => m.IsActive, "Is Active?")
    </div>

    <button type="submit" class="btn btn-primary">Save</button>
</form>

    Why use strongly-typed helpers? They automatically:

    • Bind the name attribute to the ViewModel property name (required for model binding on POST)
    • Populate the current value if the form is re-displayed (useful for validation failures)
    • Generate the correct input type (text, number, date, email, etc.)
    • Apply CSS classes from validation attributes
    • Provide compile-time IntelliSense — if you rename the property, the helpers break, catching the error early

    Modern alternative — Tag Helpers: ASP.NET Core also supports Tag Helpers like <input asp-for="Name" /> which work similarly but use HTML-like syntax instead of helper methods. Many teams prefer tag helpers for cleaner code.

    Q53. Explain Razor syntax. Give examples of @Model, @foreach, @if, and @Html. ▼

    Razor is ASP.NET's template engine. It blends C# and HTML — any line starting with @ is C# code, the rest is HTML. Razor compiles to a C# class that generates HTML at runtime.

    Core Razor syntax:

    • @Model — access the ViewModel passed from the controller
    • @{ ... } — code block (multi-line C# statements)
    • @if (...) { ... } — conditional rendering
    • @foreach (...) { ... } — loop over collections
    • @(expression) — evaluate an expression and output the result
    • @Html. — call HTML helpers
    • @DateTime.Now — inline expression output to HTML

    Complete Product catalog view example:

    html 
    <!-- Views/Products/Index.cshtml -->
@model IEnumerable<ProductListItemViewModel>

<h1>Product Catalog</h1>

@* C# code block: declare local variables *@
@{
    var totalProducts = Model.Count();
    var lowStockCount = Model.Count(p => p.Stock < 10);
    var lastUpdated = DateTime.Now.ToString("yyyy-MM-dd");
}

<!-- Output expressions -->
<p>Total: <strong>@totalProducts</strong> products (Last updated: @lastUpdated)</p>

@* Conditional rendering *@
@if (lowStockCount > 0)
{
    <div class="alert alert-warning">
        ⚠️ @lowStockCount products have low stock!
    </div>
}

@* Loop over collection *@
@if (Model.Any())
{
    <table class="table">
        <thead>
            <tr>
                <th>Name</th>
                <th>Category</th>
                <th>Price</th>
                <th>Stock</th>
                <th>Actions</th>
            </tr>
        </thead>
        <tbody>
            @foreach (var product in Model)
            {
                <tr>
                    <td>@product.Name</td>
                    <td><span class="badge">@product.Category</span></td>
                    <td>$@product.Price.ToString("F2")</td>
                    <td>
                        <!-- Nested if inside loop -->
                        @if (product.Stock < 10)
                        {
                            <span class="text-danger">⚠️ @product.Stock</span>
                        }
                        else
                        {
                            <span class="text-success">✓ @product.Stock</span>
                        }
                    </td>
                    <td>
                        <a href="@Url.Action("Edit", new { id = product.Id })">Edit</a>
                        <a href="@Url.Action("Delete", new { id = product.Id })" onclick="return confirm('Delete?')">Delete</a>
                    </td>
                </tr>
            }
        </tbody>
    </table>
}
else
{
    <p><em>No products found.</em></p>
}

<!-- Action links -->
<div class="mt-3">
    <a href="@Url.Action("Create")" class="btn btn-primary">+ New Product</a>
</div>

    Key Razor features in action:

    html 
    <!-- Partial view include (reusable component) -->
@await Html.PartialAsync("_ProductCard", product)

<!-- Display formatted values -->
Price: @($"${product.Price:N2}")

<!-- Ternary expressions -->
Status: @(product.IsActive ? "✓ Active" : "✗ Inactive")

<!-- Escape HTML (security) -->

<p>@product.Description</p>             <!-- automatically escaped (safe) -->

    Interview tip: Show confidence with Razor syntax — it's the template language of MVC. You should be able to write a loop, a conditional, and call an HTML helper without hesitation. Razor compiles to C#, so anything you can do in C#, you can do in a template.

    Q54. Explain form POST handling with ModelState.IsValid. How does validation work? ▼

    When a form is submitted via POST, ASP.NET Core automatically binds the form data to your ViewModel, runs validation, and populates ModelState. Your action checks if (ModelState.IsValid) to decide whether to proceed or re-display the form with errors.

    Validation flow:

    1. Form submitted: <form method="post"> → POST /Products/Create 2. ASP.NET binds form data to ViewModel properties (model binding) 3. Validation attributes on ViewModel run: - [Required] — field must have a value - [StringLength(100)] — text length 0-100 - [Range(1, 1000)] — numeric range - [RegularExpression(...)] — pattern matching - [Remote(...)] — call server-side validation 4. Custom IValidatableObject.Validate() runs (complex logic) 5. ModelState populated with errors (if any) 6. Controller action checks ModelState.IsValid - If TRUE: process the data, save, redirect - If FALSE: re-render the view with error messages

    Product create action with validation:

    csharp 
    // ProductsController.cs
[HttpGet]
public IActionResult Create()
{
    var vm = new ProductCreateViewModel
    {
        AvailableCategories = await _service.GetCategoriesAsync()
    };
    return View(vm);
}

[HttpPost]
[ValidateAntiForgeryToken]  // CSRF protection
public async Task<IActionResult> Create(ProductCreateViewModel vm)
{
    // ModelState automatically populated by binding + validation
    if (!ModelState.IsValid)
    {
        // Re-render the form with errors
        vm.AvailableCategories = await _service.GetCategoriesAsync();
        return View(vm);
    }

    // All validation passed — safe to proceed
    var product = new Product
    {
        Name = vm.Name.Trim(),
        Price = vm.Price,
        Category = vm.Category,
        StockQuantity = vm.StockQuantity ?? 0
    };

    await _service.CreateProductAsync(product);

    TempData["SuccessMessage"] = "Product created successfully!";
    return RedirectToAction(nameof(Index));
}

    ViewModel with validation attributes:

    csharp 
    // ViewModels/ProductCreateViewModel.cs
public class ProductCreateViewModel : IValidatableObject
{
    [Required(ErrorMessage = "Product name is required.")]
    [StringLength(100, MinimumLength = 3,
        ErrorMessage = "Name must be between 3 and 100 characters.")]
    public string Name { get; set; } = string.Empty;

    [Required(ErrorMessage = "Price is required.")]
    [Range(0.01, 999999, ErrorMessage = "Price must be between $0.01 and $999,999.")]
    [DisplayName("Product Price")]
    public decimal Price { get; set; }

    [Required(ErrorMessage = "Category is required.")]
    public string Category { get; set; } = string.Empty;

    [Range(0, 10000, ErrorMessage = "Stock must be between 0 and 10,000.")]
    [DisplayName("Initial Stock")]
    public int? StockQuantity { get; set; }

    // For display in dropdown
    [NotMapped]
    public List<string> AvailableCategories { get; set; } = new();

    // Custom validation logic (runs after attribute validation)
    public IEnumerable<ValidationResult> Validate(ValidationContext context)
    {
        // Example: category must be one of the allowed values
        if (!AvailableCategories.Contains(Category))
        {
            yield return new ValidationResult(
                "Selected category is not available.",
                new[] { nameof(Category) });
        }

        // Example: if price is very low, require a reason
        if (Price < 10 && string.IsNullOrEmpty(Name.Contains("clearance", StringComparison.OrdinalIgnoreCase) ? "" : null))
        {
            yield return new ValidationResult(
                "Very low prices should include 'clearance' in the product name.",
                new[] { nameof(Price) });
        }
    }
}

    Key validation concepts:

    • Client-side validation: JavaScript runs validation in the browser before POST (quick feedback). Built-in if you include jQuery Validate.
    • Server-side validation: Always run server-side — never trust client-side validation alone (security risk).
    • ModelState errors: ModelState.AddModelError(fieldName, message) lets you add errors in code (e.g., checking if product name already exists).
    • CSRF protection: Use [ValidateAntiForgeryToken] on POST actions and @Html.AntiForgeryToken() in forms.
    Q55. What are partial views and child actions? How do you reuse views like _ProductCard.cshtml? ▼

    Partial views are reusable view fragments — small pieces of HTML that you include in multiple places. They follow the naming convention _PartialName.cshtml (leading underscore). A child action (now called a "view component") is a controller action that returns a partial view — useful when the partial needs to fetch its own data.

    Two approaches to reuse views:

    • Simple partial: Pass data from the parent action. No data-fetching logic inside the partial.
    • View component: A self-contained unit with its own logic. The component fetches its own data and renders. No controller action needed.

    Simple partial — _ProductCard.cshtml:

    html 
    <!-- Views/Shared/_ProductCard.cshtml -->
@model ProductCardViewModel

<div class="product-card">
    <div class="card">
        <div class="card-header">
            <h5 class="card-title">@Model.Name</h5>
        </div>
        <div class="card-body">
            <p class="text-muted">@Model.Category</p>
            <p class="card-text">@Model.Description</p>
            <h6>Price: <strong>$@Model.Price.ToString("F2")</strong></h6>
        </div>
        <div class="card-footer">
            @if (Model.InStock)
            {
                <span class="badge bg-success">In Stock</span>
            }
            else
            {
                <span class="badge bg-danger">Out of Stock</span>
            }
            <a href="@Url.Action("Details", new { id = Model.Id })" class="ms-2">View Details</a>
        </div>
    </div>
</div>

    Use the partial in multiple views:

    html 
    <!-- Views/Products/Index.cshtml -->
@model IEnumerable<ProductCardViewModel>

<h1>Products</h1>

<div class="product-grid">
    @foreach (var product in Model)
    {
        <!-- Include partial view for each product -->
        @await Html.PartialAsync("_ProductCard", product)
    }
</div>

    View Component — self-contained logic:

    A view component is a C# class that handles its own data-fetching logic. Use it when a partial needs to query the database or call services independently.

    csharp 
    // Components/TopProductsViewComponent.cs
public class TopProductsViewComponent : ViewComponent
{
    private readonly IProductService _service;

    public TopProductsViewComponent(IProductService service) => _service = service;

    public async Task<IViewComponentResult> InvokeAsync(int count = 5)
    {
        // The component fetches its own data
        var topProducts = await _service.GetTopProductsAsync(count);
        return View(topProducts);  // renders Views/Shared/Components/TopProducts/Default.cshtml
    }
}
    html 
    <!-- Views/Shared/Components/TopProducts/Default.cshtml -->
@model IEnumerable<Product>

<div class="top-products">
    <h3>Top Products</h3>
    <ul>
        @foreach (var product in Model)
        {
            <li>@product.Name - $@product.Price</li>
        }
    </ul>
</div>

    Invoke the view component in any view:

    html 
    <!-- Views/Home/Index.cshtml or any other view -->

<!-- Invoke view component (it fetches its own data) -->
@await Component.InvokeAsync("TopProducts", new { count = 10 })

<!-- No need to fetch data in the controller action -->

    When to use each:

    • Simple partial @Html.PartialAsync() — data passed from parent, just formatting/display logic
    • View component — needs to fetch data independently, used across multiple unrelated views, reusable widget with its own logic

    Key difference from child actions (older .NET): In .NET Framework, child actions were controller actions that rendered partial views. ASP.NET Core introduced View Components as a cleaner, more testable pattern. View Components don't go through the full MVC pipeline — they have their own simplified flow.

    Bonus: MVC vs Web API Comparison

    Both follow the MVC pattern and both are built on ASP.NET Core. The key difference is in the response and use case.

    Aspect ASP.NET MVC ASP.NET Web API
    Response Format HTML (rendered on server) JSON or XML (data format)
    View Layer Razor templates (.cshtml) generate HTML No view — controller returns data directly
    Client Web browser receives complete HTML page JavaScript/SPA/mobile app parses JSON
    Controller Base Class Inherits Controller Inherits ControllerBase (no view support)
    HTTP Verbs Mostly POST (forms), some GET All verbs: GET, POST, PUT, DELETE, PATCH, etc.
    Routing Convention-based or attribute-based RESTful attribute routes: /api/products/{id}
    Model Binding Form data → ViewModel JSON body → DTO via [FromBody]
    Authentication Cookie-based (session) Token-based (JWT, OAuth)
    Use Case Traditional web apps, server-side rendering, dynamic HTML pages Microservices, SPA (React/Angular), mobile apps, IoT
    SEO Native — HTML sent to browser Requires extra work — client-side rendering hides content from crawlers
    Example Project Product Catalog admin panel (server-side forms) ProductCatalogAPI (JSON endpoints for mobile/SPA)
    💡

    ASP.NET Core unifies both

    In modern ASP.NET Core, you can build MVC and Web API in the same project. A single controller can have both view-returning actions (MVC) and JSON-returning actions (API). For example, ProductsController might have Get() returning HTML (MVC) and api/products returning JSON (Web API). The architecture is the same; the output differs.

    🎯

    Interview tip for MVC questions

    When asked about MVC, emphasize that you understand the server-side rendering model — controllers orchestrate business logic, services fetch data, models hold that data, and Razor templates render it into HTML. Mention that you're familiar with validation, filters, partial views, and ViewModels. Show that you can distinguish MVC (HTML) from Web API (JSON). If asked why you'd choose one over the other, say: "MVC for a traditional web app with form-based workflows; Web API for a backend that serves multiple clients (web, mobile, IoT)."
