    💎 C# & OOP Concepts

    Object-oriented programming principles and C# language features — explained for beginners with real code examples

    Interview Prep OOP Fundamentals Beginner
    💡

    What is OOP?

    Object-Oriented Programming (OOP) is a paradigm for designing software around objects — entities that bundle data (state) and methods (behavior) together. OOP is built on four pillars: Encapsulation (hiding internal details), Abstraction (exposing what matters, hiding complexity), Inheritance (sharing code through hierarchies), and Polymorphism (using the same interface with different implementations). C# is a fully object-oriented language — every line of code you write lives inside a class. Understanding these concepts deeply is essential for writing clean, maintainable code and acing technical interviews. All examples on this page use the Product Catalog scenario you are already building.

    Quick Reference: Core Concepts

    Concept Real-world Analogy C# Keyword/Feature
    Class Blueprint for a car (defines what properties and methods a car has) class Product { }
    Object/Instance An actual car (specific car with specific color and mileage) var product = new Product()
    Encapsulation Car has doors you can open; engine is hidden inside (you don't touch it directly) public, private, properties with getters/setters
    Abstraction You drive a car without knowing how the transmission works internally abstract class, interface
    Inheritance An electric car is a car — it inherits all car properties but adds battery management class ElectricCar : Car { }
    Polymorphism Different cars (sedan, truck) all respond to "accelerate" differently, same button does different things virtual, override, abstract
    Struct (value type) A simple data bundle you pass around by value (like writing down a phone number) struct Point { }
    Class (reference type) A complex entity you pass around by reference (like passing a business card to contact later) class Product { }

    Interview Questions on C# & OOP

    Q36. What is the difference between struct and class? ▼

    struct and class look similar but differ fundamentally in how they behave in memory:

    Aspect struct class
    Type Value type (stack) Reference type (heap)
    Copy behavior Copied by value — changes to copy don't affect original Copied by reference — all references point to same object
    Performance Faster, less GC pressure (small structs) Slightly slower, managed by garbage collector
    Inheritance Cannot inherit from other structs; can implement interfaces Can inherit from other classes and interfaces
    Null Cannot be null (unless nullable: Point?) Can be null

    When to use struct: small, immutable data bundles (coordinates, colors, money amounts). Example: decimal Price is actually a struct — lightweight and fast.

    When to use class: almost always for business entities like Product, Customer, or Order. They're complex, mutable, and need inheritance.

    csharp 
    // ─── STRUCT — value type, stack-allocated ───
public struct Money
{
    public decimal Amount { get; set; }
    public string Currency { get; set; }
}

// Structs are copied by value
var price1 = new Money { Amount = 99.99m, Currency = "USD" };
var price2 = price1;           // copies the struct (full copy)
price2.Amount = 50m;           // changing price2 does NOT affect price1

Console.WriteLine(price1.Amount);  // still 99.99 — unaffected

// ─── CLASS — reference type, heap-allocated ───
public class Product
{
    public string Name { get; set; }
    public decimal Price { get; set; }
}

// Classes are copied by reference
var product1 = new Product { Name = "Laptop", Price = 999.99m };
var product2 = product1;       // copies the reference (same object)
product2.Price = 50m;          // changing product2 DOES affect product1

Console.WriteLine(product1.Price);  // 50 — BOTH point to the same object

    Key takeaway: structs are for small, immutable value bundles (rare in business logic). Classes are for everything else. When in doubt, use class.

    Q37. What is encapsulation? Show an example with property validation. ▼

    Encapsulation is bundling data (fields) and methods together, and hiding the internal details behind a public interface. You expose what's necessary (properties) and hide how it's implemented (validation logic, backing fields).

    A classic example is Product.Price. The class should never allow negative prices. Without encapsulation, you'd have a public field anyone could set to -100. With proper encapsulation, the setter validates.

    csharp 
    // ❌ BAD — public field with no validation
public class Product
{
    public decimal Price;  // anyone can set this to any value, including negative
}

var product = new Product();
product.Price = -100;  // ❌ invalid state — negative price makes no business sense

// ─────────────────────────────────────

// ✅ GOOD — property with validation in the setter
public class Product
{
    private decimal _price;  // private backing field — hidden from outside

    // Public property with validation
    public decimal Price
    {
        get { return _price; }
        set
        {
            if (value < 0)
                throw new ArgumentException("Price cannot be negative");
            _price = value;
        }
    }

    // Alternative: auto-property with C# 8+ init-only setter
    public string Name { get; set; }
}

var product = new Product();
product.Price = 99.99m;           // ✅ valid
// product.Price = -100;          // ❌ throws ArgumentException — prevented

// ─────────────────────────────────────

// Modern C# 8+ — property with init-only setter (immutable)
public class ProductDto
{
    public int Id { get; init; }      // can only be set during initialization
    public string Name { get; init; }
    public decimal Price { get; init; }
}

// Can only set during construction
var dto = new ProductDto { Id = 1, Name = "Laptop", Price = 999.99m };
// dto.Price = 50;  // ❌ compile error — cannot modify init-only property

    Benefits of encapsulation: you can change the internal implementation (add validation, logging, caching) without changing the public interface. Callers only care that the property is accessible — not how it's validated behind the scenes.

    Q38. What is polymorphism? Explain method overriding with an example. ▼

    Polymorphism means "many forms" — the same method name behaves differently depending on the object type. It allows you to write code that works with a base type but actually calls specialized versions for each subtype.

    A real-world example: different product categories calculate discounts differently. Electronics gets 10% off, Clothing gets 20% off, but both use GetDiscount().

    csharp 
    // ─── Base class with virtual method ───
public class DiscountCalculator
{
    // virtual = subclasses CAN override this
    public virtual decimal GetDiscount(decimal price)
    {
        return price;  // default: no discount
    }
}

// ─── Subclass 1 — Electronics discount ───
public class ElectronicsDiscount : DiscountCalculator
{
    // override = subclass provides its own implementation
    public override decimal GetDiscount(decimal price)
    {
        return price * 0.90m;  // 10% off
    }
}

// ─── Subclass 2 — Clothing discount ───
public class ClothingDiscount : DiscountCalculator
{
    public override decimal GetDiscount(decimal price)
    {
        return price * 0.80m;  // 20% off
    }
}

// ─── Polymorphism in action ───
DiscountCalculator discount1 = new ElectronicsDiscount();
DiscountCalculator discount2 = new ClothingDiscount();

decimal price = 100m;

Console.WriteLine(discount1.GetDiscount(price));  // 90 (10% off)
Console.WriteLine(discount2.GetDiscount(price));  // 80 (20% off)

// ─── The magic: same method, different behaviour ───
List<DiscountCalculator> calculators = new()
{
    new ElectronicsDiscount(),
    new ClothingDiscount()
};

foreach (var calc in calculators)
{
    // Each calls its own GetDiscount() implementation
    var discounted = calc.GetDiscount(100m);
    Console.WriteLine($"Discount: {discounted}");
}
// Output: 90, 80

    Key keywords:

    • virtual on the base method — allows overriding in subclasses
    • override on the subclass method — provides a new implementation
    • When you call the method through a base-type reference, the actual runtime type determines which version runs (not the declared type)

    This is powerful because you can write one loop that works with any DiscountCalculator subclass without knowing what it is.

    Q39. When do you use abstract class vs interface? ▼

    Both define contracts that subclasses/implementors must follow, but they serve different purposes:

    Aspect abstract class interface
    Represents "Is-A" relationship (a subclass IS a kind of the parent) Capability/contract (a class CAN DO what the interface defines)
    State Can have fields, constructors, state Only methods/properties (no state in older C#)
    Inheritance Single inheritance only (one parent class) Multiple inheritance (implement many interfaces)
    Access modifiers Can have private, protected, public members Members are implicitly public
    When to use Shared code between related classes Define a behavior contract without implementation details
    csharp 
    // ─── ABSTRACT CLASS — "is-a" relationship ───
// Example: all repositories in Product Catalog share a base pattern
public abstract class BaseRepository<T>
{
    protected readonly AppDbContext _context;  // shared state

    public BaseRepository(AppDbContext context) => _context = context;

    // Shared concrete method — all repos use this
    public virtual async Task<IEnumerable<T>> GetAllAsync()
    {
        return await _context.Set<T>().ToListAsync();
    }

    // Abstract method — subclass MUST implement
    public abstract Task<T> GetByIdAsync(int id);
}

public class ProductRepository : BaseRepository<Product>
{
    public ProductRepository(AppDbContext ctx) : base(ctx) { }

    public override async Task<Product> GetByIdAsync(int id) =>
        await _context.Products.FirstOrDefaultAsync(p => p.Id == id);
}

// ─── INTERFACE — "can-do" capability ───
// Example: multiple unrelated classes that CAN be logged or cached
public interface IRepository<T>
{
    Task<IEnumerable<T>> GetAllAsync();
    Task<T> GetByIdAsync(int id);
}

public interface ICacheable
{
    string GetCacheKey();
    TimeSpan CacheDuration { get; }
}

// A class can implement multiple interfaces (but inherit from only one class)
public class ProductService : IRepository<Product>, ICacheable
{
    public Task<IEnumerable<Product>> GetAllAsync() { /* ... */ }
    public Task<Product> GetByIdAsync(int id) { /* ... */ }
    public string GetCacheKey() => "products:all";
    public TimeSpan CacheDuration => TimeSpan.FromMinutes(5);
}

    Rule of thumb:

    • Use abstract class when classes are closely related (all repositories, all entities). There's shared implementation and a parent-child relationship.
    • Use interface when you're defining a capability that unrelated classes might have (loggable, cacheable, disposable). Multiple classes implement it without being related.
    Q40. What is the difference between virtual, abstract, and sealed? ▼

    These three keywords control method inheritance and overriding behaviour:

    • virtual — "subclasses MAY override this method if they want to"
    • abstract — "subclasses MUST override this method" (only in abstract classes)
    • sealed — "subclasses CANNOT override this method"
    csharp 
    // ─── VIRTUAL — optional override ───
public class ProductService
{
    // virtual = subclasses CAN override this, but don't have to
    public virtual void LogProductAdded(Product product)
    {
        Console.WriteLine($"Product added: {product.Name}");
    }
}

public class DetailedProductService : ProductService
{
    // Can override if we want to customize the logging
    public override void LogProductAdded(Product product)
    {
        Console.WriteLine($"[DETAILED] Product ID: {product.Id}, Name: {product.Name}, Price: {product.Price}");
    }
}

// ─── ABSTRACT — mandatory override ───
public abstract class PaymentProcessor
{
    // abstract = subclass MUST implement this (no default implementation)
    public abstract Task<bool> ProcessPaymentAsync(decimal amount);

    // Non-abstract method — all processors share this
    public void LogPayment(decimal amount)
    {
        Console.WriteLine($"Processing: {amount:C}");
    }
}

public class CreditCardProcessor : PaymentProcessor
{
    // MUST implement this — not optional
    public override async Task<bool> ProcessPaymentAsync(decimal amount)
    {
        // Credit card-specific logic
        return await ChargeCreditCardAsync(amount);
    }

    private async Task<bool> ChargeCreditCardAsync(decimal amount) { /* ... */ }
}

// ─── SEALED — prevents override ───
public class FinalProductService : ProductService
{
    // sealed = subclasses of FinalProductService cannot override this
    public sealed override void LogProductAdded(Product product)
    {
        Console.WriteLine($"[SEALED] {product.Name}");
    }
}

public class ExtendedProductService : FinalProductService
{
    // ❌ COMPILE ERROR — cannot override a sealed method
    // public override void LogProductAdded(Product product) { }
}

    When to use each:

    • virtual — the most common for base classes. Allows flexibility without requiring override.
    • abstract — when the base class has no sensible default and subclass MUST provide the implementation.
    • sealed — rare. Use when you've perfected a method and never want anyone overriding it (can lead to inflexibility).
    Q41. What is the IDisposable pattern? When and how do you use it? ▼

    IDisposable is an interface that helps clean up unmanaged resources (database connections, file handles, network sockets). Managed resources (strings, lists) are automatically cleaned up by the garbage collector. Unmanaged resources are not — you must clean them up yourself.

    The pattern ensures resources are freed even if an exception occurs, using the using statement.

    csharp 
    // ─── Simple IDisposable implementation ───
public class ProductExporter : IDisposable
{
    private StreamWriter _fileWriter;  // unmanaged resource

    public ProductExporter(string filePath)
    {
        _fileWriter = new StreamWriter(filePath);
    }

    public async Task ExportProductAsync(Product product)
    {
        await _fileWriter.WriteLineAsync($"{product.Id},{product.Name},{product.Price}");
    }

    // IDisposable interface implementation
    public void Dispose()
    {
        // Close and free the file handle
        _fileWriter?.Close();
        _fileWriter?.Dispose();
    }
}

// ─── Using statement ensures Dispose() is called ───
using (var exporter = new ProductExporter("products.csv"))
{
    var product = new Product { Id = 1, Name = "Laptop", Price = 999m };
    await exporter.ExportProductAsync(product);
}  // Dispose() is called automatically here, even if an exception occurred

// ─── Modern C# 8+ — using declaration (cleaner) ───
using var exporter = new ProductExporter("products.csv");
var product = new Product { Id = 1, Name = "Laptop", Price = 999m };
await exporter.ExportProductAsync(product);
// Dispose() is called automatically at the end of the scope

// ─── Best practice: implement IAsyncDisposable for async cleanup ───
public class ProductDatabaseConnection : IAsyncDisposable
{
    private SqlConnection _connection;

    public async Task OpenAsync(string connectionString)
    {
        _connection = new SqlConnection(connectionString);
        await _connection.OpenAsync();
    }

    public async ValueTask DisposeAsync()
    {
        if (_connection != null)
        {
            await _connection.CloseAsync();
            _connection.Dispose();
        }
    }
}

// Use with await using
await using (var db = new ProductDatabaseConnection())
{
    await db.OpenAsync("connection-string");
    // use db
}  // DisposeAsync() is called automatically

    Rules for IDisposable:

    • Implement IDisposable if your class uses unmanaged resources (files, database connections, native handles)
    • Use the using statement or using declaration to ensure Dispose() is called
    • In async code, implement IAsyncDisposable instead, and use await using
    • Never rely on finalizers alone — they run unpredictably
    Q42. What is LINQ? Explain query syntax vs method syntax. ▼

    LINQ (Language-Integrated Query) is a C# feature that lets you query collections (lists, databases, XML) using a SQL-like syntax or fluent method calls. It dramatically reduces the code needed to filter, transform, and aggregate data.

    LINQ works on any IEnumerable<T> collection. When querying Entity Framework DbSets, LINQ is translated to SQL automatically.

    csharp 
    // Sample data
var products = new List<Product>
{
    new() { Id = 1, Name = "Laptop", Category = "Electronics", Price = 999m, Stock = 5 },
    new() { Id = 2, Name = "Mouse", Category = "Electronics", Price = 25m, Stock = 100 },
    new() { Id = 3, Name = "Shirt", Category = "Clothing", Price = 50m, Stock = 30 },
    new() { Id = 4, Name = "Jeans", Category = "Clothing", Price = 80m, Stock = 20 },
};

// ─── QUERY SYNTAX — SQL-like, reads top-to-bottom ───
var expensiveElectronics = from p in products
                           where p.Category == "Electronics" && p.Price > 100
                           select new { p.Name, p.Price };

// ─── METHOD SYNTAX — Fluent, chained method calls ───
var expensiveElectronicsMethod = products
    .Where(p => p.Category == "Electronics" && p.Price > 100)
    .Select(p => new { p.Name, p.Price });

// ─── GROUP BY ───
var categoryCounts = from p in products
                     group p by p.Category into category
                     select new { Name = category.Key, Count = category.Count() };

// Method syntax equivalent
var categoryCountsMethod = products
    .GroupBy(p => p.Category)
    .Select(g => new { Name = g.Key, Count = g.Count() });

// ─── ORDER BY ───
var sortedByPrice = from p in products
                    orderby p.Price descending
                    select p;

var sortedByPriceMethod = products
    .OrderByDescending(p => p.Price);

// ─── JOIN ───
var customers = new List<Customer>
{
    new() { Id = 1, Name = "Alice", FavoriteProductId = 1 },
    new() { Id = 2, Name = "Bob", FavoriteProductId = 3 }
};

var customerPreferences = from c in customers
                          join p in products on c.FavoriteProductId equals p.Id
                          select new { Customer = c.Name, Product = p.Name };

var customerPreferencesMethod = customers
    .Join(products, c => c.FavoriteProductId, p => p.Id,
          (c, p) => new { Customer = c.Name, Product = p.Name });

// ─── Aggregation ───
var totalValue = products.Sum(p => p.Price * p.Stock);
var avgPrice = products.Average(p => p.Price);
var maxPrice = products.Max(p => p.Price);

// ─── EXISTS / ANY / ALL ───
bool hasExpensive = products.Any(p => p.Price > 500);           // true
bool allInStock = products.All(p => p.Stock > 0);               // true
var noStock = products.FirstOrDefault(p => p.Stock == 0);        // null

    Query vs Method syntax:

    • Query syntax reads like SQL, easier for complex queries with joins and grouping.
    • Method syntax is more explicit, integrates with lambda expressions, slightly more powerful.
    • Both compile to the same IL code — choose based on readability.

    With Entity Framework: LINQ queries are translated to SQL at runtime. products.Where(...).ToList() executes on the database, not in memory.

    Q43. What is async/await? Why is it important? ▼

    async/await is a C# feature that lets you write asynchronous code that looks synchronous — easier to read and maintain. Instead of blocking a thread while waiting for I/O (database, network, file), you release the thread and resume later when the result arrives.

    This is critical for web APIs because a single thread can handle thousands of concurrent requests instead of being blocked on each one.

    csharp 
    // ❌ BAD — synchronous blocking (old way)
public class ProductService
{
    private readonly AppDbContext _context;

    public List<Product> GetAllProducts()
    {
        // Thread is BLOCKED here waiting for database response
        // During this block, this thread cannot handle other requests
        var products = _context.Products.ToList();
        return products;
    }
}

// With 1000 concurrent requests and a thread pool of 100 threads:
// 100 requests are processing, 900 are queued waiting for a free thread
// Response time is slow because threads are tied up

// ─────────────────────────────────────

// ✅ GOOD — asynchronous non-blocking
public class ProductService
{
    private readonly AppDbContext _context;

    public async Task<List<Product>> GetAllProductsAsync()
    {
        // Thread is NOT blocked here — it's released to handle other requests
        // When database responds, any available thread resumes this method
        var products = await _context.Products.ToListAsync();
        return products;
    }
}

// With 1000 concurrent requests and a thread pool of 100 threads:
// All 1000 requests are "processing" at the same time (truly concurrent)
// Response time is faster because threads aren't blocked waiting

// ─────────────────────────────────────

// Using async/await in a controller
[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
    private readonly IProductService _service;

    [HttpGet]
    public async Task<ActionResult<IEnumerable<ProductResponse>>> GetAll()
    {
        // await waits for the database query without blocking the thread
        var products = await _service.GetAllProductsAsync();
        return Ok(products);
    }
}

// ─────────────────────────────────────

// Common async patterns
public class ProductService
{
    private readonly AppDbContext _context;

    // Single async operation
    public async Task<Product> GetByIdAsync(int id) =>
        await _context.Products.FirstOrDefaultAsync(p => p.Id == id);

    // Multiple parallel operations
    public async Task<(List<Product>, int)> GetProductsAndCountAsync()
    {
        var getProducts = _context.Products.ToListAsync();
        var getCount = _context.Products.CountAsync();

        // Wait for both in parallel
        await Task.WhenAll(getProducts, getCount);

        return (await getProducts, await getCount);
    }

    // Loop with async operations
    public async Task ImportProductsAsync(IEnumerable<CreateProductRequest> requests)
    {
        foreach (var req in requests)
        {
            var product = new Product { Name = req.Name, Price = req.Price };
            _context.Products.Add(product);
        }

        // Save all at once
        await _context.SaveChangesAsync();
    }
}

    Key concepts:

    • async keyword on a method means it can use await inside.
    • await pauses execution and releases the thread; execution resumes when the awaited operation completes.
    • Task<T> represents an asynchronous operation that returns type T.
    • async Task<T> vs async Task — the latter returns no value, just completes.

    In ASP.NET Core all the built-in methods are async by default: ToListAsync(), FirstOrDefaultAsync(), SaveChangesAsync(). Use them! Not using async in a web API is leaving performance on the table.

    Q44. What is the null-coalescing operator (?? and ??=)? ▼

    The null-coalescing operator (??) provides a default value if the left side is null. It's a cleaner way to handle null values than if (x == null) statements.

    The null-coalescing assignment operator (??=) assigns a value only if the variable is currently null.

    csharp 
    // ❌ OLD WAY — verbose null checking
public string GetProductName(Product product)
{
    if (product == null)
    {
        return "Unknown";
    }
    else
    {
        return product.Name;
    }
}

// ─────────────────────────────────────

// ✅ BETTER — using ?? operator
public string GetProductName(Product product) =>
    product?.Name ?? "Unknown";  // if product or Name is null, return "Unknown"

// ─────────────────────────────────────

// Common patterns
Product product = null;
string name = product?.Name ?? "Unknown";          // "Unknown"

decimal price = product?.Price ?? 0m;              // 0

// List or collection
var productCount = products?.Count ?? 0;           // 0 if products is null

// Chaining ?? for multiple fallbacks
string description = product?.Description
                     ?? category?.DefaultDescription
                     ?? "No description available";

// ─────────────────────────────────────

// NULL-COALESCING ASSIGNMENT ??=
public class ProductCache
{
    private string _cachedName;

    public void CacheName(Product product)
    {
        // Only assign if _cachedName is null
        _cachedName ??= product?.Name;  // equivalent to: if (_cachedName == null) _cachedName = product?.Name;
    }
}

// Practical example — lazy initialization
public class ProductService
{
    private List<Product> _allProducts;

    public async Task<List<Product>> GetAllAsync()
    {
        // Load from database only if not already cached
        _allProducts ??= await _context.Products.ToListAsync();
        return _allProducts;
    }
}

// ─────────────────────────────────────

// NULLABLE REFERENCE TYPES (C# 8+) — enable stricter null checking
#nullable enable

public string GetProductPrice(Product? product)  // ? = nullable, can be null
{
    // Compiler warning if you access product.Name without null-checking
    var name = product?.Name ?? "Unknown";  // OK — using null-safe operator
    // var name = product.Name;  // ❌ Compiler warning — product might be null
    return $"Product: {name}";
}

    Advantages:

    • Cleaner code — fewer if/else branches.
    • Safer — works with nullable reference types for compile-time null-checking.
    • More readable — intention is clear: use this value or fall back to that.
    Q45. What is pattern matching? Show C# 7+ examples. ▼

    Pattern matching is a C# feature (C# 7+) that lets you test values against patterns in cleaner ways than traditional if/else chains. It works with types, constants, properties, lists, and custom patterns.

    csharp 
    // ❌ OLD WAY — verbose if/else ───
public string GetDiscountMessage(object product)
{
    if (product is Product p)
    {
        if (p.Price > 500)
            return "50% off on expensive items";
        else if (p.Price > 100)
            return "20% off on items over $100";
    }
    return "No discount";
}

// ─────────────────────────────────────

// ✅ C# 7 — Type patterns and is expressions ───
public string GetDiscountMessage(Product product) =>
    product switch
    {
        { Price: > 500 } => "50% off on expensive items",
        { Price: > 100 } => "20% off on items over $100",
        { Price: > 0 } => "10% off",
        _ => "No discount"  // _ = default case (wildcard)
    };

// ─────────────────────────────────────

// ✅ C# 8+ — Property patterns (most common) ───
public decimal ApplyDiscount(Product product) =>
    product switch
    {
        { Category: "Electronics", Price: > 500 } => product.Price * 0.90m,     // 10% off
        { Category: "Electronics" } => product.Price * 0.95m,                   // 5% off
        { Category: "Clothing", Price: > 100 } => product.Price * 0.80m,        // 20% off
        { Category: "Clothing" } => product.Price * 0.90m,                      // 10% off
        { Stock: 0 } => product.Price,                                           // no discount if out of stock
        _ => product.Price * 0.95m                                              // default: 5% off
    };

// ─────────────────────────────────────

// ✅ C# 9 — Relational patterns and logical patterns ───
public string ValidateProduct(Product product) =>
    product switch
    {
        // Relational patterns
        { Price: < 0 } => "Price cannot be negative",
        { Price: 0 } => "Price cannot be zero",
        { Stock: < 0 } => "Stock cannot be negative",

        // Logical AND pattern (and)
        { Category: "Electronics", Price: > 1000, Stock: > 0 } => "Valid expensive product",

        // Logical OR pattern (or)
        { Category: "Electronics" or "Gadgets" } => "Valid tech product",

        // Logical NOT pattern (not)
        { Price: not > 0 } => "Price must be positive",

        // Null checking
        { Name: null or "" } => "Product name is required",

        _ => "Valid product"
    };

// ─────────────────────────────────────

// ✅ C# 9 — RECORDS (immutable data classes) ───
// Records are perfect for pattern matching
public record ProductDto(int Id, string Name, decimal Price, string Category);

public string GetProductSummary(ProductDto product) =>
    product switch
    {
        (_, _, > 500, "Electronics") => "Expensive tech",
        (_, _, > 100, _) => "Premium product",
        _ => "Regular product"
    };

// Destructuring in patterns
public void ProcessOrders(IEnumerable<(int Id, decimal Total, string Status)> orders)
{
    foreach (var order in orders)
    {
        var message = order switch
        {
            (_, > 1000, "Completed") => "High-value completed order",
            (_, _, "Pending") => "Order still processing",
            _ => "Other order"
        };
        Console.WriteLine(message);
    }
}

// ─────────────────────────────────────

// List patterns (C# 11+)
public string AnalyzeScores(int[] scores) =>
    scores switch
    {
        []              => "No scores",
        [var single]    => $"One score: {single}",
        [var first, .. var rest] => $"First: {first}, Others: {rest.Length}",
        [.., var last]  => $"Last score: {last}",
    };

// ─────────────────────────────────────

// Custom type patterns with custom types
public class Order
{
    public int Id { get; set; }
    public decimal Total { get; set; }
    public bool IsPaid { get; set; }
}

public string GetOrderStatus(Order order) =>
    order switch
    {
        { Total: > 500, IsPaid: true } => "High-value paid order",
        { IsPaid: false } => "Unpaid order",
        _ => "Other order"
    };

    Pattern matching syntax:

    • property: value — property equals a specific value
    • property: > 100 — relational patterns (>, <, >=, <=)
    • and, or, not — logical combinations
    • _ — wildcard (matches anything, used as default case)
    • (_, _, value, _) — destructuring tuples/records
    • [..] — list patterns (C# 11+)

    Why use pattern matching: It reduces nesting, is more readable than large if/else chains, and catches compile-time errors when adding new cases to an exhaustive switch.

    Q46. What is the difference between List<T> and IEnumerable<T>? ▼

    List<T> is a concrete collection class. IEnumerable<T> is an interface that describes "something you can iterate over".

    AspectList<T>IEnumerable<T>
    Type Concrete class Interface/abstraction
    Memory All items loaded in memory Can be lazy (yields items one at a time)
    Indexing Supports [i] fast access No indexer (must iterate from start)
    Modification Can Add(), Remove(), Count Read-only (depends on implementation)
    LINQ LINQ executes immediately LINQ defers (lazy evaluation)
    csharp 
    // ❌ BAD — returns List, forces caller to use concrete type
public List<Product> GetProducts() { return _products.ToList(); }

// ✅ GOOD — returns IEnumerable, flexible, lazy
public IEnumerable<Product> GetProducts() { return _products; }

// LINQ difference: List vs IEnumerable
var list = new List<int> { 1, 2, 3 }.Where(x => x > 1);  // executes now
IEnumerable<int> lazy = _numbers.Where(x => x > 1);       // executes on iteration

foreach (var num in lazy)  // WHERE query runs HERE
{
    Console.WriteLine(num);
}

    Best practice: Return IEnumerable<T> or IQueryable<T> from methods. Let callers decide if they need a List. This enables lazy evaluation and performance optimization.

    Q47. What is the difference between class and record in C# 9+? ▼

    Records (C# 9+) are a new type primarily for immutable data. They provide automatic structural equality, deconstruction, and concise syntax compared to classes.

    Featureclassrecord
    Equality Reference equality (default) Structural equality (by default)
    ToString() ClassName (default) Shows all properties
    Immutability Mutable (by design) Immutable (by design)
    Deconstruction Manual implementation Automatic
    Use case Complex entities, state management DTOs, lightweight data transfer
    csharp 
    // ❌ CLASS — verbose, manual equality
public class PersonClass
{
    public string Name { get; init; }
    public int Age { get; init; }
    public override bool Equals(object? obj) { /* boilerplate */ }
    public override int GetHashCode() { /* boilerplate */ }
}

// ✅ RECORD — concise, automatic equality
public record PersonRecord(string Name, int Age);

var p1 = new PersonRecord("Alice", 30);
var p2 = new PersonRecord("Alice", 30);
Console.WriteLine(p1 == p2);  // true (structural equality)

// Automatic deconstruction
var (name, age) = p1;
Console.WriteLine($"{name}, {age}");  // Alice, 30

// Inheritance
public record EmployeeRecord(string Name, int Age, string Department)
    : PersonRecord(Name, Age);

    When to use records: DTOs, API responses, immutable data transfer objects. When to use classes: domain entities with complex behavior, mutable state.

    Q48. What is the difference between == and Equals() for string comparison? ▼

    Both == and Equals() compare string content (not reference). However, they have subtle differences in behavior and performance.

    csharp 
    string a = "Product";
string b = "Product";
string c = new string('P', 1) + "roduct";  // constructed dynamically

// Both work for strings, but different for other types
Console.WriteLine(a == b);           // true
Console.WriteLine(a.Equals(b));      // true
Console.WriteLine((object)a == (object)b);  // true (string interning)

// WHY? For strings, == operator is overloaded to call Equals()
// For other reference types, == checks reference identity

// Example with custom class
public class Product
{
    public int Id { get; set; }
    public string Name { get; set; }
}

var p1 = new Product { Id = 1, Name = "Laptop" };
var p2 = new Product { Id = 1, Name = "Laptop" };

Console.WriteLine(p1 == p2);      // false (different objects)
Console.WriteLine(p1.Equals(p2)); // false (no override)

// Override for structural equality
public class ProductFixed
{
    public int Id { get; set; }
    public string Name { get; set; }

    public override bool Equals(object? obj) =>
        obj is ProductFixed other && Id == other.Id && Name == other.Name;

    public override int GetHashCode() => HashCode.Combine(Id, Name);
}

var pf1 = new ProductFixed { Id = 1, Name = "Laptop" };
var pf2 = new ProductFixed { Id = 1, Name = "Laptop" };
Console.WriteLine(pf1.Equals(pf2)); // true

    Best practice: For strings, either works — use == for brevity. For custom types, override Equals() and GetHashCode() if you need structural equality. Better yet, use record types which do this automatically.

    Q49. What is reflection in C#? When would you use it? ▼

    Reflection is the ability to inspect and manipulate types, properties, methods, and fields at runtime. You can discover what a type contains without knowing it at compile time.

    csharp 
    public class Product
{
    public int Id { get; set; }
    public string Name { get; set; }
    public decimal Price { get; set; }
}

// Reflection: discovering type information at runtime
var type = typeof(Product);
Console.WriteLine($"Type: {type.Name}");

// Get all properties
var properties = type.GetProperties();
foreach (var prop in properties)
{
    Console.WriteLine($"  {prop.Name}: {prop.PropertyType.Name}");
}
// Output:
//   Id: Int32
//   Name: String
//   Price: Decimal

// Create instance using reflection
var instance = Activator.CreateInstance(type);
var idProp = type.GetProperty("Id");
idProp?.SetValue(instance, 1);

// Invoke method using reflection
var method = type.GetMethod("ToString");
var result = method?.Invoke(instance, null);
Console.WriteLine($"Result: {result}");

    Common uses of reflection:

    • Dependency Injection containers (discovering types that implement an interface)
    • ORM frameworks like Entity Framework (discovering properties for database mapping)
    • Serialization libraries (JSON, XML converters)
    • Unit testing frameworks (discovering test methods)
    • Validation frameworks (discovering validation attributes)

    ⚠️ Warning: Reflection is slow. Use it sparingly, typically once during application startup (not in loops). Cache reflection results when possible.

    Q50. What is a delegate? How does it relate to event? ▼

    Delegate is a type-safe function pointer — a type that represents "a method with a specific signature". Event is a wrapper around a delegate that controls who can subscribe/unsubscribe.

    csharp 
    // ─── DELEGATE: define a function signature ───
public delegate void ProductEventHandler(Product product);

// ─── CLASS that raises an event ───
public class ProductService
{
    // EVENT: controlled access to delegate
    public event ProductEventHandler? OnProductAdded;

    public void AddProduct(Product product)
    {
        // ... add product to database ...

        // Raise the event (only possible within this class)
        OnProductAdded?.Invoke(product);
    }
}

// ─── SUBSCRIBERS to the event ───
var service = new ProductService();

// Subscribe: += to register handler
service.OnProductAdded += (product) =>
{
    Console.WriteLine($"Product added: {product.Name}");
};

// Another subscriber
service.OnProductAdded += (product) =>
{
    Console.WriteLine($"Logging: {DateTime.Now} - {product.Name}");
};

// When AddProduct is called, both handlers run
service.AddProduct(new Product { Name = "Laptop", Price = 999m });

// Output:
// Product added: Laptop
// Logging: [timestamp] - Laptop

    Delegate vs Event:

    • Delegate: Just a type (function signature). Anyone can invoke it.
    • Event: Wrapper that restricts access. Only the class that owns it can invoke it. External code can only subscribe (+=) or unsubscribe (-=).

    Modern alternative: Use Action<T> and Func<T, TResult> instead of custom delegates.

    C# Modern Features (C# 9+)

    C# continues to evolve. Here are game-changing features introduced in recent versions:

    Records — Immutable Data Classes

    Records are perfect for DTOs and immutable data. They provide structural equality, pattern matching, and concise syntax.

    csharp 
    // ✅ RECORD — immutable, structural equality, pattern matching
public record ProductDto(int Id, string Name, decimal Price);

var product1 = new ProductDto(1, "Laptop", 999.99m);
var product2 = new ProductDto(1, "Laptop", 999.99m);

Console.WriteLine(product1 == product2);  // true — structural equality (same values)
// With a class, this would be false (different instances)

// Deconstruction
var (id, name, price) = product1;
Console.WriteLine($"{id}: {name} @ {price:C}");

// Inheritance — record can inherit from another record
public record DiscountedProduct(int Id, string Name, decimal Price, decimal Discount)
    : ProductDto(Id, Name, Price);

    Init-Only Properties

    Properties that can only be set during object initialization, ensuring immutability after construction.

    csharp 
    public class Product
{
    public int Id { get; init; }              // can only set during new
    public string Name { get; init; }
    public decimal Price { get; set; }        // regular property — can be set anytime
}

var product = new Product { Id = 1, Name = "Laptop", Price = 999m };
// product.Id = 2;        // ❌ error — cannot set init-only property
product.Price = 899m;     // ✅ ok — regular property can be set

    Nullable Reference Types

    Enable at the top of a file with #nullable enable to get compile-time warnings about null-related bugs.

    csharp 
    #nullable enable

public class Product
{
    public string Name { get; set; } = "";     // non-nullable — must have a value
    public string? Description { get; set; }   // nullable — can be null
}

public string GetProductInfo(Product? product)  // parameter can be null
{
    // ❌ Compiler warning — product might be null
    var name = product.Name;

    // ✅ OK — null-safe operator
    var name = product?.Name ?? "Unknown";
}

    Top-Level Statements

    Write code directly in a file without wrapping it in a class or Main method (useful for scripts and learning).

    csharp 
    // ✅ This is valid at the top level (no class/Main needed)
using System;

var products = new List<string> { "Laptop", "Mouse", "Monitor" };
Console.WriteLine($"Products: {string.Join(", ", products)}");

// The compiler automatically wraps this in Program.Main()

    File-Scoped Types

    Mark a class as visible only within its own file, reducing namespace pollution.

    csharp 
    // This type is ONLY visible in this file, hidden from other files
file class InternalProductMapper
{

        new(product.Id, product.Name, product.Price);
