📡 Apache Kafka

Distributed event streaming platform handling millions of messages per second. Think of it as a super-powered message queue on steroids.

💭

Real-World Analogy: Post Office

Mail arrives: Letters go to post office (Kafka), sorted by zip code (Topic). Multiple postal workers (Consumers) pick up letters from same zip (Consumer Group). Letters stored at post office for days if no one picks them up.

Unlike RabbitMQ: RabbitMQ is like a delivery service (deletes after delivery). Kafka is like an archive (stores forever for replay).

Why Kafka?

The Streaming Problem:

  • ❌ Your app generates 1 MILLION events per second
  • ❌ RabbitMQ starts struggling (bottleneck)
  • ❌ Need to replay events from yesterday (RabbitMQ deletes after delivery)
  • ❌ Want multiple services reading same stream simultaneously
  • ❌ Need disaster recovery (all messages backed up)

Key Concepts

1️⃣ Topics & Partitions

Topic: Category of events (like email folders)

Topic: "orders"          → All order events
Topic: "user-signups"    → User registration events
Topic: "payment-events"  → Payment processing events

Each Topic has Partitions (for parallel processing):
orders: Partition 0 [Event 1, 2, 3, 4]
orders: Partition 1 [Event 5, 6, 7, 8]
orders: Partition 2 [Event 9, 10, 11, 12]

✅ Partitions allow 3 consumers to read in parallel
✅ Same consumer always reads same partition (order guaranteed)

2️⃣ Producers (Event Sources)

Services publishing events to Kafka:

var producer = new KafkaProducer(new ProducerConfig
    { BootstrapServers = "localhost:9092" });

// Publish event
await producer.ProduceAsync("orders", new Message
{
    Key = "order-123",    // Route to partition based on key
    Value = "{orderId: 123, amount: 99.99}"
});

✅ Kafka stores this message immediately
✅ Other services can read it now and forever

3️⃣ Consumers (Event Readers)

Services reading events from Kafka:

var consumer = new KafkaConsumer(new ConsumerConfig
{
    BootstrapServers = "localhost:9092",
    GroupId = "email-service",  // Consumer group name
    AutoOffsetReset = AutoOffsetReset.Earliest
});

consumer.Subscribe(new[] { "orders" });

// Process messages
while (true)
{
    var message = consumer.Consume();
    Console.WriteLine($"Order: {message.Value}");

    // Send email, save to DB, etc

    consumer.Commit();  // Mark as processed
}

✅ Consumer remembers position (offset)
✅ Can restart and resume from where it left off
✅ Multiple consumers same topic = duplicate processing (by design)

4️⃣ Consumer Groups

Multiple consumers reading same topic independently:

Scenario: Order topic with 3 partitions

Consumer Group A (Email Service):
  - Consumer A1 reads Partition 0
  - Consumer A2 reads Partition 1
  - Consumer A3 reads Partition 2
  → Each consumer processes full event stream

Consumer Group B (Analytics):
  - Consumer B1 reads Partition 0
  - Consumer B2 reads Partition 1
  - Consumer B3 reads Partition 2
  → Gets ALL same events independently

Same order event processed by:
  ✅ Email service (send confirmation)
  ✅ Analytics (record metrics)
  ✅ Inventory (reserve stock)

Each consumer group maintains own position!

5️⃣ Retention & Replay

Kafka keeps messages for days (configurable):

Day 1:  100,000 orders published
Day 2:  200,000 more orders
Day 3:  New analytics service joins
        Reads from Day 1 (replays all messages!)
        Gets 300,000 historical orders

Or if system crashes:
        Old consumer restarted
        Reads from last offset (where it stopped)
        No lost messages!

Configuration:
retention.ms = 604800000  (7 days)
retention.bytes = 1099511627776  (1TB)

Code Example: .NET Kafka Producer & Consumer

Order Service - Kafka Producer

using Confluent.Kafka;

public class KafkaOrderService
{
    private readonly IProducer _producer;

    public KafkaOrderService()
    {
        var config = new ProducerConfig
        {
            BootstrapServers = "localhost:9092"
        };
        _producer = new ProducerBuilder(config).Build();
    }

    public async Task CreateOrder(Order order)
    {
        // Save to database
        var savedOrder = await _db.Orders.AddAsync(order);

        // Publish to Kafka
        var orderJson = JsonSerializer.Serialize(new
        {
            order.Id,
            order.CustomerId,
            order.TotalAmount,
            order.CreatedAt
        });

        var deliveryReport = await _producer.ProduceAsync(
            "orders",  // Topic
            new Message
            {
                Key = order.Id.ToString(),  // Partition key
                Value = orderJson
            }
        );

        Console.WriteLine($"Message delivered to {deliveryReport.TopicPartitionOffset}");
    }
}

Email Service - Kafka Consumer

using Confluent.Kafka;

public class KafkaEmailService : BackgroundService
{
    private readonly IConsumer _consumer;
    private readonly EmailService _emailService;

    public KafkaEmailService(EmailService emailService)
    {
        var config = new ConsumerConfig
        {
            BootstrapServers = "localhost:9092",
            GroupId = "email-service-group",
            AutoOffsetReset = AutoOffsetReset.Earliest,
            EnableAutoCommit = false
        };

        _consumer = new ConsumerBuilder(config).Build();
        _consumer.Subscribe(new[] { "orders" });
        _emailService = emailService;
    }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            try
            {
                var message = _consumer.Consume(stoppingToken);

                if (message == null) continue;

                // Parse event
                var orderData = JsonSerializer.Deserialize(message.Value);

                // Send email
                await _emailService.SendOrderConfirmationAsync(
                    orderData.CustomerId,
                    orderData.Id,
                    orderData.TotalAmount
                );

                // Mark as processed
                _consumer.Commit(message);

                Console.WriteLine($"Email sent for order {orderData.Id}");
            }
            catch (Exception ex)
            {
                Console.WriteLine($"Error: {ex.Message}");
                // Don't commit = message reprocessed on restart
            }
        }
    }

    public override void Dispose()
    {
        _consumer?.Dispose();
        base.Dispose();
    }
}

public class OrderEvent
{
    public string Id { get; set; }
    public string CustomerId { get; set; }
    public decimal TotalAmount { get; set; }
    public DateTime CreatedAt { get; set; }
}

Best Practices

✅ Key-Based Partitioning

Use consistent key so related events go to same partition. Maintains order per key.

✅ Consumer Groups

Multiple independent consumers = same topic gets processed multiple ways.

✅ Offset Management

Commit offsets only after successful processing. Prevents message loss on crash.

✅ Topic Naming Convention

Use clear names: "order-events", "user-signups" not "topic1", "data"

✅ Schema Validation

Use Avro or Protobuf for event schemas. Prevent corrupted messages.

✅ Monitoring

Track consumer lag (how far behind vs latest message).

Common Mistakes

⚠️

❌ Mistake: Not Committing Offsets

If you don't commit, consumer restarts and reprocesses all messages. Either double-process or lose data!

⚠️

❌ Mistake: Using Same Group ID for Different Services

Each service should have own group ID. Same group = they share partitions (only one processes message).

⚠️

❌ Mistake: Ignoring Partition Key

No key = random partition. Events for same order go different partitions = out of order!

Kafka vs RabbitMQ (Quick Comparison)

KAFKA                           RABBITMQ
────────────────────────────────────────────────────
✅ High throughput (1M msgs)    ✅ Low latency
✅ Stores messages (replay)     ✅ Deletes after delivery
✅ Consumer groups (elegant)    ✅ Message routing (complex)
✅ Streaming/Big data           ✅ Task queues
✅ Partition order              ✅ Direct service calls

Use Kafka for:                Use RabbitMQ for:
- Analytics pipelines         - Job queues
- Event sourcing             - RPC calls
- Audit logs                 - Simple pub/sub
- Million+ events/sec        - Lightweight messaging

Interview Q&A

Q1: How is Kafka different from RabbitMQ?

A: Core differences:

Message Retention: Kafka keeps for days, RabbitMQ deletes after delivery

Throughput: Kafka built for millions/sec, RabbitMQ for thousands/sec

Consumer Groups: Kafka has elegant consumer groups, RabbitMQ has exchange/binding complexity

Replay: Kafka can replay messages, RabbitMQ can't

Use Case: Kafka for streaming/analytics, RabbitMQ for task queues

Q2: What happens if consumer crashes in Kafka?

A: Two scenarios:

If consumer committed offset before crash:

  • Consumer restarts
  • Reads last committed offset
  • Only processes new messages (after crash)
  • No message loss, no reprocessing

If consumer crashed before committing:

  • Consumer restarts
  • Message wasn't marked as processed
  • Reprocesses from before crash
  • Risk of duplicate processing (must be idempotent!)

Q3: How do partitions ensure order?

A: Only within partition!

Order is guaranteed: Messages in same partition processed sequentially by single consumer

Order is NOT guaranteed: Messages in different partitions can be out of order

Solution: Use consistent key (order ID) so related messages go to same partition

Example: Order updates
Key = "order-123" → Always goes to Partition 0
  Message 1: Order created
  Message 2: Payment processed
  Message 3: Shipped
  ✅ Processed in order by same consumer

Key = "order-456" → Always goes to Partition 1
  Messages processed by different consumer, own order
