🔀 Git & Version Control - CI/CD Pipeline

Collaborate on code, review changes safely, and auto-deploy when you push. Git + GitHub Actions = automated production deployments.

⏱️

Real-World Analogy: Time Machine + Team Collaboration

❌ Without Git: 10 developers editing same code file, emailing each other versions, Alice overwrites Bob's changes, app breaks.

✅ With Git: Each developer has own branch. Alice's changes in feature/product-filter, Bob's in feature/add-sorting. Merge branches only after code review. Full history = go back to any moment.

Why Version Control?

Problems Without Git:

  • ❌ Lost code: Deleted file by mistake? No backup. Lost forever.
  • ❌ Conflicts: Alice && Bob both modify same file → Who wins? Manual merge nightmare
  • ❌ No history: Code broke yesterday. What changed? Guess and pray
  • ❌ No rollback: Production broken by latest deploy. Can't revert to yesterday's working version
  • ❌ No code review: Anyone pushes anything → Bad code reaches production

Key Concepts

1️⃣ Commits (Snapshots in Time)

Commit = "Save game" checkpoint with message explaining change

git add ProductService.cs
git commit -m "feat: add product filter by category"
# Creates: Commit hash abcd1234 with timestamp, author, message

Good commit: One logical change (not "fixed stuff" + "refactored" + "updated").

Example history:

abcd1234 feat: add product filter by category
xyz5678  fix: handle null category ids
klm9101  test: add unit tests for filter
→ Each commit is reversible, reviewable, testable

2️⃣ Branches (Parallel Development)

Branch = separate copy of code (independent change)

# Alice works on new feature
git checkout -b feature/product-filter
# Bob works on bug fix independently
git checkout -b bugfix/null-reference

# Both can commit to their branches without interfering
# Later: Merge when both complete

Why? Main branch stays stable. Features developed separately. Only merge when tested.

3️⃣ Pull Requests (Code Review)

PR = "Can I merge my feature into main?"

# Alice pushes feature branch
git push origin feature/product-filter

# Creates Pull Request on GitHub
# → Bob reviews code: "Looks good! +1"
# → CI/CD pipeline runs tests automatically
# → All pass? → Merge button enabled
# → Alice clicks merge → Code integrated to main

Benefits: Code review catches bugs, knowledge sharing, audit trail.

4️⃣ CI/CD Pipeline (Automated Testing & Deployment)

On every push: Auto-test, auto-build, auto-deploy

# .github/workflows/deploy.yml
name: CI/CD Pipeline
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Run tests
        run: dotnet test
      - name: Build Docker image
        run: docker build -t productcatalogapi:latest .
      - name: Push to registry
        run: docker push myregistry.azurecr.io/productcatalogapi:latest
      - name: Deploy to Kubernetes
        run: kubectl apply -f k8s-deployment.yaml

Result: Push code → Tests pass → Build image → Deploy to production (all automated!)

5️⃣ Tags (Release Markers)

Tag = label on specific commit (version marker)

# Mark a release
git tag -a v1.2.0 -m "Release version 1.2.0 - Added product filter"
git push origin v1.2.0

# Later: If v1.2.0 has bug → Deploy v1.2.1 instead
# Can always check out: git checkout v1.2.0 → Exact code from that release

Why? Track which code is in production. Audit trail. Quick rollback to previous version.

Complete Git Workflow (Team Scenario)

== DEVELOPER: Alice ==
# 1. Create feature branch
git checkout -b feature/add-product-filter

# 2. Make changes
# Edit ProductService.cs, ProductController.cs
git add ProductService.cs ProductController.cs
git commit -m "feat: add product filter by category

- Added ProductService.filter(categoryId) method
- Updated ProductController.GetProducts() endpoint
- Supports null categoryId (returns all products)"

# 3. Push to remote
git push origin feature/add-product-filter

== GITHUB: Pull Request ==
# Alice creates PR on GitHub
# Title: "feat: add product filter by category"
# Description: "Users can now filter products by category from the UI"

# Link issues: Closes #123 (issue tracker)

== CI/CD PIPELINE: Automatic ==
# GitHub Actions triggered:
✓ Linting ✓ Unit tests ✓ Integration tests ✓ Build Docker image
# All pass → "All checks passed"

== CODE REVIEW: Bob ==
# Bob reviews PR
# Comments: "Nice! Have you considered edge case when categoryId is 0?"
# Alice commits fix: git commit -m "test: add edge case for categoryId == 0"
# Bob approves: "Looks great! 👍"

== DEPLOYMENT: Automatic ==
# Alice clicks "Merge pull request"
git merge feature/add-product-filter → main

# GitHub Actions triggered again:
✓ Tests pass ✓ Build image: productcatalogapi:v1.2.1
✓ Push to registry: myregistry.azurecr.io/productcatalogapi:v1.2.1
✓ Update Kubernetes: kubectl apply -f deployment.yaml (image: v1.2.1)

# Result: ProductCatalogAPI v1.2.1 deployed to production
# Zero downtime rolling update (Kubernetes handles gradual Pod replacement)

Branching Strategies

Git Flow (Traditional) 
main (releases)
  ↑ merge release
release/ (release candidates)
  ↑ merge develop
develop (next version)
  ↑ merge feature
feature/xxx (features)

Good for: Complex apps with scheduled releases

GitHub Flow (Modern) 
main (always deployable)
  ↑ merge feature
feature/xxx (features)

Good for: Continuous deployment, web apps, startups

Trunk-Based (Advanced) 
main (everyone commits)
All commits to main
Feature flags hide incomplete features

Good for: High-throughput teams, microservices

GitOps: Kubernetes Deployment from Git

Super advanced: Git becomes source of truth for infrastructure

Developer commits to Git
  ↓
Kubernetes manifests updated in Git repo
  ↓
ArgoCD watches the Git repo
  ↓
ArgoCD detects change: "manifests.yaml changed from v1.1 to v1.2"
  ↓
ArgoCD auto-syncs: kubectl apply -f manifests.yaml
  ↓
Kubernetes cluster updated
  ↓
ProductCatalogAPI running v1.2 in production

Result: Git = single source of truth for entire system

Best Practices

  • ✅ Small focused commits: One feature/fix per commit. Easier to review, revert if needed
  • ✅ Meaningful commit messages: "fix: handle null product ids" > "fixed stuff"
  • ✅ .gitignore everything sensitive: bin/, obj/, .env, appsettings.Secrets.json
  • ✅ Branch protection on main: Require code review + tests pass before merge
  • ✅ Tag releases: v1.2.0, v1.3.0. Easy to track versions
  • ✅ Atomic commits: Commit only compiles = safe to revert

Common Mistakes

  • ❌ Large feature branches: Develop for 2 weeks → Merge conflicts nightmare. Merge daily instead
  • ❌ Committing secrets: API keys, passwords in code → Visible forever in history. Use environment variables!
  • ❌ Bad commit messages: "asdf", "fixed", "updated" → Impossible to understand history. Use conventional commits: feat:, fix:, test:
  • ❌ Force push to main: NEVER. Rewrites history, everyone confused. Only amend unpushed commits
  • ❌ Merging without tests: Tests haven't run → Broken code reaches production. Require CI/CD pass!

Interview Q&A

Q: Why use branches instead of committing directly to main?

A: Safety + code review + parallel development. Main stays stable. Features developed independently. Only merge proven code.

Q: What's difference between merge and rebase?

A: Merge = creates merge commit (preserves history, non-linear). Rebase = replays commits on top (linear history). Use merge for shared branches, rebase for local cleanup.

Q: How do you handle merge conflicts?

A: Edit file, mark both sections as resolved, test, commit. Or use git mergetool. Or abort (git merge --abort) and rebase instead.
