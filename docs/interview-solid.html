    📐 SOLID Design Principles

    5 principles to write better code. All explained with the simplest possible example: NotificationService sending emails, SMS, and push notifications.

    Advanced Architecture 5 Principles
    💡

    What are SOLID Principles?

    5 rules that help you write clean code. Follow them = easy to test, easy to change, easy to reuse. Ignore them = code becomes a nightmare to maintain.

    SOLID At A Glance

    Letter Principle Simple Rule
    SSingle ResponsibilityOne class = one job
    OOpen/ClosedExtend without editing
    LLiskov SubstitutionSubclass = parent works
    IInterface SegregationSmall, focused interfaces
    DDependency InversionDepend on abstractions

    S — Single Responsibility Principle

    Rule: A class should have ONE reason to change.

    ❌ BAD Example: One Class Does Everything

    // ❌ BAD: NotificationService does 3 things
public class NotificationService
{
    // Reason to change #1: Email provider changes
    public void SendEmail(string to, string subject, string body)
    {
        using var smtp = new SmtpClient("smtp.gmail.com");
        var mail = new MailMessage("app@company.com", to)
        {
            Subject = subject,
            Body = body
        };
        smtp.Send(mail);
    }

    // Reason to change #2: SMS provider changes
    public void SendSms(string phone, string message)
    {
        Console.WriteLine($"SMS: {message}");
        // Call Twilio API...
    }

    // Reason to change #3: Push notification provider changes
    public void SendPush(string deviceId, string title, string message)
    {
        Console.WriteLine($"Push: {title}");
        // Call Firebase API...
    }
}

// PROBLEMS:
// ❌ 3 REASONS TO CHANGE: Gmail issue, Twilio issue, Firebase issue
// ❌ Testing email = must mock SMS + Push
// ❌ Huge, unmaintainable class

    ✅ GOOD Example: One Class Per Responsibility

    // ✓ GOOD: Split into 3 focused classes

// #1 Email responsibility ONLY
public interface IEmailService
{
    void SendEmail(string to, string subject, string body);
}
public class EmailService : IEmailService
{
    public void SendEmail(string to, string subject, string body)
    {
        using var smtp = new SmtpClient("smtp.gmail.com");
        var mail = new MailMessage("app@company.com", to)
        {
            Subject = subject,
            Body = body
        };
        smtp.Send(mail);
        Console.WriteLine("✓ Email sent");
    }
}

// #2 SMS responsibility ONLY
public interface ISmsService
{
    void SendSms(string phone, string message);
}
public class SmsService : ISmsService
{
    public void SendSms(string phone, string message)
    {
        Console.WriteLine($"✓ SMS sent: {message}");
        // Twilio API call...
    }
}

// #3 Push responsibility ONLY
public interface IPushService
{
    void SendPush(string deviceId, string title, string message);
}
public class PushService : IPushService
{
    public void SendPush(string deviceId, string title, string message)
    {
        Console.WriteLine($"✓ Push sent: {title}");
        // Firebase API call...
    }
}

// BENEFITS:
// ✓ EmailService has 1 reason to change: Gmail changes
// ✓ SmsService has 1 reason to change: Twilio changes
// ✓ PushService has 1 reason to change: Firebase changes
// ✓ Test email WITHOUT SMS or Push
// ✓ Swap Gmail for SendGrid = edit ONLY EmailService
    ✓

    SRP in 1 Sentence

    Instead of one class doing email+SMS+push, have three classes: EmailService, SmsService, PushService. Each has one job.

    O — Open/Closed Principle

    Rule: Open for EXTENSION, closed for MODIFICATION. Add new features without editing old code.

    ❌ BAD Example: If/Else Explosion

    // ❌ BAD: Adding notification type = editing this class
public class Notifier
{
    public void Send(string type, string recipient, string message)
    {
        if (type == "email")
            Console.WriteLine($"Email to {recipient}: {message}");
        else if (type == "sms")
            Console.WriteLine($"SMS to {recipient}: {message}");
        else if (type == "push")
            Console.WriteLine($"Push to {recipient}: {message}");
        // Adding WhatsApp? Edit this method!
        // Adding Slack? Edit this method!
    }
}

// PROBLEMS:
// ❌ Every new type requires MODIFYING Notifier
// ❌ Risk of breaking email or SMS
// ❌ Not extensible

    ✅ GOOD Example: Use Interfaces (Strategy Pattern)

    // ✓ GOOD: Extend without modifying

// Base contract
public interface INotificationChannel
{
    void Send(string recipient, string message);
}

// Email implementation
public class EmailChannel : INotificationChannel
{
    public void Send(string recipient, string message)
    {
        Console.WriteLine($"✓ Email to {recipient}: {message}");
    }
}

// SMS implementation
public class SmsChannel : INotificationChannel
{
    public void Send(string recipient, string message)
    {
        Console.WriteLine($"✓ SMS to {recipient}: {message}");
    }
}

// Push implementation
public class PushChannel : INotificationChannel
{
    public void Send(string recipient, string message)
    {
        Console.WriteLine($"✓ Push to {recipient}: {message}");
    }
}

// ADDING NEW CHANNEL: NO EDITING!
public class WhatsAppChannel : INotificationChannel
{
    public void Send(string recipient, string message)
    {
        Console.WriteLine($"✓ WhatsApp to {recipient}: {message}");
    }
}

// Notifier just uses the interface
public class Notifier
{
    private readonly INotificationChannel _channel;

    public Notifier(INotificationChannel channel)
    {
        _channel = channel;
    }

    public void Send(string recipient, string message)
    {
        _channel.Send(recipient, message);
    }
}

// BENEFITS:
// ✓ Add WhatsApp = create WhatsAppChannel
// ✓ No modifications to Notifier
// ✓ Closed for modification, open for extension!
    ✓

    OCP in 1 Sentence

    Use interfaces. To add a new notification type, create a new class implementing the interface — don't edit existing code.

    L — Liskov Substitution Principle

    Rule: A subclass must work wherever the parent is expected. No surprises.

    ❌ BAD Example: Subclass Breaks Contract

    // ❌ BAD: EmailChannel breaks contract
public interface INotificationChannel
{
    void Send(string recipient, string message);
}

public class EmailChannel : INotificationChannel
{
    public void Send(string recipient, string message)
    {
        Console.WriteLine($"Sending email to {recipient}");
    }
}

// ❌ PROBLEM: CachedChannel pretends to send but doesn't
public class CachedChannel : INotificationChannel
{
    public void Send(string recipient, string message)
    {
        // ❌ It just logs, doesn't actually send!
        Console.WriteLine("Cached (not sent)");
        throw new NotImplementedException("Cached channel doesn't send");
    }
}

// Code expects all INotificationChannel to send
public void ProcessNotifications(INotificationChannel channel)
{
    channel.Send("user@example.com", "Hello");
    // CRASH if CachedChannel!
}

// PROBLEMS:
// ❌ Caller expects Send() to work, but it throws exception
// ❌ LSP violated: CachedChannel is NOT substitutable for INotificationChannel

    ✅ GOOD Example: Honor the Contract

    // ✓ GOOD: Separate interface for different needs

// Contract: MUST actually send
public interface INotificationChannel
{
    void Send(string recipient, string message);
}

// Contract: For storage/caching only
public interface INotificationLogger
{
    void Log(string recipient, string message);
}

// Email SENDS
public class EmailChannel : INotificationChannel
{
    public void Send(string recipient, string message)
    {
        Console.WriteLine($"✓ Sending email to {recipient}");
    }
}

// Cache ONLY LOGS
public class CacheLogger : INotificationLogger
{
    public void Log(string recipient, string message)
    {
        Console.WriteLine($"✓ Cached: {recipient}");
    }
}

// Now safe and clear
public void ProcessNotifications(INotificationChannel channel)
{
    channel.Send("user@example.com", "Hello");
    // Safe! All INotificationChannel implementations actually send
}

public void LogNotifications(INotificationLogger logger)
{
    logger.Log("user@example.com", "Hello");
    // Safe! All INotificationLogger implementations log
}

// BENEFITS:
// ✓ No surprise exceptions
// ✓ Caller knows what to expect
// ✓ Safe polymorphism
    ✓

    LSP in 1 Sentence

    If you pass EmailChannel where INotificationChannel is expected, it works perfectly. Never throw exceptions or do unexpected things.

    I — Interface Segregation Principle

    Rule: Classes should not be forced to implement methods they don't use.

    ❌ BAD Example: Fat Interface

    // ❌ BAD: One interface, too many methods
public interface INotificationService
{
    void SendEmail(string to, string subject, string body);
    void SendSms(string phone, string message);
    void SendPush(string deviceId, string title, string message);
    void LogNotification(string recipient, string type);
    void RetryFailedNotification(int notificationId);
    void ArchiveNotification(int notificationId);
}

// ❌ BasicEmailSender must implement ALL 6 methods
public class BasicEmailSender : INotificationService
{
    public void SendEmail(string to, string subject, string body)
    {
        Console.WriteLine("Sending email...");
    }

    // ❌ Forced to implement SMS (it doesn't send SMS!)
    public void SendSms(string phone, string message)
    {
        throw new NotImplementedException();
    }

    // ❌ Forced to implement Push (it doesn't send push!)
    public void SendPush(string deviceId, string title, string message)
    {
        throw new NotImplementedException();
    }

    // ... 3 more not-implemented methods!
}

// PROBLEMS:
// ❌ Fat interface = lots of NotImplementedException
// ❌ Developer confused: which methods actually work?
// ❌ ISP violated

    ✅ GOOD Example: Segregated Interfaces

    // ✓ GOOD: Small, focused interfaces

// Senders
public interface IEmailSender
{
    void SendEmail(string to, string subject, string body);
}

public interface ISmsSender
{
    void SendSms(string phone, string message);
}

public interface IPushSender
{
    void SendPush(string deviceId, string title, string message);
}

// Storage & Logging
public interface INotificationRepository
{
    void Save(Notification notification);
    void Retry(int notificationId);
    void Archive(int notificationId);
}

// Now BasicEmailSender implements ONLY what it needs
public class BasicEmailSender : IEmailSender
{
    public void SendEmail(string to, string subject, string body)
    {
        Console.WriteLine("✓ Email sent");
    }
}

// AdvancedService can implement multiple
public class AdvancedService : IEmailSender, ISmsSender, IPushSender, INotificationRepository
{
    public void SendEmail(string to, string subject, string body) { /* ... */ }
    public void SendSms(string phone, string message) { /* ... */ }
    public void SendPush(string deviceId, string title, string message) { /* ... */ }
    public void Save(Notification n) { /* ... */ }
    public void Retry(int id) { /* ... */ }
    public void Archive(int id) { /* ... */ }
}

// BENEFITS:
// ✓ BasicEmailSender only implements what it needs
// ✓ No dummy NotImplementedException methods
// ✓ Clear contracts
// ✓ Type-safe code
    ✓

    ISP in 1 Sentence

    Split fat interfaces into small ones. EmailSender implements IEmailSender only (not SMS or Push).

    D — Dependency Inversion Principle

    Rule: Depend on abstractions, not concrete implementations. Inject dependencies.

    ❌ BAD Example: Tight Coupling

    // ❌ BAD: UserController creates EmailService directly
public class UserController
{
    private readonly EmailService _emailService;

    public UserController()
    {
        _emailService = new EmailService(); // ❌ Tight coupling!
    }

    public void RegisterUser(User user)
    {
        // ... save user ...
        _emailService.SendEmail(user.Email, "Welcome", "Hi!");
    }
}

// PROBLEMS:
// ❌ UserController depends on concrete EmailService
// ❌ Can't test without real email (slow, fragile)
// ❌ Can't swap EmailService for a mock or test double
// ❌ Change EmailService constructor? Edit UserController!

    ✅ GOOD Example: Dependency Injection

    // ✓ GOOD: Inject the dependency

// UserController depends on INTERFACE, not concrete class
public class UserController
{
    private readonly IEmailService _emailService;

    // Inject interface via constructor
    public UserController(IEmailService emailService)
    {
        _emailService = emailService;
    }

    public void RegisterUser(User user)
    {
        // ... save user ...
        _emailService.SendEmail(user.Email, "Welcome", "Hi!");
    }
}

// Real implementation
public class EmailService : IEmailService
{
    public void SendEmail(string to, string subject, string body)
    {
        Console.WriteLine($"Sending email to {to}");
    }
}

// NOW TESTING IS EASY!
public class UserControllerTests
{
    [Test]
    public void RegisterUser_Should_SendEmail()
    {
        // Mock the email service
        var mockEmail = new Mock();

        // Inject mock
        var controller = new UserController(mockEmail.Object);

        // Call the method
        controller.RegisterUser(new User { Email = "test@example.com" });

        // Verify email was sent
        mockEmail.Verify(x => x.SendEmail("test@example.com", "Welcome", "Hi!"), Times.Once);
    }
}

// BENEFITS:
// ✓ No real emails during testing
// ✓ Test is fast and isolated
// ✓ Easy to swap EmailService for SendgridService
// ✓ UserController doesn't know (or care) about implementation details
    ✓

    DIP in 1 Sentence

    Inject IEmailService via constructor, not new EmailService(). This lets you test with mocks and swap implementations.

    SOLID Summary

    Principle Problem Solution
    S One class does email+SMS+push Three classes: EmailService, SmsService, PushService
    O Adding feature = editing existing code Use interfaces, add new implementation
    L Subclass throws exceptions, breaks contract Separate interfaces for different needs
    I Fat interface with NotImplementedException Split into focused interfaces
    D Tight coupling, hard to test Inject interfaces via constructor
