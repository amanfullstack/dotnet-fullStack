    📐 SOLID Design Principles

    Master the 5 SOLID principles with real-world code examples from ProductCatalogAPI. Build maintainable, scalable, and professional code.

    Advanced Architecture 5 Principles
    💡

    What is SOLID?

    SOLID is an acronym for 5 design principles that help create clean, maintainable, and extensible code. Violating SOLID leads to rigid code that breaks easily. Following SOLID makes code flexible, testable, and professional — exactly what senior developers do.

    S — Single Responsibility Principle (SRP)

    Definition: A class should have one, and only one, reason to change. Each class should have a single responsibility.

    ❌ Problem: Violating SRP

    When a class handles multiple responsibilities, it becomes difficult to test, maintain, and modify:

    csharp 
    // BAD: ProductManager violates SRP — does validation, persistence, AND notifications
public class ProductManager
{
    private readonly string _connString;

    public void AddProduct(Product product)
    {
        // 1. VALIDATION — one responsibility
        if (string.IsNullOrWhiteSpace(product.Name))
            throw new ArgumentException("Product name is required");
        if (product.Price <= 0)
            throw new ArgumentException("Price must be positive");

        // 2. PERSISTENCE — another responsibility
        using (var conn = new SqlConnection(_connString))
        {
            var cmd = new SqlCommand("INSERT INTO Products (Name, Price) VALUES (@Name, @Price)", conn);
            cmd.Parameters.AddWithValue("@Name", product.Name);
            cmd.Parameters.AddWithValue("@Price", product.Price);
            conn.Open();
            cmd.ExecuteNonQuery();
        }

        // 3. NOTIFICATION — yet another responsibility
        var client = new SmtpClient("smtp.gmail.com");
        client.Send("admin@company.com", "admin@company.com", "New Product Added", product.Name);
    }
}

// Problems:
// - Testing requires real DB connection + real email server
// - Changing email template requires modifying ProductManager
// - Class has 3 reasons to change (validation logic, DB schema, email format)

    ✅ Solution: Separate into Focused Classes

    Split the monolithic class into three focused classes, each with ONE reason to change:

    csharp 
    // Responsibility 1: VALIDATION
public interface IProductValidator
{
    void Validate(Product product);
}

public class ProductValidator : IProductValidator
{
    public void Validate(Product product)
    {
        if (string.IsNullOrWhiteSpace(product.Name))
            throw new ArgumentException("Product name is required");
        if (product.Price <= 0)
            throw new ArgumentException("Price must be positive");
    }
}

// Responsibility 2: PERSISTENCE
public interface IProductRepository
{
    Task SaveAsync(Product product);
}

public class SqlProductRepository : IProductRepository
{
    private readonly string _connectionString;

    public SqlProductRepository(IConfiguration config)
    {
        _connectionString = config.GetConnectionString("DefaultConnection")!;
    }

    public async Task SaveAsync(Product product)
    {
        using var conn = new SqlConnection(_connectionString);
        var cmd = new SqlCommand("INSERT INTO Products (Name, Price) VALUES (@Name, @Price)", conn);
        cmd.Parameters.AddWithValue("@Name", product.Name);
        cmd.Parameters.AddWithValue("@Price", product.Price);
        await conn.OpenAsync();
        await cmd.ExecuteNonQueryAsync();
    }
}

// Responsibility 3: NOTIFICATION
public interface IEmailService
{
    Task SendAsync(string to, string subject, string body);
}

public class EmailService : IEmailService
{
    public async Task SendAsync(string to, string subject, string body)
    {
        using var client = new SmtpClient("smtp.gmail.com");
        var mail = new MailMessage("admin@company.com", to) { Subject = subject, Body = body };
        await client.SendMailAsync(mail);
    }
}

// ORCHESTRATOR: ProductService - remains simple, coordinates responsibilities
public class ProductService
{
    private readonly IProductValidator _validator;
    private readonly IProductRepository _repository;
    private readonly IEmailService _emailService;

    public ProductService(IProductValidator validator, IProductRepository repository, IEmailService emailService)
    {
        _validator = validator;
        _repository = repository;
        _emailService = emailService;
    }

    public async Task AddProductAsync(Product product, string notificationEmail)
    {
        _validator.Validate(product);
        await _repository.SaveAsync(product);
        await _emailService.SendAsync(notificationEmail, "New Product Added", $"Product {product.Name} added successfully");
    }
}

// BENEFITS:
// ✓ Each class has ONE reason to change
// ✓ ProductValidator can be tested without DB or Email
// ✓ SqlProductRepository can be swapped for EfProductRepository
// ✓ EmailService can be swapped for SendgridEmailService
// ✓ ProductService logic is clear and orchestrates the workflow
// ✓ High cohesion, loose coupling

    Interview Questions on SOLID

    Q1. Explain Single Responsibility Principle with a real example from ProductCatalogAPI. ▼

    Answer: SRP states that a class should have only one reason to change. In ProductCatalogAPI, ProductService should ONLY contain business logic for managing products. Database access (SqlProductRepository), validation (ProductValidator), and email notifications (EmailService) should be separate classes.

    Why it matters: If you need to change how emails are sent, you modify EmailService — not ProductService. Each class is independently testable and reusable. If ProductService does validation, persistence, AND notification, it has three reasons to change and becomes unmaintainable.

    Q2. How does Open/Closed Principle help when adding new payment methods? ▼

    Answer: OCP says code should be OPEN for extension but CLOSED for modification. If we have a giant PaymentProcessor with if/else for each payment type, adding Bitcoin requires modifying existing code — violating OCP.

    Solution: Use Strategy pattern. Create an IPaymentStrategy interface. Each payment type (CreditCard, PayPal, Bitcoin) implements it. PaymentProcessor registers strategies. Adding Bitcoin = create BitcoinPayment class implementing IPaymentStrategy — no modification to existing code.

    Q3. Give an example of violating Liskov Substitution Principle. ▼

    Classic example: Bird base class with Fly() method. Sparrow extends Bird and implements Fly(). But Ostrich extends Bird — it can't fly! If we call bird.Fly() on an Ostrich, it throws NotSupportedException. This violates LSP because Ostrich is not safely substitutable for Bird.

    Fix: Don't put Ostrich in a Bird hierarchy that requires flying. Use interface segregation: IFlyingBird interface that Sparrow implements but Ostrich doesn't.

    Q4. Why is Interface Segregation important for maintainability? ▼

    Answer: ISP says clients shouldn't be forced to depend on methods they don't use. If IMachine has Print(), Scan(), Fax(), a SimplePrinter must implement all three — even though it can only print. If someone calls Scan(), it throws NotImplementedException.

    Solution: Segregate into focused interfaces: IPrinter, IScanner, IFaxMachine. SimplePrinter only implements IPrinter. MultifunctionPrinter implements all three. Now client code is type-safe — if you have IPrinter, you KNOW Print() works.

    Q5. How does Dependency Inversion enable unit testing? ▼

    Problem: If ProductService instantiates FileLogger directly (new FileLogger()), unit tests create actual files on disk. Test is slow, fragile, and depends on file system.

    Solution (DIP): Inject ILogger interface via constructor. Test passes a mock ILogger that does nothing. No files created. Test runs instantly and is isolated.

    Code: public ProductService(ILogger logger) { _logger = logger; } Test: var mockLogger = new Mock<ILogger>(); var service = new ProductService(mockLogger.Object);

    Q6. How do SOLID principles work together? ▼

    SOLID principles are interconnected:

    • SRP + DIP: When you split ProductManager into focused classes (SRP), you naturally end up injecting their interfaces (DIP)
    • OCP + LSP: OCP uses Strategy pattern — all strategies implement IPaymentStrategy. LSP ensures every strategy honors the contract
    • ISP + SRP: Segregating interfaces (IPrinter, IScanner) keeps classes focused on single responsibility
    • DIP + OCP: Depending on abstractions lets you swap implementations without modifying code — new payment type doesn't require changing PaymentProcessor
