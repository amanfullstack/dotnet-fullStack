    📐 SOLID Design Principles

    All 5 SOLID principles explained using one consistent scenario: an E-Commerce Order System. Every principle shows exactly what goes wrong (BAD) and how to fix it (GOOD).

    Advanced Architecture 5 Principles
    🛒

    Scenario: E-Commerce Order System

    All 5 principles use the same codebase: placing an order, calculating discounts, processing payment, sending notifications, and accessing order data. Watch how each SOLID violation breaks the system — and how applying the principle fixes it cleanly.

    SOLID at a Glance

    S — Single Responsibility Principle

    Definition: A class should have one, and only one, reason to change. Split concerns into separate classes.

    ❌ BAD: Multiple Responsibilities in One Class

    OrderProcessor does validation, discount calculation, payment, email, AND database storage. Six reasons to change = guaranteed to break!

    csharp 
    // ❌ BAD: One class, six responsibilities — MESS!
public class OrderProcessor
{
    private readonly string _connString;

    public void ProcessOrder(Order order, Customer customer)
    {
        // Responsibility 1: VALIDATION
        if (order.Items.Count == 0) throw new Exception("Empty order");
        if (customer.Email == null) throw new Exception("No email");

        // Responsibility 2: DISCOUNT CALCULATION
        decimal discount = customer.IsVip ? 0.10m : order.Total > 1000 ? 0.05m : 0m;
        order.Total -= order.Total * discount;

        // Responsibility 3: PAYMENT PROCESSING
        using var http = new HttpClient();
        var result = http.PostAsync("https://api.payment.com/charge",
            new StringContent(order.Total.ToString())).Result;
        if (!result.IsSuccessStatusCode) throw new Exception("Payment failed");

        // Responsibility 4: EMAIL NOTIFICATION
        using var smtp = new SmtpClient("smtp.gmail.com");
        var mail = new MailMessage("orders@store.com", customer.Email);
        mail.Subject = "Order Confirmed";
        smtp.Send(mail);

        // Responsibility 5: DATABASE STORAGE
        using var conn = new SqlConnection(_connString);
        var cmd = new SqlCommand(
            "INSERT INTO Orders (CustomerId, Total) VALUES (@id, @total)", conn);
        cmd.Parameters.AddWithValue("@id", customer.Id);
        cmd.Parameters.AddWithValue("@total", order.Total);
        conn.Open();
        cmd.ExecuteNonQuery();

        // Responsibility 6: LOGGING
        File.AppendAllText("orders.log", $"Order {order.Id} processed\n");
    }
}

// PROBLEMS:
// ❌ Test needs: real DB, real email server, real payment API, real file system
// ❌ Change discount logic? Edit ProcessOrder (risky, might break payment!)
// ❌ Change email format? Edit ProcessOrder (risky, might break validation!)
// ❌ 6 reasons to change = class will break constantly

    ✅ GOOD: One Responsibility Per Class

    Split into 6 focused classes. Each has ONE job. Each testable independently.

    csharp 
    // ✓ GOOD: Each class has ONE responsibility

// 1. VALIDATION ONLY
public interface IOrderValidator
{
    void Validate(Order order, Customer customer);
}

public class OrderValidator : IOrderValidator
{
    public void Validate(Order order, Customer customer)
    {
        if (order.Items.Count == 0)
            throw new ArgumentException("Order must contain items");
        if (customer.Email == null)
            throw new ArgumentException("Email required");
    }
}

// 2. DISCOUNT ONLY
public interface IDiscountCalculator
{
    decimal Calculate(Order order, Customer customer);
}

public class DiscountCalculator : IDiscountCalculator
{
    public decimal Calculate(Order order, Customer customer)
    {
        return customer.IsVip ? 0.10m : order.Total > 1000 ? 0.05m : 0m;
    }
}

// 3. PAYMENT ONLY
public interface IPaymentService
{
    Task<bool> ChargeAsync(decimal amount);
}

public class PaymentService : IPaymentService
{
    private readonly HttpClient _http;
    public PaymentService(HttpClient http) => _http = http;

    public async Task<bool> ChargeAsync(decimal amount)
    {
        var result = await _http.PostAsync("https://api.payment.com/charge",
            new StringContent(amount.ToString()));
        return result.IsSuccessStatusCode;
    }
}

// 4. EMAIL ONLY
public interface IEmailService
{
    Task SendAsync(string to, string subject, string body);
}

public class EmailService : IEmailService
{
    public async Task SendAsync(string to, string subject, string body)
    {
        using var smtp = new SmtpClient("smtp.gmail.com");
        var mail = new MailMessage("orders@store.com", to)
        {
            Subject = subject,
            Body = body
        };
        await smtp.SendMailAsync(mail);
    }
}

// 5. DATABASE ONLY
public interface IOrderRepository
{
    Task SaveAsync(Order order);
}

public class OrderRepository : IOrderRepository
{
    private readonly string _connString;
    public OrderRepository(IConfiguration config)
        => _connString = config.GetConnectionString("Default");

    public async Task SaveAsync(Order order)
    {
        using var conn = new SqlConnection(_connString);
        var cmd = new SqlCommand(
            "INSERT INTO Orders (CustomerId, Total) VALUES (@id, @total)", conn);
        cmd.Parameters.AddWithValue("@id", order.CustomerId);
        cmd.Parameters.AddWithValue("@total", order.Total);
        await conn.OpenAsync();
        await cmd.ExecuteNonQueryAsync();
    }
}

// 6. LOGGING ONLY
public interface IOrderLogger
{
    Task LogAsync(string message);
}

public class OrderLogger : IOrderLogger
{
    public Task LogAsync(string message)
        => File.AppendAllTextAsync("orders.log", $"{message}\n");
}

// 7. ORCHESTRATOR: OrderService (only coordinates, logic elsewhere)
public class OrderService
{
    private readonly IOrderValidator _validator;
    private readonly IDiscountCalculator _discount;
    private readonly IPaymentService _payment;
    private readonly IEmailService _email;
    private readonly IOrderRepository _repository;
    private readonly IOrderLogger _logger;

    public OrderService(IOrderValidator validator, IDiscountCalculator discount,
        IPaymentService payment, IEmailService email, IOrderRepository repository, IOrderLogger logger)
    {
        _validator = validator;
        _discount = discount;
        _payment = payment;
        _email = email;
        _repository = repository;
        _logger = logger;
    }

    public async Task ProcessOrderAsync(Order order, Customer customer)
    {
        _validator.Validate(order, customer);

        decimal discount = _discount.Calculate(order, customer);
        order.Total -= order.Total * discount;

        bool paid = await _payment.ChargeAsync(order.Total);
        if (!paid) throw new Exception("Payment failed");

        await _email.SendAsync(customer.Email, "Order Confirmed",
            $"Thank you! Order #{order.Id}");
        await _repository.SaveAsync(order);
        await _logger.LogAsync($"Order {order.Id} completed");
    }
}

// BENEFITS:
// ✓ Test DiscountCalculator without payment/email/database
// ✓ Change email? Edit only EmailService
// ✓ Add new payment method? Create new IPaymentService, don't edit existing code
// ✓ Each class has ONE reason to change
// ✓ Reusable everywhere: DiscountCalculator, EmailService can be used in other features

    O — Open/Closed Principle

    Definition: Classes should be open for extension but closed for modification. Add features by creating new classes, not by editing existing ones.

    ❌ BAD: Massive If/Else for New Features

    Adding a new shipping method requires editing PaymentProcessor. Modify existing code = risk breaking it!

    csharp 
    // ❌ BAD: Every new payment method requires EDITING existing class
public class PaymentProcessor
{
    public bool ProcessPayment(string method, decimal amount)
    {
        if (method == "CreditCard")
        {
            Console.WriteLine($"Charging ${amount} to credit card");
            return true;
        }
        else if (method == "PayPal")
        {
            Console.WriteLine($"Charging ${amount} via PayPal");
            return true;
        }
        else if (method == "Bitcoin")
        {
            Console.WriteLine($"Charging ${amount} via Bitcoin");
            return true;
        }
        // Add Apple Pay? EDIT this file again!
        // Add Google Pay? EDIT this file again!
        return false;
    }
}

// PROBLEMS:
// ❌ Every new payment method requires modifying PaymentProcessor
// ❌ Risk: editing might break CreditCard or PayPal processing
// ❌ Class grows indefinitely — unmaintainable
// ❌ Violates OCP: not closed for modification

    ✅ GOOD: Strategy Pattern (Interfaces + Polymorphism)

    Create new payment classes without touching the processor. EXTENSION without MODIFICATION.

    csharp 
    // ✓ GOOD: Open for extension, closed for modification

// Base contract (never changes!)
public interface IPaymentMethod
{
    Task<bool> ChargeAsync(decimal amount);
}

// Existing implementations
public class CreditCardPayment : IPaymentMethod
{
    public async Task<bool> ChargeAsync(decimal amount)
    {
        Console.WriteLine($"Charging ${amount} to credit card");
        await Task.Delay(50);
        return true;
    }
}

public class PayPalPayment : IPaymentMethod
{
    public async Task<bool> ChargeAsync(decimal amount)
    {
        Console.WriteLine($"Charging ${amount} via PayPal");
        await Task.Delay(50);
        return true;
    }
}

public class BitcoinPayment : IPaymentMethod
{
    public async Task<bool> ChargeAsync(decimal amount)
    {
        Console.WriteLine($"Charging ${amount} via Bitcoin");
        await Task.Delay(50);
        return true;
    }
}

// NEW PAYMENT METHOD? Just create a new class (no edits to existing code!)
public class ApplePayPayment : IPaymentMethod
{
    public async Task<bool> ChargeAsync(decimal amount)
    {
        Console.WriteLine($"Charging ${amount} via Apple Pay");
        await Task.Delay(50);
        return true;
    }
}

// PaymentProcessor — NEVER CHANGES!
public class PaymentProcessor
{
    private readonly Dictionary<string, IPaymentMethod> _methods;

    public PaymentProcessor()
    {
        _methods = new Dictionary<string, IPaymentMethod>
        {
            { "card", new CreditCardPayment() },
            { "paypal", new PayPalPayment() },
            { "bitcoin", new BitcoinPayment() },
            { "applepay", new ApplePayPayment() }
        };
    }

    // Add new payment method to dictionary dynamically
    public void RegisterMethod(string name, IPaymentMethod method)
    {
        _methods[name] = method;
    }

    public async Task<bool> ProcessAsync(string method, decimal amount)
    {
        if (_methods.TryGetValue(method.ToLower(), out var payment))
            return await payment.ChargeAsync(amount);
        throw new ArgumentException($"Unknown method: {method}");
    }
}

// BENEFITS:
// ✓ PaymentProcessor CLOSED for modification (never change it!)
// ✓ OPEN for extension (add GooglePayPayment, CryptocurrencyPayment, etc.)
// ✓ New developers can add payment types without touching existing code
// ✓ Lower risk of breaking existing functionality
// ✓ Clean, scalable architecture

    L — Liskov Substitution Principle

    Definition: Derived classes must be substitutable for their base. If code expects a parent type, the child type must work correctly without surprises.

    ❌ BAD: Subclass Breaks Parent's Contract

    CashPayment implements IPaymentMethod but throws NotImplementedException. Code trusts it not to throw!

    csharp 
    // ❌ BAD: CashPayment violates LSP — breaks contract!
public interface IShippingMethod
{
    Task<bool> ShipAsync(Order order, string address, DateTime deadline);
}

public class StandardShipping : IShippingMethod
{
    public async Task<bool> ShipAsync(Order order, string address, DateTime deadline)
    {
        Console.WriteLine($"Shipping to {address}, arrives by {deadline}");
        await Task.Delay(100);
        return true;
    }
}

public class ExpressShipping : IShippingMethod
{
    public async Task<bool> ShipAsync(Order order, string address, DateTime deadline)
    {
        Console.WriteLine($"Express shipping to {address}, arrives by {deadline}");
        await Task.Delay(50);
        return true;
    }
}

// ❌ Problem: PickupOnly doesn't support shipping (violates LSP!)
public class PickupOnly : IShippingMethod
{
    public async Task<bool> ShipAsync(Order order, string address, DateTime deadline)
    {
        // ❌ This method doesn't make sense for pickup!
        // ❌ Violates contract: CallingCode expects this to work
        if (!string.IsNullOrEmpty(address))
            throw new NotSupportedException("Pickup doesn't need an address!");

        throw new NotImplementedException("Pickup requires store visit");
    }
}

// Calling code trusts IShippingMethod — CRASH when it gets PickupOnly!
public class OrderService
{
    public async Task ShipOrderAsync(IShippingMethod shipping, Order order)
    {
        // This might crash with PickupOnly!
        bool success = await shipping.ShipAsync(order, "123 Main St", DateTime.Now.AddDays(5));
        if (!success)
            throw new Exception("Shipping failed");
    }
}

// PROBLEMS:
// ❌ Code assumes all IShippingMethod support address + deadline
// ❌ PickupOnly breaks the assumption — throws exception
// ❌ Runtime crash — not caught at compile time
// ❌ Liskov substitution is violated

    ✅ GOOD: Separate Concerns with Different Interfaces

    PickupOnly doesn't claim to support shipping. Use separate interfaces for what's actually supported.

    csharp 
    // ✓ GOOD: Separate interfaces for different delivery types

// Core fulfillment (every order needs one)
public interface IFulfillmentMethod
{
    Task<bool> FulfillAsync(Order order);
}

// For shipped orders
public interface IShippedFulfillment : IFulfillmentMethod
{
    Task<bool> ShipAsync(string address, DateTime deadline);
}

// Standard shipping
public class StandardShipping : IShippedFulfillment
{
    public async Task<bool> FulfillAsync(Order order)
    {
        Console.WriteLine("Fulfilling order: standard shipping");
        await Task.Delay(100);
        return true;
    }

    public async Task<bool> ShipAsync(string address, DateTime deadline)
    {
        Console.WriteLine($"Shipping to {address}, arrives by {deadline}");
        await Task.Delay(100);
        return true;
    }
}

// Express shipping
public class ExpressShipping : IShippedFulfillment
{
    public async Task<bool> FulfillAsync(Order order)
    {
        Console.WriteLine("Fulfilling order: express shipping");
        await Task.Delay(50);
        return true;
    }

    public async Task<bool> ShipAsync(string address, DateTime deadline)
    {
        Console.WriteLine($"Express shipping to {address}, arrives by {deadline}");
        await Task.Delay(50);
        return true;
    }
}

// Pickup only (doesn't implement IShippedFulfillment!)
public class PickupOnly : IFulfillmentMethod
{
    private readonly string _storeLocation;

    public PickupOnly(string storeLocation)
    {
        _storeLocation = storeLocation;
    }

    public async Task<bool> FulfillAsync(Order order)
    {
        Console.WriteLine($"Order ready for pickup at {_storeLocation}");
        await Task.Delay(50);
        return true;
    }
}

// Type-safe calling code
public class OrderService
{
    // Any fulfillment method works here
    public async Task FulfillOrderAsync(IFulfillmentMethod method, Order order)
    {
        bool success = await method.FulfillAsync(order);
        if (!success)
            throw new Exception("Fulfillment failed");
    }

    // Only for shipped orders
    public async Task ShipOrderAsync(IShippedFulfillment shipping, Order order, string address)
    {
        bool success = await shipping.ShipAsync(address, DateTime.Now.AddDays(5));
        if (!success)
            throw new Exception("Shipping failed");
    }
}

// Usage
var pickup = new PickupOnly("Store #5");
var express = new ExpressShipping();

var service = new OrderService();

// ✓ Works perfectly — PickupOnly honors IFulfillmentMethod contract
await service.FulfillOrderAsync(pickup, order);

// ✓ Works perfectly — ExpressShipping honors IShippedFulfillment contract
await service.ShipOrderAsync(express, order, "123 Main St");

// ✓ Compile-time safety — can't pass PickupOnly to ShipOrderAsync!
// await service.ShipOrderAsync(pickup, order, "123 Main St"); // COMPILER ERROR!

// BENEFITS:
// ✓ LSP honored: all IFulfillmentMethod are safely substitutable
// ✓ PickupOnly not forced to implement shipping it doesn't support
// ✓ Type-safe: compiler prevents misuse
// ✓ Clear contracts: if you have IShippedFulfillment, you KNOW ShipAsync works

    I — Interface Segregation Principle

    Definition: Clients should never depend on methods they don't use. Split fat interfaces into smaller, focused ones.

    ❌ BAD: Fat Interface Forces Dummy Implementations

    IOrder has 10 methods. DigitalOrder must implement Print/Ship which make no sense for digital.

    csharp 
    // ❌ BAD: Fat interface — forces dummy implementations
public interface IOrder
{
    Task ValidateAsync();
    Task ApplyDiscountAsync(decimal percent);
    Task ChargePaymentAsync(decimal amount);
    Task SendEmailAsync(string email);

    // Physical order operations
    Task PrintLabelAsync();
    Task SchedulePickupAsync();
    Task TrackShipmentAsync();

    // Digital order operations
    Task SendAccessKeyAsync(string email);
    Task GenerateLicenseAsync();
}

// Digital order: stuck implementing shipping methods it doesn't need!
public class DigitalOrder : IOrder
{
    public async Task ValidateAsync() { /* useful */ }
    public async Task ApplyDiscountAsync(decimal percent) { /* useful */ }
    public async Task ChargePaymentAsync(decimal amount) { /* useful */ }
    public async Task SendEmailAsync(string email) { /* useful */ }

    // ❌ Forced to implement — throws exceptions!
    public async Task PrintLabelAsync()
    {
        throw new NotImplementedException("Digital orders don't print");
    }

    public async Task SchedulePickupAsync()
    {
        throw new NotImplementedException("Digital orders don't ship");
    }

    public async Task TrackShipmentAsync()
    {
        throw new NotImplementedException("Digital orders don't ship");
    }

    public async Task SendAccessKeyAsync(string email) { /* useful */ }
    public async Task GenerateLicenseAsync() { /* useful */ }
}

// Physical order: stuck implementing digital methods it doesn't need!
public class PhysicalOrder : IOrder
{
    public async Task ValidateAsync() { /* useful */ }
    public async Task ApplyDiscountAsync(decimal percent) { /* useful */ }
    public async Task ChargePaymentAsync(decimal amount) { /* useful */ }
    public async Task SendEmailAsync(string email) { /* useful */ }
    public async Task PrintLabelAsync() { /* useful */ }
    public async Task SchedulePickupAsync() { /* useful */ }
    public async Task TrackShipmentAsync() { /* useful */ }

    // ❌ Forced to implement — throws exceptions!
    public async Task SendAccessKeyAsync(string email)
    {
        throw new NotImplementedException("Physical orders don't download");
    }

    public async Task GenerateLicenseAsync()
    {
        throw new NotImplementedException("Physical orders don't license");
    }
}

// PROBLEMS:
// ❌ Throwing NotImplementedException defeats the purpose of interfaces!
// ❌ Developers confused: which methods are actually supported?
// ❌ Maintenance nightmare: adding method affects unrelated classes
// ❌ No compile-time guarantee what methods work on which order type

    ✅ GOOD: Segregated Interfaces (Each Class Uses Only What It Needs)

    csharp 
    // ✓ GOOD: Small, focused interfaces

// Core (all orders need these)
public interface IOrder
{
    Task ValidateAsync();
    Task ApplyDiscountAsync(decimal percent);
    Task ChargePaymentAsync(decimal amount);
}

// Shipped orders need these
public interface IShippableOrder : IOrder
{
    Task PrintLabelAsync();
    Task SchedulePickupAsync();
    Task TrackShipmentAsync();
}

// Digital orders need these
public interface IDownloadableOrder : IOrder
{
    Task SendAccessKeyAsync(string email);
    Task GenerateLicenseAsync();
}

// Digital: no shipping methods!
public class DigitalOrder : IDownloadableOrder
{
    public async Task ValidateAsync() { Console.WriteLine("Validating..."); }
    public async Task ApplyDiscountAsync(decimal percent) { Console.WriteLine($"Discount: {percent}%"); }
    public async Task ChargePaymentAsync(decimal amount) { Console.WriteLine($"Charging ${amount}"); }
    public async Task SendAccessKeyAsync(string email) { Console.WriteLine($"Sending key to {email}"); }
    public async Task GenerateLicenseAsync() { Console.WriteLine("License generated"); }
}

// Physical: no digital methods!
public class PhysicalOrder : IShippableOrder
{
    public async Task ValidateAsync() { Console.WriteLine("Validating..."); }
    public async Task ApplyDiscountAsync(decimal percent) { Console.WriteLine($"Discount: {percent}%"); }
    public async Task ChargePaymentAsync(decimal amount) { Console.WriteLine($"Charging ${amount}"); }
    public async Task PrintLabelAsync() { Console.WriteLine("Printing label"); }
    public async Task SchedulePickupAsync() { Console.WriteLine("Scheduling pickup"); }
    public async Task TrackShipmentAsync() { Console.WriteLine("Tracking shipment"); }
}

// Type-safe calling code
public class OrderProcessor
{
    // Works with any order
    public async Task ProcessAsync(IOrder order)
    {
        await order.ValidateAsync();
        await order.ApplyDiscountAsync(0.10m);
        await order.ChargePaymentAsync(99.99m);
    }

    // Works ONLY with shippable orders
    public async Task ShipAsync(IShippableOrder order)
    {
        await ProcessAsync(order); // Core processing
        await order.PrintLabelAsync();
        await order.SchedulePickupAsync();
    }

    // Works ONLY with downloadable orders
    public async Task DownloadAsync(IDownloadableOrder order)
    {
        await ProcessAsync(order); // Core processing
        await order.SendAccessKeyAsync("customer@email.com");
        await order.GenerateLicenseAsync();
    }
}

// Compile-time safety!
var physical = new PhysicalOrder();
var digital = new DigitalOrder();

// ✓ Works — DigitalOrder implements IDownloadableOrder
await processor.DownloadAsync(digital);

// ✓ Works — PhysicalOrder implements IShippableOrder
await processor.ShipAsync(physical);

// ❌ COMPILER ERROR! Can't pass DigitalOrder to ShipAsync!
// await processor.ShipAsync(digital);

// BENEFITS:
// ✓ No throwaway NotImplemented methods
// ✓ Clear contracts: if you have IShippableOrder, you KNOW PrintLabel works
// ✓ Compile-time verification of what methods are available
// ✓ Easy to add new order types
// ✓ Focused, reusable interfaces

    D — Dependency Inversion Principle

    Definition: Depend on abstractions, not concretions. Inject dependencies via constructor; don't create them inside the class.

    ❌ BAD: Tight Coupling (Creates Dependencies Directly)

    OrderService directly instantiates SqlOrderRepository and FileLogger. Can't test without real DB/files!

    csharp 
    // ❌ BAD: OrderService depends on concrete implementations
public class SqlOrderRepository
{
    public void Save(Order order)
    {
        Console.WriteLine($"Saving to SQL database: Order {order.Id}");
        // Real database connection...
    }
}

public class FileLogger
{
    public void Log(string message)
    {
        Console.WriteLine($"Writing to file: {message}");
        File.AppendAllText("orders.log", message + "\n");
    }
}

// ❌ TIGHT COUPLING: OrderService creates dependencies directly
public class OrderService
{
    private readonly SqlOrderRepository _repository;
    private readonly FileLogger _logger;

    public OrderService()
    {
        _repository = new SqlOrderRepository(); // ❌ Hard-coded dependency!
        _logger = new FileLogger(); // ❌ Hard-coded dependency!
    }

    public void PlaceOrder(Order order)
    {
        _repository.Save(order);
        _logger.Log($"Order {order.Id} placed");
    }
}

// TEST NIGHTMARE
public class OrderServiceTests
{
    [Test]
    public void PlaceOrder_Should_SaveAndLog()
    {
        // ❌ PROBLEMS:
        // - OrderService creates REAL SqlOrderRepository (connects to real DB!)
        // - OrderService creates REAL FileLogger (writes to real file!)
        // - Test is SLOW (database operations)
        // - Test is FRAGILE (depends on DB availability)
        // - How to verify it saved? Query the real database!?
        // - How to verify it logged? Read the real file!?

        var service = new OrderService();

        service.PlaceOrder(order); // Creates real DB connection + file I/O!

    }

// PROBLEMS:
// ❌ Can't test without real database
// ❌ Can't test without file system
// ❌ Can't swap SQL for MongoDB, PostgreSQL, in-memory, etc.
// ❌ Can't swap FileLogger for ConsoleLogger, EventLogger, etc.
// ❌ Depends on low-level implementation details
// ❌ High coupling = fragile code

    ✅ GOOD: Dependency Injection (Depends on Abstractions)

    Inject interfaces via constructor. Test with mocks. Swap implementations freely.

    csharp 
    // ✓ GOOD: OrderService depends on abstractions

// Abstractions (interfaces)
public interface IOrderRepository
{
    Task SaveAsync(Order order);
}

public interface IOrderLogger
{
    Task LogAsync(string message);
}

// Concrete implementations
public class SqlOrderRepository : IOrderRepository
{
    private readonly string _connString;

    public SqlOrderRepository(IConfiguration config)
    {
        _connString = config.GetConnectionString("Default");
    }

    public async Task SaveAsync(Order order)
    {
        Console.WriteLine($"Saving to SQL: Order {order.Id}");
        await Task.Delay(100); // Simulate DB
    }
}

public class FileLogger : IOrderLogger
{
    public async Task LogAsync(string message)
    {
        Console.WriteLine($"Writing to file: {message}");
        await File.AppendAllTextAsync("orders.log", message + "\n");
    }
}

// ✓ GOOD: OrderService depends on interfaces (abstractions), not concrete classes
public class OrderService
{
    private readonly IOrderRepository _repository;
    private readonly IOrderLogger _logger;

    // ✓ Inject dependencies via constructor
    public OrderService(IOrderRepository repository, IOrderLogger logger)
    {
        _repository = repository ?? throw new ArgumentNullException(nameof(repository));
        _logger = logger ?? throw new ArgumentNullException(nameof(logger));
    }

    public async Task PlaceOrderAsync(Order order)
    {
        await _repository.SaveAsync(order);
        await _logger.LogAsync($"Order {order.Id} placed");
    }
}

        // ✓ CLEAR: Verify exactly what was called!
    }

    [Test]
    public async Task PlaceOrder_Should_HandleRepositoryError()
    {
        // ✓ Easy to test error scenarios with mocks!
        var mockRepository = new Mock<IOrderRepository>();

            .ThrowsAsync(new Exception("DB unavailable"));

    Interview Questions

    Q1. Explain SRP with your E-Commerce Order System example. ▼

    Answer: SRP means one class, one responsibility. In a bad design, OrderProcessor does validation, discount, payment, email, database, and logging — 6 reasons to change. The good design splits it: OrderValidator only validates, DiscountCalculator only calculates, PaymentService only processes payment, etc. Now each class has ONE reason to change, is independently testable, and reusable.

    Why it matters: If payment rules change, you modify only PaymentService. Risk is isolated. Without SRP, editing one thing risks breaking everything.

    Q2. How does OCP help when adding new payment methods? ▼

    Answer: OCP: "open for extension, closed for modification." Bad design: massive if/else in PaymentProcessor. Adding Apple Pay requires editing that class — risky. Good design: IPaymentMethod interface. Each payment type is a separate class (CreditCardPayment, PayPalPayment, ApplePayPayment). Adding a new payment method = create new class implementing IPaymentMethod. PaymentProcessor never changes!

    Why it matters: New features don't require editing existing, tested code. Risk is zero.

    Q3. Explain LSP violation with the shipping example. ▼

    Answer: LSP: subclass must be safely substitutable for parent. Bad: PickupOnly implements IShippingMethod but throws NotImplementedException because it doesn't support address + deadline. Code expects any IShippingMethod to work — crash! Good: PickupOnly implements IFulfillmentMethod (core only), not IShippedFulfillment. Different interfaces for different fulfillment types. Type-safe — compiler prevents misuse.

    Why it matters: Polymorphism only works if subclass honors parent's contract.

    Q4. What's the benefit of Interface Segregation? ▼

    Answer: ISP: don't force implementations to depend on unused methods. Bad: IOrder has 10 methods. DigitalOrder forced to implement PrintLabel, ScheduleShip (throws NotImpl). Confusing — what methods actually work? Good: segregate into focused interfaces: IOrder (core), IShippableOrder (physical), IDownloadableOrder (digital). Each class implements only what it needs. Compile-time safety: if you have IShippableOrder, you KNOW PrintLabel works.

    Why it matters: Clear contracts, no dummy methods, compile-time verification.

    Q5. How does DIP enable unit testing? ▼

    Answer: DIP: depend on abstractions (interfaces), inject via constructor. Bad: OrderService creates SqlOrderRepository and FileLogger directly. Test needs real DB + file I/O — slow, fragile, can't test without external systems. Good: OrderService takes IOrderRepository and IOrderLogger in constructor. Test passes mock implementations. No real DB/files. Lightning-fast tests, complete isolation, easy to test error scenarios.

    Why it matters: Unit tests run in milliseconds, not seconds. You can run 1000s of tests locally without infrastructure.

    Q6. How do SOLID principles work together? ▼

    Answer: They're interconnected:

    • SRP + DIP: Split OrderProcessor into specialized classes (SRP), then inject them (DIP)
    • OCP + LSP: Use Strategy pattern (OCP: open for extension) with interfaces (LSP: honor contracts)
    • ISP + SRP: Segregate interfaces (ISP) keeps classes focused (SRP)
    • DIP + OCP: Depend on abstractions (DIP) lets you swap implementations (OCP) without modifying existing code

    Result: code that's maintainable, testable, extensible, and professional.
