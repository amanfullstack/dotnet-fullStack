👁️ Views & Razor

Learn Razor syntax, view layouts, and HTML templating in ASP.NET MVC.

What Are Views & Razor?

Views are HTML files that display data to users. They're the "V" in MVC. Razor is a templating syntax that lets you mix C# code with HTML to generate dynamic pages.

Instead of returning raw HTML from your controller, you return a View with data (a model). Razor processes your template, fills in the dynamic content, and sends back pure HTML to the browser.

Why Views Matter

  • Separation of concerns: UI logic stays in views, business logic in controllers
  • Reusability: Layouts and partial views eliminate code duplication
  • Dynamic content: Show different data without creating separate HTML files
  • Type-safe: Strongly-typed views catch errors at compile time
  • Easy maintenance: Change UI without touching controller code

Key Concepts

📝 Razor Syntax Basics

@ is the magic symbol that starts C# code in Razor:

  • @Model.PropertyName - Access model property
  • @{ var x = 5; } - Execute C# code block
  • @if (...) { } - Conditional logic
  • @foreach (var item in list) { } - Loop through collections
  • @Html.ActionLink("Text", "Action", "Controller") - Create links

🎨 Layouts & Master Pages

Layouts provide the common HTML structure (header, navigation, footer) for multiple views. Define once, use everywhere.

  • _Layout.cshtml - Master template with @RenderBody()
  • Views inherit layout automatically
  • Reduces duplication, easier to update styles site-wide

🔌 Partial Views

Partial views are reusable view components (like a product card). Include them in multiple places:

@Html.Partial("_ProductCard", product)
  • Keep views DRY (Don't Repeat Yourself)
  • Easier to test and maintain
  • File name starts with underscore: _ComponentName.cshtml

📦 Strongly-Typed Views

Declare your model type at the top of the view for IntelliSense and compile-time checking:

@model ProductViewModel
@* Now you can use @Model.Name with full IntelliSense *@
<h1>@Model.Name</h1>

📚 View Data & Bag

ViewData and ViewBag pass data from controller to view for non-model data:

  • ViewData["key"] - Dictionary access, requires casting
  • ViewBag.Key - Dynamic property access, cleaner syntax
  • Use for small bits of data (not full models)

Code Examples

Example 1: Strongly-Typed View with Display & Forms

@* Product listing view *@
@model List<ProductViewModel>

<div class="products-grid">
    @if (Model != null && Model.Count > 0)
    {
        @foreach (var product in Model)
        {
            <div class="product-card">
                <h3>@product.Name</h3>
                <p>@Html.DisplayFor(m => product.Description)</p>
                <p class="price">@product.Price.ToString("C")</p>

                @* Create links using HTML helpers *@
                <a href="@Url.Action("Details", new { id = product.Id })">
                    View Details
                </a>

                @* Show edit button only if user is admin *@
                @if (User.IsInRole("Admin"))
                {
                    <a href="@Url.Action("Edit", new { id = product.Id })">
                        Edit
                    </a>
                }
            </div>
        }
    }
    else
    {
        <p>No products found.</p>
    }
</div>

Example 2: Form Submission with Validation

@* Create/Edit product form view *@
@model ProductViewModel

<div class="form-container">
    <h2>@(Model.Id > 0 ? "Edit" : "Create") Product</h2>

    <!-- Display validation summary (shows all errors) -->
    @Html.ValidationSummary(false, "", new { @class = "error-summary" })

    @using (Html.BeginForm("Save", "Product", FormMethod.Post))
    {
        <!-- CSRF token protection -->
        @Html.AntiForgeryToken()

        <div class="form-group">
            @Html.LabelFor(m => m.Name)
            @Html.TextBoxFor(m => m.Name, new { @class = "form-control" })
            @Html.ValidationMessageFor(m => m.Name) <!-- Show errors for this field -->
        </div>

        <div class="form-group">
            @Html.LabelFor(m => m.Price)
            @Html.TextBoxFor(m => m.Price, new { @class = "form-control", type = "number" })
            @Html.ValidationMessageFor(m => m.Price)
        </div>

        <div class="form-group">
            @Html.LabelFor(m => m.CategoryId)
            @Html.DropDownListFor(m => m.CategoryId,
                new SelectList(ViewBag.Categories, "Id", "Name"),
                "-- Select Category --",
                new { @class = "form-control" })
            @Html.ValidationMessageFor(m => m.CategoryId)
        </div>

        <button type="submit" class="btn btn-primary">Save</button>
        <a href="@Url.Action("Index")" class="btn btn-secondary">Cancel</a>
    }
</div>

Example 3: Layout & Partial Views

@* _Layout.cshtml - Master template *@
<!DOCTYPE html>
<html>
<head>
    <title>@ViewData["Title"] - My Site</title>
    <link rel="stylesheet" href="~/css/site.css" />
</head>
<body>
    <header>
        @Html.Partial("_Header") <!-- Include partial view -->
    </header>

    <nav>
        @Html.Partial("_Navigation")
    </nav>

    <main>
        @RenderBody() <!-- Page-specific content goes here -->
    </main>

    <footer>
        @Html.Partial("_Footer")
    </footer>

    <script src="~/js/site.js"></script>
</body>
</html>

---

@* _ProductCard.cshtml - Reusable partial view *@
@model ProductViewModel

<div class="product-card">
    <img src="@Model.ImageUrl" alt="@Model.Name" />
    <h3>@Model.Name</h3>
    <p class="description">@Model.Description</p>
    <span class="price">@Model.Price.ToString("C")</span>
    <a href="@Url.Action("Details", new { id = Model.Id })">View Details</a>
</div>

---

@* Index.cshtml - Uses layout and partial views *@
@model List<ProductViewModel>
@{
    ViewData["Title"] = "Products";
    Layout = "~/Views/Shared/_Layout.cshtml";
}

<h1>@ViewData["Title"]</h1>

<div class="products-grid">
    @foreach (var product in Model)
    {
        @Html.Partial("_ProductCard", product) <!-- Reuse card component -->
    }
</div>

✅ Best Practices

1. Always Use Strongly-Typed Views

Why: Compile-time errors, IntelliSense support, safer refactoring

❌ Bad:

<h1>@ViewBag.ProductName</h1> <!-- No IntelliSense, runtime errors possible -->

✅ Good:

@model ProductViewModel
<h1>@Model.Name</h1> <!-- Full IntelliSense, compile-time checking -->

2. Use HTML Helpers for Dynamic Content

Why: Cleaner code, automatic URL routing, CSRF protection

❌ Bad:

<!-- Hard-coded URLs break when routes change -->
<a href="/Product/Details/5">View Details</a>

✅ Good:

<!-- Auto-generates correct URL based on routing -->
@Html.ActionLink("View Details", "Details", "Product", new { id = Model.Id })

3. Extract Reusable Parts into Partials

Why: DRY principle, easier to test, consistent UI

❌ Bad: Repeat product card HTML in 10 different views

✅ Good: Create _ProductCard.cshtml, include with @Html.Partial()

4. Always Include @Html.AntiForgeryToken()

Why: Prevents CSRF (Cross-Site Request Forgery) attacks

@using (Html.BeginForm("Create", "Product", FormMethod.Post))
{
    @Html.AntiForgeryToken() <!-- Always include this -->

    <!-- form fields -->
}

5. Display Validation Messages

Why: Users know what's wrong with their input

<!-- Show all errors at top -->
@Html.ValidationSummary()

<!-- Show error for specific field -->
<div class="form-group">
    @Html.LabelFor(m => m.Email)
    @Html.TextBoxFor(m => m.Email)
    @Html.ValidationMessageFor(m => m.Email) <!-- Error appears here -->
</div>

🎓 Interview Q&A

Q1: What's the difference between HTML Helpers and Tag Helpers?

Answer:

  • HTML Helpers: Old syntax, C# method calls, less intuitive
  • Tag Helpers: Modern syntax, looks like HTML attributes, more readable
<!-- HTML Helper (old) -->
@Html.TextBoxFor(m => m.Name)

<!-- Tag Helper (modern, preferred) -->
<input asp-for="Name" class="form-control" />

Q2: When should you use ViewBag vs strongly-typed models?

Answer:

  • Models: For main data (List of products, user details). Strongly typed, safe, preferred.
  • ViewBag: For small, secondary data (page title, list of categories for dropdown). No IntelliSense, less safe.
// Controller
public IActionResult Create()
{
    // Main data - use model
    var model = new ProductViewModel();

    // Secondary data - use ViewBag
    ViewBag.Categories = _service.GetCategories();

    return View(model);
}

Q3: What happens when you use @Html.Partial() vs @await Html.PartialAsync()?

Answer:

  • @Html.Partial(): Synchronous, blocks page rendering until partial loads
  • @await Html.PartialAsync(): Asynchronous, doesn't block, better performance. Use this for I/O operations.
<!-- Sync - simple partials with no async logic -->
@Html.Partial("_ProductCard", product)

<!-- Async - if partial does database queries or API calls -->
@await Html.PartialAsync("_RelatedProducts", product)
