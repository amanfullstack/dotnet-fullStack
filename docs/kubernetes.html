☸️ Kubernetes - Container Orchestration

Automatically scale, heal, and deploy thousands of containers across multiple servers. Think of it as an orchestra conductor for Docker containers.

🎭

Real-World Analogy: Orchestra Conductor

❌ Without Kubernetes: 100 musicians (containers) on stage, each following their own sheet music. If violinist gets sick → play stops. Manual reschedule → chaos.

✅ With Kubernetes: Conductor (K8s) watches orchestra. Violinist gets sick? → Auto-replace from backup. Need louder sound? → Auto-add 3 more violinists. Rain storm? → Move everyone inside. Perfect harmony without conductor intervening.

Why Kubernetes?

Problems at Scale (100s of containers):

  • ❌ Manual Management: SSH into 50 servers, manually start containers → Slow, error-prone
  • ❌ Container Dies: If ProductAPI-Pod-3 crashes → Customer gets 504. Need someone to restart it NOW
  • ❌ No Auto-Scaling: Black Friday traffic spike? Manually spin up 20 more instances? Too slow!
  • ❌ No Orchestration: How to deploy v2.0 without downtime? Manually kill old ones, start new ones → Risk of downtime
  • ❌ Networking Hell: How do containers securely communicate? Hard-coded IP addresses? Nightmare when containers move

Solutions WITH Kubernetes:

  • ✅ Self-Healing: Container crashes? K8s auto-restarts it
  • ✅ Auto-Scaling: CPU usage 80%? K8s auto-creates more Pods
  • ✅ Rolling Updates: Deploy v2.0: Start new Pod, drain traffic from old Pod, delete old Pod → Zero downtime
  • ✅ Service Discovery: Containers can find each other by name (internal DNS)
  • ✅ Load Balancing: Traffic auto-distributed across replicas

Key Concepts

1️⃣ Pod (Smallest K8s Unit)

Pod = Wrapper around container(s)

Usually: 1 container per Pod
Sometimes: 2-3 containers (tightly coupled only!)

Example:
┌────────────────────────────┐
│ Pod (unique IP shared)      │
├────────────────────────────┤
│ Container: ProductAPI      │
│ Container: Logging Sidecar │
└────────────────────────────┘

Why not just containers? K8s Pods are Linux networking namespace → All containers in Pod share IP/localhost. Easy inter-process communication.

2️⃣ Deployment (Desired State)

Declarative: "I want 3 replicas of ProductAPI v1.2.0"

apiVersion: apps/v1
kind: Deployment
metadata:
  name: product-api
spec:
  replicas: 3              # "I want 3 copies running"
  selector:
    matchLabels:
      app: product-api
  template:
    metadata:
      labels:
        app: product-api
    spec:
      containers:
      - name: product-api
        image: myregistry.azurecr.io/productcatalogapi:1.2.0
        ports:
        - containerPort: 5000
        resources:
          requests:
            memory: "128Mi"
            cpu: "100m"
          limits:
            memory: "256Mi"
            cpu: "500m"
        livenessProbe:
          httpGet:
            path: /health
            port: 5000
          initialDelaySeconds: 10
          periodSeconds: 10

K8s watches & maintains: If one Pod crashes → K8s auto-creates new one. Always 3 replicas.

3️⃣ Service (Network Expose)

Problem: Pods are temporary. IP changes when Pod restart. How does frontend find backend?

Solution: Service = Stable DNS name + Load Balancer

apiVersion: v1
kind: Service
metadata:
  name: product-api
spec:
  type: LoadBalancer          # Expose outside cluster
  selector:
    app: product-api          # Route traffic to Pods with this label
  ports:
  - port: 80                  # External port (what users see)
    targetPort: 5000          # Internal Pod port
    protocol: TCP

Result: Frontend calls http://product-api → Automatically routed to one of 3 Pods (round-robin load balanced)

4️⃣ ConfigMap & Secrets

Problem: How to configure apps without rebuilding image?

ConfigMap = Config as YAML (not in image)

apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  LOG_LEVEL: "INFO"
  MAX_CONNECTIONS: "100"
  DATABASE_HOST: "postgres.default.svc.cluster.local"

Secret = Encrypted config (passwords, tokens)

apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
stringData:
  password: "super-secret-password-123"

Reference in Deployment:

env:
- name: LOG_LEVEL
  valueFrom:
    configMapKeyRef:
      name: app-config
      key: LOG_LEVEL
- name: DB_PASSWORD
  valueFrom:
    secretKeyRef:
      name: db-credentials
      key: password

5️⃣ Ingress (HTTP Routing)

Service gives internal routing. Ingress gives external HTTP routing.

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-gateway
spec:
  ingressClassName: nginx
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /products
        pathType: Prefix
        backend:
          service:
            name: product-api
            port:
              number: 80
      - path: /orders
        pathType: Prefix
        backend:
          service:
            name: order-api
            port:
              number: 80

Result: User visits api.example.com/products → Routed to product-api Service

Best Practices

1. Set resource limits

Memory & CPU limits prevent one greedy Pod from starving others.

2. Use health checks

Liveness probes detect dead Pods. Readiness probes prevent traffic before ready.

3. Namespaces for isolation

Dev/staging/prod in separate namespaces. RBAC controls access.

4. Use ConfigMap + Secrets

Never hardcode config in Deployment. Update ConfigMap, K8s redeploys.

5. StatelessPods

Pods should be replaceable. Persist data to PersistentVolumes, not Pod disk.

6. Monitor with Prometheus

Scrape metrics from Pods. Alert on high CPU/memory usage.

Common Mistakes

  • ❌ No resource limits: One Pod eats 100% CPU → Starves other Pods. Always set requests/limits!
  • ❌ Missing health checks: Broken Pod processes traffic → Cascading issues. Use livenessProbe + readinessProbe!
  • ❌ Hardcoding config in image: Change one value = rebuild image. Use ConfigMap instead!
  • ❌ Not using Namespaces: Dev/prod mixed → Dev test breaks production. Use namespaces!
  • ❌ Stateful Pods: Pod restarts = data lost. Use PersistentVolumes for databases!

Interview Q&A

Q: What's the difference between a Pod and a container?

A: Container = single process. Pod = one or more containers in same network namespace (shared IP). Pod is K8s smallest unit.

Q: How does Kubernetes keep Deployments running?

A: Controller watches actual vs desired state. If Pod dies: controller detects (Pod missing) → creates new Pod. If too many Pods → kills extras. Continuous reconciliation.

Q: When to use Kubernetes vs Docker Compose?

A: Compose = single machine, dev/staging. K8s = multi-node cluster, production (auto-scaling, self-healing, rolling updates).
