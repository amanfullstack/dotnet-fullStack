    🗄️ SQL & Databases

    Database design, query optimization, and SQL fundamentals — essential for backend developers

    Interview Prep Database Design Intermediate
    ⚠️

    SQL is NOT optional — even with EF Core

    Many junior developers think "I use Entity Framework Core, so I don't need to know SQL." This is a critical mistake. EF Core is an abstraction — but when your queries run slowly, EF Core generates the SQL behind the scenes, and you must understand that SQL to fix the problem. Knowing SQL is how you spot N+1 queries, inefficient JOINs, and missing indexes. Every production backend developer must be comfortable reading and writing SQL directly. This page assumes you already know SQL basics (SELECT, WHERE, INSERT) and dives into the concepts you'll be asked about in interviews.

    Quick Reference

    SQL Concept Purpose When to Use
    INNER JOIN Return only rows with matches in both tables When you need related data from multiple tables (e.g., Product + Category)
    LEFT JOIN All rows from left table + matching rows from right table When you want all left-side data even if no match (e.g., all Products, even discontinued ones)
    Normalization Remove data redundancy, reduce update anomalies Design time — before creating tables
    Index Speed up SELECT queries and WHERE filters On columns frequently used in WHERE, ORDER BY, JOIN conditions
    Transaction Group statements so all succeed or all fail (ACID) When multiple operations must be atomic (e.g., money transfer)
    View Reusable SELECT query stored in the database When you run the same complex SELECT repeatedly
    GROUP BY Aggregate data — combine rows with the same value When you need totals, counts, averages by category
    Stored Procedure Pre-compiled SQL logic stored in the database For complex multi-step operations or when you need database-side business logic

    Q42. INNER JOIN vs LEFT JOIN vs RIGHT JOIN

    Explain the difference between INNER JOIN, LEFT JOIN, and RIGHT JOIN with examples.

    JOINs are how you combine data from multiple tables. The key difference is which rows are included when a match doesn't exist.

    Visual Comparison

    Imagine two tables:

    sql 
    -- Products table
ProductId | Name        | CategoryId
1         | Laptop      | 1
2         | Phone       | 2
3         | Monitor     | NULL (category missing)
4         | Keyboard    | 1

-- Categories table
CategoryId | CategoryName
1          | Electronics
2          | Mobile
3          | Furniture (no products)

    INNER JOIN — Only matching rows

    Returns rows where a match exists in both tables. Products without a category and categories with no products are excluded.

    sql 
    SELECT p.ProductId, p.Name, c.CategoryName
FROM Products p
INNER JOIN Categories c ON p.CategoryId = c.CategoryId;

-- Result: 3 rows (Laptop, Phone, Keyboard — but NOT Monitor because CategoryId is NULL)

    LEFT JOIN — All from left table, matching from right

    Returns all rows from the left table (Products) plus matching rows from the right table (Categories). If no match, the right-side columns are NULL.

    sql 
    SELECT p.ProductId, p.Name, c.CategoryName
FROM Products p
LEFT JOIN Categories c ON p.CategoryId = c.CategoryId;

-- Result: 4 rows
-- ProductId | Name      | CategoryName
-- 1          | Laptop    | Electronics
-- 2          | Phone     | Mobile
-- 3          | Monitor   | NULL (no matching category)
-- 4          | Keyboard  | Electronics

    RIGHT JOIN — All from right table, matching from left

    Opposite of LEFT JOIN. Returns all rows from the right table plus matching rows from the left table.

    sql 
    SELECT p.ProductId, p.Name, c.CategoryName
FROM Products p
RIGHT JOIN Categories c ON p.CategoryId = c.CategoryId;

-- Result: 4 rows
-- ProductId | Name      | CategoryName
-- 1          | Laptop    | Electronics
-- 4          | Keyboard  | Electronics
-- 2          | Phone     | Mobile
-- NULL       | NULL      | Furniture (category with no products)
    💡

    Best practice: prefer LEFT JOIN

    SQL developers typically use LEFT JOIN more often than RIGHT JOIN. A RIGHT JOIN can always be rewritten as a LEFT JOIN by swapping table order. LEFT JOIN is more intuitive: "give me all the main things, and attach details if they exist."

    Q43. Normalization — 1NF, 2NF, 3NF

    What is database normalization? Explain 1NF, 2NF, and 3NF with violations and fixes.

    Normalization is the process of organizing a database to reduce redundancy and dependency anomalies. There are levels, numbered 1NF through 5NF. We focus on the first three.

    1NF — First Normal Form: Atomic Values

    Rule: Every column must contain only atomic (indivisible) values. No arrays, lists, or repeating groups.

    Violation:

    sql 
    -- ❌ VIOLATES 1NF — Tags column contains multiple values
CREATE TABLE Products (
    ProductId INT,
    Name VARCHAR(100),
    Tags VARCHAR(MAX)  -- might contain "electronics, gadget, popular"
);

-- ✅ FIXED — Create a separate Tags table
CREATE TABLE Products (
    ProductId INT PRIMARY KEY,
    Name VARCHAR(100)
);

CREATE TABLE ProductTags (
    ProductId INT,
    TagName VARCHAR(50),
    FOREIGN KEY (ProductId) REFERENCES Products(ProductId)
);

    2NF — Second Normal Form: No Partial Dependencies

    Rule: First, achieve 1NF. Then, every non-key column must depend on the entire primary key, not just part of it.

    Violation (composite key example):

    sql 
    -- ❌ VIOLATES 2NF — Primary key is (OrderId, ProductId)
-- But ProductPrice depends only on ProductId, not the full key
CREATE TABLE OrderDetails (
    OrderId INT,
    ProductId INT,
    Quantity INT,
    ProductPrice DECIMAL,  -- This depends ONLY on ProductId, not (OrderId, ProductId)
    PRIMARY KEY (OrderId, ProductId)
);

-- ✅ FIXED — Move product-dependent data to Products table
CREATE TABLE Products (
    ProductId INT PRIMARY KEY,
    ProductPrice DECIMAL
);

CREATE TABLE OrderDetails (
    OrderId INT,
    ProductId INT,
    Quantity INT,
    PRIMARY KEY (OrderId, ProductId),
    FOREIGN KEY (ProductId) REFERENCES Products(ProductId)
);

    3NF — Third Normal Form: No Transitive Dependencies

    Rule: First, achieve 2NF. Then, non-key columns must not depend on other non-key columns.

    Violation:

    sql 
    -- ❌ VIOLATES 3NF — CategoryName depends on CategoryId, which is a non-key column
CREATE TABLE Products (
    ProductId INT PRIMARY KEY,
    Name VARCHAR(100),
    CategoryId INT,
    CategoryName VARCHAR(100)  -- Transitive dependency: depends on CategoryId, not ProductId directly
);

-- ✅ FIXED — Move category data to its own table
CREATE TABLE Categories (
    CategoryId INT PRIMARY KEY,
    CategoryName VARCHAR(100)
);

CREATE TABLE Products (
    ProductId INT PRIMARY KEY,
    Name VARCHAR(100),
    CategoryId INT,
    FOREIGN KEY (CategoryId) REFERENCES Categories(CategoryId)
);
    ✅

    Interview tip: Why normalization matters

    Normalization prevents update anomalies. If you have CategoryName duplicated in every Products row, you must update it everywhere when the category changes — risk of inconsistency. By storing it once in the Categories table, you change it once and the change is consistent everywhere.

    Q44. Stored Procedures — Pros, Cons, and When to Use

    What is a stored procedure? When would you use one instead of just writing queries in your application code?

    A stored procedure is a reusable block of SQL logic compiled and stored in the database. You call it from your application like a function.

    Example Stored Procedure

    sql 
    -- Stored procedure: get all products in a category and apply a discount
CREATE PROCEDURE GetCategoryProductsWithDiscount
    @CategoryId INT,
    @DiscountPercent DECIMAL
AS
BEGIN
    SELECT
        ProductId,
        Name,
        Price,
        Price * (1 - @DiscountPercent / 100) AS DiscountedPrice
    FROM Products
    WHERE CategoryId = @CategoryId;
END;

-- Call it from C# or SQL:
EXEC GetCategoryProductsWithDiscount @CategoryId = 1, @DiscountPercent = 10;

    Pros of Stored Procedures

    • Performance: Pre-compiled and cached by the database — faster for complex operations
    • Centralized logic: Database logic lives in one place, not scattered across your application
    • Security: Can grant users permission to execute the procedure without exposing underlying table structure
    • Reduced network traffic: If the procedure does a lot of work, you send one request instead of many

    Cons of Stored Procedures

    • Harder to test: You need a database and stored procedure management tools (not easy in unit tests)
    • Version control: Hard to version-control SQL procedures the same way you version-control code
    • Language mismatch: Procedure logic is in T-SQL or PL/SQL, not your application language — splits the team's knowledge
    • Debugging: Debugging a procedure requires SQL debuggers, not your IDE
    • Deployment: Deploying a procedure change requires coordinating with DBAs

    When to Use

    • Complex, multi-step operations that hit the database many times and benefit from batching
    • Heavy aggregations or reporting queries that run slowly in the application
    • Compliance or audit requirements where you must log database changes at the database level
    • Legacy systems already heavy on stored procedures (rewrite incrementally)

    Modern Reality: ORM (EF Core) vs Stored Procedures

    Most modern backend developers use ORMs like Entity Framework Core. EF can call stored procedures, but the trend is toward application-level logic because it's easier to test, deploy, and maintain. Stored procedures are used when you really need the database-side performance or when you're dealing with complex multi-step transactions that can't wait for application-level logic.

    Q45. Database Indexes — Structure, When to Add, Performance Tradeoffs

    What is a database index? How do B-tree indexes work? When should you add an index, and what are the tradeoffs?

    What is an Index?

    An index is a data structure that speeds up SELECT queries and WHERE filters. Without an index, SQL must scan every row (table scan). With an index, SQL can jump directly to matching rows.

    B-Tree Structure

    Most databases use B-tree indexes (balanced tree). The tree is ordered so that:

    • Left subtree has smaller values, right subtree has larger values
    • Tree stays balanced — no branch is much deeper than others
    • This ensures O(log n) lookup time even on huge tables

    Example: If you have an index on ProductId, the database builds a tree that lets it find ProductId = 42 in ~4-5 disk reads instead of scanning 1 million rows.

    When to Add an Index

    sql 
    -- ✅ Index on foreign keys (always used in JOINs)
CREATE INDEX IX_Products_CategoryId ON Products(CategoryId);

-- ✅ Index on columns used in WHERE frequently
CREATE INDEX IX_Products_Name ON Products(Name);

-- ✅ Composite index if you filter by two columns together
CREATE INDEX IX_Products_CategoryPrice ON Products(CategoryId, Price);
-- This speeds up: WHERE CategoryId = 1 AND Price > 100

-- ❌ Avoid indexes on columns with low cardinality (few distinct values)
-- Example: a "Gender" column with only 2-3 values — scanning is faster than index lookup

-- ❌ Too many indexes slow down INSERT/UPDATE/DELETE because the index must be maintained

    Performance Tradeoffs

    Operation Without Index With Index
    SELECT (WHERE) Slow — scan all rows Fast — O(log n) lookup
    INSERT Fast — just add row Slower — must update index
    UPDATE Slow — scan to find row Medium — index finds row, then update index
    DELETE Slow — scan to find row Medium — index finds row, then delete index entry
    Disk space Minimal Extra disk for index structure
    💡

    Golden rule: Index for reads, not for writes

    In a typical web application, reads vastly outnumber writes (often 10:1 or higher). So you add indexes to speed up the frequent reads, accepting the overhead on the rare writes. But if you have a write-heavy system, every index is a cost.

    Q46. DISTINCT vs GROUP BY vs Aggregate Functions

    What is the difference between DISTINCT, GROUP BY, COUNT, SUM, AVG, and MAX?

    DISTINCT — Remove duplicate rows

    Returns unique rows. Useful for finding out "how many different categories exist?"

    sql 
    -- Get all unique category names (duplicates removed)
SELECT DISTINCT CategoryName
FROM Products;

-- Result: If you have 100 products but only 5 categories, you get 5 rows

    GROUP BY — Aggregate rows by a value

    Combines multiple rows that have the same value into a single row. Always used with an aggregate function.

    sql 
    -- Count products per category
SELECT CategoryId, COUNT(*) AS ProductCount
FROM Products
GROUP BY CategoryId;

-- Result:
-- CategoryId | ProductCount
-- 1          | 25 (Electronics)
-- 2          | 10 (Mobile)
-- 3          | 5  (Furniture)

    Aggregate Functions — Combine values

    Functions that work on a set of rows and return a single value.

    COUNT(*) — Count rows

    sql 
    SELECT COUNT(*) FROM Products;  -- Result: 100 (total rows)
SELECT COUNT(CategoryId) FROM Products;  -- Result: 95 (ignores NULL)

    SUM — Add up values

    sql 
    SELECT SUM(Price) FROM Products;  -- Total cost of all inventory

    AVG — Calculate average

    sql 
    SELECT AVG(Price) FROM Products;  -- Average product price

    MAX / MIN — Find highest or lowest

    sql 
    SELECT MAX(Price), MIN(Price) FROM Products;
-- Result: 9999.99 (max), 9.99 (min)

    Complex Example: GROUP BY with Aggregates

    sql 
    -- For each category, get count, total price, average price
SELECT
    c.CategoryName,
    COUNT(*) AS ProductCount,
    SUM(p.Price) AS TotalInventoryValue,
    AVG(p.Price) AS AvgPrice
FROM Products p
JOIN Categories c ON p.CategoryId = c.CategoryId
GROUP BY c.CategoryId, c.CategoryName
ORDER BY TotalInventoryValue DESC;

    DISTINCT vs GROUP BY

    Feature DISTINCT GROUP BY
    Purpose Remove duplicates from result Aggregate data and combine rows
    With aggregates Cannot use — doesn't make sense Always used with aggregates like COUNT, SUM
    Example SELECT DISTINCT CategoryId FROM Products SELECT CategoryId, COUNT(*) FROM Products GROUP BY CategoryId

    Q47. Transactions — What are they? ACID Properties

    What is a database transaction? Explain ACID: Atomicity, Consistency, Isolation, Durability.

    A transaction is a group of SQL statements that either all succeed or all fail together. It's the database's guarantee that your data stays correct even if things go wrong (power outage, network failure, etc.).

    Basic Transaction

    sql 
    BEGIN TRANSACTION;

UPDATE Accounts SET Balance = Balance - 100 WHERE AccountId = 1;
UPDATE Accounts SET Balance = Balance + 100 WHERE AccountId = 2;

-- If all statements succeed, commit:
COMMIT;

-- If ANY statement fails, rollback (undo all changes):
ROLLBACK;

    ACID Properties

    A — Atomicity (All or Nothing)

    A transaction is atomic — either all statements execute or none do. In the money transfer example above: you cannot take $100 from account 1 and then crash before adding it to account 2. Either both happen or neither does.

    C — Consistency (Data Validity)

    After a transaction, the database is in a valid state. All constraints are satisfied. If a transfer violates a business rule (e.g., balance can't go negative), the transaction fails and rolls back.

    I — Isolation (No Interference)

    Two transactions don't interfere with each other. If Transaction A is transferring money and Transaction B is reading the balance, B doesn't see partial/inconsistent data from A.

    Isolation levels (increasing strictness):

    • READ UNCOMMITTED: Can see uncommitted data from other transactions (dirty read)
    • READ COMMITTED: Only sees committed data (default in most databases)
    • REPEATABLE READ: Prevents dirty reads and non-repeatable reads
    • SERIALIZABLE: Transactions run as if they were serial (strictest, slowest)

    D — Durability (Permanent)

    Once a transaction is committed, it's permanent. Even if the server crashes, the data survives. The database writes to disk before confirming the commit.

    Transactions in EF Core

    csharp 
    using var transaction = await _db.Database.BeginTransactionAsync();

try
{
    var product = await _db.Products.FindAsync(productId);
    product.Stock -= 100;

    var order = new Order { ProductId = productId, Quantity = 100 };
    _db.Orders.Add(order);

    await _db.SaveChangesAsync();
    await transaction.CommitAsync();  // All-or-nothing
}
catch
{
    await transaction.RollbackAsync();  // Undo both operations
    throw;
}

    Q48. UNION, UNION ALL, INTERSECT, EXCEPT — Set Operations

    Explain set operations: UNION, UNION ALL, INTERSECT, EXCEPT with examples.

    Set operations combine results from two or more SELECT queries.

    UNION — Combine and remove duplicates

    Returns all rows from both queries, with duplicates removed.

    sql 
    -- Get all products and categories (combined list, no duplicates)
SELECT Name AS Item FROM Products
UNION
SELECT CategoryName AS Item FROM Categories;

-- Result might be:
-- Item
-- Laptop
-- Phone
-- Monitor
-- Electronics
-- Mobile
-- Furniture

    UNION ALL — Combine and keep duplicates

    Faster than UNION because it doesn't remove duplicates. Use this if you know there are no duplicates or you want to keep them.

    sql 
    SELECT ProductId, Name FROM Products
UNION ALL
SELECT ProductId, Name FROM ArchivedProducts;

-- Result includes all products even if some ProductIds are in both tables

    INTERSECT — Rows in both queries

    Returns only rows that appear in both queries (the overlap).

    sql 
    -- Find product names that appear in BOTH Products and PopularItems
SELECT Name FROM Products
INTERSECT
SELECT Name FROM PopularItems;

-- Result: Only names that are in BOTH tables

    EXCEPT — Rows in first query but not second

    Returns rows from the first query that do NOT appear in the second query.

    sql 
    -- Find products that are NOT in any orders (never been sold)
SELECT ProductId FROM Products
EXCEPT
SELECT DISTINCT ProductId FROM Orders;

-- Result: ProductIds in Products but not in Orders

    Q49. Views — What are they? When to use?

    What is a database view? When would you create one instead of just writing the query in your application?

    A view is a reusable SELECT query stored in the database. It acts like a virtual table — you query it like a table, but the data is computed on-the-fly from the underlying tables.

    Creating a View

    sql 
    -- Create a view that joins Products and Categories
CREATE VIEW vw_ProductDetails AS
SELECT
    p.ProductId,
    p.Name AS ProductName,
    p.Price,
    c.CategoryName,
    p.Stock
FROM Products p
LEFT JOIN Categories c ON p.CategoryId = c.CategoryId
WHERE p.IsActive = 1;

-- Now use it like a table:
SELECT * FROM vw_ProductDetails;
SELECT * FROM vw_ProductDetails WHERE Price > 100;

    Pros of Views

    • Reusability: Write the complex JOIN once, use it everywhere
    • Simplicity: Hides complexity — consumers see a simple table-like interface
    • Security: You can expose a view to users without exposing the underlying tables
    • Centralized logic: If the JOIN changes, you change it in one place

    Cons of Views

    • Performance: Every query against the view re-executes the underlying query (not materialized by default)
    • Update complexity: Updating through a view is tricky or impossible if it involves JOINs
    • Hidden cost: Developers might not realize a "simple table" actually runs a complex JOIN

    When to Create a View

    • The same complex query is used in multiple places in your application
    • You want to simplify and document a commonly-needed result
    • You need to hide table structure from certain users for security
    • You have a reporting query that's run frequently and is resource-intensive

    Materialized Views (Advanced)

    Some databases (like SQL Server) support materialized views — the view result is pre-computed and stored on disk, making queries instant but requiring manual refreshes. Use this for reporting views that are expensive to compute but don't need real-time data.

    Q50. Query Optimization — Indexes, EXPLAIN PLAN, N+1 Problem

    How do you optimize slow queries? Explain EXPLAIN PLAN and the N+1 problem.

    Finding Slow Queries: EXPLAIN PLAN

    EXPLAIN shows how the database will (or did) execute your query, including which indexes were used and how many rows were scanned.

    sql 
    -- View the execution plan BEFORE running the query
EXPLAIN SELECT * FROM Products WHERE CategoryId = 1;

-- Output might show:
-- ┌─ Nested Loop (cost=0.29..8.35 rows=5)
-- │  ├─ Index Scan on idx_Products_CategoryId (rows=5)
-- │  └─ Filter: CategoryId = 1
-- └─ Total cost: 8.35

    Key metrics to look for:

    • Cost: Lower is better. High cost means the query takes a long time.
    • Rows: How many rows the database expects (or found). Huge row counts often mean missing indexes or a bad JOIN.
    • Index Scan vs Table Scan: "Index Scan" is good (uses an index). "Table Scan" is bad (reads every row).
    • Nested Loop vs Hash Join: "Nested Loop" is fast for small sets, "Hash Join" is better for large joins.

    The N+1 Query Problem

    You run 1 query to get a list of items, then run N more queries inside a loop — one for each item. Classic mistake with ORMs.

    BAD Example: N+1 Queries

    csharp 
    // ❌ N+1 PROBLEM
// 1 query: get all products
var products = _db.Products.ToList();  // Query 1

// Then N queries: for each product, get its category
foreach (var product in products)
{
    var category = _db.Categories.FirstOrDefault(c => c.CategoryId == product.CategoryId);
    // Query 2, 3, 4, ... N+1
    Console.WriteLine($"{product.Name} — {category?.CategoryName}");
}

// Total: 1 + N queries. If you have 1000 products, that's 1001 database round-trips!

    GOOD Example: Use Include() to eager-load

    csharp 
    // ✅ GOOD — Only 1 query via JOIN + Include
var products = _db.Products
    .Include(p => p.Category)  // Eager-load the category
    .ToList();

// No loop queries — all data is already in memory
foreach (var product in products)
{
    Console.WriteLine($"{product.Name} — {product.Category?.CategoryName}");
}

// Total: 1 query. Same data, but 1000x faster!
    ⚠️

    N+1 queries are a performance killer

    In interviews, when asked "How would you optimize this code?", look for loops that query the database. That's almost always N+1. The fix: use eager loading (Include), JOIN in your query, or batch multiple items together in a single WHERE IN clause.

    SQL Best Practices

    💡

    The fundamentals that will save your career

    These practices are not optional — they're how production systems stay fast and reliable. Every one of these mistakes has crashed a real system.

    1. Always use indexes on Foreign Keys and frequently queried columns

    sql 
    -- ✅ GOOD — indexes on columns used in WHERE and JOIN
CREATE INDEX IX_Products_CategoryId ON Products(CategoryId);  -- FK
CREATE INDEX IX_Products_Price ON Products(Price);            -- frequently filtered

-- ❌ BAD — no indexes, queries scan all rows
SELECT * FROM Products WHERE CategoryId = 5;  -- Linear scan!

    2. Avoid SELECT * — specify columns needed

    sql 
    -- ❌ BAD — transfers unnecessary data
SELECT * FROM Products;  -- includes Description (10MB), LargeBlob (100MB), etc.

-- ✅ GOOD — only what you need
SELECT ProductId, Name, Price FROM Products;  -- lean and fast

    3. Use JOINs instead of multiple queries

    sql 
    -- ❌ BAD — two round-trips
SELECT * FROM Products WHERE CategoryId = 1;
SELECT * FROM Categories WHERE CategoryId = 1;

-- ✅ GOOD — one JOIN, one round-trip
SELECT p.*, c.CategoryName
FROM Products p
JOIN Categories c ON p.CategoryId = c.CategoryId
WHERE p.CategoryId = 1;

    4. Use composite indexes for multi-column filters

    sql 
    -- ❌ BAD — two separate indexes or table scan
CREATE INDEX IX_CategoryId ON Products(CategoryId);
CREATE INDEX IX_Price ON Products(Price);

-- ✅ GOOD — one composite index for common WHERE patterns
CREATE INDEX IX_Products_CategoryPrice ON Products(CategoryId, Price);

-- This speeds up:
SELECT * FROM Products WHERE CategoryId = 1 AND Price > 100;

    5. Profile queries with EXPLAIN output

    sql 
    -- Always check your query plan BEFORE it goes to production
EXPLAIN
SELECT p.ProductId, p.Name, c.CategoryName, COUNT(o.OrderId) AS OrderCount
FROM Products p
LEFT JOIN Categories c ON p.CategoryId = c.CategoryId
LEFT JOIN Orders o ON p.ProductId = o.ProductId
GROUP BY p.ProductId, p.Name, c.CategoryName;

-- Look for:
--   - Table Scans (bad) vs Index Scans (good)
--   - High cost numbers (anything > 1000 usually means you need an index)
--   - Row estimates that are way off (hint: update statistics)
    ✅

    Interview tip: SQL optimization mindset

    When an interviewer asks "How would you speed up this query?", the answer almost always involves: (1) adding indexes on WHERE/JOIN columns, (2) removing SELECT *, (3) checking for N+1 problems, or (4) using EXPLAIN to spot table scans. These three ideas cover 90% of real performance problems. Practice spotting them in code and you'll ace database questions.
