🐳 Docker - Containerization & Deployment

Package your app + dependencies in a shipping container. Works the same on your laptop, CI/CD, and production. No more "works on my machine" problems.

🏗️

Real-World Analogy: Shipping Containers

❌ Without Docker: You mail your app on your laptop with all its dependencies scattered around. Recipient's environment different? App breaks.

✅ With Docker: You pack your app + Node.js 18 + MongoDB into a standardized shipping container. Load it anywhere, runs identically. No surprises.

Why Containerization Matters

The Problem Without Docker:

  • ❌ "Works on my machine": Your Windows laptop has Node 18, server has Node 16 → Different behavior
  • ❌ Environment Hell: Dev team uses PostgreSQL 12, production uses PostgreSQL 14 → Bugs only appear in prod
  • ❌ Slow Onboarding: New team member takes 2 days to install all dependencies & get it working
  • ❌ Dependency Conflicts: App A needs Java 11, App B needs Java 17 → Both can't run on same server
  • ❌ Risky Deployments: Manual setup on production = high chance of missing a step

The Solution WITH Docker:

  • ✅ Consistency: Image built once, runs identically everywhere
  • ✅ Isolation: Each container has its own runtime, no conflicts
  • ✅ Reproducibility: Anyone can run `docker run myapp:1.0.0` and get exact same environment
  • ✅ Speed: Deploy in seconds instead of hours of manual setup

Key Concepts

1️⃣ Images vs Containers

Docker Image: Template/Blueprint (like a .zip file)

FROM mcr.microsoft.com/dotnet/aspnet:8.0
COPY app .
ENTRYPOINT ["dotnet", "app.dll"]
↓
Result: myapp:1.0.0 (370 MB image file)

Docker Container: Running instance of image (like executable running in memory)

docker run myapp:1.0.0
↓
Result: Container PID 1234 running, consuming 50 MB RAM

Analogy: Image = Class, Container = Object instance

2️⃣ Multi-Stage Builds (Keep Images Small)

❌ WRONG (Bloated image):

FROM mcr.microsoft.com/dotnet/sdk:8.0
COPY . .
RUN dotnet publish -c Release
ENTRYPOINT ["dotnet", "ProductCatalogAPI.dll"]
# Result: 2 GB image (includes SDK, build tools, everything)

✅ RIGHT (Optimized):

# Stage 1: Build (need SDK)
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS builder
COPY . .
RUN dotnet publish -c Release -o /app

# Stage 2: Runtime (only need runtime)
FROM mcr.microsoft.com/dotnet/aspnet:8.0
COPY --from=builder /app .
ENTRYPOINT ["dotnet", "ProductCatalogAPI.dll"]
# Result: 370 MB image (only runtime, no SDK)

Why? Build stage needs full SDK (800 MB). Runtime stage only needs ASP.NET runtime (160 MB). Final image: 370 MB instead of 2 GB!

3️⃣ Base Images (FROM clause)

Base Image = Operating System + Runtime

FROM mcr.microsoft.com/dotnet/aspnet:8.0
# = Ubuntu 22.04 + .NET 8.0 runtime
# Size: ~160 MB (precompiled, optimized for .NET)

FROM node:18-alpine
# = Alpine Linux + Node.js 18
# Size: ~170 MB (lighter than node:18-bookworm which is 1.1 GB)

Pick wisely:

  • mcr.microsoft.com/dotnet/aspnet:8.0 → Optimized for .NET apps
  • node:18-alpine → Lightweight (Alpine = 5 MB base, Bookworm = 120 MB)
  • ubuntu:22.04 → Universal but heavier (77 MB)

4️⃣ Docker Layers (How images are built)

Each instruction = new layer (like git commits)

FROM mcr.microsoft.com/dotnet/aspnet:8.0                    # Layer 1
COPY . /app                                                  # Layer 2
RUN dotnet restore                                           # Layer 3
RUN dotnet publish -c Release -o /bin                        # Layer 4
ENTRYPOINT ["dotnet", "/bin/ProductCatalogAPI.dll"]          # Layer 5

Docker caches layers: If you rebuild and only change code, it reuses layers 1-3, rebuilds only layer 4!

Optimization: Order instructions by how often they change (rarely-changing first, frequently-changing last):

FROM mcr.microsoft.com/dotnet/sdk:8.0 AS builder    # Base image (never changes)
COPY *.csproj .                                      # Project file (rarely changes)
RUN dotnet restore                                   # Dependencies (rarely changes)
COPY . .                                             # Source code (always changes)
RUN dotnet publish -c Release

5️⃣ Environment Variables & Port Exposure

Production-Ready Dockerfile:

FROM mcr.microsoft.com/dotnet/aspnet:8.0
WORKDIR /app
COPY --from=builder /app/publish .

# Configure runtime
ENV ASPNETCORE_ENVIRONMENT=Production
ENV ASPNETCORE_URLS=http://+:5000
ENV DOTNET_RUNNING_IN_CONTAINER=true

# Expose port
EXPOSE 5000

# Security: Don't run as root
USER app

ENTRYPOINT ["dotnet", "ProductCatalogAPI.dll"]

When you run: docker run -p 8080:5000 productcatalogapi:1.0.0

  • Container port 5000 → Host port 8080
  • Visit http://localhost:8080/api/products

Complete ProductCatalog Example

Step 1: Create Dockerfile in ProductCatalogAPI folder

# syntax=docker/dockerfile:1
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS builder
WORKDIR /src

# Copy project files
COPY ["ProductCatalogAPI.csproj", "."]

# Restore NuGet packages
RUN dotnet restore "ProductCatalogAPI.csproj"

# Copy source code
COPY . .

# Publish to Release folder
RUN dotnet publish "ProductCatalogAPI.csproj" -c Release -o /app/publish

# ===== Runtime Stage =====
FROM mcr.microsoft.com/dotnet/aspnet:8.0
WORKDIR /app

# Copy compiled app from builder
COPY --from=builder /app/publish .

# Configuration
ENV ASPNETCORE_ENVIRONMENT=Production
ENV ASPNETCORE_URLS=http://+:5000

# Security: don't run as root
RUN groupadd -r appuser && useradd -r -g appuser appuser
USER appuser

EXPOSE 5000

ENTRYPOINT ["dotnet", "ProductCatalogAPI.dll"]

Step 2: Build the image

cd d:\dotnet-fullStack\ProductCatalogAPI-WebAPI-EFCore
docker build -t productcatalogapi:1.0.0 .
# Output: Successfully built image with tag productcatalogapi:1.0.0

Step 3: Run the container

# Basic run
docker run -p 5000:5000 productcatalogapi:1.0.0

# Background run with name
docker run -d --name product-api -p 5000:5000 productcatalogapi:1.0.0

# With environment variables
docker run -d --name product-api \
  -p 5000:5000 \
  -e DATABASE_URL="Server=localhost;Database=ProductCatalog" \
  productcatalogapi:1.0.0

# Check logs
docker logs -f product-api

# Stop container
docker stop product-api

Best Practices

1. Use multi-stage builds

Separate build and runtime stages to minimize final image size.

2. Pin base image versions

Use node:18.19.0 not node:latest. Latest changes without notice!

3. Use .dockerignore

Exclude node_modules, .git, bin, obj → faster builds

4. Don't run as root

Use USER appuser for security. Containers breaking out = less damage

5. Keep layers minimal

Combine RUN commands: RUN apt-get && apt-get clean prevents bloat

6. Use health checks

HEALTHCHECK CMD curl http://localhost/health → Orchestrators know if container is sick

Common Mistakes

  • ❌ Installing dev dependencies in production: RUN npm install --save-dev webpack in final image bloats it. Use build stage.
  • ❌ Using :latest tag: FROM node:latest means different build = different environment. ALWAYS tag versions!
  • ❌ Not caching dependencies: Copy package.json BEFORE source code so docker caches dependency layer
  • ❌ Running as root: Container escapes = attacker has root access. Use non-root user!
  • ❌ Storing secrets in image: RUN export API_KEY=secret123 is visible in image history. Use environment variables at runtime!

Interview Q&A

Q: What's the difference between a Docker image and a container?

A: Image is a blueprint (immutable template), container is a running instance. Like class vs object.

Q: Why use multi-stage builds?

A: Build stage needs compilers/SDKs (heavy). Runtime stage only needs runtime. Multi-stage lets you discard build stage files, keeping final image small (370 MB vs 2 GB).

Q: How do you reduce Docker image size?

A: (1) Use multi-stage builds, (2) Use alpine base images, (3) Remove cache after package installs, (4) Pin exact dependency versions, (5) Use .dockerignore to exclude unnecessary files.
