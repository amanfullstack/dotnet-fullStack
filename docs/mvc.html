What is ASP.NET MVC?

MVC (Model-View-Controller) separates your application into three layers. The server renders HTML and sends complete web pages to the browser. Perfect for traditional web applications with rich user experiences.

The MVC Pattern Explained

MVC is a design pattern that separates an application into three interconnected components:

🎮 Model

Represents the application's data and business logic. Models are responsible for:

  • Defining data structures (Product, Category, Order)
  • Implementing business rules and validation
  • Communicating with the database through repositories/services
  • Example: Product { Id, Name, Price, Category }

👁️ View

The user interface - HTML/CSS/JavaScript that the user sees and interacts with. Views are responsible for:

  • Displaying data received from the controller
  • Capturing user input (forms, buttons)
  • No business logic - just presentation
  • Uses Razor syntax to render dynamic content

🎯 Controller

The middleman that handles user requests and coordinates between Model and View:

  • Receives HTTP requests from the browser
  • Processes user input and calls business logic (services)
  • Retrieves data from the model/database
  • Passes data to the appropriate view
  • Returns the rendered HTML to the browser

Request/Response Flow Diagram

1. USER makes HTTP request
   └─> GET /products/5

2. ROUTING matches request to controller action
   └─> ProductsController.Details(id=5)

3. CONTROLLER processes request
   ├─> Validates input
   ├─> Calls service/repository
   └─> Gets data from MODEL/DATABASE

4. MODEL/DATABASE returns data
   └─> Product { Id=5, Name="Widget", Price=9.99 }

5. CONTROLLER passes data to VIEW
   └─> ViewBag.Product = product

6. VIEW renders HTML with data
   └─> HTML: <h1>Widget - $9.99</h1>

7. RESPONSE sent to browser
   └─> Browser displays complete HTML page

Real-World Example: ProductCatalog

Here's how the ProductCatalog MVC application handles viewing a product:

1. User Request

User clicks: <a href="/products/details/5">View Product</a>

2. Controller Action

// ProductsController.cs
public class ProductsController : Controller {
    private readonly IProductService _service;

    public ProductsController(IProductService service) {
        _service = service;
    }

    // Handle GET /products/details/5
    public async Task<IActionResult> Details(int id) {
        try {
            // Step 1: Get data from service (which calls API/database)
            var product = await _service.GetProductByIdAsync(id);

            if (product == null) {
                return NotFound(); // Return 404 if product doesn't exist
            }

            // Step 2: Pass data to view
            return View(product); // Renders Details.cshtml with product data
        } catch (Exception ex) {
            // Handle errors gracefully
            return RedirectToAction("Error");
        }
    }
}

3. Model (Data Structure)

// Models/Product.cs
public class Product {
    public int Id { get; set; }
    public string Name { get; set; }
    public string Description { get; set; }
    public decimal Price { get; set; }
    public string Category { get; set; }
    public int Stock { get; set; }
    public DateTime CreatedDate { get; set; }
}

4. Service Layer (Business Logic)

// Services/ProductService.cs
public class ProductService : IProductService {
    private readonly HttpClient _httpClient;

    public async Task<Product> GetProductByIdAsync(int id) {
        try {
            // Call the ProductCatalogAPI
            var response = await _httpClient.GetAsync(
                $"http://localhost:5000/api/products/{id}"
            );

            if (!response.IsSuccessStatusCode) {
                return null;
            }

            var json = await response.Content.ReadAsStringAsync();
            var product = JsonConvert.DeserializeObject<Product>(json);
            return product;
        } catch {
            return null;
        }
    }
}

MVC vs Other Patterns

Pattern Pros Cons Best For
MVC Clear separation, easy to test, good for traditional web apps Slower (server rendering), more server load Content-heavy sites, traditional web apps
SPA (React/Angular) Fast (client rendering), responsive, rich interactivity Complex setup, API needed, harder SEO Highly interactive apps, dashboards
Microservices Scalable, independent services, easy to update Complex setup, network latency, debugging harder Large-scale enterprise applications

MVC Best Practices

  1. Keep Controllers Thin: Controllers should only handle HTTP concerns (getting requests, returning responses). Business logic belongs in services. 
    // ❌ Bad - business logic in controller
public IActionResult Create(Product model) {
    if (model.Price < 0) return BadRequest("Invalid price");
    model.DiscountedPrice = model.Price * 0.9m;
    _context.Products.Add(model);
    _context.SaveChanges();
    return Ok();
}

// ✅ Good - business logic in service
public IActionResult Create(Product model) {
    var created = _service.CreateProduct(model);
    return Ok(created);
}
  2. Use Dependency Injection: Inject services into controllers, not creating with new. 
    // ✅ Correct - dependency injection
public class ProductsController : Controller {
    private readonly IProductService _service;

    public ProductsController(IProductService service) {
        _service = service; // Injected by ASP.NET
    }
}
  3. Validate in Models: Use data annotations and validation attributes. 
    public class Product {
    [Required(ErrorMessage = "Name is required")]
    [StringLength(100, MinimumLength = 3)]
    public string Name { get; set; }

    [Range(0.01, 999999.99)]
    public decimal Price { get; set; }
}
  4. Use ViewModels for Complex Views: Create specific models for views instead of using domain models directly. 
    // View-specific model
public class ProductDetailViewModel {
    public Product Product { get; set; }
    public List<Review> Reviews { get; set; }
    public RelatedProducts[] RelatedItems { get; set; }
}
  5. Handle Errors Gracefully: Use try-catch, return appropriate status codes, log errors.
  6. Use Routing Correctly: Make URLs RESTful and meaningful. 
    // ✅ Good routes
GET /products          // List all
GET /products/5        // Get single
POST /products         // Create
PUT /products/5        // Update
DELETE /products/5     // Delete

🎓 Interview Q&A

Q1: What is the MVC pattern and why use it?

Answer: MVC (Model-View-Controller) separates an application into three interconnected components:

  • Model: Data and business logic, no UI concerns
  • View: User interface (HTML/CSS), displays data from model
  • Controller: Handles user input, coordinates between model and view

Why use it? Clear separation of concerns makes code easier to maintain, test, and scale.

Q2: What's the difference between MVC and SPA (Single Page Application)?

Answer:

  • MVC (ASP.NET MVC): Server renders complete HTML, sends to browser. Page reload on navigation.
  • SPA (React/Angular): Server provides API, client (browser) renders HTML. No page reload, fast navigation.

Which to choose?

  • MVC: Traditional web apps, content-heavy sites, SEO important
  • SPA: Highly interactive apps, dashboards, real-time features

Q3: How does routing work in ASP.NET MVC?

Answer: Routing maps URLs to controller actions:

// URL: /products/details/5
// Route: /[controller]/[action]/[id]
// Maps to: ProductsController.Details(id: 5)

// Conventional routing in Program.cs
app.MapControllerRoute(
    name: "default",
    pattern: "{controller=Home}/{action=Index}/{id?}");

// Attribute routing (recommended)
[Route("api/[controller]")]
public class ProductsController : Controller {
    [HttpGet("{id}")]
    public IActionResult GetById(int id) { ... }
}
